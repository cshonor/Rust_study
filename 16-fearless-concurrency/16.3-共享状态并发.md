# 16.3 共享状态并发

对应 demo 项目：`16-fearless-concurrency/16.3-mutex-arc-demo/`

消息传递类似**单所有权**（值进通道后发送端不再持有）；**共享内存**则类似**多所有权**——多线程同时访问同一内存，需要额外同步原语。

---

## 互斥量 `Mutex<T>`

**Mutex**（*mutual exclusion*）保证任意时刻**只有一个**线程能访问被保护的数据。访问前 **`lock()`** 取锁；`lock()` 可能阻塞，若持锁线程 panic 导致“毒化”，`lock()` 返回 `Err`（常见写法仍是 `unwrap`/`expect`）。

`lock()` 返回 **`MutexGuard<T>`**（智能指针）：`Deref` 到内部数据，`Drop` 时**自动解锁**，避免忘记释放锁。

---

## 多线程共享 `Mutex`：需要 `Arc` 而非 `Rc`

- **示例 16-13**：`counter` 不能多次 `move` 进不同 `spawn`。
- **示例 16-14**：`Rc<Mutex<T>>` 可克隆，但 **`Rc<T>` 不是 `Send`**，不能跨 `thread::spawn`。

**`Arc<T>`**（*atomically reference-counted*）与 `Rc` API 类似，引用计数更新是**线程安全**的；与 `Mutex` 组合可在线程间共享可变状态：`Arc<Mutex<T>>`。

---

## 与 `RefCell` / `Rc` 的类比

- `Mutex<T>` 提供**内部可变性**（在持有 `&` 的情况下修改内部），类似 `RefCell`，但 **`Mutex` 用于跨线程**，在**运行时**保证互斥。
- `Arc<Mutex<T>>` 类似 “多线程版” 的 `Rc<RefCell<T>>`（语义上类比，API 不同）。

Rust 仍无法防止**逻辑层**错误：如 **死锁**（两锁交叉等待），需自行设计锁顺序或避免嵌套锁。

---
