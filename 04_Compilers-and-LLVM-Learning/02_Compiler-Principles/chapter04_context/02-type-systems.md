**类型检查**是上下文相关分析的**核心任务之一**。

---

## 目的

| 目标 | 说明 |
|------|------|
| **安全性** | 编译期发现类型错误，避免部分运行时崩溃 |
| **表达能力** | 精确描述数据结构与操作 |
| **代码生成** | 推断具体类型 → 后端选指令、布局、优化（如去装箱） |

**HFT / Rust**：静态类型 + 单态化 → 无运行时类型分派开销（对比 `dyn`）。

---

## 类型分类（本章体系）

| 类别 | 例子 |
|------|------|
| **基本类型** | 整数、浮点、字符、布尔 |
| **合成 / 结构类型** | 数组、字符串、枚举、结构体/记录、指针 |

每种类型有**内部构造**与**推断/兼容规则**（如 `int + float` 是否允许、如何 promotion）。

---

## 强/弱 · 静/动

| 维度 | 对比 | 编译期 vs 运行时 |
|------|------|------------------|
| **强 vs 弱** | 隐式转换是否宽松 | 弱类型易隐藏 bug |
| **静 vs 动** | 类型何时确定 | **Rust/C**：静态 — ch4 主战场；**Lox/Python**：动态 — 运行时检查 |

| 语言 | 策略 |
|------|------|
| **Rust** | 静态强类型 — `rustc` 在语义阶段拒绝 ill-typed 程序 |
| **Lox (jlox)** | 动态 — ch7 解释执行时再报类型错误 |
| **C** | 静态弱类型 — 许多检查留到运行时或未定义行为 |
