## 原书要点

### 为何用现成解释器

| 选项 | 说明 |
|------|------|
| 手写 VM | 教学价值高（CI / Ch1），工程成本极大 |
| **`wasmi`** | Parity 团队为**以太坊客户端**等场景开发的 **Wasm 解释器**（Rust） |
| 现代替代 | **wasmtime** / **wasmer** 更常用于生产 — Layer 3 宿主可优先 wasmtime → [layer03](../../layer03_quant-ma-strategy/README.md) |

> 原书 2019 用 **wasmi**；概念（Module → Instance → call export）与 wasmtime 一致。

### wasmi 基本流程

```text
读取 add.wasm 字节流
  → Module::from_buffer / 解析
  → 链接 imports（本例 add 模块可能无 import）
  → ModuleInstance::new
  → instance.call("add", &[RuntimeValue::I32(1), I32(2)])
  → 控制台打印 3
```

概念代码（API 随 wasmi 版本略有差异）：

```rust
// 伪代码 — 表达流程
let wasm_bytes = std::fs::read("add.wasm")?;
let module = wasmi::Module::from_buffer(&wasm_bytes)?;
let instance = wasmi::ModuleInstance::new(&module, &imports_ref)?;
let result = instance.invoke_export("add", &[I32(1), I32(2)])?;
```

### 与 [6.1](./6.1-good-host.md) 五项职责的对应

| 职责 | wasmi 中的体现 |
|------|----------------|
| Load/Validate | `Module::from_buffer` 解析 + 验证 |
| Expose Exports | `invoke_export("add", ...)` |
| Satisfy Imports | 构造 `imports` 表（add 模块可为空） |
| Execute | 解释器逐条执行 |
| Isolation | 模块独立 linear memory + trap 边界 |

### 与 Layer 1 衔接

同一 `add` 函数：

```text
Ch1 WAT 手写  →  Ch3 Rust cdylib  →  Ch6 wasmi 控制台调用
Layer 1 还可并排 .ll 对照
```

→ [layer01](../../layer01_rust-llvm-to-wasm/README.md)
