## 一句话

**比赛引擎** = **Game Loop** 给每个 Wasm 机器人分配时间片；初版「每 Wasm 一线程 + 全局状态数组」触发 **RwLock 死锁与 Borrow Checker 地狱** → 用 **ECS** 拆状态、限写锁，实现安全高并发。

---

## 原书要点

### Game Loop 与并发痛点

| 初版方案 | 问题 |
|----------|------|
| **每个 Wasm 机器人一个线程** | 多线程同时读写 **共享机器人状态数组** |
| **`RwLock` 全局状态** | 读写冲突 · **卡死** · 借用检查器 **compile/runtime 痛苦** |

→ 对照 [05-atomic · 锁与并发](../../../05-Async-Concurrency-Network/01-atomic/README-学习区.md)

### ECS（Entity-Component-System）架构

| 概念 | WARoS 中的含义 |
|------|----------------|
| **Entity 实体** | 一个 **Wasm 机器人实例** 的唯一 ID |
| **Component 组件** | 极细状态碎片：**运动** · **伤害** · **扫描** · … |
| **System 系统** | 只修改**特定组件**的逻辑；**唯一**持写锁的入口 |

```text
Entity (robot #2)
  ├─ MotionComponent      ← DriveSystem 独占写
  ├─ DamageComponent      ← CombatSystem 独占写
  └─ ScanComponent        ← RadarSystem 独占写

规则：没有「全局 RobotState 大锁」— 只有系统级细粒度锁
```

| ECS 收益 | 说明 |
|----------|------|
| **并发安全** | 系统之间减少锁竞争 |
| **Borrow Checker 友好** | 拆分后 Rust 可证明互斥访问 |
| **可扩展** | 新行为 = 新 Component + System |

### 引擎 Runtime（wasmi）

| 职责 | 说明 |
|------|------|
| **加载** | 多个 `.wasm` 机器人 → 各自 **ModuleInstance** |
| **Satisfy imports** | 实现 [8.2 WARoS API](./8.2-waros-api.md) · 读/写 ECS 组件 |
| **Execute** | 每帧调用机器人 export · trap 隔离单机器人 |
