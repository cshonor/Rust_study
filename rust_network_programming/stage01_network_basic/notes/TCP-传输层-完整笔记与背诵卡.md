# TCP（传输控制协议）｜考试 / 面试完整版笔记

> 对齐 6 条主线：连接管理 → 可靠交付 → 流量控制 → 拥塞控制 → 首部开销 → 典型场景。  
> 每条：**原理 + 考点 + 一句话总结**。

---

## 1）面向连接：三次握手建连接、四次挥手释放

**核心**：通信前必须先「约好」，结束要「礼貌分手」，全程有状态。

### 三次握手（建立连接）

| 次序 | 方向 | 关键字段 | 含义 |
|------|------|----------|------|
| ① | C → S | **SYN=1，seq=x** | 我想连，我的初始序号是 x |
| ② | S → C | **SYN=1，ACK=1，seq=y，ack=x+1** | 同意连，确认 x，我的初始序号是 y |
| ③ | C → S | **ACK=1，ack=y+1** | 确认 y，连接建立 |

**状态变迁（常考）**

- 客户端：CLOSED → **SYN_SENT** → **ESTABLISHED**
- 服务端：LISTEN → **SYN_RCVD** → **ESTABLISHED**

```text
     C                          S
     |---- SYN, seq=x --------->|
     |<--- SYN+ACK, seq=y -----|
     |---- ACK, ack=y+1 ------>|
     |      ESTABLISHED         |
```

**考点**：为何三次而不是两次？—— 防止**历史重复 SYN** 导致服务端误建连（需客户端最终 ACK 确认「我要的是这次连接」）。

### 四次挥手（断开连接）

| 次序 | 方向 | 关键字段 | 含义 |
|------|------|----------|------|
| ① | C → S | **FIN=1，seq=u** | 我这边不再发数据了 |
| ② | S → C | **ACK=1，ack=u+1** | 收到；我可能还在发 |
| ③ | S → C | **FIN=1，seq=v** | 我这边也发完了 |
| ④ | C → S | **ACK=1，ack=v+1** | 确认；连接彻底关闭 |

**TIME-WAIT（2MSL）**：主动关闭方在发完最后一个 ACK 后进入，等待可能迟到的报文消失，并确保对端若重传 FIN 仍能 ACK。

```text
     C                          S
     |---- FIN, seq=u --------->|
     |<--- ACK ----------------|
     |<--- FIN, seq=v ---------|
     |---- ACK ---------------->|
     |   TIME-WAIT (2MSL)       |
```

**一句话**：握手 = 双方确认收发能力；挥手 = 双方各自收尾，**可靠断开**。

## 2）可靠交付：无丢失、无重复、按序到达

TCP 在**不可靠的 IP** 上提供**字节流级可靠**，四大机制：

| 机制 | 作用 |
|------|------|
| **序号 seq** | 每个字节有编号，接收方按序重组、去重 |
| **确认 ACK** | 已收到 → 回 ACK（**ack = 期望的下一个字节序号**） |
| **超时重传** | 发出去迟迟无 ACK → 超时后重传（丢数据或丢 ACK 都能救） |
| **快速重传** | 连续 **3 个重复 ACK** → 不等超时，立刻重传缺失段 |

**一句话**：**序号保序 + ACK 确认 + 超时 / 快速重传防丢**，做到不丢、不重、不乱序。

---

## 3）流量控制：滑动窗口，防止发送方把接收方「撑爆」

**问题**：发送太快 → 接收缓冲区满 → 丢包。  
**手段**：**接收窗口 rwnd**（接收方在 ACK 里通告「我还能收多少字节」）。

- 发送方发送量 **≤ rwnd**（不能超过接收方水位线）
- rwnd **动态变化**：接收忙则变小，闲则变大
- 与拥塞无关，是**端到端**「接收能力」匹配

```text
  发送方                          接收方
  [已发未确认] [可发窗口≤rwnd]     [缓冲区]
       |---------- 数据 ---------->|
       |<--------- ACK + rwnd -----|
```

**一句话**：**流量控制 = 点对点速度匹配，靠 rwnd 滑动窗口，防接收方溢出**。

---

## 4）拥塞控制：防止网络过载丢包

**问题**：发送太快 → 路由器队列满 → 延迟↑、丢包↑、全网拥堵。  
**手段**：**拥塞窗口 cwnd**（发送方根据网络状况自行限速）。

| 阶段 | 行为 |
|------|------|
| **慢启动** | cwnd 从 1 MSS 起，每 RTT **指数增**（1→2→4→8…），试探带宽 |
| **拥塞避免** | cwnd ≥ **ssthresh** 后，每 RTT **线性 +1 MSS**，平稳占带宽 |
| **快速重传 / 快速恢复** | 3 个重复 ACK → 快速重传 + cwnd 下调后进入恢复，避免一刀砍到 1 |
| **超时** | 严重拥塞 → cwnd 常重置为 1，**重新慢启动** |

**发送窗口（实际可发量）**：

```text
  有效发送窗口 = min(rwnd, cwnd)
```

同时尊重**接收能力**与**网络能力**。

**一句话**：**拥塞控制 = 全网负载控制，靠 cwnd 动态试探，避免网络堵车**。

## 5）开销大、首部 20–60 字节

TCP 首部 **最小 20 字节，最大 60 字节**（含选项）；UDP 仅 **8 字节**。

**必考字段**

| 字段 | 说明 |
|------|------|
| 源 / 目的端口 | 各 16 bit |
| seq / ack | 各 32 bit |
| 窗口 | 16 bit（rwnd，可选项扩大） |
| 标志位 | **URG / ACK / PSH / RST / SYN / FIN**（握手挥手重点 **SYN、ACK、FIN**） |
| 校验和 | 端到端差错检测 |
| 选项 | MSS、SACK、时间戳、窗口扩大等 |

**一句话**：**为可靠与控制多字段，首部重，换强语义服务**。

## 6）典型场景：不能丢、不能乱的应用

| 场景 | 为何用 TCP |
|------|------------|
| **HTTP/HTTPS** | 页面、API 体必须完整有序 |
| **FTP/SFTP** | 文件字节级正确 |
| **SSH** | 命令与回显可靠 |
| **SMTP/POP3/IMAP** | 邮件正文与附件不能丢 |
| **数据库** | 查询结果、事务语义要准确 |

**对比 UDP**：直播、DNS、游戏状态等可容忍丢包、要低延迟时多用 UDP。

**一句话**：**凡要求完整、准确、有序的字节流，用 TCP**。

## 面试 / 笔试万能一段（背这一段即可）

**TCP** 是面向连接、可靠、全双工的传输层协议：用**三次握手 / 四次挥手**管理连接；用**序号、ACK、超时重传、快速重传**保证可靠；用**滑动窗口 rwnd** 做流量控制、用**拥塞窗口 cwnd**（慢启动、拥塞避免、快速恢复）做拥塞控制；首部 **20–60 字节**开销较大；适用于 **HTTP、文件、邮件、SSH、数据库** 等对可靠性要求高的场景。
