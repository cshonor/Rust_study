## 1. 使用 Serde 进行序列化与反序列化

### Serde 简介

**Serde**（**Ser**ialize / **De**serialize）是 Rust 生态里序列化的事实标准：

| 格式 | 常见 crate |
|------|------------|
| JSON | `serde_json` |
| YAML | `serde_yaml` |
| TOML | `toml` + `serde` |
| CSV | `csv` + `serde` |
| MessagePack、Bincode 等 | 对应 `rmp-serde`、`bincode` 等 |

网络服务里 **JSON**（人机可读、调试方便）和 **二进制**（体积小、解析快）最常见。

### 基本用法：`serde_derive`

```toml
# Cargo.toml
[dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"
```

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Serialize, Deserialize)]
struct Point3D {
    x: f64,
    y: f64,
    z: f64,
}

fn main() -> serde_json::Result<()> {
    let p = Point3D { x: 1.0, y: 2.0, z: 2.0 };
    let json = serde_json::to_vec(&p)?;           // 序列化 → Vec<u8>
    let back: Point3D = serde_json::from_slice(&json)?;  // 反序列化
    Ok(())
}
```

过程宏 **`#[derive(Serialize, Deserialize)]`** 自动生成实现，无需手写样板代码。

### 网络应用示例（书中 TCP + JSON）

典型流程（配合 [stage03](../../stage03_std_tcp_udp/) 的 `TcpStream`）：

```text
客户端                          服务器
  读控制台输入 (x,y,z)
  构建 Point3D
  serde_json::to_vec → write_all
                              read → from_slice → Point3D
                              计算到原点距离 √(x²+y²+z²)
                              序列化结果 → write_all
  read → 显示距离
```

**工程注意（TCP 粘包）**：

- `TcpStream` 是**字节流**，一次 `read` 不一定对应一条 JSON。  
- 常见做法：**长度前缀**（先读 4 字节 `len`，再读 `len` 字节）、**换行分隔**（`read_line`）、或 **HTTP 式 Content-Length**。  
- Serde 只负责「这一段字节 ↔ 结构体」；**分帧**要在 I/O 层自己设计。

```rust
// 示例：长度前缀（u32 大端）+ JSON body
fn send_json(stream: &mut TcpStream, msg: &impl Serialize) -> std::io::Result<()> {
    let body = serde_json::to_vec(msg)?;
    let len = (body.len() as u32).to_be_bytes();
    stream.write_all(&len)?;
    stream.write_all(&body)?;
    Ok(())
}
```

## 2. 自定义序列化与反序列化

宏能覆盖绝大多数类型；以下情况常需**手动实现**：

- 输出字段名与 Rust 字段不一致（也可用 `#[serde(rename = "...")]`）  
- 特殊编码（枚举 externally/internally tagged）  
- 与遗留二进制布局逐字段对齐  

### 实现 `Serialize`

手动调用序列化器的结构化 API，例如 `serialize_struct`、`serialize_field`：

```rust
use serde::ser::{Serialize, SerializeStruct, Serializer};

impl Serialize for MyType {
    fn serialize<S: Serializer>(&self, serializer: S) -> Result<S::Ok, S::Error> {
        let mut st = serializer.serialize_struct("MyType", 2)?;
        st.serialize_field("custom_x", &self.x)?;
        st.serialize_field("custom_y", &self.y)?;
        st.end()
    }
}
```

### 实现 `Deserialize` 与访问者模式（Visitor）

反序列化侧更复杂：输入可能是 map、seq、字符串等，需实现 **`Visitor`**，在 `visit_map`、`visit_str`、`visit_f64` 等回调里**逐步构造**目标类型。

```text
JSON/二进制输入 → Deserializer → Visitor::visit_* → 你的 Struct
```

书中为自定义类型实现 `visit_map` 读键值对、`visit_str` 读字符串等，是理解 Serde 插件与自定义格式的关键。

### 单元测试：`serde_test`

用 **令牌流（Token stream）** 断言序列化输出是否符合预期，而不必真的 round-trip JSON：

```rust
// 概念示例
use serde_test::{assert_tokens, Token};

assert_tokens(
    &MyType { x: 1 },
    &[Token::Struct { name: "MyType", len: 1 }, Token::I64(1)],
);
```

适合验证**手写 `Serialize`/`Deserialize`** 是否与协议规范一致。

## 3. 文本数据解析

有些协议（如 **HTTP 请求行**）用**原始文本**切分比「整包反序列化成 giant struct」更自然。

### PEG 与 `nom`

**PEG（Parsing Expression Grammar）** 无歧义、可组合；**`nom`** 是 Rust 里常用的**解析器组合子**库（自底向上：小解析器拼成大解析器）。

| 宏 / 组合子 | 作用 |
|-------------|------|
| **`tag!`** | 匹配固定字节/字符串（如 `b"GET "`） |
| **`take_until!`** | 读到某标记为止 |
| **`do_parse!`** / 组合子链 | 顺序执行，保留中间结果 |
| **`alt`** / **`|`** | 多分支选一（GET 或 POST） |

### HTTP 请求行（书中示例结构）

解析目标示例：

```http
GET /path HTTP/1.1
```

得到：**方法**、**URL**、**HTTP 版本**。

```rust
// 概念：nom 7+ 多为函数式组合子，书中多为 macro 风格
// use nom::{bytes::complete::tag, sequence::tuple, IResult, &str};
// fn parse_request_line(input: &str) -> IResult<(&str, Method, &str, &str)> { ... }
```

### 调试 `nom` 宏

- **`dbg!` / `dbg_dump!`**：打印解析失败位置与剩余输入。  
- **`cargo expand`**：展开宏，看清实际生成的解析链。  
- 策略：**先写最小子解析器单测**，再组合。

### 对比：`pom`

| | **nom** | **pom** |
|---|---------|---------|
| 风格 | 宏 + 组合子为主 | **Trait + 运算符**（`\|` 或，`*` 顺序） |
| 学习曲线 | 宏调试较难 | 更接近「用 Rust 写解析表达式」 |

两者都是 PEG；选一种深入即可，HTTP/协议行 `nom` 生态更大。

**与 Stage 06**：完整 HTTP 会用 **hyper** 等库；本章是**理解请求行如何被拆开**。

## 4. 二进制数据解析

底层协议（IP/TCP/自定义帧）常是**定长或按位布局**的二进制，不适合 JSON。

### 用 `nom` 解析二进制

以 **IPv6 首部**为例（书中）：字段按**位**划分，需按 bit 读取：

| 字段 | 典型宽度 |
|------|----------|
| 版本 | 4 bit |
| 流量类别 | 8 bit |
| 流标签 | 20 bit |
| 有效载荷长度 | 16 bit |
| … | … |

`nom` 提供 **`bits!`**、**`take_bits!`** 等，在比特流上切分，再填入强类型结构体（如 `Ipv6Addr`）。

```text
原始 &[u8] → bits! 子解析器 → (version, tc, flow_label, ...)
           → 组装为 Ipv6Header { ... }
```

**与 `libpnet`（Ch3）**：`libpnet` 偏抓包与链路层；`nom` 偏**你自己定义**的字节布局解析，两者可配合。

### 二进制 vs Serde

| 方式 | 适用 |
|------|------|
| **Serde + bincode/postcard** | 双方都是 Rust（或约定 schema），结构体直接映射 |
| **nom 手写布局** | 标准协议头、跨语言、位域、非字节对齐字段 |
| **固定头 + Serde 体** | 头用 `nom`/手动 `Read`，负载用 JSON/MessagePack |
