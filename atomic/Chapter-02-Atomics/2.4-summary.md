## 3. 日常开发规避数据竞争（实操法则）

### 3.1 优先安全抽象（约 99% 场景）

| 需求 | 优先选用 |
|------|----------|
| 局部并发、可限定线程寿命 | `thread::scope` |
| 共享可变状态 | `Mutex<T>` / `RwLock<T>` |
| 消息传递 | `mpsc::channel` 等 |

**不要**默认手写 `Atomics` + `unsafe`。

### 3.2 必须用 Atomics 时的逻辑陷阱

1. **原子性 ≠ 业务一致性**  
   两个原子变量（如 `num_done` 与 `total_time`）分别 `fetch_add` 时，他线程可能看到「数量已增、耗时未增」的中间态。不可接受 → 用 **`Mutex`** 打包更新。

2. **整数溢出**  
   `fetch_add` / `fetch_sub` **回绕**。全局 ID、索引 → 用 **`compare_exchange_weak` + 循环** 做越界检查与更新一体（Fetch-Update 模式），见 `2.3-compare-and-exchange/2.3-compare-and-exchange-id-allocator-demo.rs`。

### 3.3 使用 `unsafe` 的底线

- 读文档 **Safety** 段，写清你向编译器保证的不变量（注释在 `unsafe` 块上方）。  
- 记住：**DR 引发 UB** 时，错误影响可被优化器「渗透到」看似无关的前序代码。

## 4. 核心术语（本章 + 跨章）

| 术语 | 含义 |
|------|------|
| **Data Race** | 并发非原子访问同一内存 → UB |
| **Race（竞争）** | 多线程竞态执行，不一定是 DR（如幂等懒加载） |
| **Relaxed** | 单变量原子性，**无**跨变量顺序保证 |
| **Fetch-Update** | `fetch_*` 或 CAS 循环实现的读-改-写 |
| **CAS / compare_exchange** | 期望值匹配才写入 |
| **Wrapping** | 原子算术溢出回绕 |
| **Visibility** | 他线程何时能看到本线程写入（第 3 章用内存序解决） |

---

## 5. 面试 / 笔试一句话

**Rust 用借用默认消除数据竞争；跨线程无锁共享单变量用原子操作保证不可分割；`Relaxed` 只保单变量原子性；多变量一致性与可见顺序靠 `Mutex` 或第 3 章内存序；`fetch_add` 会溢出回绕，关键资源分配要用 CAS 循环把「检查+更新」打成一体。**
