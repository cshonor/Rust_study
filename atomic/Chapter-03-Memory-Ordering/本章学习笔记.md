## 整体脉络

**本章核心问题**：**为什么原子操作必须指定 `Ordering`？**

| 前置（第 2 章） | 本章（第 3 章） |
|-----------------|-----------------|
| 原子 API：`load` / `store` / `fetch_*` / CAS | **重排与可见性**：编译器 / CPU 会乱序 |
| **`Relaxed`** 只保证**单变量**原子性、总修改顺序 | **`Release` / `Acquire`** 建立 **happens-before**，同步**普通内存** |
| 数据竞争由借用 / 原子消除 | **Ordering** 解决「别的变量何时对谁可见」 |

**两条主线**：

1. **3.1 → 3.2**：先懂**为什么**要限制重排（重排无害于单线程、有害于多线程）→ 再懂 **Rust/C++ 内存模型** 与 **Happens-Before**；
2. **3.3 → 3.4 → 3.5**：五种 **`Ordering`** 怎么选 → **`fence`** 边界 → **常见误区**（x86、SeqCst、Relaxed 当标志）。

**一句话**：**原子性消除该原子变量上的数据竞争；Ordering 解决其他内存的可见性与顺序。**
