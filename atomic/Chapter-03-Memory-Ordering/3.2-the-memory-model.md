## 一、内存模型是什么？

Rust 的并发内存语义**继承 C++ 内存模型**：用一套**抽象规则**描述「在多线程下，哪些写对哪些读可见、以什么顺序可见」，从而统一 x86、ARM 等不同硬件行为。

| 作用 | 说明 |
|------|------|
| **跨架构** | 同一份 Rust 源码 + 正确的 `Ordering`，在不同 CPU 上语义一致 |
| **可推理** | 用 **Happens-Before** 等关系证明程序无数据竞争外的 UB |
| **与借用分工** | [第 1 章](../Chapter-01-Rust-Concurrency-Basics/1.4-borrowing-data-races/1.4-borrowing-data-races.md) 编译期防 DR；原子 + Ordering 处理**已允许**的跨线程共享 |

## 二、Happens-Before（发生于…之前）

多线程同步的**基石**：若 **A happens-before B**，则 A 的所有**副作用**（对内存的写）对 B **可见**。

典型建立方式（本书重点）：

```
线程 A：写普通内存 data
       → Release store 到原子 flag
              ════════════════════  HB  ════════════════════
线程 B：Acquire load 从 flag 读到 A 写入的值
       → 此后读 data 可见 A 的写
```

**记忆**：**Release 发布，Acquire 订阅** — 成对出现，中间的原子写/读是「同步点」。

## 三、原子性 vs 内存序（Ordering）

| | **原子性（Atomicity）** | **内存序（Ordering）** |
|---|-------------------------|------------------------|
| 解决什么 | 对**单个原子变量**的读/写不可分割 | **其他内存**在跨线程间的**可见性**与**顺序** |
| 谁保证 | 硬件 + `Atomic*` API | 你选的 `Relaxed` / `Acquire` / … |
| 一句 | 消除该变量上的 DR | 约束重排，建立 HB |

> **原子性消除该原子变量上的数据竞争；Ordering 解决其他内存的可见性与顺序。**（全书第 3 章主题句）

## 四、学习本节要带走什么

1. **内存模型** = 抽象规则，不是某块物理内存；
2. **Happens-Before** = 跨线程「谁先谁后、谁对谁可见」的推理工具；
3. 下一节 [3.3 Ordering 选项](./3.3-memory-order-options.md) 的五种 `Ordering`，就是向编译器 / CPU 下达的**不同强度**的重排限制。
