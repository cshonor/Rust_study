## 8.3.1 POSIX：`pthread_cond` / `pthread_mutex`

> 书 §8.3 · 索引：[8.3-condition-variables.md](./8.3-condition-variables.md) · 后：[8.3.2 Windows](./8.3.2-windows-condvar.md)

---

## 核心类型

| 类型 | 作用 |
|------|------|
| **`pthread_mutex_t`** | 互斥锁 |
| **`pthread_cond_t`** | 条件变量 |

Rust 在 **类 Unix**（Linux、macOS 等）上 `std::sync::Mutex` / `Condvar` 常基于 pthread 族或 futex 优化路径实现。

---

## 关键 API

| C API | 语义 | ≈ Rust |
|-------|------|--------|
| **`pthread_cond_wait(cond, mutex)`** | **原子地**释放 mutex + 休眠；醒后**重新加锁** | `Condvar::wait(guard)` |
| **`pthread_cond_timedwait`** | 带超时 wait | `wait_timeout` |
| **`pthread_cond_signal`** | 唤醒**一个**等待者 | `notify_one` |
| **`pthread_cond_broadcast`** | 唤醒**全部**等待者 | `notify_all` |

---

## 与 futex 的关系（Linux）

现代 glibc 等实现中，pthread cond/mutex **底层仍可能用 futex**（WAIT/WAKE/REQUEUE）。对应用开发者：

- **POSIX 层**：`cond_wait` + `signal/broadcast`；
- **Linux 内核层**：往往是 futex  syscall。

---

## 使用约束（与 Rust 相同）

- wait **必须**在**循环**里检查条件（虚假唤醒）；
- wait 传入的 mutex **必须已持有**；
- signal/broadcast 通常应在**改完共享状态**后调用。
