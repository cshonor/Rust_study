## 1.2.3 `.join()` 与 scope 自动 join

> 书 §1.2 · 索引：[1.2](./1.2-scoped-threads.md) · 前：[1.2.2 两种 spawn](./1.2.2-two-spawns.md) · 后：[1.2.4 ScopedJoinHandle](./1.2.4-scoped-join-handle.md)

---

| 概念 | 是什么 | 谁有 |
|------|--------|------|
| **`.join()` 方法** | 阻塞**当前**线程，等待目标子线程结束，取 `Result<T, _>` / 传播 panic | `JoinHandle` 与 `ScopedJoinHandle` **都有** |
| **scope 自动 join** | `thread::scope` 闭包**走完后**，标准库对**尚未 join** 的 `ScopedJoinHandle` **批量 join** | **仅** `thread::scope` |

### 1. `.join()` 方法（两种句柄通用）

```rust
// 普通 spawn
let h = thread::spawn(move || 42);
h.join().unwrap(); // JoinHandle::join()

// scope 内
thread::scope(|s| {
    let h = s.spawn(|| 42);
    h.join().unwrap(); // ScopedJoinHandle::join() — 语义类似，生命周期绑在 'scope
});
```

作用：阻塞 caller、回收 OS 线程资源、取回返回值或捕获 panic（[1.1.2 §二](../1.1-threads-in-rust/1.1.2-join-handle.md#二err-类型三层拆解-boxdyn-any--send--static)）。

### 2. scope 的自动 join（语法糖，不是第三种 API）

这是 **`thread::scope` 独有**的保障机制：**程序流离开 scope 闭包**时，运行库保证所有 `s.spawn` 创建的子线程已结束，再销毁被借用的局部变量。

- **不等于**「不需要 `.join()` 这个概念」——自动 join 本质上就是在 scope 末尾替你调用了未处理的 handle 的 join；
- **你仍可以**在 scope 内**提前** `.join()` 拿返回值，不必等到闭包最后一行；
- **`thread::spawn` 完全没有**这层自动 join：drop 未 join 的 `JoinHandle` → 线程 **detach**；`main` 直接退出 → 子线程可能被强杀（[1.1.2 §四](../1.1-threads-in-rust/1.1.2-join-handle.md#四不用-join-会怎样)）。

**背诵**：

1. 单词是 **Join**，不是 Enjoy；
2. `thread::scope`：**离开 scope 自动 join 全部** + 可选手动 `.join()`；
3. `thread::spawn`：**必须**手动 `.join()`（或承担 detach / 被强杀风险）。

### 3. 分层拆解：自动 join 做了什么？是不是封装系统调用？

先分两层：**Rust 标准库逻辑** 与 **OS 系统调用**。

#### （1）第一层：scope 自动 join = std 库逻辑，不是内核「自动行为」

`thread::scope` 内部会登记每次 `s.spawn` 返回的 **`ScopedJoinHandle`**。当 scope 闭包结束、内部的 **`Scope` 对象析构**时，标准库会**遍历尚未手动 join 的句柄，逐个调用 `.join()`**。

**伪代码（心智模型，非源码逐行对应）**：

```rust
struct Scope {
    handles: Vec<ScopedJoinHandle<()>>,
}

impl Drop for Scope {
    fn drop(&mut self) {
        for h in self.handles.drain(..) {
            let _ = h.join(); // 自动 join：与你手写 h.join() 走同一条路
        }
    }
}
```

要点：

- 「自动阻塞」的**触发**来自 std 在 scope 收尾时的**库逻辑**（常见实现是 `Scope` 析构里批量 join）；
- **操作系统不知道** Rust 的 `Scope`、`Drop`、作用域——这层是上层封装；
- **借用栈上数据的安全性**不靠「Drop 一定执行」（Leakpocalypse 教训），而靠 **`'scope` + handle 不能逃出 scope**（见 [1.2.7 Leakpocalypse](./1.2.7-leakpocalypse-pitfalls.md)）。

#### （2）第二层：`.join()` 内部才下沉到 OS 阻塞 API

无论**手动** `h.join()`，还是 **scope 收尾自动**调用的 `h.join()`，标准库底层最终都会走到跨平台线程等待原语（[1.1.2 §六](../1.1-threads-in-rust/1.1.2-join-handle.md#六底层原理os--标准库--join)）：

| 平台 | 典型底层 |
|------|----------|
| Linux / Unix | `pthread_join()`（POSIX；内核侧常经 futex 等机制阻塞） |
| Windows | `WaitForSingleObject()` 等待线程内核句柄 |

**流程串联**：

```text
离开 scope 闭包
  → Scope 析构 / std 批量 join
  → ScopedJoinHandle::join()
  → std::thread 跨平台封装
  → pthread_join / WaitForSingleObject
  → 内核阻塞当前线程，直到目标子线程退出
```

#### （3）三个常见疑问

1. **「自动阻塞」不是 OS 在线程作用域结束时自动做的**  
   普通 `thread::spawn` 无 scope：`main` 直接退出时子线程可能被强杀——说明 **OS 不会替你 join**。

2. **「自动」= Rust 帮你调了 `.join()`，不是第三种等待 API**  
   执行链路与手写 `join` **一致**，最终都会下沉到**阻塞型系统调用**。

3. **阻塞发生在哪条线程上？**  
   scope 收尾触发 join 时，**当前正在执行 scope 闭包的那条线程**（通常是主线程）会**逐个阻塞**等待未完成的子线程；全部结束后，`Scope` 析构完成，控制流才走出 `thread::scope(...)` 块。

#### （4）手动 join vs 自动 join：底层无差别

```rust
use std::thread;

// 场景 1：手动 join，显式阻塞
thread::scope(|s| {
    let h = s.spawn(|| {});
    h.join().unwrap(); // 你触发 → 同样走 OS 等待 API
});

// 场景 2：自动 join，隐式阻塞
thread::scope(|s| {
    s.spawn(|| {});
}); // Scope 收尾替你 join → OS 等待 API 相同
```

唯一区别：**谁调用 `.join()`**——你手写，还是 `Scope` 收尾自动调用。

#### （5）scope 自动 join 时，`Result<T, _>` 怎么处理？

Scope 收尾批量 `.join()` 时，标准库**不会把 `Ok(T)` 交给你**——返回值被丢弃：

| 子线程结局 | 自动 join 行为 |
|------------|----------------|
| **正常返回 `T`** | join 成功，**`T` 被丢弃**；要拿值须 scope 内**手动** `.join()` |
| **panic** | join 得到 **`Err`** → std **继续 panic**（不会静默吞掉） |

```rust
use std::thread;

// 要拿到 T：必须手动 join
thread::scope(|s| {
    let handle = s.spawn(|| 100);
    let val = handle.join().unwrap(); // T = 100
    println!("{val}");
});

// 自动 join：Ok(T) 丢弃；若子线程 panic → 整个程序 panic
thread::scope(|s| {
    s.spawn(|| 100); // 离开 scope 时 join，你看不到 100
});
```

与 [1.1.2 §二 `T` vs `Result`](../1.1-threads-in-rust/1.1.2-join-handle.md#joinhandlet-的-t-vs-join-的-result两个独立层次) 对照：**自动 join 只保证「等线程跑完 + panic 不静默」**，不替你保存业务返回值 **`T`**。
