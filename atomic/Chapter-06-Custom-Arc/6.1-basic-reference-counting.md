## 1. 为什么手写 Arc？

标准库 `Arc<T>` 已是生产级实现；本章目的是理解：

- **共享所有权**如何用 **`AtomicUsize`** 维护引用计数  
- **`clone`** 与 **`Drop`** 各自需要什么 **Ordering**  
- 控制块布局、**`Send`/`Sync`** 约束与 **`Weak`** 的扩展方向

## 2. 控制块与指针

典型布局（教学简化版）：

```text
ArcInner<T> {
    ref_count: AtomicUsize,   // 强引用计数
    data: T,
}
```

`CustomArc<T>` 持有 `NonNull<ArcInner<T>>`（或 `*const ArcInner<T>`）。  
**不变量**：计数为 0 时内存必须已被释放；计数 > 0 时 `data` 对任一持有者可见。

---

## 3. `new` 与 `clone`

| 操作 | 原子语义 | Ordering | 原因 |
|------|----------|----------|------|
| **`new`** | `ref_count = 1` | 普通写即可 | 尚未共享 |
| **`clone`** | `fetch_add(1)` | **`Relaxed`** | 只改计数；不通过计数建立对 `data` 的新 happens-before（已有指针可读） |

`clone` 后新旧 `CustomArc` 指向**同一块**控制块；`Relaxed` 足够，因为读 `data` 不依赖「计数加一」的可见顺序，只依赖「指针仍然有效」。

---

## 4. `Drop` 与最后释放

| 步骤 | 操作 | Ordering |
|------|------|----------|
| 1 | `fetch_sub(1, Release)` | **Release**：保证本线程对 `data` 的写在该减一之前完成 |
| 2 | 若旧值 == 1 | 当前是**最后一个**持有者 |
| 3 | `fence(Acquire)` | 与所有其它线程的 **Release** 减一配对，看到它们对 `data` 的最后修改 |
| 4 | `dealloc` | 释放 `ArcInner` |

**为何不能全用 Relaxed？** 最后释放者必须看到其它持有者通过 `&T` 做的写入；Release/Acquire 在「计数归零 → 回收」边界建立同步。
