# 11.3 测试的组织结构

← [11.2 控制测试运行](./11.2-控制测试如何运行.md) · demo：`11.3-test-organization-demo/`

Rust 测试固定二分：**单元测试（unit）+ 集成测试（integration）** — 目录、位置、权限、写法完全不同。

```bash
cd Book/11-testing/11.3-test-organization-demo
cargo test
cargo test --test integ_case
```

---

## 一、单元测试 Unit Test（写在 `src/**.rs` 内部）

### 1. 存放位置

和源码同文件，**源码末尾用 `#[cfg(test)] mod tests {}` 包裹**。

```rust
// src/lib.rs
fn private_add(a: i32, b: i32) -> i32 { a + b }
pub fn pub_add(a: i32, b: i32) -> i32 { a + b }

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_private() {
        // ✅ 单元测试可以直接测私有函数
        assert_eq!(private_add(1, 2), 3);
    }

    #[test]
    fn test_pub() {
        assert_eq!(pub_add(2, 3), 5);
    }
}
```

### 2. `#[cfg(test)]` 核心作用

- **`cargo build`**：整块 tests **不编译**进最终产物
- **`cargo test`**：编译并执行里面所有 `#[test]`

### 3. 单元测试特点

1. 和源码同文件，粒度小，针对单个函数/模块
2. `use super::*` → **能访问私有方法、私有字段**
3. `cargo test` 自动全部执行

## 二、集成测试 Integration Test（项目根 `tests/`，与 `src/` 同级）

```
项目根目录
├── src/
│   └── lib.rs
├── tests/          ← 集成测试全部放这里
│   └── integ_case.rs
└── Cargo.toml
```

### 1. 关键规则

1. `tests/` 下**每个 `.rs` 是一个独立测试 crate**
2. 站在**外部使用者视角**，只能 `use 库名;` 引入
3. **只能调用 `pub` API**，访问不到私有函数

```rust
// tests/integ_case.rs
// Cargo.toml 里 name = "test_organization_demo"
use test_organization_demo::pub_add;

#[test]
fn test_inte_pub() {
    assert_eq!(pub_add(10, 20), 30);
    // private_add() 编译报错
}
```

### 2. 单独跑某一个集成测试文件

```bash
cargo test --test integ_case
```

### 3. 集成测试抽取公共代码（踩坑）

#### ❌ 错误：`tests/common.rs`

Cargo 会把 `common.rs` 当成**独立集成测试 crate**，输出里出现 `running 0 tests` 冗余日志。

#### ✅ 标准：`tests/common/mod.rs`

```
tests/
├── common/
│   └── mod.rs
└── integ_case.rs
```

```rust
// tests/common/mod.rs
pub fn helper() -> i32 { 10 }

// tests/integ_case.rs
mod common;
use common::helper;

#[test]
fn use_common() {
    assert_eq!(helper(), 10);
}
```

目录形式**不会**被 Cargo 识别成独立测试 crate。

## 三、二进制项目（main-only）特殊问题

### 问题

只有 `src/main.rs`、无 `src/lib.rs` 时，`tests/` **无法 `use` main 里的代码** — 二进制 crate 不能被外部引用。

### 工程标准拆分

```
src/
├── lib.rs   // 业务逻辑、pub API
└── main.rs  // 入口：参数解析、调用 lib
```

```rust
// src/lib.rs
pub fn calc(a: i32) -> i32 { a * 2 }

// src/main.rs
use test_organization_demo::calc;
fn main() {
    println!("{}", calc(5));
}
```

此时 `tests/*.rs` 可正常 `use test_organization_demo::calc;`。

## 四、单元 vs 集成对比表

| 项目 | 单元测试 | 集成测试 |
|------|----------|----------|
| 存放位置 | `src` 同文件内部 | 项目根 `tests/` |
| 访问权限 | 私有 + 公有 | **仅 `pub`** |
| 编译控制 | `#[cfg(test)]` 打包剔除 | 仅 `cargo test` 编译 |
| 测试目标 | 单模块/函数细节 | 整体链路、多模块配合 |
| 执行 | `cargo test` 自动 | 全部：`cargo test`；单文件：`cargo test --test xxx` |

## demo 目录

```
11.3-test-organization-demo/
├── Cargo.toml
├── src/
│   ├── lib.rs      # private_add / pub_add / calc + 单元测试
│   └── main.rs     # 二进制入口，调用 lib
└── tests/
    ├── common/
    │   └── mod.rs  # helper()
    └── integ_case.rs
```

| 命令 | 说明 |
|------|------|
| `cargo test` | 单元 + 全部集成 |
| `cargo test --test integ_case` | 只跑 `tests/integ_case.rs` |
| `cargo run` | 运行 main，打印 `calc(5)` → 10 |
