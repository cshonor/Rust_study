# 14.3 Cargo 工作空间

**工作空间（workspace）** 是一组共享**同一个 `Cargo.lock`** 和**同一个 `target/` 输出目录**的包。适合把大库拆成多个协同演进的 crate，同时避免重复编译、统一依赖版本。

对应 demo：`Book/14-cargo-crates/14.3-workspace-demo/`

---

## 根目录 `Cargo.toml`

根配置以 **`[workspace]`** 开头，列出 **`members`** 路径；**不包含** `[package]`（根目录本身通常不是一个 crate）。

```toml
[workspace]
members = ["adder", "add-one", "add-two"]
```

在根目录执行 `cargo build`，产物在 **`workspace-root/target`**，成员 crate 下一般**不再**各自一份 `target/`（在成员目录里跑 `cargo build` 也会写到工作空间根的 `target`）。

---

## 成员之间依赖：路径依赖

工作空间**不会**自动假设成员互相依赖，必须在对应 crate 的 `Cargo.toml` 里写明：

```toml
[dependencies]
add-one = { path = "../add-one" }
```

Rust 里 crate 名 `add-one` 对应导入名 **`add_one`**（连字符变下划线）。

---

## 运行 / 测试指定包

在 workspace 根目录：

- **`cargo run -p adder`**：运行名为 `adder` 的包（`Cargo.toml` 里 `[package].name`）
- **`cargo test`**：运行工作空间内**所有** crate 的测试
- **`cargo test -p add-one`**：只测 `add-one`

---

## 共享 `Cargo.lock` 与外部依赖

整个工作空间**只有根目录一份 `Cargo.lock`**。多个成员都依赖同一外部 crate（如 `rand`）时，Cargo 会解析为**同一版本**，各成员仍需在**自己的** `Cargo.toml` 里声明依赖后才能 `use`（不会在成员间“自动可见”）。

demo 里 `add-one` 使用 `rand = "0.8"`（与书中 `0.5.5` 仅是版本差异，语义相同）。

---

## 发布

向 crates.io 发布时，**每个 crate 单独** `cargo publish`；没有“一次发布整个 workspace”的 `--all` 开关。

---

## demo 命令

```bash
cd Book/14-cargo-crates/14.3-workspace-demo
cargo build
cargo run -p adder
cargo test
cargo test -p add-one
```
