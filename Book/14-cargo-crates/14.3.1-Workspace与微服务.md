# 14.3.1 Workspace 里的 Package ≠ 微服务

← 主笔记：[14.3 Cargo 工作空间](./14.3-Cargo工作空间.md) · [7.1.4 `src/bin` vs Workspace](../07-packages-modules/7.1.4-src-bin与多exe.md)

---

## 核心区分

> **Cargo Workspace 里的多个 Package ≠ 天然微服务**

| | Workspace 多 Package（默认） | 真正微服务 |
|---|-------------------------------|------------|
| 调用方式 | **编译期链接**，`use engine::…` **进程内函数调用** | 各 Package **独立 exe、独立进程、独立部署** |
| 通信 | 内存跳转，**无网络** | 跨进程 / 跨机器 IPC、RPC、MQ … |
| 本质 | **代码库拆分、源码组织** | **运行时隔离、服务边界** |

---

## 一、第一种：同 Workspace、未拆微服务（单进程）

```
workspace/
 ├ engine/     lib：交易核心
 └ gateway/    bin：入口 exe
```

`gateway/Cargo.toml`：

```toml
[dependencies]
engine = { path = "../engine" }
```

```rust
// gateway/src/main.rs
use engine::trade::order_submit;

fn main() {
    order_submit(); // 直接函数调用，内存跳转，无 IO、无网络
}
```

> **最后通常编译成 1 个（或少数几个）exe，跑在同一进程里**——只是源码分 Package 管理，**和微服务无关**。

demo → [14.3-hft-workspace-demo](./14.3-hft-workspace-demo/) 的 `gateway` → `engine` 就是这种。

---

## 二、第二种：拆成微服务（各 Package 独立进程，HFT 常用）

每个子 Package **各自 `main.rs` → 独立 exe**，分开启动、分开部署，进程之间才需要通信。

### 跨服务通信（HFT 常用度排序）

| # | 方式 | 场景 | 特点 |
|---|------|------|------|
| 1 | **本机 IPC** | 同机：行情 → 策略 → 网关 | Unix 域套接字、**共享内存 mmap**、消息队列；不走网卡，**微秒级**，HFT 本机标配 |
| 2 | **内网 TCP/UDP** | 多机微服务 | **UDP** 行情广播；**TCP / gRPC（`tonic`）** 下单、RPC |
| 3 | **中间件** | 中低频量化 | RabbitMQ、Kafka：解耦、异步，延迟高于直连 IPC |
| 4 | **共享存储** | 配置、持仓落地 | Redis、SQL：多服务读写同一份数据 |

### Rust 生态提示

- **gRPC 微服务**：[`tonic`](https://github.com/hyperium/tonic)（async，需 `.proto` + server/client）
- **本机低延迟**：共享内存 crate（如 `shared_memory`、`iceoryx2` 等，按项目选型）

---

## 三、划清边界（最易混）

| 层面 | 是什么 | 算不算微服务 |
|------|--------|--------------|
| **源码分包（Workspace + `path`）** | 编译链接、函数调用 | ❌ 只是代码组织 |
| **产物分包（各 Package 独立 exe + 分开部署）** | 进程隔离 + 跨进程通信 | ✅ 才是微服务 |

**Workspace 解决「怎么拆代码、怎么统一编译」；微服务解决「怎么拆进程、怎么跨边界通信」。**

---

## 四、HFT 落地实例

```
workspace/
├ market-feed/      bin：行情接收（独立进程）
├ strategy-core/    bin：策略引擎（独立进程）
└ order-gateway/    bin：券商网关（独立进程）
```

部署后 **3 个 exe = 3 个进程**：

```
market-feed  ←──共享内存/UDP IPC──→  strategy-core
strategy-core ←── TCP / tonic RPC ──→  order-gateway
```

| 链路 | 通信 |
|------|------|
| 行情 → 策略 | 本机共享内存 / UDP |
| 策略 → 网关 | TCP / **tonic** RPC 发委托 |

---

## 五、tonic RPC 最简示意（跨进程才需要）

> 仅示意结构；完整工程需 `tonic-build`、async runtime（tokio）。**同 Workspace 内 `path` 依赖时不要走 RPC，直接 `use`。**

`order.proto`（节选）：

```protobuf
service OrderGateway {
  rpc Submit (OrderReq) returns (OrderAck);
}
```

服务端（`order-gateway` 进程）：

```rust
// tonic 生成代码后
async fn submit(req: Request<OrderReq>) -> Result<Response<OrderAck>, Status> {
    // 对接券商 API …
    Ok(Response::new(OrderAck { id: 1 }))
}
```

客户端（`strategy-core` 进程）：

```rust
let mut client = OrderGatewayClient::connect("http://127.0.0.1:50051").await?;
client.submit(OrderReq { /* … */ }).await?;
```

## 六、怎么选（口诀）

| 需求 | 做法 |
|------|------|
| 同进程、要低延迟 | Workspace + **`path` + 函数调用** |
| 同机多进程、HFT 热路径 | 独立 bin Package + **IPC / 共享内存** |
| 多机、要标准接口 | 独立部署 + **gRPC / tonic** |
| 解耦、可丢一点延迟 | **Kafka / MQ** |
