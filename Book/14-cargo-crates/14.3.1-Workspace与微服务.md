# 14.3.1 两种打包形态：单 exe vs 多 exe

← 主笔记：[14.3 Cargo 工作空间](./14.3-Cargo工作空间.md) · demo：[14.3-packaging-demo](./14.3-packaging-demo/) · [7.1.4](../07-packages-modules/7.1.4-src-bin与多exe.md)

---

## 结论前置

Workspace 里每个 Package 都是**独立项目**，但**打包结果**分两种，完全分开：

| 形态 | 结构 | 产物 | 运行时 |
|------|------|------|--------|
| **形态 1** | lib Package + bin 依赖 lib | **1 个 exe** | **单进程**，函数调用，无通信 |
| **形态 2** | 多个 bin Package（各有 `main.rs`） | **N 个 exe** | **多进程**，必须 IPC / 网络 |

> **Workspace 多 Package ≠ 天然微服务**——取决于最后出几个 exe、开几个进程。

---

## 一、形态 1：lib + bin 依赖 → 只产出【单个 exe】

```
workspace/
├ Cargo.toml
├ core_lib/          ← Package A：仅 lib.rs，**无 main，不出 exe**
│  ├ Cargo.toml
│  └ src/lib.rs
└ app_bin/           ← Package B：main.rs，二进制 crate
   ├ Cargo.toml
   └ src/main.rs
```

`app_bin/Cargo.toml`：

```toml
[dependencies]
core_lib = { path = "../core_lib" }
```

### 编译逻辑

1. 编译 `core_lib` → 生成库（`.rlib` 等）
2. 编译 `app_bin` 时 **把 core_lib 链接进来**
3. **最终只有 `app_bin.exe` 一个可执行文件**

### 运行时

- `core_lib` **永远不会单独出 exe**（没有 `main.rs`）
- `core_lib` + `app_bin` 代码 **全在同一进程**
- 互相调用 = **普通函数调用**，内存直接访问，**不需要 socket / IPC**

**B 能用的内容，严格限于 A 的 `lib.rs` 里 `pub` 导出的项** → [7.1.5](../07-packages-modules/7.1.5-lib.rs是Crate出口.md)

demo → [14.3-packaging-demo](./14.3-packaging-demo/)（`common` lib + `app_bin`）

## 二、形态 2：多个 bin Package → 【N 个 exe】= N 个进程

```
workspace/
├ Cargo.toml
├ market_server/     ← main.rs → market_server.exe
└ strategy_app/      ← main.rs → strategy_app.exe
```

```bash
cargo build -p market_server   # exe 1
cargo build -p strategy_app    # exe 2
```

两个 exe **分开启动** → **两个 OS 进程**，地址空间隔离，**不能直接 `use` 函数调用**。

### 跨进程通信（HFT 常用）

| # | 方式 | 场景 |
|---|------|------|
| 1 | **IPC**（Unix 套接字、共享内存 mmap、本机 MQ） | 同机，微秒级，HFT 本机内部 |
| 2 | **TCP/UDP / tonic(gRPC)** | 多机 RPC、UDP 行情广播 |
| 3 | **Kafka / Redis** | 中低频解耦、配置与持仓落地 |

demo → `14.3-packaging-demo/feed/` + `trade/`（各 1 exe，注释标明 IPC 位）

---

## 三、一句话区分

| | 打包结果 | 进程 | 调用 |
|---|----------|------|------|
| **Lib + Bin 依赖** | **1 个 exe** | 单进程 | 函数调用，无通信 |
| **Bin + Bin** | **N 个 exe** | 多进程 | IPC / 网络 |

---

## 四、补充：同一 Package 的 `src/bin/`

同一 Package 里 `src/main.rs` + `src/bin/*.rs` → 也是 **多个 exe、多进程**，同样需要 IPC。

区别只是 **没拆成独立 Package**，逻辑与 Workspace 多 bin 一致 → [7.1.4](../07-packages-modules/7.1.4-src-bin与多exe.md)

## 五、HFT 组合范例

```
workspace/
├ common/        ← lib：结构体、工具，**永不出 exe**，被链接
├ feed/          ← bin → feed.exe
└ trade/         ← bin → trade.exe
```

- `common` 被 `feed`、`trade` **各自链接进各自 exe**（编译期）
- `feed.exe` ↔ `trade.exe`：**两个进程**，本机用 **共享内存** 等发盘口（运行时）

```rust
// feed/src/main.rs（形态 2：进程内可调 common，跨进程需 IPC）
use common::format_tick;
fn main() {
    println!("{}", format_tick("BID", 100.0));
    // TODO: 写入共享内存 / Unix socket → trade 进程
}
```

## 六、与「微服务」的关系

| 层面 | 算不算微服务 |
|------|--------------|
| Workspace + `path` 链接（形态 1） | ❌ 代码组织 |
| 多 exe 分开部署 + 跨进程通信（形态 2） | ✅ 微服务运行时 |

tonic RPC 示意 → 见旧版扩展；**形态 1 内不要上 RPC**。
