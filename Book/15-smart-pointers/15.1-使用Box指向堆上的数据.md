## 核心结论（先背）

1. **递归 enum 直接嵌套自身** → 类型大小无限递归 → 编译报错；**`Box<List>`** 用固定宽指针打破循环，节点放堆上。
2. **`Box<T>`** = 栈上指针 + 堆上 `T`；无引用计数，零额外运行时开销。
3. **`Box` 靠 `Deref` + `Drop` 两大 trait 成为智能指针基石**：隐式当 `&T` 用；出作用域自动释堆。

## 一、`Box<T>` 本质

| 位置 | 存什么 |
|------|--------|
| **栈** | `Box` 实例（指针宽，64 位通常 8 字节） |
| **堆** | `T` 的真实数据 |

### 三大使用场景

| # | 场景 | 要点 |
|---|------|------|
| 1 | **递归类型** | 打破 `Sized` 要求；Cons 链表经典例 |
| 2 | **大数据上堆** | move 只拷栈上指针，不整块复制 |
| 3 | **`Box<dyn Trait>`** | trait 对象，抹平 impl 尺寸差 → [17.2](../17-oop/17.2-为使用不同类型的值而设计的trait对象.md) |

> 小类型（`i32`/`bool`）**不必**无脑 `Box`；优势在**间接存储**与**递归**。

## 二、递归类型报错 + Cons 链表（完整）

### 1. 报错原因：无限递归，算不出大小

Rust 要求**栈上类型必须 `Sized`**，编译期要确定占用字节数。

#### 错误代码

```rust
enum List {
    Cons(i32, List), // ❌
    Nil,
}
```

**拆解：**

- `Cons` = `i32` + `List`
- 内层 `List` 又可能是 `Cons(i32, List)` …
- **大小无限膨胀** → 编译器永远算不出 → 报错

形象理解：**盒子套盒子，没有尽头，没法确定单个盒子多大。**

### 2. 用 `Box<T>` 修复（标准解法）

```rust
enum List {
    Cons(i32, Box<List>), // ✅ 指针宽固定
    Nil,
}

use List::{Cons, Nil};
let list = Cons(1, Box::new(Cons(2, Box::new(Nil))));
```

#### 为什么 `Box` 能解决？

1. **指针本身大小固定**（64 位下 8 字节）
2. `Cons` = `i32` + `Box<List>` → **整体 `Sized`**
3. 下一节点在**堆**上，栈上只留指针

#### 内存分布

```text
栈：Cons 实例（i32 + 指向堆的指针）
堆：指针 → 下一个 List 节点（可能又是 Cons…）
```

层层节点在堆上，打破栈上无限递归。

> **业务列表优先 `Vec<T>`**；Cons 链表用于学习递归、`Box`、内存布局。→ demo：`cargo run`

## 三、基础语法与 `Deref`

```rust
let b = Box::new(100);
println!("{}", b);   // Deref：Box<i32> → &i32
println!("{}", *b);  // 手动解引用，等价思路
assert_eq!(*b, 100);
```

### `Deref` 做了什么？

标准库为 `Box<T>` 实现 `Deref<Target = T>`：

```rust
// 伪代码逻辑
impl<T> Deref for Box<T> {
    type Target = T;
    fn deref(&self) -> &T { /* 返回堆上 T 的引用 */ }
}
```

**简单记：`Deref` 让智能指针「伪装」成普通引用，直接访问内部数据。**

→ 全解：[15.2 Deref](./15.2-通过Deref将智能指针当作引用.md) · coercion · 方法自动解引用

## 四、`Drop` 与堆内存释放

```rust
{
    let b = Box::new(String::from("hello"));
    println!("{b}");
} // 离开作用域 → Box::drop → 先 drop String，再释放堆块
println!("作用域结束");
```

| 要点 | 说明 |
|------|------|
| **自动触发** | 出作用域 / `mem::drop(b)`；**禁止** `b.drop()` |
| **`Box` 行为** | 回收堆上 `T` 占用的内存 |
| **泛化** | `Rc`/`Arc`/`Vec`/锁守卫都依赖 `Drop` → [15.3](./15.3-使用Drop运行清理代码.md) |

→ demo：`cargo run -- drop`（带日志的堆上类型）

---

## 五、`Deref` + `Drop` = `Box` 为何是基石

```text
Box::new(x)  →  堆分配
使用 b / *b  →  Deref 当 &T
出作用域     →  Drop 释堆
```

后续 `Rc`/`Arc`/`RefCell` 都在此之上叠加引用计数、内部可变性等能力。

---

## 六、易错点

1. **`Box<T>` 拥有堆数据** — 不是 `&T` 借用
2. **move `Box`** — 只移动栈上指针，堆块不逐字节拷贝
3. **Cons 仅教学** — 工程用 **`Vec`**
4. **`Box::new` 必堆分配** — 无法强制放栈

---

## 七、一句话总结

1. 直接嵌套 `List` → **无限大小**；`Box<List>` → **固定指针 + 堆节点**。
2. **`Deref`**：智能指针 → 隐式 `&T`；**`Drop`**：生命周期结束 → 自动释堆。
3. **`Box<T>` = `Deref` + `Drop`**，是 Rust 智能指针链的起点。
