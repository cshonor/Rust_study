# 7.1.3 工业范例：Actix-web 项目目录（1 个 Package）

← 主笔记：[7.1 包与 Crate](./7.1-包和crate.md) · 前置：[7.1.1](./7.1.1-二进制与库crate.md) · [7.1.2 main 调分文件模块](./7.1.2-main调用分文件模块.md) · demo：[webserver_layout/](./7.1-packages-crates-demo/webserver_layout/)

以 Zero To Production 风格的 **Actix-web RESTful** 项目为例，把 7.1 规则落到真实目录。

---

## 一、整体属性

整个 `webserver/` 文件夹 = **1 个 Package**，同时包含：

| Crate | 入口 | 作用 |
|-------|------|------|
| **1 个 lib crate** | `src/lib.rs` | 业务、路由、工具；统一 `pub` 导出 |
| **1 个默认 bin crate** | `src/main.rs` | 读配置 → 启动 Web 服务 exe |

> 符合 Cargo：**一个 Package 最多 1 个 lib；二进制可 1 个或多个**（此处 1 lib + 1 main）。

```
webserver/                    ← Package 根目录
├ Cargo.toml
├ Cargo.lock
├ src/
│  ├ lib.rs                   ← 库 crate 入口：注册 + 导出总关口
│  ├ main.rs                  ← 二进制 crate 入口：只负责启动
│  ├ startup.rs               ← 服务初始化
│  ├ configuration.rs         ← 配置读取
│  ├ utils.rs                 ← 通用工具
│  ├ domain/
│  │  ├ mod.rs                 ← 文件夹模块入口
│  │  ├ email_client.rs
│  │  └ …
│  └ routes/
│     └ mod.rs
│  （无 src/bin/ — 只需 main.rs 一个 exe）
└ tests/
   ├ api.rs                    ← 集成测试 crate 入口（mod 子模块）
   └ api/
      └ health_check.rs
```

demo 骨架（无 Actix 依赖，结构相同）→ `7.1-packages-crates-demo/webserver_layout/`

## 二、逐个拆解

### 1. `src/lib.rs`（库 crate 核心）

| 职责 | 做法 |
|------|------|
| **注册子模块** | `pub mod domain;` · `pub mod routes;` · `pub mod utils;` … |
| **对外暴露 API** | 模块 / 函数 / 结构体加 `pub`；可选 `pub use` 缩短路径 |

`main.rs` 用法：

```rust
use webserver::configuration::get_configuration;
use webserver::startup::run;
```

→ **[7.1.2 有 lib 过 lib](7.1.2-main调用分文件模块.md)** 的工业标准写法。

#### `lib.rs` 标准导出模板

```rust
// 注册子模块（对应 src/ 下 .rs 或 文件夹/mod.rs）
pub mod configuration;
pub mod domain;
pub mod routes;
pub mod startup;
pub mod utils;

// 可选：再导出常用入口，main 路径更短
pub use startup::run;
```

> 新建 `xxx.rs` 或 `xxx/` 文件夹 → **必须在 `lib.rs` 里 `pub mod xxx;`**，否则 `main` / `tests` 无法 `use 包名::xxx`。

### 2. `src/main.rs`（二进制 crate）

**只做组装启动**，不写业务：

1. 读配置（`configuration`）
2. 调 `startup` 初始化
3. 启动 Actix-web 监听

业务、路由全在 lib 里 → **入口与业务分离**。

### 3. `domain/`（领域层）

- **`domain/mod.rs`**：文件夹模块入口；`pub mod email_client;` 等注册同目录各文件
- 由 **`lib.rs`** 的 `pub mod domain;` 对整个 crate 生效
- 放订阅、邮件等业务（DDD 领域层）

→ 详解见 **[§五 `domain/mod.rs`](#五domainmodrs-是什么不是-mobo)**

### 4. `routes/`（HTTP 接口层）

绑定 URL → handler；handler 调 `domain` 业务逻辑。

### 5. `tests/`（集成测试）

Cargo 规范：**项目根 `tests/` 下每个顶层 `.rs` = 独立集成测试 crate**；子目录用 `tests/api.rs` + `#[path = "api/…"]` 引入（见 demo）。

测试代码可直接：

```rust
use webserver_layout_demo::routes::health_check;
```

无需额外配置——测的是 **lib crate 对外 `pub` 的 API**。

## 三、知识点复盘

| # | 对应 7.1 |
|---|----------|
| 1 | 有 `lib.rs` → 业务在 lib 注册导出，`main` 依赖本包库 |
| 2 | 全项目 **1 个 `lib.rs`** → 遵守「Package 最多 1 个 lib crate」 |
| 3 | `main.rs` = 默认二进制 crate → 编译 web 服务 exe |
| 4 | 新文件 / 文件夹 → `lib.rs` 里 `pub mod xxx;`，否则无法 `use` |

## 四、编译与测试

| 命令 | 发生什么 |
|------|----------|
| `cargo build` | 先编 **lib crate** → 再编 **main**，链接 lib → 生成 exe |
| `cargo run` | 运行默认 bin（`main.rs`） |
| `cargo test` | 单元测试（lib 内 `#[cfg(test)]`）+ **`tests/` 集成测试** |

```
cargo build 顺序
  lib.rs（库 crate）→ .rlib
       ↓ 链接
  main.rs（bin crate）→ webserver.exe
```

## 五、`domain/mod.rs` 是什么（不是 mobo）

> **`mod.rs` = 文件夹的模块声明文件**：`domain/` 整个文件夹 = 名叫 `domain` 的一个 Rust 模块。

```
src/domain/
 ├ mod.rs              ← 标记 domain 是完整模块
 ├ email_client.rs
 ├ new_subscriber.rs
 ├ subscriber_email.rs
 └ subscriber_name.rs
```

| | 无 `mod.rs` | 有 `mod.rs` |
|---|-------------|-------------|
| `domain/` | 只是普通文件夹，编译器**不会**自动打包成模块 | **整个文件夹 = `domain` 模块** |

### 两层注册

| 层级 | 文件 | 写法 |
|------|------|------|
| **文件夹内** | `domain/mod.rs` | `pub mod email_client;` · `pub mod new_subscriber;` … |
| **crate 根** | `lib.rs` | `pub mod domain;` → 整个 domain 对外开放 |

最终使用：

```rust
use webserver::domain::email_client::EmailClient;
```

### `domain/mod.rs` 完整示例

```rust
pub mod email_client;
pub mod new_subscriber;
pub mod subscriber_email;
pub mod subscriber_name;
```

### `lib.rs` 侧

```rust
pub mod domain;
pub mod routes;
pub mod utils;
```

> **一句话**：单个 `.rs` 文件用 `lib.rs` 里 `mod xxx;` 即可；**文件夹要变成模块，必须放 `mod.rs`**（或在 crate 根用 `domain.rs` + `domain/` 子目录，见 [7.5](./7.5-将模块分割进不同文件.md)）。

demo → `webserver_layout/src/domain/mod.rs`

## 六、为什么没有 `src/bin/`？

### 二进制 crate 两种写法

| 写法 | 何时用 |
|------|--------|
| **`src/main.rs`** | **默认主二进制**（至多 1 个）；单体 web 项目 **只需要 1 个 exe** → **绝大多数 REST 后端只用这个** |
| **`src/bin/xxx.rs`** | 需要**多个独立 exe** 时才建（服务端 + CLI 初始化 + 数据迁移 …） |

### 本项目现状

- 只启动 **一个 web 服务** → 只用 `src/main.rs`，**不必建 `src/bin/`**，完全合规
- 结构：**1 Package → 1 lib crate（`lib.rs`）+ 1 bin crate（`main.rs`）**

> 无 `src/bin` ≠ 缺东西；是「只要一个 exe」的正常选择。多 exe 范例 → [7.1 §三](./7.1-包和crate.md) · demo `multi_bin/`

## 七、完整调用链路

```
email_client.rs 等
       ↑ pub mod … 注册
domain/mod.rs
       ↑ pub mod domain;
src/lib.rs（lib crate root）
       ↑ use webserver::domain::email_client::…
src/main.rs（bin crate root）→ cargo run → webserver.exe
```

1. domain 内各业务文件 → 在 **`domain/mod.rs`** 注册为子模块  
2. domain 整体 → 在 **`lib.rs`** 写 `pub mod domain;` 对外导出  
3. **`main.rs`**：`use webserver::domain::email_client::…` 调用（经 lib 中转）

## 八、小总结

| # | 要点 |
|---|------|
| 1 | `mod.rs` 绑定**整个文件夹**为一个模块，收拢文件夹内各 `.rs` |
| 2 | 单个 `.rs` 不需要 `mod.rs`，在上一级 `mod xxx;` 即可 |
| 3 | 无 `src/bin` = 只编译 **一个 exe（`main.rs`）**，REST 后端最常见 |
