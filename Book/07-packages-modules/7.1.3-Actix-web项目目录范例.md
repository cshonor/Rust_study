# 7.1.3 工业范例：Actix-web 项目目录（1 个 Package）

← 主笔记：[7.1 包与 Crate](./7.1-包和crate.md) · 前置：[7.1.1](./7.1.1-二进制与库crate.md) · [7.1.2](./7.1.2-main调用分文件模块.md) · demo：[webserver_layout/](./7.1-packages-crates-demo/webserver_layout/)

以 Zero To Production 风格的 **Actix-web RESTful** 项目为例，把 7.1 规则落到真实目录。

---

## 一、整体属性

整个 `webserver/` = **1 个 Package**：

| Crate | 入口 | 作用 |
|-------|------|------|
| **1 个 lib crate** | `src/lib.rs` | 业务、路由、工具；统一 `pub` 导出 |
| **1 个 bin crate** | `src/main.rs` | 读配置 → 启动 Web exe |

```
webserver/
├ Cargo.toml
├ src/
│  ├ lib.rs
│  ├ main.rs
│  ├ startup.rs · configuration.rs · utils.rs
│  ├ domain/          ← lib 内部子模块，不是独立 crate
│  │  ├ mod.rs
│  │  └ …
│  └ routes/
│     └ mod.rs
│  （无 src/bin/ — 只需 main.rs 一个 exe）
└ tests/
   ├ api.rs
   └ api/…
```

demo → `webserver_layout/`（无 Actix，结构相同）

## 二、Crate vs Module（domain 不是独立 crate）

> ✅ **domain 不是独立 crate，只是当前 lib crate 内部的子模块。**

### 层级（从上到下）

```
Package（webserver/）
└ lib crate（src/lib.rs = 整箱，编译单元）
   └ domain（箱内一格 = 模块，不是箱子）
      └ domain/mod.rs + 各业务 .rs
```

| 概念 | 比喻 | 说明 |
|------|------|------|
| **crate** | **整箱** | 编译最小单元；`lib.rs` = 1 个 lib 箱，`main.rs` = 1 个 bin 箱 |
| **module** | **箱内格子** | `domain` / `routes` / `utils` 全是 **crate 内代码分组** |

> module **永远升不成**独立 crate；要独立 crate → **拆成新 Package**（新文件夹 + 新 `Cargo.toml`）。

模块私有性条文 → [7.2](./7.2-定义模块来控制作用域与私有性.md)

## 三、三步生效流程（mod + pub）

### 第一步：`domain/mod.rs` 收拢文件夹内文件

```rust
// src/domain/mod.rs
pub mod email_client;
pub mod new_subscriber;
pub mod subscriber_email;
pub mod subscriber_name;
```

→ 告诉编译器：**这些零散 `.rs` 都属于 `domain` 模块**。

### 第二步：`lib.rs` 把 domain 挂进 lib crate

```rust
// src/lib.rs
mod domain;      // 收录进 crate，仅 lib 内部可访问
pub mod domain;  // 加 pub → main / tests / 外部依赖可 use
```

| 写法 | 效果 |
|------|------|
| `mod domain;` | domain **私有** → 只有 lib crate 内部能用 |
| `pub mod domain;` | domain **公开** → `main`、tests、`use 包名::domain` |

### 第三步：`main.rs` / `tests` 调用

```rust
use webserver::domain::email_client::EmailClient;
```

包名 = `Cargo.toml` 的 `name`；路径从 **lib 导出的公开模块** 进入。

### 口诀

**文件夹成模块靠 `mod.rs` 收拢内部文件；`lib.rs` 里 `mod` 收录进 crate，加 `pub` 才能外面调用。**

## 四、各目录职责（简表）

| 路径 | 角色 |
|------|------|
| **`lib.rs`** | lib crate root；`pub mod domain;` 等统一导出 |
| **`main.rs`** | bin crate root；只组装启动，不写业务 |
| **`domain/mod.rs`** | 文件夹模块入口；注册 `email_client` 等 |
| **`routes/`** | HTTP 层；调 domain 业务 |
| **`tests/`** | 集成测试；`use 包名::…` 测 lib 公开 API |

#### `lib.rs` 导出模板

```rust
pub mod configuration;
pub mod domain;
pub mod routes;
pub mod startup;
pub mod utils;

pub use startup::run;
```

## 五、为什么没有 `src/bin/`？

| 写法 | 何时用 |
|------|--------|
| **`src/main.rs`** | 只要 **1 个 exe**（单体 REST 后端）→ **常态** |
| **`src/bin/*.rs`** | 要多个独立 exe 时才建 |

本项目：**1 lib + 1 main**，无 `src/bin` 完全合规。多 bin 范例 → [7.1 §三](./7.1-包和crate.md) · `multi_bin/`

---

## 六、完整调用链

```
email_client.rs …
      ↑ pub mod …（domain/mod.rs）
domain 模块
      ↑ pub mod domain;（lib.rs / lib crate）
main.rs / tests
      use webserver::domain::email_client::…
      ↓
cargo run → webserver.exe（bin crate 链接 lib crate）
```

---

## 七、何时把 domain 拆成独立 Package（独立 lib crate）？

**默认不用拆**——同项目内 `pub mod domain;` 即可。

考虑**新 Package**（如 `domain_core/` + `path` 依赖）当：

| 场景 | 原因 |
|------|------|
| **多项目复用** | 多个 web / CLI Package 共用同一套 domain |
| **独立发布** | 要上 crates.io，版本与主服务分开 |
| **编译隔离** | domain 极大，希望单独 `cargo test` / 单独 CI |
| **依赖边界** | domain 不应依赖 Actix/HTTP，物理隔离防循环依赖 |

拆法：新建文件夹 + `Cargo.toml` + `src/lib.rs` → 原项目 `[dependencies] domain_core = { path = "../domain_core" }`。

日常单体 web：**domain 留在 lib 内当模块即可**（本笔记范例）。

## 八、编译与测试

| 命令 | 发生什么 |
|------|----------|
| `cargo build` | 先编 lib crate → 再编 main，链接 → exe |
| `cargo run` | 运行 `main.rs` |
| `cargo test` | lib 内单元测试 + `tests/` 集成测试 |

## 九、知识点复盘

| # | 要点 |
|---|------|
| 1 | **domain = module**，不是 crate |
| 2 | 全项目 **1 个 `lib.rs`** → 最多 1 个 lib crate |
| 3 | `main.rs` = 默认 bin crate |
| 4 | 新文件夹 → `mod.rs` + `lib.rs` 里 `pub mod` |
| 5 | 无 `src/bin` = 单 exe，REST 后端常见 |
