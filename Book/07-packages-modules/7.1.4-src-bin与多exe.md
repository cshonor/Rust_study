# 7.1.4 `src/bin/`：多 exe、多工具 vs 微服务 vs HFT

← 主笔记：[7.1 包与 Crate](./7.1-包和crate.md) · demo：`multi_bin/` · HFT 架构 → [14.3 Cargo 工作空间](../14-cargo-crates/14.3-Cargo工作空间.md)

---

## 一、`.rs` 是什么

**`.rs` = Rust 源文件后缀**（Rust Source）。  
Cargo 按**路径规则**决定它是 lib、main 还是 `src/bin/` 下的独立二进制——**不要求**文件名叫 `main.rs`。

---

## 二、`src/bin/` 目录：每个 `.rs` → 一个 exe

只要放在 **`src/bin/`** 下，**每个 `.rs` 文件 = 一个独立的 binary crate**，编译出**各自的 exe**：

```
src/bin/
 ├ cli.rs      → cli.exe     cargo run --bin cli
 ├ migrate.rs  → migrate.exe cargo run --bin migrate
 └ report.rs   → report.exe  cargo run --bin report
```

| 要点 | 说明 |
|------|------|
| 文件名 | **不必**叫 `main.rs`；文件名 = 二进制 crate 名（`cli.rs` → `--bin cli`） |
| 入口 | 每个文件里仍要有 **`fn main()`** |
| 共用代码 | 同 Package 的 **`lib.rs`** 可被各 bin `use 包名::…` |

demo → `7.1-packages-crates-demo/multi_bin/`

---

## 三、没有 `src/bin/` 时：默认只有一个 exe

| 情况 | 结果 |
|------|------|
| 只有 **`src/main.rs`**，无 `src/bin/` | **默认唯一**二进制入口 → 编译 **1 个 exe** |
| 典型例子 | Actix-web 后端：只跑一个 web 服务，**不必建 `src/bin/`** |

→ 见 [7.1.3](./7.1.3-Actix-web项目目录范例.md)

## 四、`src/bin/` 是干什么的？（不是微服务）

### ✅ 正确理解：一个 Package 里的「多工具 / 多命令行程序」

**集中管理多个独立可执行程序**，共用同一套 `lib` 代码，各自单独运行：

| exe | 用途示例 |
|-----|----------|
| `main.rs` | Web 服务 / 主程序 |
| `bin/migrate.rs` | 数据库迁移工具 |
| `bin/report.rs` | 离线报表生成 |
| `bin/replay.rs` | 日志回放调试 |

→ **像一套工具箱**：同一仓库、同一 lib、多个入口，**不是**按服务拆部署。

### ❌ 不是：微服务架构

| | `src/bin/`（同 Package） | 微服务（多 Package 部署） |
|---|--------------------------|---------------------------|
| 部署 | 多个 exe，常在同一机器 / 同一发布包 | **独立进程、独立部署单元**，常不同机器/容器 |
| 边界 | 共享 lib，编译期绑在一起 | 网络 API、独立生命周期 |
| 典型场景 | CLI + 服务端 + 批处理脚本 | 订单服务、行情服务、用户服务各自一个项目 |

> **微服务**通常 = **多个独立 Package / 仓库 + 独立部署**；不是靠在一个项目里多加几个 `src/bin/*.rs` 就算微服务。

## 五、HFT 场景：更常用 Workspace，而不是堆 `src/bin/`

高频交易更倾向 **Cargo Workspace**：把核心拆成**多个独立 Package（crate）**，统一根 `Cargo.toml` 管理：

```
hft-workspace/
├ Cargo.toml          # [workspace] members = [...]
├ hft-core/           # 基础类型、协议、零拷贝工具
├ hft-market-data/    # 行情接入、解码
├ hft-trading-engine/ # 撮合 / 发单引擎（独立 exe）
└ hft-gateway/        # 对外网关（独立 exe）
```

| 对比 | `src/bin/`（单 Package 多 exe） | Workspace（多 Package） |
|------|--------------------------------|-------------------------|
| 隔离 | 弱：同一 Package、同一 lib 边界 | **强**：crate 级依赖边界、可单独编译测试 |
| 延迟 / 可靠 | 适合小工具共用 lib | 引擎、行情、核心类型**物理拆分**，避免无关代码进热路径 |
| 依赖 | 所有 bin 共享同一 `Cargo.toml` 依赖 | 各 crate **按需**声明依赖（引擎不必拉 web 框架） |

**HFT 常见做法**：

- **热路径引擎** → 独立 Package，`release` 优化、依赖最小化  
- **工具链**（回放、对账、配置生成）→ 可同 workspace 其他 member，或单独 Package  
- **少用大 Package + 很多 `src/bin/`** 扛架构拆分；用 **workspace + 多 crate** 做模块解耦与编译隔离  

详解 → [14.3 Cargo 工作空间](../14-cargo-crates/14.3-Cargo工作空间.md) · demo `14.3-workspace-demo/`

---

## 六、怎么选（速查）

| 需求 | 推荐 |
|------|------|
| 只要 1 个程序（web 服务） | 仅 `src/main.rs`，无 `src/bin/` |
| 同项目多个小工具，共用 lib | `src/main.rs` + `src/bin/*.rs` |
| 要多套**库**、严格边界、HFT / 大单体拆分 | **多个 Package + workspace** |
| 微服务独立部署 | **多个 Package**（各服务一个 repo 或 workspace member），网络通信 |

---

## 口诀

- **`src/bin/` 里每个 `.rs` = 一个 exe**，名字不必是 `main.rs`  
- **无 `src/bin/` → 默认只有 `main.rs` 那一个 exe**  
- **多工具用 `src/bin/`；多服务 / HFT 拆边界用 workspace + 多 Package**
