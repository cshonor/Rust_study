## 核心结论（先背）

1. **unsafe ≠ 关闭安全检查** — 程序员承诺维护不变式；`&T`/`&mut T` 借用规则仍有效
2. **五大 unsafe 超能力**：裸指针解引用 · `unsafe fn` · `static mut` · `unsafe impl` · `union` 字段
3. **原则**：缩小 `unsafe` 块，封装为**安全 API**（如 `split_at_mut`）
4. **典型场景**：FFI、底层数据结构、性能关键路径（仍须证明安全）

---

常规 Rust 在编译期检查内存安全；**unsafe** 允许五类编译器**无法代为证明**安全的操作，由开发者承担契约责任。

## 为何需要 unsafe

静态分析偏保守：宁可拒绝部分合法程序，也不误放行 UB。若你能证明局部合法，可用 `unsafe { ... }` 向编译器说明“此处由我负责”。**unsafe 不会关闭借用检查**：对 `&T` / `&mut T` 的规则仍适用。

五类 **unsafe 超能力**：

1. **解引用裸指针** `*const T` / `*mut T`
2. **调用 `unsafe fn` 或 unsafe 方法**
3. **读/写 `static mut`**
4. **`unsafe impl` 不安全 trait**
5. **访问 `union` 字段**

实践上应尽量：**缩小 unsafe 块**，并把 unsafe 封装成**安全 API**（如标准库的 `split_at_mut`）。

---

## 裸指针

- 可在**安全代码里创建**裸指针；**解引用**必须在 `unsafe` 中。
- 不保证有效、可为 null、可同时存在多个可变指针——易数据竞争，需自行保证不变式。

---

## `unsafe fn` 与封装

`unsafe fn` 表示调用者须满足额外契约，须在 `unsafe { }` 或另一个 `unsafe fn` 体中调用（函数体本身视作 unsafe 上下文）。

---

## `split_at_mut` 思路

借用检查器无法证明“同一 slice 的两段可变借用不重叠”；可用 `as_mut_ptr` + `slice::from_raw_parts_mut` 在**已证明 `mid <= len` 且两段不重叠**的前提下构造两段 slice（见 demo）。

---

## FFI：`extern "C"`

在 `extern "C" { ... }` 中声明的外部函数**在 Rust 侧一律视为 unsafe 调用**（他种语言不保证 Rust 规则）。`"C"` 指定 **ABI**。

从其它语言调用 Rust：在函数上使用 `pub extern "C" fn` 并常配合 `#[no_mangle]`；若做成动态库供 C 链接，需 `crate-type = ["cdylib"]` 等（见 Cargo 文档）。

---

## 静态变量

- **不可变 `static`**：全局固定地址；访问安全。
- **`static mut`**：读/写均须在 **unsafe** 中（多线程下极易数据竞争，优先用第 16 章并发原语）。

---

## `unsafe trait`

当 trait 的不变量无法由编译器验证时，可声明 `unsafe trait`，实现须 `unsafe impl`（例如手动为类型实现 `Send`/`Sync` 时）。

---

## `union`

与 C 互操作时常见；访问字段须在 **unsafe** 中，因 Rust 不知当前激活的变体。
