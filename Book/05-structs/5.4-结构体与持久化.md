# 5.4 结构体与持久化（HFT 向）

← 结构体基础：[5.1](./5.1-定义并实例化结构体.md) · Serde/JSON：[3.2 JSON 与编码](../03-common-concepts/3.2-JSON与编码.md)

**一句话**：落地存盘/入库的**结构化数据**几乎都用 `struct` 规整字段；定义放 **`model/`**，读写放 **`db/`**（或 `storage/`）。

对应 demo：[5.4-persist-json-demo](./5.4-persist-json-demo/)（结构体 → JSON 文件 → 读回）

---

## 一、为什么持久化优先用结构体？

持久化 = 内存里的结构化数据 → 字节（JSON / Protobuf / 自定义二进制）→ 磁盘或数据库。

```rust
/// HFT 成交记录：入库、写日志文件时常用这样一个 struct 打包字段
struct Trade {
    order_id: u64,
    code: String,      // 持久化用 String；协议常量可用 &'static str
    price: f64,
    volume: u64,
    timestamp: u64,
}
```

- **零散变量**不好整体序列化；**一张表 / 一条记录 = 一个 struct**，列 ↔ 字段一一映射。
- **不是只能用 struct**：单个标量可直接写；**一批关联字段**必用 struct。

---

## 二、工程里 struct 放哪？

### 规范目录（HFT 常见）

```
src/
├── model/          # 只放数据模型 struct，不写 IO
│   ├── trade.rs
│   ├── quote.rs
│   └── mod.rs
├── db/             # SQL、文件 IO、序列化、落地
└── main.rs
```

| 层 | 职责 |
|----|------|
| **model** | `Trade`、`Quote` 等字段定义 |
| **db** | `import model::*`，序列化后写 MySQL / 文件 |

小工具/练习：struct 写在当前 `*.rs` 顶部，下面跟读写逻辑即可（见 demo）。

---

## 三、完整链路

1. 收行情 → 填进 `Quote`（内存 struct）
2. **序列化**：`serde` / Protobuf → 字节
3. **db 层**：写字节到文件或数据库 → **持久化**
4. 重启：**反序列化** → 还原为 struct

---

## 四、派生宏（工业标配）

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
struct Quote {
    stock_id: String,
    bid: f64,
    ask: f64,
}
```

- **`Serialize`**：struct → JSON/二进制
- **`Deserialize`**：字节 → struct

本地文件示例见 demo：`cargo run` 生成 `trade.json` 再读回。

## 六、编码：Rust UTF-8 ↔ 数据库 / 文件

### Rust 端（硬性）

`String` / `&str` / `str` **语言层固定 UTF-8**；`serde_json`、写文件输出的文本字节默认就是 UTF-8。**Rust 里不能把字符串改成 GBK 存储**，只能在**字节边界**用 `encoding_rs` 等转码（见 [3.2 JSON 与编码](../03-common-concepts/3.2-JSON与编码.md)）。

```rust
#[derive(Serialize, Deserialize)]
struct Trade {
    stock_name: String, // 内存 UTF-8 → JSON/入库全程 UTF-8 字节
}
```

### 数据库（推荐统一 utf8mb4）

| 项 | 建议 |
|----|------|
| MySQL | **`utf8mb4`** + `utf8mb4_unicode_ci`（勿用阉割 3 字节 `utf8`） |
| PostgreSQL / SQLite | 多数默认 UTF-8，一般开箱即用 |
| 建表 | `CHARSET=utf8mb4` |
| 连接串 | `charset=utf8mb4`，驱动传输 UTF-8 |

```sql
CREATE TABLE trade (
    order_no   VARCHAR(32),
    stock_name VARCHAR(64)
) CHARSET=utf8mb4;
```

**三层统一**：实例默认编码 → 表 CHARSET → 客户端连接 charset，与 Rust 对齐 → **零转码**。

### 非 UTF-8 库（GBK 等）？

技术上可在入库前 UTF-8→GBK、出库再转回，但**不推荐**：易乱码、多一次转码、HFT 日志/中文备注增加线上风险。**工业项目：库表连接全 utf8mb4**。

### 文件

JSON / 文本日志：直接写 `String`（UTF-8）即可；二进制协议字段用 `Vec<u8>` / `&[u8]`，与文本编码无关。

---

## 七、HFT 提醒

- 热路径日志/入库：**model 定义清晰**，冷路径再序列化；热路径避免频繁 **`clone` 整包**。
- 线上报文解析多用 **`&[u8]` 零拷贝**（见 [4.3](../04-ownership/4.3-切片slice.md)）；**落地存档**再用 struct + 序列化。

---

## 八、和 C `struct` 的区别

| | C `struct` | Rust `struct` |
|--|------------|-----------------|
| 布局 | 内存排列 | 同样可控制布局（`#[repr(C)]` 等） |
| 持久化 | 手写序列化 | 常配 **`serde`** 等派生，一键互转 |
