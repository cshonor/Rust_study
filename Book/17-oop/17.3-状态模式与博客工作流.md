# 17.3 状态模式与博客工作流

对外只暴露 **`Post`**（或工作流入口）时，可用**状态模式**把「草案 / 待审 / 已发布」的规则收拢在状态对象里，避免调用方误操作。本章给出两种常见取舍：

1. **运行时状态**：`Post` 内持有 `Option<Box<dyn State>>`，状态转移由 `request_review` / `approve` 等驱动（trait 对象 + **动态分发**）。
2. **编译期状态**：用 **`DraftPost` / `PendingReviewPost` / `Post`** 等不同类型表示阶段，非法操作直接**不能通过编译**（类型级状态机）。

对应 demo：`Book/17-oop/17.3-blog-workflow-demo/`（模块 `blog_state` 与 `blog_typestate`）

---

## 1. 状态模式（trait 对象）

- **`state: Option<Box<dyn State>>`**：`take()` 取出旧状态所有权，再写入新状态；`Option` 避免结构体字段“暂时为空”与借用规则冲突。
- **`State` 方法签名**常用 **`self: Box<Self>`**，在 **Box** 上消费当前状态并返回 **`Box<dyn State>`**，完成转移。
- **`content` 委托**：`Post::content` 通过 `state.as_ref().unwrap().content(self)` 把规则放到各 `State` 实现里；`Published` 覆盖默认空串，返回 `&post.content`。
- **生命周期**：`fn content<'a>(&self, post: &'a Post) -> &'a str` 表明返回的切片与 `post` 正文同生命期。

**权衡**：规则集中在各 `State` 实现；但状态之间可能耦合（例如新增中间状态要改相邻状态的转移）。`request_review` / `approve` 在 `Post` 上往往形态相似（可宏消除重复，见第 19 章）。

---

## 2. 将状态编码为类型（typestate）

- **`Post::new()` → `DraftPost`**：对外不能直接构造已发布的 `Post`（字段私有且无其它入口）。
- **`DraftPost::request_review(self) -> PendingReviewPost`**：消费草案，进入待审。
- **`PendingReviewPost::approve(self) -> Post`**：消费待审，得到**唯一**带 `content()` 的已发布类型。
- 草案/待审类型**不提供** `content()`，误读未发布内容会在**编译期**失败。

**权衡**：状态转移不再完全封装在一个 `Post` 类型内部，调用方需用 `let post = post.request_review()` 等形式保存新类型；换来更少运行时错误。

---

## 3. trait 对象与 Rust

- 状态模式常用 **动态分发**；**类型状态**更多依赖 **所有权移动** 与 **泛型/具体类型**，常得静态检查之利。
- OOP 模式在 Rust 中可行，但不必处处照搬；**所有权 + 类型系统**往往提供更便宜的正确性保证。

## 小结（第 17 章与面向对象）

**Trait 对象**（如 `Box<dyn Trait>`）能在运行时提供类似部分面向对象设计所需的灵活性（动态分发），适合状态模式等场景。Rust 同时还有**所有权**、**将状态编码为类型**等工具，不必处处沿用传统 OOP；权衡在于可维护性、性能与编译期保证之间的取舍。
