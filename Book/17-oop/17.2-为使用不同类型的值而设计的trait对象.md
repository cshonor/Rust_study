# 17.2 为使用不同类型的值而设计的 trait 对象

编译期若已知“固定几种类型”，可用枚举（如第 8 章 `SpreadsheetCell`）在同一条 `Vec` 里存不同形态的数据。若希望**用户日后扩展类型集合**（例如 GUI 里新增 `Image`、`SelectBox`），库方无法事先枚举所有具体类型，这时常用 **trait 对象** 表达“只要实现了某 trait，就能放进同一集合”。

对应 demo：`Book/17-oop/17.2-trait-objects-demo/`（`Draw` + `Screen` + `Button`，使用者在 `main` 里实现 `SelectBox`）

---

## trait 对象：`dyn Trait` + 指针

通过**指针**（如 `&dyn Trait`、`Box<dyn Trait>`）与 **`dyn`** 关键字，把“实现了指定 trait 的具体类型”**擦除**为同一 trait 对象类型，在运行时再通过 **vtable** 解析要调用的方法。

→ **胖指针两字段如何分工**：[10.2.6](../10-generics-traits-lifetimes/10.2.6-dyn胖指针与虚表.md) · **通俗全解**：[10.2.8 vtable](../10-generics-traits-lifetimes/10.2.8-vtable通俗全解.md)

典型集合类型：

```rust
Vec<Box<dyn Draw>>
```

`Screen::run` 对每条 `component` 调用 `draw()`，无需知道具体是 `Button` 还是 `SelectBox`。

---

## 与泛型 + trait bound 的对比

```rust
pub struct Screen<T: Draw> {
    pub components: Vec<T>,
}
```

泛型 `T` **一次**只能对应**一个**具体类型，因此 `Vec<T>` 里通常是**同质**组件（全是 `Button` 或全是 `TextField`）。单态化带来 **静态分发**。

`Vec<Box<dyn Draw>>` 可在**同一** `Vec` 里混放多种实现 `Draw` 的类型，代价是 **动态分发**（运行时查方法）。

---

## 鸭子类型与类型安全

“能走能叫就当鸭子”——在 Rust 里，若类型未实现 `Draw`，则无法放进 `Box<dyn Draw>`，**编译期**报错，而不是运行时才发现没有方法。

---

## 静态分发 vs 动态分发

→ **完整对比表（6 维 + 口诀）**：[10.2.4 impl vs dyn](../10-generics-traits-lifetimes/10.2.4-impl与dyn-Trait对比.md#五完整对比表impl-vs-dyn)

- **静态分发**：单态化，编译期已知调用哪个实现，便于内联优化。
- **动态分发**：trait 对象，编译期不知所有具体类型，运行时决议；部分优化受限。

---

## 对象安全（object safe）

只有 **对象安全** 的 trait 才能作为 `dyn Trait`。实践中常记两条：

1. 方法返回类型**不是** `Self`（否则无法在未知的具体类型上构造返回值）。
2. 方法**没有**额外泛型参数（泛型参数会参与具体类型的“形状”，trait 对象已抹掉具体类型）。

因此 `Clone`（`fn clone(&self) -> Self`）**不能**作为 `dyn Clone` 使用。
