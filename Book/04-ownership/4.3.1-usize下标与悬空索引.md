# 4.3.1 `usize` 下标与悬空索引

← 主笔记：[4.3 切片](./4.3-切片slice.md) · demo：[4.3-slices-demo](./4.3-slices-demo/)

---

## 旧方案：返回 `usize`

```rust
fn first_word_index(s: &String) -> usize {
    let bytes = s.as_bytes();
    for (i, &b) in bytes.iter().enumerate() {
        if b == b' ' {
            return i;
        }
    }
    s.len()
}
```

```rust
let mut s = String::from("hello world");
let idx = first_word_index(&s);
s.clear(); // 清空原字符串
// idx 仍是 5，但字符串已空 —— 悬空索引，编译器查不出（The Book 4.3 原话）
```

---

## 为啥 `idx` 是「裸数字」

1. **`usize` 只是栈上的普通整数**：不存指针、不绑定 `s`、不带生命周期。
2. **借用检查只管 `&`，不管普通数字**：编译器不会追踪「5 对应哪段串、串有没有被 `clear()`」。

**澄清**：数字 5 本身不是悬空；**在 `clear()` 后用 `&s[0..idx]` 访问**才是悬空访问（类似 C：`pos=5` 后 `memset` 再 `arr[pos]`）。

```c
char arr[] = "hello world";
int pos = 5;
memset(arr, 0, sizeof(arr));
// pos 仍是 5；arr[5] 越界
```

---

## 什么是悬空索引

- 当初 `5` 表示：原串第 5 字节处是空格；
- `s.clear()` 后 `len() == 0`，**数字仍是 5，语义已失效**；
- 再 `&s[0..idx]` → 越界；**`clear` 与 `idx` 可同存编译通过**，常到**运行时**才 panic。

---

## 对照：`usize` vs 切片

| 对比 | `usize` 裸下标 | `&str` / `&[u8]` |
|------|----------------|------------------|
| 本质 | 普通整数 | 绑定原容器的引用 |
| 原数据被改 | 数字不变，编译器不管 | 有切片存活时禁止可变修改 |
| 出错时机 | 往往**运行时** | 往往**编译期** |

`&str` 为何更安全 → [4.3.2 切片与借用规则](./4.3.2-切片与借用规则.md)。

## HFT：`Vec` 报文缓冲区

```rust
// ❌ 裸 usize
let mut buf: Vec<u8> = receive_udp();
let split_pos: usize = 16;
buf.clear(); // split_pos 仍是 16 → 后续按下标解析悬空

// ✅ &[u8] 子切片
let part: &[u8] = &buf[0..16];
// buf.clear(); // part 仍存活 → 编译错误
```

## 小结

| 要点 | 一句 |
|------|------|
| `usize` | 只是数；用来访问已清空数据 → 悬空**访问** |
| 切片 | 绑定原内存，编译期拦非法修改 |
