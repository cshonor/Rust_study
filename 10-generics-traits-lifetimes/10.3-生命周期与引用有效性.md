# 10.3 生命周期与引用有效性

对应 demo 项目：`10-generics-traits-lifetimes/10.3-lifetimes-demo/`  
所有权背景（栈/堆、字面量、无 drop）：[4.1-什么是所有权.md](../04-ownership/4.1-什么是所有权.md)

---

## 核心结论（先记这句）

生命周期标记（如 `'a`、`'static`）**不会创建新引用**，而是给引用**绑定「指向的数据能活多久」的约束**——**不是**描述引用变量本身活多久。

> 入参标 `'a`、返回值也标 `'a`  
> → 返回的引用**一定来自**带同一 `'a` 的入参（或与之同寿命的数据），**不能比入参活得更久**。

---

## 一、两个概念：变量生命周期 vs 生命周期标注

### 1. 普通变量的「生命周期」（客观时长）

从**声明**到**离开作用域被销毁**，这段实际存活时间就是它的真实生命周期——运行时客观行为。

```rust
fn main() {
    let num = 10;
    let r = &num;
} // num、r 一同销毁
```

### 2. 生命周期标注 `'a`（编译器的约束标签）

`'a` **不是**描述变量本身活多久，而是**给引用加的一套规则约定**：

> 所有标了同一个 `'a` 的引用，它们**指向的数据**，存活时间必须**互相兼容**（取最短上限）。

它是编译期的**语法约束**，用来排查悬垂引用，**不改变**变量本身的销毁规则。

---

## 二、生命周期标注 vs 作用域：两套独立规则

| 规则 | 管什么 |
|------|--------|
| 生命周期标签（`'a`、`'static`） | **被指向的数据**能存活多久 |
| 代码块 `{}` | **引用变量本身**什么时候销毁 |

```rust
fn main() {
    let s: &'static str = "test";
} // 变量 s（引用本身）消失；"test" 仍在只读段
```

两者独立：`s` 可以消失，但 `'static` 数据仍在；`r` 和 `num` 可以同生共死，由 `'a` 保证引用不越界。

---

## 三、类型拆解：`&'static str`

| 部分 | 含义 |
|------|------|
| `&` | **引用**——借用，不持有所有权 |
| `'static` | **生命周期标注**——指向的数据与程序同生共死 |
| `str` | 被引用的原生字符串类型 |

> 不存在单独的 `'static str`，一定是 **`&'static str`**。

```rust
let s: &'static str = "abc";
// s 是栈上局部引用，指向只读数据段的 "abc"
// 'static 修饰的是 "abc" 这份数据，不是 s 本身
```

**合法返回示例**：

```rust
fn get_str() -> &'static str {
    let s = "hello";
    s // 数据在只读段，局部 s 消失后数据仍在
}
```

**非法返回对比**：

```rust
fn bad() -> &str {
    let st = String::from("hi");
    &st // ❌ st 出函数 drop
}
```

## 四、`'a` 到底约束什么

```rust
fn borrow<'a>(data: &'a i32) -> &'a i32 {
    data
}

fn main() {
    let num = 10;
    let r = borrow(&num);
} // r、num 一同销毁——引用不能比数据活得更久
```

**函数签名解读**：

| 部分 | 含义 |
|------|------|
| `data: &'a i32` | 入参引用指向的数据打上标签 `'a` |
| `-> &'a i32` | 返回引用指向的数据也必须是 `'a` 那一份 |

### 违反约束 → 编译报错

```rust
fn bad<'a>() -> &'a i32 {
    let tmp = 20;
    &tmp // ❌ tmp 出函数销毁，无入参提供 'a 数据源
}
```

### 三条约束规则（好记）

1. `'a` 是**时长分组标签**，给「引用能合法使用的最长时间」分组——**不是变量绑定标签**；
2. 同标签 `'a` 的引用，数据源**存活时长须互相兼容**（可以是不同变量，不要求是同一个）；
3. 引用**可用时长 ≤ 原始数据的生命周期**（整组 `'a` 取**最短**数据源为上限）。

### 核心疑问速答

- **`'a` 是什么？** **时长分组**——同组引用的**最晚可用时间**必须一致（取最短数据源为准）。
- **引用变量会被销毁吗？** **会。** `'a` 只管：在使用引用期间，**指向的数据必须还活着**。

---

## 五、同 `'a` 约束什么：最晚可用时间

**核心**：`'a` 是**时长分组标签**，不是**变量绑定标签**。  
同 `'a` ≠ 必须指向同一个变量——强制的是：**组内所有引用的「最晚可用时间」必须是同一个节点**。

### 什么是「最晚可用时间」

对一个**引用**来说：

> **最晚可用时间** = 它指向的数据**被销毁的那一刻**。

数据一旦销毁，引用就变成悬垂指针，Rust 禁止。  
生命周期 `'a` 的作用：**所有标了 `'a` 的引用，最晚可用时间必须相同**；编译器以**最短**的那个为准。

**为什么 `'a` 写在引用上，却要管数据？**

1. `'a` 标在引用类型上（`&'a i32`），约束的是引用；
2. 但引用能不能用，**完全依赖指向的数据**；
3. 所以约束引用 = 保证：**引用活着的整个期间，数据一定还活着**。

### 一句话定规则

同一组 `'a` 的所有引用：

> **它们指向的数据，最晚销毁时间必须一致**——不能有的先消失、有的后消失。

| 情况 | 数据销毁 | 共用 `'a` |
|------|----------|-----------|
| 合规 | `a`、`b` 在 `main` 末尾**同时**销毁 | ✅ |
| 违规 | `inner` 内层先销毁，`outer` 外层后销毁 | ❌ |

```rust
fn test<'a>(x: &'a i32, y: &'a i32) {}

fn main() {
    let a = 1;
    let b = 2;
    test(&a, &b); // ✅ 同时销毁
}

fn main_bad() {
    let outer = 1;
    {
        let inner = 2;
        // test(&outer, &inner); // ❌ inner 先销毁
    }
}
```

### 易混淆：约束的是数据，不是引用变量

不是要求**引用变量本身**同时销毁，而是：

👉 **引用所指向的数据源**，必须**同一时刻**销毁（最晚销毁时间一致）。

- 引用变量是栈上普通变量，出 `{}` 该怎么销毁就怎么销毁，`'a` **不管它**；
- `'a` 只管：在引用还能用的区间里，**数据源不能提前消失**。

一长一短的数据源 → 用 **`'a` / `'b` 不同标签**，各自按自己的销毁时间约束（见场景 3）。

### 取最短为上限：整组跟着「短命」走

**核心**：

> 同一组 `'a` 里，**整组引用的合法使用上限 = 组内寿命最短的那个数据**。  
> 共用 `'a` 的所有引用，**可用范围强制对齐到最短寿命**——最终效果就是**所有人的有效区间必须一致**。

只要组里有一个「短命」数据，全组引用都**不能活过它**——这就是「取最短为上限」。

#### 统一逻辑：红线与越线（两句话不矛盾）

**只记一条规则**——同一函数内多参数都标 `'a`：

> 所有被引用的数据，**实际存活区间必须完全落在「最短数据的存活区间」里面**。  
> 人话：**全员都要比最短的「同时死或更早死」**，不能有任何数据活得更久。

**合法：全员都不越线**

```rust
fn test<'a>(x: &'a i32, y: &'a i32) {}

fn main() {
    let a = 1;
    let b = 2;
    test(&a, &b);
} // a、b 同时销毁；最短 = 末尾 = 所有人 → ✅
```

**违规：有数据越过红线**

```rust
fn main() {
    let outer = 100; // 活到整个 main
    {
        let inner = 200; // 组内最短，活到内层 }
        // test(&outer, &inner); // ❌
    }
}
```

编译器步骤：

1. 组内最短 = `inner` → **红线** = 内层 `}`
2. `inner` 寿命 = 内层 `}` → 在红线内 ✅
3. `outer` 寿命 = 整个 main → **越过红线** ❌ → 报错

**两处说法为何不矛盾？（递进关系）**

| 说法 | 含义 |
|------|------|
| 上限 = 组内**最短**数据的寿命 | **先画红线** |
| 一长一短塞进同组 `'a` 必报错 | **拿红线校验**：长数据越线 → 违规 |

完整链：**画红线 → 长数据越线 → 判定违规 → 报错**。

**关键纠正**：❌ 不是「长数据主动缩短寿命跟着红线走就没事」。  
Rust **不能**让变量缩短寿命——生死**只由 `{}` 决定**，编译器改不了。  
`outer` **天生**活到外层末尾 → **天生越线** → 编译拒绝。

**带返回值同理**：

```rust
fn test<'a>(x: &'a i32, y: &'a i32) -> &'a i32 { x }

fn main() {
    let outer = 100;
    {
        let inner = 200;
        // let r = test(&outer, &inner); // ❌ 根源仍是 outer 越线
    }
}
```

**背三句**：① 共用 `'a` → 以最短数据**画红线**；② 所有数据源寿命**须在红线内**；③ 有数据长于红线 → 越线 → 报错。

#### 不是手动「主动销毁」

Rust **没有**手动提前删掉变量的写法。局部变量只在**所属 `{}` 结束时自动销毁**。

下面两句话是一回事：

1. 同组 `'a` 的合法上限 = 组内最短数据的寿命  
2. 同组 `'a` 要求：所有数据源的**有效区间必须对齐**（物理销毁位置须一致）

这是**编译期静态规则**，不是运行时手动操作。

```rust
fn same_group<'a>(p1: &'a i32, p2: &'a i32) {}

fn main() {
    let outer = 10; // 寿命 = 整个 main
    {
        let inner = 20; // 寿命 = 仅内层 {}
        // same_group(&outer, &inner); // ❌
    } // inner 自动销毁
} // outer 自动销毁
```

**编译器判断**：

1. `inner` 是组内**最短** → 整组 `'a` 最大边界钉在「内层 `}`」；
2. `&outer` 能活到最外层 `}`，**长于 `'a` 边界**；
3. `&outer` 打了 `'a` 就不能超出该边界 → `outer` 实际存活 > `'a` 上限 → **报错**。

**合法对齐的两种情况**：

| 情况 | 说明 |
|------|------|
| 同一代码块 | `a`、`b` 同时销毁，最短 = 所有人 → ✅ |
| 内层块内 `a`、`b` 同销毁 | 如 `print_both(&a, &b)` 均在内层 `}` 结束 → ✅ |

**误区：内层用完就不再使用 `&outer`？**

```rust
// 即使在内层块内调用后不再用 &outer，依然 ❌
fn main() {
    let outer = 10;
    {
        let inner = 20;
        // same_group(&outer, &inner);
    }
}
```

生命周期是**编译期静态检查**：不看运行时「用没用」，只看**变量的物理销毁位置**。销毁位置一先一后 → 共用 `'a` 必冲突。

**两个核心疑问**：

| 疑问 | 答案 |
|------|------|
| 是不是手动放弃/销毁变量？ | ❌ 不能；靠 `{}` 自动销毁；「取最短」是编译期限制 |
| 「上限 = 最短」≈ 「必须寿命一样」？ | ✅ 共用 `'a` 时，被引用数据的**物理销毁位置必须相同** |

#### 限制的是引用，不是原变量

**核心分清**：`'a` 规则限制的是 **`'a` 分组下的引用**，**不限制、不销毁原变量本身**。

1. **「不能再使用」指什么？**  
   带 `'a` 的引用到截止线后**不能再出现、再使用**；原变量（如 `outer`）内存还在，**直接用变量本身完全合法**。

2. **函数形参引用，函数里不用也报错吗？**  
   **会。** 生命周期是编译期**静态契约**——签名定下规则后，编译器对比传参时各数据源的实际寿命；函数体空着、完全不读参数，契约不成立照样报错。

```rust
fn demo<'a>(x: &'a i32, y: &'a i32) {} // 函数体为空也须满足契约

fn main() {
    let outer = 100;
    {
        let inner = 200;
        // demo(&outer, &inner); // ❌ 契约 vs 实际寿命不匹配
    } // 'a 组引用使用截止线；inner 在此销毁
    println!("{}", outer); // ✅ 原变量 outer，与引用分组无关
}
```

**分步解释**：

| | `&inner` | `&outer`（带 `'a`） | `outer` 原变量 |
|---|----------|---------------------|----------------|
| 截止线 | 内层 `}` | 被 `'a` 限制在内层 | 无此限制 |
| 出内层块后 | 引用非法 | `'a` 引用非法 | **仍可用** ✅ |

> 把「长寿数据的引用」塞进「短命 `'a` 组」→ 编译器判定越界风险 → 非法。  
> 逻辑：**先定契约（签名），再验传参是否匹配**——与运行时「用不用」无关。

**无 `'a` 分组时**：出内层块后 `println!("{}", outer)` 无任何限制——没有共用 `'a` 的引用契约，原变量照常使用。

#### 三层分清：原变量、`'a` 引用、新引用

核心代码（全程对照）：

```rust
fn demo<'a>(x: &'a i32, y: &'a i32) {}

fn main() {
    let outer = 100;
    {
        let inner = 200;
        // demo(&outer, &inner); // ❌ 两个引用都属于 'a 组，契约不匹配
    } // inner 销毁；'a 组引用使用截止线
    println!("{}", outer);
}
```

| 层次 | 是什么 | 出内层 `}` 后 |
|------|--------|---------------|
| **原变量** `outer` | 变量本身 | ✅ 随便读写，与 `'a` 无关 |
| **`'a` 分组引用** `&'a i32` | 调用时传入的 `&outer`、`&inner` | ❌ 不能再使用任何 `'a` 引用 |
| **新建引用** | 出块后另建 | 见下 |

**出内层块后，能给 `outer` 新建引用吗？**

| 新建类型 | 可否 | 原因 |
|----------|------|------|
| **普通引用** `let r = &outer` | ✅ | 全新生命周期，跟着 `outer` 活到最后，**不属于旧 `'a` 组** |
| **`&'a i32` 再指向 `outer`** | ❌ | `'a` 截止点已过，不能再出现 `'a` 类型引用 |

**为什么 `demo(&outer, &inner)` 一调用就报错？**

签名 `demo<'a>(x: &'a i32, y: &'a i32)` = **编译期契约**：`x`、`y` 有效范围**完全一样**。

1. 传入 `&inner`（只到内层 `}`）→ 整组 `'a` 截止点钉在内层 `}`；
2. `&outer` 被贴 `'a` 标签，却指向活得更长的数据 → **越界风险** → 契约失效 → 报错；
3. 与函数内部**用不用** `x`、`y` **无关**。

**三个对照例子**：

```rust
// 例1：出块直接用原变量 ✅
fn main() {
    let outer = 100;
    { let inner = 200; }
    println!("{}", outer);
}

// 例2：出块新建普通引用 ✅
fn main() {
    let outer = 100;
    { let inner = 200; }
    let r = &outer; // 新生命周期，与旧 'a 无关
    println!("{}", r);
}

// 例3：出块后再用 &'a 指向 outer ❌
// fn bad<'a>(r: &'a i32) {}
// 在内层块外：bad(&outer) 若 'a 截止于内层 } → 非法
```

**大白话**：

1. **原变量** — 嵌套块结束后仍完好，直接读写不受限；
2. **`'a` 分组引用** — 有效期被最短者限到内层 `}`；跨过此线不能再使用旧 `'a` 引用，也不能再建 `&'a` 指向 `outer`；
3. **全新普通引用** — 不受旧 `'a` 约束，可正常创建使用。

**记忆口诀**：同 `'a` → 绑最短寿命；**限制引用，不限制原变量**；出块可 `&outer`，不可再 `&'a`。

#### 两套代码对比：报错 vs 正常运行（别混成一段）

> **报错 = 写了违规调用 → 整程序编译失败，根本跑不起来。**  
> **能正常使用 `outer` = 删掉了那句违规调用 → 另一套代码。**

**第一套：会编译报错（问题行在内层块）**

```rust
fn demo<'a>(x: &'a i32, y: &'a i32) {}

fn main() {
    let outer = 100;
    {
        let inner = 200;
        demo(&outer, &inner); // ❌ 编译在此终止，程序无法运行
    }
    // 下面永远执行不到——因为上面整段编译失败
    println!("{}", outer);
}
```

1. `demo` 要求两引用共用 `'a`；`outer`/`inner` 销毁一长一短 → 不满足；
2. **编译直接失败**，不是运行时错误；
3. 后面的 `println!` **没机会执行**——不是「报错后还能用 outer」。

**第二套：删掉违规调用，正常运行**

```rust
fn main() {
    let outer = 100;
    {
        let inner = 200;
        // 无 'a 分组、无违规调用
    } // inner 销毁
    println!("{}", outer); // ✅ 编译通过，正常执行
}
```

1. 没有共用 `'a` 的引用分组；
2. 只是普通嵌套变量：`inner` 先销毁，`outer` 继续存活；
3. **完全正常**——和第一套是两套独立代码，不是同一段程序。

**困惑解答**：「都报错了，`outer` 怎么还能用？」→ **不是同一段代码**。写 `demo(&outer, &inner)` 就炸；删掉这句就过。

**浓缩规则**：

| | 违规 `'a` 分组调用 | 纯嵌套变量 |
|---|-------------------|------------|
| 结果 | 编译报错，跑不起来 | 内层先销毁，外层继续用 ✅ |
| `outer` 出块后 | （程序未生成，谈不到执行） | 正常使用 |

**大白话**：报错是「长短不一塞进同一 `'a` 组」导致的；`outer` 能不能用，取决于**有没有写那句违规调用**。

```rust
fn use_both<'a>(a: &'a i32, b: &'a i32) {
    println!("{} {}", a, b);
}

fn main() {
    let long_live = 100; // 活到 main 末尾
    {
        let short_live = 200; // 只在内层块存活
        // use_both(&long_live, &short_live); // ❌
    } // short_live 在此失效 — 组内最短
}
```

**编译器校验逻辑**：

1. `&long_live`、`&short_live` 都归入 `'a`；
2. 组内最短寿命 = `short_live` → **`'a` 上限 = 内层 `}`**；
3. `&long_live` 原本能活到外层 `}`，**超过组上限**；
4. 引用真实寿命不能超过所属 `'a` 组的上限 → **冲突，拒绝编译**。

**合法：全员都不超过组内最短上限**

```rust
fn print_both<'a>(x: &'a i32, y: &'a i32) {
    println!("{} {}", x, y);
}

fn main() {
    {
        let a = 1;
        let b = 2;
        print_both(&a, &b); // ✅ a、b 同时在内层 } 销毁，上限匹配
    }
}
```

组内最短 = 内层 `}`；两个引用的真实寿命**刚好等于**这个上限 → 通过。

### 场景 1：同代码块 — 最晚可用时间相同（可共用 `'a`）

```rust
fn cmp<'a>(x: &'a i32, y: &'a i32) -> &'a i32 {
    if *x > *y { x } else { y }
}

fn main() {
    let v1 = 10;
    let v2 = 20;
    cmp(&v1, &v2);
} // ← v1、v2 在这里一起销毁
```

| 引用 | 指向 | 最晚可用时间 |
|------|------|--------------|
| `&v1` | `v1` | `main` 块末尾 |
| `&v2` | `v2` | `main` 块末尾 |

二者**完全一致** → 可打同一个 `'a`，编译器放行。  
`v1` 先定义、`v2` 后定义，**不改变「一起销毁」**。

#### 声明先后 ≠ 生命周期长短

Rust 看**销毁时机**，不看定义顺序。同一块内变量**统一在块末尾回收**；栈上销毁顺序 LIFO（`v2` → `v1`）只是回收瞬间，**不影响** `'a`——二者最晚可用节点都是块末尾。

### 场景 2：嵌套块 — 最晚可用时间不同（不能共用 `'a`）

```rust
fn print_two<'a>(x: &'a i32, y: &'a i32) {}

fn main() {
    let outer = 100;
    {
        let inner = 200;
        // print_two(&outer, &inner); // ❌
    } // ← inner 在这里销毁
} // outer 在这里才销毁
```

| 引用 | 指向 | 最晚可用时间 |
|------|------|--------------|
| `&outer` | `outer` | **外层** `}` |
| `&inner` | `inner` | **内层** `}` |

👉 **两个引用的最晚可用时间不一样！**  
函数签名却强制归入同一组 `'a` → **`'a` 被锁死在 `inner` 失效处，但 `&outer` 活得更久** → 取最短为上限规则冲突 → 编译失败。

### 场景 3：不同标签 — 各自独立的最晚可用时间

```rust
fn two_group<'a, 'b>(x: &'a i32, y: &'b i32) {}

fn main() {
    let outer = 100;
    {
        let inner = 200;
        two_group(&outer, &inner); // ✅
    }
}
```

- `'a` → `&outer` 的最晚可用时间
- `'b` → `&inner` 的最晚可用时间  
两组独立，互不干涉。

### 也合法：同 `'a`，指向同一变量（只是特例）

```rust
fn take<'a>(a: &'a i32, b: &'a i32) {}

fn main() {
    let num = 99;
    take(&num, &num); // ✅ 合法，但不是 'a 的硬性要求
}
```

### 三个独立标签 `'a` / `'b` / `'c`

```rust
fn triple<'a, 'b, 'c>(x: &'a i32, y: &'b i32, z: &'c i32) {
    println!("{} {} {}", x, y, z);
}
```

### 返回值标签与入参对应

**只返回其中一个**——谁作返回值，复用谁的标签：

```rust
fn pick_one<'a, 'b>(x: &'a i32, y: &'b i32) -> &'a i32 { x }
```

**返回多个引用**——元组每个位置分别对应：

```rust
fn return_two<'a, 'b>(x: &'a i32, y: &'b i32) -> (&'a i32, &'b i32) {
    (x, y)
}
```

**入参共用 `'a`，返回任意一个**：

```rust
fn pick_any<'a>(x: &'a i32, y: &'a i32) -> &'a i32 {
    if *x > *y { x } else { y }
}
```

### 三条铁律 + 误区纠正

| | |
|---|---|
| 共用同一 `'a` | 所有引用**最晚可用时间须一致**；变量可同可不同 |
| 最晚可用时间不一还共用 `'a` | **编译报错** |
| 想互不约束 | 用不同标签 `'a`/`'b`/`'c` |

❌ 同 `'a` → 必须同一变量  
✅ 同 `'a` → 归为**同一时长组**，变量可以相同也可以不同

**五条总结**：

1. **最晚可用时间** = 引用指向的数据**被销毁的那一刻**；
2. 同写 `'a` = 组内数据源**最晚销毁时间必须一致**；
3. **取最短为上限**：整组可用范围对齐最短寿命；**物理销毁位置须相同**；
4. 静态检查：内层用完不用也不行；嵌套一先一后 → 报错；
5. `'a` 约束**数据源**与**引用**的匹配；**不限制原变量**出块后继续使用。
6. 函数签名是**编译期契约**；形参引用函数内不用，传参违背契约仍报错。

## 六、`'a` 是函数局部标签

**核心**：不同函数里的 `'a` **互相独立、互不干扰**——不是全局共用标记；`'a` 只在**当前函数签名内部**做分组约束。

### 同名 `'a`，不同函数，互不影响

```rust
fn demo1<'a>(x: &'a i32, y: &'a i32) {
    println!("{} {}", x, y);
}

fn demo2<'a>(p: &'a i32) { // 这是 demo2 自己的 'a，与 demo1 无关
    println!("{}", p);
}

fn main() {
    let a = 10;
    let b = 20;
    demo1(&a, &b);
    demo2(&a);
}
```

1. `demo1` 的 `'a`、`demo2` 的 `'a` 只是**名字碰巧一样**，两套独立规则；
2. 不会因为别的函数也用了 `'a` 就连带报错；
3. 生命周期标签作用域 = **定义它的那个函数**。

### 嵌套块 + 多函数调用

```rust
fn demo1<'a>(x: &'a i32, y: &'a i32) {}
fn demo2<'a>(p: &'a i32) {}

fn main() {
    let outer = 100;
    {
        let inner = 200;
        // demo1(&outer, &inner); // ❌ 本函数内两源寿命不同，共用 'a
        demo2(&inner);           // ✅ 单参，'a 只绑 inner（内层块）
    }
    // demo2(&inner); // ❌ inner 已销毁
    demo2(&outer);    // ✅ 'a 绑 outer（整个 main）
}
```

- `demo1` 报错原因不变：**同一函数内**两数据源共用 `'a`；
- `demo2` 每次调用只有一个数据源，`'a` 只匹配那一个，无「长短打架」。

### 带返回值：`&'a` 跟着本组上限走

**规则**：返回值的 `'a` 须与来源入参的 `'a` 对齐；返回引用有效期 = 本组 `'a` 上限（取最短）。

**同块合法**：

```rust
fn get_short<'a>(x: &'a i32, y: &'a i32) -> &'a i32 { x }

fn main() {
    let v1 = 1;
    let v2 = 2;
    let ret = get_short(&v1, &v2);
    println!("{}", ret);
} // v1、v2、ret 同时失效
```

**嵌套块报错**：

```rust
fn main() {
    let outer = 100;
    {
        let inner = 200;
        // let ret = get_short(&outer, &inner); // ❌ 'a 上限 = 内层 }
    }
}
```

**单入参返回引用 — 每次调用独立合法**：

```rust
fn return_ref<'a>(p: &'a i32) -> &'a i32 { p }

fn main() {
    let outer = 100;
    {
        let inner = 200;
        let r1 = return_ref(&inner); // 'a 跟随 inner
    }
    let r2 = return_ref(&outer);   // 'a 跟随 outer
}
```

每次调用里 `'a` 只对应**唯一数据源**，无冲突。

### 两个疑问速答

| 疑问 | 答案 |
|------|------|
| 不同函数都用 `'a`，会互相影响吗？ | **不会**；各函数 `'a` 专属，同名只是巧合 |
| 返回值 `'a` 怎么被最短寿命限制？ | 返回 `&'a T` 与同组入参绑定；上限 = 最短数据源；入参不统一 → 报错 |

**极简记忆**：`'a` 作用域 = **当前函数**；同函数多参共用 `'a` → 寿命须一致；返回 `&'a` → 跟着本组上限；跨函数同名 `'a` 无关。

## 七、为什么需要生命周期？

Rust 要杜绝**悬垂引用**。编译器必须知道引用**能活多久**；用 `'a`、`'b` 等当「时间标签」。

```rust
// let r;
// {
//     let x = 5;
//     r = &x;
// }
// println!("{}", r); // ❌ x 已离开作用域
```

被引用者必须活得比引用更久。

---

## 八、最简示例：`get_first` 与 `longest`

### 无生命周期标注（编译失败）

```rust
// fn get_first(s: &str) -> &str {
//     &s[0..1]   // ❌ missing lifetime specifier
// }
```

### 加上 `'a`（正确）

```rust
fn get_first<'a>(s: &'a str) -> &'a str {
    &s[0..1]
}
```

1. 所有 `'a` **共享同一存活约束**
2. `s: &'a str`：入参引用存活时间为 `'a`
3. 返回 `&'a str`：返回值**来自 `s` 内部**，存活 ≤ `s` 的存活

### `longest`：两个入参同一 `'a`

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

- 两者须**能重叠到同一 `'a`**（实际有效长度取**较短**那个）
- 返回 `'a`：结果**要么是 `x` 要么是 `y`**

## 九、结构体中的引用

```rust
struct ImportantExcerpt<'a> {
    part: &'a str,
}
```

`ImportantExcerpt` 的实例不能比 `part` 指向的数据活得更久。

## 十、生命周期省略规则（Lifetime Elision）

1. 每个引用参数都有自己的输入生命周期
2. 只有一个输入生命周期时，赋给所有输出生命周期
3. 方法有 `&self` / `&mut self` 时，输出生命周期默认取 self 的生命周期

因此 `fn first_word(s: &str) -> &str` 可省略标注。

---

## 十一、`'static` 生命周期

`'static` 表示引用指向的数据可活到整个程序结束。字符串字面量都是 `'static`：

```rust
let s: &'static str = "I have a static lifetime.";
```

- 字面量本体在**只读数据段**，程序全程存在
- 局部引用变量 `s` 出 `{}` 仍消失，但指向的数据不受影响
- 不要用 `'static` 逃避错误；通常应修复真实的生命周期不匹配

---

## 十二、结合泛型 + trait bound + 生命周期

```rust
use std::fmt::Display;

fn longest_with_an_announcement<'a, T>(x: &'a str, y: &'a str, ann: T) -> &'a str
where
    T: Display,
{
    println!("Announcement! {}", ann);
    if x.len() > y.len() { x } else { y }
}
```
