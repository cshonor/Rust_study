## 1. 网络的简史

| 阶段 | 要点 |
|------|------|
| 电报时代 | 现代互联网的远祖之一 |
| 以太网（Ethernet） | 代表局域网（LAN）技术 |
| **ARPANET（1969）** | 现代互联网的雏形 |
| 协议标准化 | 各地 LAN / WAN 互联 → 今天的全球互联网 |

**记忆**：不是一夜之间出现的「一张大网」，而是多种网络按标准协议逐步拼起来。

## 2. 网络的分层模型

不同架构的网络要能互通，需要**标准协议**和**协议栈分层**。

### OSI 七层（自下而上）

物理层 → 数据链路层 → 网络层 → 传输层 → 会话层 → 表示层 → **应用层**

### TCP/IP 四层（常用对照）

| TCP/IP 层 | 大致对应 OSI | 职责 |
|-----------|--------------|------|
| **硬件接口层** | 物理 + 数据链路 | 帧在链路上传输 |
| **IP 层（网络层）** | 网络层 | 寻址、路由、转发 |
| **传输层** | 传输层 | TCP / UDP 等端到端交付 |
| **应用层** | 会话 + 表示 + 应用 | 进程语义、数据格式（HTTP、DNS…） |

### 封装（Encapsulation）

数据从上层往下层传递时，每一层都会加上本层的 **PDU 头部**（有时还有尾部），这个过程叫**封装**。接收端逐层剥头，还原出应用数据。

```text
应用数据
  → + 应用层头  → 段/报文 + TCP/UDP 头  → + IP 头  → + 帧头/尾  → 比特流上线
```

写 Rust 网络程序时，你多数时间在**应用层 + 传输层**（`std::net` / Tokio）打交道，但排错时要能想到下面还有 IP、链路层。

## 3. 网络寻址

在全球互联网里**唯一标识**主机、网络和进程的方式。

| 类型 | 长度 / 形式 | 作用 |
|------|-------------|------|
| **MAC 地址** | 48 位以太网地址 | 同一**网段（链路）**内标识网卡/设备 |
| **IP 地址** | IPv4 32 位；IPv6 128 位 | 在 IP 网络中标识主机；常用 **CIDR** 表示网段，如 `192.168.100.1/26` |
| **ASN（自治系统编号）** | 32 位，IANA 分配 | 标识一个**自治系统**（一组由同一策略管理的路由器） |
| **端口号（Port）** | 0～65535 | 区分同一主机上的**不同应用进程**；**0～1024** 多为系统保留服务 |

**层次关系（通俗）**：

- 同网段找机器 → 常先看 MAC（ARP 把 IP 映射到 MAC）  
- 跨网段找机器 → 用 IP + 路由  
- 找到机器后找「哪个程序」→ **IP + 端口**（套接字五元组里的关键部分）

---

## 4. IP 路由的工作原理

**路由器**根据**路由表**决定从哪个接口把包转发出去，通常用掩码做 **最长前缀匹配（Longest prefix match）**：多条路由都能匹配时，选前缀最长的那条。

| 机制 | 说明 |
|------|------|
| **TTL（生存时间）** | 每经一个路由器减 1；减到 0 则丢弃，避免包在网络里死循环 |
| **内部网关协议** | OSPF、RIP 等，在自治系统**内部**交换路由信息 |
| **外部网关协议** | **BGP**，在自治系统**之间**交换可达性信息 |

应用开发很少直接改路由表，但要理解：**连不上远程 IP** 时，可能是路由、防火墙或本机默认网关问题，而不只是「代码写错」。

## 5. DNS（域名系统）

把人类可读的域名（如 `www.google.com`）解析成机器使用的 **IP 地址**。

### 典型解析路径（由近到远）

1. **本机缓存** / 操作系统解析器  
2. **本地 DNS 解析器**（常由 ISP 或路由器提供）  
3. 若无缓存 → 向 **根域名服务器（Root）** 查询  
4. → **顶级域服务器（TLD，如 `.com`）**  
5. → **权威域名服务器（Authoritative）**  
6. 得到 IP 后返回，并往往写入缓存

### 为何常用 UDP？

查询报文小、对偶发丢包容忍度高（可超时重试），因此 DNS 查询**极其适合 UDP**（注意：也有 DNS over TCP 等场景，但经典模型以 UDP 为主）。

**与 Rust 编程的关系**：`TcpStream::connect("example.com:80")` 之前，底层通常会先做 DNS 解析；异步框架里 DNS 也可能单独成为阻塞/线程池热点。

## 6. 常见的数据传输服务模型

### 面向连接：TCP

- 发送数据前必须先**建立连接**  
- 靠**源/目的端口、序列号、确认号**等保证**按序、可靠**交付（在丢包、乱序时重传）  
- 建立连接：**三次握手**（SYN → SYN+ACK → ACK）  
- 代价：更高延迟、更复杂状态机；收益：**可靠**、**拥塞控制**、适合大多数「不能丢字节」的业务  

状态名（第一章常出现）：`SYN_SENT`、`ESTABLISHED`、`TIME_WAIT` 等 — 详见 [TCP 完整笔记](./TCP-传输层-完整笔记与背诵卡.md)。

### 无连接：UDP、以及 IP 层的转发

- 各报文**相互独立**，无需先建连  
- 头部比 TCP 小得多，**无确认、无重传**（由应用自己决定要不要补）  
- 适合：视频/语音流、游戏位置同步、DNS 查询等**可丢包、要低延迟**的场景  

| 对比 | TCP | UDP |
|------|-----|-----|
| 连接 | 要三次握手 | 无连接 |
| 可靠/有序 | 协议保证 | 不保证 |
| 开销 | 首部大、状态多 | 轻量 |
| 典型场景 | HTTP、SSH、邮件 | DNS、QUIC 底层、实时流 |

## 7. Linux 中的网络编程接口（套接字）

POSIX/Linux 下最核心的原语是 **套接字（Socket）**。Unix「一切皆文件」：套接字常表现为**文件描述符**，并记录本地/远程 **IP 与端口**。

### 服务端典型流程（TCP）

| 系统调用 | 作用 |
|----------|------|
| `socket` | 创建套接字 |
| `bind` | 绑定本机 IP + 端口 |
| `listen` | 进入监听状态 |
| `accept` | 接受一个已完成握手的连接，得到**新** fd |
| `recv` / `read` | 收数据 |
| `send` / `write` | 发数据 |
| `close` / `shutdown` | 关闭或半关闭连接 |

### 客户端典型流程

`socket` → `connect`（向服务器 IP:端口发起连接）→ `send` / `recv` → `close`

### 阻塞 vs 异步非阻塞

上述调用默认多为**阻塞**：线程卡在 `accept`/`recv` 直到有数据或超时。

为高并发，OS 提供：

| 机制 | 说明 |
|------|------|
| `select` / `poll` | 在一组 fd 上等待「可读/可写」 |
| `epoll`（Linux） | 更高性能的多路复用 |
| `libevent` 等库 | 封装上述机制 |

Rust 路径对照：

- **Stage 03**：`std::net` 阻塞式套接字  
- **Stage 07**：Tokio 等在 `epoll` 等之上封装异步 I/O  

第一章末尾的 **TCP 状态转换图**（`LISTEN`、`SYN_RCVD`、`ESTABLISHED`、`FIN_WAIT`、`TIME_WAIT`…）是读懂抓包与连接泄漏的基础，与下一章写 `bind`/`listen`/`accept` 直接相关。
