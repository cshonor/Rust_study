## 一、全章在解决什么问题

[Ch3](../stage03_std_tcp_udp/) 传的是**字节流**；本章把字节变成应用能用的**结构**：

```text
应用结构体  ←→  Serde（JSON/二进制）  ←→  Vec<u8>
文本行协议  ←→  nom PEG（tag、take_until）  ←→  字段切片
定长/位域帧  ←→  nom（be_u32、take、bits!）  ←→  强类型头 + payload
```

**分帧**（长度前缀、换行、HTTP Content-Length）在 I/O 层；**Serde / nom** 只管「已切好的一段」如何编码或解码。

## 三、技术选型速查

| 场景 | 推荐工具 | 理由 |
|------|----------|------|
| REST / 调试友好消息体 | Serde + `serde_json` | 人机可读、跨语言 |
| Rust 服务间紧凑消息 | Serde + `bincode` / postcard | 体积小、derive 即用 |
| wire 字段名 ≠ Rust 字段 | `#[serde(rename)]` 或手写 Serialize | 见 [4.1.1](./4.1.1-custom-serde.md) |
| HTTP 请求行、SMTP 文本命令 | `nom` 文本组合子 | 比硬套 JSON 自然 |
| 标准二进制头、位域 | `nom` `be_u32`、`bits!` | 精确对齐协议文档 |
| TCP 多条 JSON | 长度前缀 + Serde body | [4.1](./4.1-serde-serialization.md) + [4.3](./4.3-parsing-binary.md) |
| 完整 HTTP 服务 | hyper 等 | [stage06](../stage06_http_programming/) |

## 五、本章自检

- [ ] 会用 `#[derive(Serialize, Deserialize)]` + `serde_json::to_vec` / `from_slice`
- [ ] 能说明 TCP 流上为何需要**长度前缀**或**分隔符**（Serde 不解决粘包）
- [ ] 知道何时用 `#[serde(rename)]`，何时才手写 `Serialize` / `Visitor`
- [ ] 能解释 `nom` 里 `tag`、`take_until1`、`tuple` 的作用
- [ ] 能区分文本 PEG（HTTP 行）与二进制 `be_u32` + `take`（长度前缀帧）
- [ ] 知道 `complete` 解析器假设输入已完整，流式 TCP 要缓冲或 `streaming`

## 六、新手易错（全章汇总）

1. **只学 Serde 不做分帧**：`from_slice` 在粘包环境必踩坑。
2. **一种工具包打天下**：JSON 体 + 二进制头很常见，应组合 [4.1](./4.1-serde-serialization.md) 与 [4.3](./4.3-parsing-binary.md)。
3. **nom 与 Serde 角色混淆**：nom 切**语法**；Serde 映射**已识别的一段**到类型。
4. **忽略字节序与 `len` 校验**：二进制协议安全第一。
5. **过早引入完整 HTTP 库**：先 [4.2](./4.2-parsing-textual.md) 理解请求行，再上 stage06。
