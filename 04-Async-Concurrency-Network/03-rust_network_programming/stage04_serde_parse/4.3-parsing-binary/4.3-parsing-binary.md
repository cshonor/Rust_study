## 4.3 Parsing binary data（解析二进制数据）

## 一、本节在学什么

底层与自定义协议常是**定长字段**或**按位布局**的二进制，不适合整包 JSON：

- IP/TCP/UDP 首部、自定义帧头
- **长度前缀 + payload**（与 [Ch3](../stage03_std_tcp_udp/) TCP 粘包方案一致）
- 位域（如 IPv6 首部：版本 4 bit + 流量类别 8 bit + …）

本节用 **`nom` 解析 `&[u8]`**（`be_u32`、`take` 等）；书中 IPv6 示例用 **`bits!` / `take_bits!`** 切比特流。

---

## 二、长度前缀分帧（TCP 常见形态）

[4.1](../4.1-serde-serialization/4.1-serde-serialization.md) 提到：先传 4 字节长度，再传 body。发送与解析应对称。

### 发送侧（概念）

```rust
let payload = b"hello binary frame";
let mut buf = Vec::new();
buf.extend_from_slice(&(payload.len() as u32).to_be_bytes()); // 大端 u32
buf.extend_from_slice(payload);
// buf → write_all 到 TcpStream
```

### 解析侧（本仓库 demo）

```rust
use nom::{bytes::complete::take, number::complete::be_u32, IResult};

fn parse_frame(input: &[u8]) -> IResult<&[u8], &[u8]> {
    let (input, len) = be_u32(input)?;  // 读 4 字节大端长度
    take(len)(input)                     // 再读 len 字节 payload
}
```

```rust
match parse_frame(&buf) {
    Ok((rest, body)) => {
        // body 即 payload；rest 为帧后剩余字节（粘包时下一条帧）
    }
    Err(e) => { /* 数据不足或长度非法 */ }
}
```

仓库对应：[4.3-parsing-binary-length-prefix-demo.rs](./code/4.3-parsing-binary-length-prefix-demo.rs)

| 字段 | 宽度 | 字节序 |
|------|------|--------|
| `len` | 4 字节 `u32` | 本 demo：**大端**（`be_u32`） |
| `payload` | `len` 字节 | 可以是 JSON、Protobuf、任意二进制 |

## 三、`nom` 二进制 API 速查

| 函数 / 宏 | 作用 |
|-----------|------|
| `be_u32` / `le_u32` | 读 4 字节无符号整数（大端 / 小端） |
| `be_u16`、`u8` 等 | 其它定长整数 |
| `take(n)` | 读恰好 `n` 字节 |
| `bits!` | 进入按位子解析器 |
| `take_bits!(n)` | 读 `n` 个 bit |

与 [4.2](../4.2-parsing-textual/4.2-parsing-textual.md) 相同：**子解析器组合**，只是输入从 `&str` 换成 `&[u8]`。

## 四、按位解析：IPv6 首部（书中）

标准协议头字段**不按字节对齐**，需比特级切分：

| 字段 | 典型宽度 |
|------|----------|
| 版本 | 4 bit |
| 流量类别 | 8 bit |
| 流标签 | 20 bit |
| 有效载荷长度 | 16 bit |
| 下一首部 | 8 bit |
| 跳数限制 | 8 bit |
| 源/目的地址 | 各 128 bit |

```text
原始 &[u8]
  → bits! 子解析器
  → (version, traffic_class, flow_label, ...)
  → 组装 Ipv6Header { ... }
```

**与 `libpnet`（Ch3 抓包）**：`libpnet` 偏链路层抓包与构造；`nom` 偏**你自己定义**或精读标准布局。两者可配合：抓到的包用 `nom` 验证字段含义。

---

## 五、对照：二进制解析 vs Serde

| 方式 | 适用 | 本 stage 关联 |
|------|------|---------------|
| **Serde + bincode / postcard** | 双方都是 Rust（或共享 schema），结构体直接映射 | [4.1](../4.1-serde-serialization/4.1-serde-serialization.md) 换格式 crate |
| **nom 手写布局** | 标准协议头、跨语言、位域 | 本节 + `be_u32`/`take`/`bits!` |
| **固定头 + Serde 体** | 头定长二进制，负载 JSON | 头用本节，体用 `serde_json` |

```text
[ u32 len ][ JSON body ]     ← 头 nom，体 Serde（常见 RPC）
[ IPv6 fixed header ][ ... ] ← 头 bits!，体可能再分层
```

---

## 六、流式读取注意

TCP 上 `read` 可能只拿到**半帧**：

| 情况 | `nom` `complete` 行为 | 做法 |
|------|------------------------|------|
| 缓冲区少于 4+len 字节 | `Err`（数据不足） | 累积到 `Vec`，或改用 `streaming` 解析器 |
| 一次 read 多帧 | 第一帧 `Ok((rest, body))`，`rest` 继续解析 | 循环 `parse_frame` |

生产代码常在连接层维护 **read buffer**，而不是假设一次 `read` 刚好一帧。

---

## 七、新手易错

1. **字节序不一致**：发送 `to_be_bytes` 却用 `le_u32` 解析，长度会错到离谱，可能 OOM 或 panic。
2. **`len` 不信任对端**：应对 `len` 设上限，防止恶意超大分配。
3. **`take(len)` 与 `usize`**：`len` 来自 `u32`，在 32 位平台注意转换；非法 `len` 要返回协议错误。
4. **把 JSON 当二进制帧**：JSON 是文本 UTF-8；长度前缀里可以是 JSON 字节，但解析 JSON 内容仍用 `serde_json`（[4.1](../4.1-serde-serialization/4.1-serde-serialization.md)）。
5. **混淆 `complete` 与半包**：demo 在内存 `Vec` 上跑；真实 socket 要处理 `Incomplete`。
