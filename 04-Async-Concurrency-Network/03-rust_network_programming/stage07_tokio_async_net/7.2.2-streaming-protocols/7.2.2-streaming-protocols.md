## 7.2.2 Writing streaming protocols（编写流式协议）

> 书 §7.2.2 · 分帧与解析：[Ch04](../stage04_serde_parse/notes/Ch04-数据序列化反序列化与解析-学习笔记.md) · 贯通稿：[notes/Ch07-…](../notes/Ch07-使用Tokio进行异步网络编程-学习笔记.md)

---

## 一、本节在学什么

大 payload 不必**全部收完**才开始处理：

- **Header 完整** → 立刻路由、鉴权、开管道
- **Body 仍在传输** → Stream 继续 `read`，边收边处理

配合 **长度前缀**（或定长头）分帧，与 Ch4 Codec / nom 思路一致。

---

## 二、流式 vs 缓冲整包

| 模式 | 行为 | 适用 |
|------|------|------|
| 缓冲整包 | 收齐 Body 再 dispatch | 小消息、简单 API |
| **流式 Header/Body** | Header 一到就决策 | 大上传、实时管道 |
| **长度前缀帧** | 先读 4 字节 len，再读 body | 通用二进制协议 |

```text
[ Header 已完整 ] → 可先 dispatch（鉴权、选 handler）
[ Body  仍在传输 ] → AsyncRead 继续 read，可选 Stream 消费
```

---

## 三、长度前缀分帧

**线格式**：`[ u32 BE length ][ payload bytes… ]`

```rust
fn decode_frame(buf: &[u8]) -> Option<(&[u8], &[u8])> {
    if buf.len() < 4 {
        return None; // 头不完整
    }
    let len = u32::from_be_bytes(buf[..4].try_into().ok()?) as usize;
    if buf.len() < 4 + len {
        return None; // body 未收齐
    }
    Some((&buf[4 + len..], &buf[4..4 + len]))
}
```

编码侧：

```rust
let payload = b"{\"header\":\"ok\",\"body\":\"streaming...\"}";
let mut wire = Vec::new();
wire.extend_from_slice(&(payload.len() as u32).to_be_bytes());
wire.extend_from_slice(payload);
```

| 步骤 | 说明 |
|------|------|
| 读 4 字节 | 解析大端 `u32` 得 `len` |
| 读 `len` 字节 | 得到一帧 payload |
| 剩余 buffer | 可能含下一帧（粘包） |

生产环境用 **`tokio_util::codec::LengthDelimitedCodec`** 或自定义 `Decoder` 处理**半包/粘包**；demo 为离线 buffer 演示。

仓库对应：[7.2.2-streaming-length-prefix-demo.rs](./code/7.2.2-streaming-length-prefix-demo.rs)

## 四、与 Tokio Codec 层衔接

```text
TcpStream (AsyncRead)
    ↓ Framed + LengthDelimitedCodec
    ↓ Stream<Item = Bytes>  每项一帧
    ↓ 首帧当 Header 解析 → spawn 处理 Body 流
```

| Ch4 概念 | Tokio 7.2 对应 |
|----------|----------------|
| nom 解析 HTTP 行 | 首帧 JSON/二进制 Header |
| 长度前缀（4.3） | `LengthDelimitedCodec` |
| serde 反序列化 | Header 帧 `serde_json::from_slice` |

---

## 五、新手易错

1. **`u32` 长度无上限**：恶意对端发 `len=0xFFFFFFFF` 会 OOM；需 **max frame size**。
2. **短读**：一次 `read` 不一定凑齐一帧；Decoder 要维护 `BytesMut` 累积 buffer。
3. **字节序**：协议固定 BE/LE；与主机序混用会错。
4. **JSON 里嵌长度**：长度前缀是**传输层分帧**；JSON 内部字段长度由 JSON 语法决定，两层不要混。
