## 7.2.1 Socket multiplexing in tokio（套接字多路复用）

> 书 §7.2.1 · Collatz 服务器示例 · 贯通稿：[notes/Ch07-…](./notes/Ch07-使用Tokio进行异步网络编程-学习笔记.md)

---

## 一、本节在学什么

单连接上**多个请求同时在飞**时，响应顺序不必与到达顺序一致。书中 Collatz 服务器用 **RequestId** 做关联：

- 客户端发 req(id=1)、req(id=2)、req(id=3)
- 服务器并行处理；**id=2 可能先于 id=1 返回**
- 客户端按 id 匹配响应，而非假设 FIFO

思想类似 **HTTP/2 Stream ID**、RPC 的 call id。

---

## 二、为何需要多路复用

| 问题（串行处理） | 多路复用做法 |
|------------------|--------------|
| 长任务阻塞后续短任务 | 每请求独立 task |
| 吞吐差、延迟高 | 短任务先完成先返回 |
| 单连接只能一问一答 | 单连接多 in-flight 请求 |

```text
Client A  req id=1 ─┐
Client B  req id=2 ─┼→ Server 并行 spawn 处理
Client C  req id=3 ─┘
         ← resp 2 可能先于 resp 1 返回
```

---

## 三、Tokio 中的实现要点

| 组件 | 作用 |
|------|------|
| **`RequestId`** | 请求/响应配对键（u32、UUID 等） |
| **`tokio::spawn`** | 每请求一个 task，互不阻塞 |
| **`mpsc` / 写锁** | 多 task 安全写回同一 socket（需串行化写或分 channel） |
| **协议层** | 帧头带 id；Codec 解码后 dispatch |

```rust
use tokio::time::{sleep, Duration};

async fn handle(id: u32, delay_ms: u64) {
    sleep(Duration::from_millis(delay_ms)).await;
    println!("response id={id} (delay {delay_ms}ms)");
}

#[tokio::main]
async fn main() {
    let mut handles = vec![];
    for (id, delay) in [(1, 300), (2, 100), (3, 200)] {
        handles.push(tokio::spawn(handle(id, delay)));
    }
    for h in handles {
        let _ = h.await;
    }
}
```

Demo 模拟：**id=2（100ms）先于 id=3（200ms）先于 id=1（300ms）** 打印完成顺序。

仓库对应：[7.2.1-socket-multiplexing-demo.rs](./7.2.1-socket-multiplexing/7.2.1-socket-multiplexing-demo.rs)

---

## 四、与 HTTP/2 / gRPC 对照

| 机制 | 多路复用方式 |
|------|--------------|
| HTTP/1.1 | 通常多连接或 pipelining（受限） |
| **HTTP/2** | 单 TCP 多 **stream id** |
| **gRPC**（Ch5） | HTTP/2 上多并发 RPC |
| 书中 Collatz | 自定义二进制/文本协议 + RequestId |

---

## 五、新手易错

1. **响应乱序 ≠ 错误**：协议必须带 id；客户端不能假设「第二个发的第二个回」。
2. **多 task 写同一 `TcpStream`**：需 `Mutex` 或 dedicated writer task，避免写交错。
3. **无上限 spawn**：恶意客户端可打满 task 数；需连接级/全局并发限制。
4. **与 [7.2.2 流式](./7.2.2-streaming-protocols.md) 混淆**：多路复用是**多请求**；流式是**单请求分 Header/Body 阶段**。
