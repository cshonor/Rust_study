## 3.2.1 UDP multicasting（UDP 多播）

> 书 §3.2.1 · 单播 UDP 见 [3.2](../3.2-simple-udp-server-client/3.2-simple-udp-server-client.md) · 地址类型见 [3.3](../3.3-std-net-utilities/3.3-std-net-utilities.md) · 贯通稿：[notes/Ch03-…](../notes/Ch03-使用Rust进行TCP和UDP编程-学习笔记.md)

---

## 一、什么是 UDP 多播

**一句话**：向 **组播 IP** 发一个 UDP 报文，**同一组播组内**所有已「加入」该组的 socket 都能收到——**一对多**，无需逐个 `send_to` 每个客户端。

| | **单播（3.2）** | **多播（本节）** | **广播** |
|---|----------------|------------------|----------|
| 目标 | 单个 `SocketAddr` | 组播地址（一组接收者） | 子网所有主机（如 `255.255.255.255`） |
| 典型 API | `send_to(addr, …)` | `send_to(239.x.x.x:port, …)` + 接收端 `join_multicast_v4` | 受限多、路由器常不转发 |
| 场景 | 客户端—服务器 | 服务发现、日志汇聚、局域网通知 | 老式发现协议（少用） |

与 TCP 无关：多播建立在 **UDP datagram** 之上。

---

## 二、组播地址（IPv4）

IANA 为 IPv4 多播保留 **`224.0.0.0`～`239.255.255.255`**：

| 范围 | 用途 |
|------|------|
| `224.0.0.0/24` | 本地链路（路由不转发） |
| **`239.0.0.0/8`** | **管理范围**（私网组播，类似私网单播；本书 demo 用此段） |

本书 demo：**`239.0.0.1:39191`**（仅局域网/本机学习；勿与生产组播冲突）。

> **IPv6** 对应 `Ipv6Addr` 组播 + `join_multicast_v6`；本书以 **IPv4** 为主。

---

## 三、核心 API：`join_multicast_v4`

接收方必须**显式加入组播组**，内核才会把发往该组播地址的包交给你的 socket：

```rust
use std::net::{Ipv4Addr, SocketAddrV4, UdpSocket};

let bind = SocketAddrV4::new(Ipv4Addr::UNSPECIFIED, 39191); // 0.0.0.0:39191
let socket = UdpSocket::bind(bind)?;
socket.join_multicast_v4(
    &Ipv4Addr::new(239, 0, 0, 1),  // 组播组地址
    &Ipv4Addr::UNSPECIFIED,        // 从哪块网卡加入（0.0.0.0 = 由 OS 选）
)?;
```

| 参数 | 含义 |
|------|------|
| **multicast** | 要加入的组播 IP（如 `239.0.0.1`） |
| **interface** | 本地接口地址；`UNSPECIFIED` 表示默认网卡 |

加入之后，对该 **IP:端口** 的 UDP 报文会进入 **`recv_from`**，与单播用法相同。

离开组播组（可选）：**`leave_multicast_v4`**。

---

## 四、接收端流程（recv demo）

```rust
use std::net::{Ipv4Addr, SocketAddrV4, UdpSocket};
use std::time::Duration;

const MULTICAST: Ipv4Addr = Ipv4Addr::new(239, 0, 0, 1);
const PORT: u16 = 39191;

let socket = UdpSocket::bind(SocketAddrV4::new(Ipv4Addr::UNSPECIFIED, PORT))?;
socket.join_multicast_v4(&MULTICAST, &Ipv4Addr::UNSPECIFIED)?;
socket.set_read_timeout(Some(Duration::from_secs(30)))?;

let mut buf = [0u8; 1500];
let (len, src) = socket.recv_from(&mut buf)?;
println!("from {src}: {}", String::from_utf8_lossy(&buf[..len]));
```

**要点**：

1. **`bind` 端口**须与发送方 `send_to` 的端口一致（`39191`）。
2. **`bind` 到 `0.0.0.0`** 表示在所有接口上监听该端口。
3. 学习 demo 设 **30s 读超时**，避免无发送方时一直阻塞。

仓库对应：[3.2.1-udp-multicasting-recv-demo.rs](./code/3.2.1-udp-multicasting-recv-demo.rs)

## 五、发送端流程（send demo）

发送方**不必** `join_multicast_v4`（除非也要收同组报文）；向组播地址 **`send_to`** 即可：

```rust
use std::net::{Ipv4Addr, SocketAddrV4, UdpSocket};

let socket = UdpSocket::bind("0.0.0.0:0")?;
let target = SocketAddrV4::new(Ipv4Addr::new(239, 0, 0, 1), 39191);
socket.send_to(b"hello multicast", target)?;
```

**TTL（生存时间）**：跨路由器转发组播时，可用 **`set_multicast_ttl_v4`** 限制跳数；本机/局域网 demo 通常默认即可。

仓库对应：[3.2.1-udp-multicasting-send-demo.rs](./code/3.2.1-udp-multicasting-send-demo.rs)


| 环境 | 说明 |
|------|------|
| **Linux / macOS** | 本机 loop 一般可通 |
| **Windows** | 可能需管理员权限、防火墙放行 UDP，或组播路由未就绪 → **recv 超时仍算正常**；以 API 笔记为主 |
| **防火墙** | 组播端口需放行；云主机常不支持组播 |

## 七、典型应用场景

- **局域网服务发现**（mDNS 等用不同地址，思路类似）
- **日志 / 指标汇聚**（多生产者 → 多分析器）
- **流媒体**（常配合 RTP；本书只到 UDP 组播 API）

---

## 八、对照：3.2 单播回显 vs 3.2.1 多播

| | **3.2 UDP echo** | **3.2.1 多播** |
|---|------------------|----------------|
| 地址 | `127.0.0.1:39090` | `239.0.0.1:39191` |
| 接收前提 | `bind` 即可 | `bind` + **`join_multicast_v4`** |
| 发送 | `send_to(服务器地址)` | `send_to(组播地址)` |
| 一对多 | 否（需多次 send） | 是 |

---

## 九、新手易错

1. **只 bind 不 join**：收不到组播包。
2. **端口不一致**：发送 `239.0.0.1:39191`，接收也必须 bind **`39191`**。
3. **把组播地址当普通单播 bind**：应对 **`0.0.0.0:port` bind**，组播 IP 只出现在 **`join`** / **`send_to` 目标**。
4. **期望跨公网组播**：多数互联网路径不支持；限局域网或专用网络。
5. **Windows 上 demo 失败就怀疑代码**：先确认 recv 终端已启动、防火墙、权限。
