## 1. Web 安全与 HTTPS / TLS 原理

### 通信加密基础

- **HTTP**：应用层**明文**（头 + 体可被中间人读取/修改）。  
- **HTTPS**：HTTP 跑在 **TLS**（或历史 **SSL**）之上，线路上的载荷为密文。

### 握手与身份验证（简化流程）

```text
1. TCP connect（仍从 Ch3 的三次握手开始）
2. TLS ClientHello / ServerHello — 协商版本、密码套件、随机数
3. 服务器出示 X.509 证书链
4. 客户端用信任的 CA 根验证链 → 确认「真的是这个域名/组织」
5. 协商会话密钥（常含 ECDHE 等前向保密）
6. 应用数据（HTTP）在加密通道内传输
```

**最关键一步**：**验证服务器身份** — 依赖 **CA 签发的数字证书**（X.509），而非仅「连上某个 IP」。

| 术语 | 含义 |
|------|------|
| **CA** | 证书颁发机构，用私钥给站点证书签名 |
| **证书链** | 站点证书 ← 中间 CA ← 根 CA（根预装在系统/浏览器） |
| **主机名校验** | 证书 `Subject Alternative Name` 须包含你访问的域名 |

**开发环境**：自签名证书 → 客户端默认**不信任**，需手动导入或关闭校验（仅本地调试）。

## 2. 使用 Let's Encrypt 与 Rustls

### Let's Encrypt 与 ACME

| 项目 | 说明 |
|------|------|
| **Let's Encrypt** | 非营利 CA，**免费**短期证书（常 90 天） |
| **ACME** | 自动化协议：通过 **HTTP-01 / DNS-01** 等「质询」证明你控制域名 |
| 续期 | 到期前自动 re-issue（certbot、acme 客户端） |

Rust 侧：

- 命令行 / crate：**`acme-client`** 等获取证书文件（`fullchain.pem` + `privkey.pem`）。  
- **Rocket**：启用 **`tls` feature**，配置证书路径启动 HTTPS（书中示例；其他框架见各自文档）。

### 安全客户端：`rustls` + `webpki`

**`rustls`**：纯 Rust TLS 实现，**不链接 OpenSSL**（减少 C 依赖与许可证纠缠）。

典型客户端流程：

```text
DNS 解析域名 → rustls 配置根证书（webpki-roots）
→ TLS 握手验证服务器证书
→ 加密的 HTTP（可配合 reqwest 的 rustls feature）
```

| 组件 | 作用 |
|------|------|
| **`rustls`** | TLS 协议实现 |
| **`webpki`** | 证书路径校验、名称匹配 |
| **`reqwest`** | `rustls-tls` feature 一键 HTTPS GET |

**选型**：新项目客户端/服务端常优先 **rustls**；需与老系统 OpenSSL 行为完全一致时再考虑 **native-tls**。

## 3. 在 Rust 中使用 OpenSSL

### X.509 证书主要字段

| 字段 | 含义 |
|------|------|
| Version | 证书版本 |
| Serial Number | 序列号 |
| Signature Algorithm | 签名算法（如 SHA256withRSA） |
| Issuer | 颁发者 DN |
| Validity | 生效 / 过期时间 |
| Subject | 主体（CN、O 等） |
| Subject Public Key Info | 公钥及算法 |
| Extensions | SAN（域名列表）、Key Usage 等 |

### 编程生成：`rust-openssl`

书中示例：**代码内生成**测试用证书：

1. 生成 **RSA 3072** 私钥  
2. 构造 X.509，**自签名**（Issuer = Subject）  
3. 签名算法如 **SHA512**，有效期 **365 天**  

用途：**本地集成测试**、内网服务、mTLS 实验 — **不要**把自签名当生产公网方案。

```text
rust-openssl  →  与系统 OpenSSL 库绑定
适合：需要 OpenSSL 特有 API、PKCS#12 转换、遗留互操作
```

## 4. 保护 Tokio 异步应用

### `tokio-tls`（书中）→ 现代栈

书中 **`tokio-tls`** + **`TlsAcceptor`** 在异步 TCP 流上包装 TLS。

| 现代常见组合 | 说明 |
|--------------|------|
| **`tokio-rustls`** | Tokio `AsyncRead/Write` + **rustls** |
| **`tokio-native-tls`** | Tokio + 系统 OpenSSL/Schannel |

```rust
// 概念：accept 后 upgrade 到 TLS
let tls_stream = tls_acceptor.accept(tcp_stream).await?;
// 再在 tls_stream 上跑 HTTP（Hyper）或自定义协议
```

### PKCS12（`.pfx`）与 PEM

书中要点：**`tokio-tls` → `native-tls`** 加载服务端身份时，常要求 **PKCS#12** 打包（证书 + 私钥 + 链），而不能直接丢两个 PEM 文件。

```bash
# 概念：OpenSSL 将 PEM 转为 PKCS12（书中流程）
openssl pkcs12 -export -in cert.pem -inkey key.pem -out server.pfx
```

| 格式 | 内容 |
|------|------|
| **PEM** | Base64 文本，`-----BEGIN CERTIFICATE-----` |
| **PKCS#12** | 二进制容器，常 `.pfx` / `.p12` |

**rustls 路径**：通常直接配置 **PEM 或 DER**，不必转 PKCS12 — 选型不同，步骤不同。

**延伸**：[Ch07](../../stage07_tokio_async_net/) Hyper 服务 + `tokio-rustls` 是常见生产组合。

## 5. 使用 `ring` 进行底层密码学与 DH 密钥交换

TLS 握手内部已包含密钥协商；本章用 **`ring`** 单独演示 **Diffie-Hellman** 思想（理解「共享密钥从哪来」）。

### 问题

在不安全信道上，如何让双方得到**相同的对称密钥**，且窃听者无法从公开信息推导出？

### DH / ECDH 流程（书中：X25519）

| 步骤 | 客户端 | 服务器 |
|------|--------|--------|
| 1 | 生成 ephemeral 私钥 `c_priv` | 生成 `s_priv` |
| 2 | 计算公钥 `c_pub` | 计算 `s_pub` |
| 3 | 交换 `c_pub` ↔ `s_pub`（明文可传） | 同左 |
| 4 | `shared = DH(c_priv, s_pub)` | `shared = DH(s_priv, c_pub)` |
| 5 | 两端 **shared 相同** | 用于后续 AEAD 加密 |

**`ring`**：提供 **`agreement`**（如 **X25519**）、哈希、AEAD 等原语 — **不是**完整 TLS 栈，适合学习或自定义协议（需谨慎，勿重复发明 TLS）。

```text
应用层自定义协议（少见）
    可能：ring 做 X25519 + ChaCha20-Poly1305
生产 Web/API：
    用 TLS（rustls）即可，内部已含 ECDHE
```

**与 `01-atomic/`**：无直接重叠；与 **Ch5 gRPC** 常默认 TLS。
