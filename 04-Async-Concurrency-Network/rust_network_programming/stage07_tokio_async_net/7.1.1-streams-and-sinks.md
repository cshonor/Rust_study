## 7.1.1 Working with streams and sinks（流与接收器）

> 书 §7.1.1 · 贯通稿：[notes/Ch07-…](./notes/Ch07-使用Tokio进行异步网络编程-学习笔记.md) · 阻塞线程对照：[atomic §1.2 scoped threads](../../atomic/Chapter-01-Rust-Concurrency-Basics/1.2-scoped-threads/1.2-scoped-threads.md)

---

## 一、本节在学什么

Future 处理**单次**结果；网络 I/O 常是**连续多帧**。本节引入：

- **Stream**：异步版 `Iterator`（`.next().await`）
- **Sink**：异步版「写入端」，消化一系列项
- **异步同步原语**：`tokio::sync::mpsc` 等，替代阻塞版 `std::sync`

---

## 二、Stream：异步 Iterator

**一句话**：一系列**异步产生**的项；`None` 表示流结束。

| Iterator | Stream |
|----------|--------|
| `next()` → `Option<Item>` | `poll_next` / `.next().await` → `Option<Item>` |
| 同步拉取 | 未就绪时 Pending |
| `for x in iter` | `while let Some(x) = stream.next().await` |

用途：TCP 连续读帧、WebSocket 消息、定时器 tick、从 channel **消费**消息。

```rust
// 概念：从 Stream 读帧
while let Some(frame) = stream.next().await {
    process(frame);
}
```

套接字侧常用 **`AsyncRead` + `AsyncWrite`**（`tokio::io`）或 **`Framed<IO, Codec>`**（`tokio_util::codec`）把字节流拆成帧级 Stream。

---

## 三、Sink：异步写入端

**一句话**：异步**接收**项并发送出去（socket、channel、协议编码器）。

```text
Stream  ←── 读路径（入站数据）
Sink    ──→  写路径（出站数据）
```

| 场景 | 典型类型 |
|------|----------|
| task 间传消息 | `mpsc::Sender`（`send().await`） |
| 编码后写 socket | `Framed` 的 Sink 侧 |
| 背压 | bounded channel 容量、`poll_ready` |

**与 Ch4**：Codec 解码后 often 得到 `Stream<Item = Frame>`；编码走 Sink。

---

## 四、异步同步原语

标准库 `std::sync::Mutex`、`std::sync::mpsc` 在 async 里 **`.lock().await` 会阻塞 worker 线程**（或死锁），需 **async 感知**类型：

| 原语 | 说明 |
|------|------|
| **`oneshot`** | 单次完成通知 |
| **`mpsc`（tokio）** | 多生产者单消费者，task 间 message passing |
| **`tokio::sync::Mutex`** | 持锁期间让出线程（`.lock().await`） |
| **`watch` / `broadcast`** | 状态广播、多订阅者 |

**原则**：在 `.await` 点之间**少持锁**；优先 **message passing** 划分 task 边界。

```rust
use tokio::sync::mpsc;

#[tokio::main]
async fn main() {
    let (tx, mut rx) = mpsc::channel(4);
    for i in 0..3 {
        tx.send(format!("frame-{i}")).await.unwrap();
    }
    drop(tx); // 关闭发送端，rx 才能结束
    while let Some(item) = rx.recv().await {
        println!("sink got: {item}");
    }
}
```

仓库 demo 用 **`mpsc::recv`** 模拟 Stream 消费：[7.1.1-streams-and-sinks-mpsc-demo.rs](./7.1.1-streams-and-sinks/7.1.1-streams-and-sinks-mpsc-demo.rs)

## 五、对照：`std::sync::mpsc` vs `tokio::sync::mpsc`

| | **`std::sync::mpsc`** | **`tokio::sync::mpsc`** |
|---|----------------------|-------------------------|
| `send` / `recv` | 阻塞 | `.await` 异步 |
| 使用场景 | OS 线程间 | async task 间 |
| 在 `async fn` 里 | 阻塞 worker | 正确让出 |

书中 **`BiLock`**：两 future 间仲裁共享状态；现代多直接用 `tokio::sync::Mutex` + 短临界区。

---

## 六、新手易错

1. **忘记 `drop(tx)`**：多个 producer 时，最后一个 sender  drop 后 receiver 才收到 `None` 结束循环。
2. **无界 channel 掩盖背压**：生产远快于消费会 OOM；网络协议应配合 bounded channel 或 flow control。
3. **`std::sync::Mutex` 跨 `.await`**：易死锁；换 `tokio::sync::Mutex` 或把锁内逻辑改成 message passing。
4. **Stream 与字节流混淆**：TCP 是字节流，Stream 的「项」是你协议定义的**帧**，需 Codec 分帧（见 [7.2.2](./7.2.2-streaming-protocols.md)）。
