## 一、本节在学什么

[Ch03](../stage03_std_tcp_udp/) 用套接字传**原始字节**；本节把视角升到**应用层**的第一类协议——**RPC（Remote Procedure Call）**：

- 调用方像调**本地函数**一样，通过网络调用远端过程
- 不再手写「第 3 字节是命令号」，而是按**接口定义**（IDL）生成客户端/服务端代码
- 书中重点 **gRPC** = **HTTP/2** + **Protocol Buffers** + `.proto` 代码生成

本章 demo 用 **JSON over TCP** 演示 RPC 的 **stub → 网络 → skeleton** 形状，避免引入 `protoc` / `tonic` 完整工具链；完整 gRPC 按原书在独立项目练习。

---

## 二、RPC 是什么

**一句话**：客户端 **stub** 序列化参数、发请求；服务端 **skeleton** 收请求、执行业务、返回结果；调用方只看见「函数返回值」。

| 角色 | 职责 |
|------|------|
| **调用方（客户端 stub）** | 序列化参数 → 发请求 → 收响应 → 反序列化 |
| **服务方（服务端 skeleton）** | 收请求 → 调用真实业务逻辑 → 序列化结果并返回 |
| **价值** | 微服务拆分、**语言无关**接口（配合 IDL）、隐藏网络细节 |

与「自己定义 TCP 字节协议」的区别：

| | 手写 TCP 协议 | RPC / gRPC |
|---|---------------|------------|
| 接口 | 文档 + 约定偏移量 | **IDL（`.proto`）** 有 schema |
| 代码 | 手写编解码 | **生成** client stub / server trait |
| 演进 | 字段变更易踩坑 | protobuf 字段号 + 向后兼容规则 |
| 传输 | 任意 | gRPC 默认 **HTTP/2 on TCP** |

## 三、gRPC 与 Protocol Buffers

**gRPC** 是 Google 推出的 RPC 框架，本书重点技术栈：

| 层次 | 技术 |
|------|------|
| 传输 | 默认 **HTTP/2**（跑在 TCP 上） |
| 能力 | 一元调用、**服务端流 / 客户端流 / 双向流** |
| IDL | **Protocol Buffers（`.proto`）** |
| 载荷 | protobuf **二进制**编码（比 JSON 更紧凑、有 schema） |

```protobuf
// 概念示例：打车应用
syntax = "proto3";

message CabLocationRequest {
  double latitude = 1;
  double longitude = 2;
  string cab_id = 3;
}

service FooBarService {
  rpc RecordLocation(CabLocationRequest) returns (Empty);
  rpc GetNearbyCabs(NearbyRequest) returns (NearbyResponse);
}
```

```text
.proto  →  build.rs + protoc  →  生成的 Rust 模块
                                    ↓
                          server impl / client stub
                                    ↓
                          HTTP/2 上 gRPC 帧（TCP）
```

### 与现代生态对照（书外补充）

| 书中 / 较老栈 | 当前常见栈 |
|---------------|------------|
| `protoc-rust-grpc` | **`tonic`** + **`prost`**（Rust gRPC 主流） |
| 手写 HTTP/2 | 仍由库处理；你主要写 `.proto` 与业务 impl |

学习本节重在 **IDL → 代码生成 → 客户端/服务端分工**；具体 crate 名可随项目选型替换。

**延伸**：[stage06 HTTP](../stage06_http_programming/README.md) · [stage07 Tokio](../stage07_tokio_async_net/README.md)（异步 gRPC 常基于 Tokio）。

## 四、书中 Rust 实现流程（`protoc-rust-grpc`）

1. **编写 `.proto`**：定义 `message` 类型 + `service` 与 `rpc` 方法。
2. **构建时生成代码**：`build.rs` 调用 **`protoc-rust-grpc`**（或同类工具）+ `protoc`，在 **`cargo build` 时**生成 Rust 模块。
3. **实现服务端**：为生成的 trait 编写具体逻辑（如记录位置、查询附近车辆）。
4. **实现客户端**：调用生成的 stub 发起 RPC。

你写的业务代码只关心「实现 trait / 调 stub」，帧格式、HTTP/2 多路复用由框架处理。

## 五、本章 Demo：最小 JSON-RPC（Serde + TCP）

仓库用 [Ch04 Serde](../stage04_serde_parse/) 的 **JSON** 在 TCP 上模拟 RPC，对应 [Ch03](../stage03_std_tcp_udp/3.1-simple-tcp-server-client.md) 的 **`TcpListener` / `TcpStream` + `thread::spawn`** 模型。

**请求 / 响应结构**（与 gRPC 的「方法名 + 参数 → 结果」同形）：

```rust
#[derive(Deserialize)]
struct RpcRequest {
    method: String,
    a: i32,
    b: i32,
}

#[derive(Serialize)]
struct RpcResponse {
    result: i32,
}
```

### 服务端（skeleton）

| 步骤 | 做法 |
|------|------|
| 监听 | `TcpListener::bind("127.0.0.1:48081")?` |
| 接受 | `incoming()` → 每连接 **`thread::spawn`** |
| 处理 | `read` JSON → 按 `method` 分发 → `write_all` JSON 响应 |

```rust
fn handle(mut stream: TcpStream) -> std::io::Result<()> {
    let mut buf = [0u8; 1024];
    let n = stream.read(&mut buf)?;
    let req: RpcRequest = serde_json::from_slice(&buf[..n])?;
    let resp = if req.method == "add" {
        RpcResponse { result: req.a + req.b }
    } else {
        RpcResponse { result: -1 }
    };
    stream.write_all(&serde_json::to_vec(&resp)?)?;
    Ok(())
}
```

仓库对应：[5.1-introduction-to-rpc-server-demo.rs](./5.1-introduction-to-rpc/5.1-introduction-to-rpc-server-demo.rs) · 监听 **`127.0.0.1:48081`**

### 客户端（stub）

| 步骤 | API |
|------|-----|
| 连接 | `TcpStream::connect("127.0.0.1:48081")?` |
| 调用 | 组装 `RpcRequest` → `serde_json::to_vec` → `write_all` |
| 结果 | `read` → `serde_json::from_slice` → 打印 `result` |

```rust
let req = RpcRequest {
    method: "add".into(),
    a: 40,
    b: 2,
};
stream.write_all(&serde_json::to_vec(&req)?)?;
// read → RpcResponse → println!("RPC add(40,2) => {}", resp.result);
```

仓库对应：[5.1-introduction-to-rpc-client-demo.rs](./5.1-introduction-to-rpc/5.1-introduction-to-rpc-client-demo.rs)

### 与真实 gRPC 的对照

| | 本章 demo | 生产 gRPC |
|---|-----------|-----------|
| 编码 | JSON 文本 | protobuf 二进制 |
| 分帧 | 单次 read（教学简化） | HTTP/2 帧 + gRPC 状态码 |
| 工具 | `serde` / `serde_json` | `tonic` + `prost` + `.proto` |
| 价值 | 理解 stub/skeleton 分工 | 微服务 API、流式 RPC |

## 六、新手易错

1. **TCP 是字节流**：demo 假设「一次 read = 一条 JSON」；生产需长度前缀或 HTTP/2 分帧（见 [Ch04 粘包](../stage04_serde_parse/4.1-serde-serialization.md)）。
2. **先起 server 再起 client**；`48081` 被占用时 `bind` 失败。
3. **RPC ≠ REST**：RPC 强调「像函数调用」；REST 强调资源与 HTTP 动词；gRPC 走 HTTP/2 但语义仍是 RPC。
4. **demo 未实现错误码 / 超时**：真实 gRPC 有 `Status`、deadline；阻塞 `read` 可设 `set_read_timeout`（见 [3.1 读超时](../stage03_std_tcp_udp/3.1-simple-tcp-server-client.md)）。
5. **完整 gRPC 需 `protoc` 工具链**：本章刻意用 JSON 降低环境门槛；练手 gRPC 请另建项目并加 `tonic-build`。
