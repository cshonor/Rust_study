## 1.1.3 spawn 的 `Send + 'static` 约束

> 书 §1.1（续）· [1.1.1 spawn/join](./1.1.1-spawn-join-basics.md) · [1.1.2 JoinHandle](./1.1.2-join-handle.md) · 绕过 `'static`：[1.2 scope](../1.2-scoped-threads/1.2-scoped-threads.md) · [1.3 Arc](../1.3-shared-ownership/1.3-shared-ownership.md) · [1.6 Send/Sync](../1.6-send-sync/1.6-send-sync.md)

**结论**：

1. **`T: 'static` 不只修饰引用**——自有类型、`Arc`、结构体等都可以满足；含义是类型内部**不含任何短于 `'static` 的借用**（[§3.4](#34-tstatic-到底是什么)）。
2. **`spawn` 要防的是场景 B**：创建线程的**函数先 return**、局部栈释放，子线程仍可能被调度（不是「`main` 跑完子线程还在跑」）。
3. **`move` 自有类型进闭包** → 闭包内无短命引用 → **`F: 'static` 天然成立**（[§3.7](#37-spawn-两种传参借用-vs-move)）；借栈 → **`thread::scope`**（[§七](#七绕过-staticthreadscope-为什么可以借局部变量)）。

---

## 一、`spawn` 完整签名（逐条拆解）

> 是 **`spawn`**（创建线程），不是 swap（交换）。

```rust
pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T,  // 约束 1：闭包特征 + 返回 T
    F: Send,            // 约束 2：闭包可跨线程发送
    F: 'static,         // 约束 3：闭包捕获物须「活得够久」
    T: Send + 'static,  // 约束 4：返回值可跨线程传回
```

（标准库把 1–3 合并写成 `F: FnOnce() -> T + Send + 'static`。）

| # | 约束 | 对象 | 干什么 |
|---|------|------|--------|
| **1** | `F: FnOnce() -> T` | 闭包 | 无参、**只执行一次**的线程入口；`T` = 闭包返回值 → `JoinHandle<T>` 里的 `T`（见 [1.1.2](./1.1.2-join-handle.md)） |
| **2** | `F: Send` | 闭包 | 闭包**整体所有权**可安全 **move 到子线程**；`Rc`、裸指针等 **非 Send** → 编译拒绝 |
| **3** | `F: 'static` | 闭包 | 捕获的数据**不能是短生命周期借用**；子线程可能比**创建它的函数栈帧**活得更久（见 [§三](#三为什么必须-static)） |
| **4** | `T: Send + 'static` | 返回值 | `join` 把 **`T` move 回**等待线程；`JoinHandle` 可存到函数外 → **`T` 也不能带悬垂引用** |

### 约束 1：`FnOnce() -> T`

线程入口**只跑一遍**，故是 **`FnOnce`**，不是 `Fn` / `FnMut`（语义见 [1.1.4](./1.1.4-closure-traits-fnonce.md)）。

### 约束 2：`F: Send`

`spawn` 在**当前线程**构造闭包，再 **move 给新 OS 线程**执行 → 闭包及其捕获物必须 **`Send`**。

### 约束 3：`F: 'static` — **来自 `spawn` 签名，为防悬垂引用**

#### `'static` 从哪来？

`std::thread::spawn` 大致长这样（简化）：

```rust
pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,  // 👈 这里写死了 F: 'static
    T: Send + 'static,
{
    // ...
}
```

只要你调用 `thread::spawn(move || { ... })`，闭包类型 **`F`** 就必须满足 **`FnOnce` + `Send` + `'static`**。**约束 3 是 spawn API 自带的，不是凭空来的。**

#### 为什么 spawn 要 `F: 'static`？

**常见误解**：「`main` 跑完、主线程栈销毁，子线程还在访问栈引用。」

**更严谨的说法**：

| 场景 | 会发生什么 | 是不是 `'static` 要防的主因？ |
|------|----------|------------------------------|
| **A：`main` 全部代码执行完** | **整个进程立刻销毁**，所有子线程被 OS **强制终止** | ❌ 不是主因（进程退出后不存在「子线程长期跑」） |
| **B：普通函数 `create_thread()` 提前 `return`，`main` 仍在跑** | 函数栈上局部变量 **已 drop**，detach 或未 join 的子线程 **仍可能被调度** | ✅ **这才是核心风险** |

`spawn` 要求 **`F: 'static`**，防的是 **场景 B**：**创建者函数栈帧销毁 ≠ 子线程结束**。编译器无法预判调度顺序（先跑子线程还是先销毁栈），只能用最保守规则：**禁止闭包捕获任何非 `'static` 的借用**。

#### 核心反例（与 `main` 退出无关）

```rust
use std::thread;

fn create_thread() {
    let s = String::from("局部字符串，仅存活于本函数");
    // ❌ 闭包借用 s，F 不满足 'static
    let _h = thread::spawn(|| println!("{s}"));
    // 函数返回 → s 被 drop；子线程可能还没 println → 悬垂引用 / UB
}

fn main() {
    create_thread();
    // create_thread 已返回，s 早已释放；main 仍在 sleep
    std::thread::sleep(std::time::Duration::from_secs(2));
}
```

| 致命点 | 说明 |
|--------|------|
| 不是进程退出 | `main` 还没结束，但 **`create_thread` 的栈已销毁** |
| 调度不确定 | OS 可能先切走子线程、也可能先跑完函数 drop `s` |
| Rust 对策 | 编译期直接拒绝；须 **`move`**、**`'static` 数据** 或 **`thread::scope`** |

#### `JoinHandle` 被 drop → detach 加剧风险

`spawn` 返回的 **`JoinHandle`** 若**未 `join` 就被丢弃**，线程 **detach（分离）**：

- 子线程脱离句柄管控，创建它的**函数栈仍可先销毁**；
- 在进程最终退出、OS 强杀线程**之前**，子线程完全可能执行到**已释放栈**上的引用 → 崩溃、乱码等 **UB**。

因此 **`'static` 约束与 detach 风险是同一套模型**：线程寿命不受创建者栈帧约束时，闭包内绝不能有短生命周期借用。

**`'static` 在此的实用含义**：闭包**不能借**「可能比线程活得短」的数据；只能 **move 所有权** 或使用 **`'static` / 全局** 数据。详见 [§三](#三为什么必须-static)。

### 约束 4：`T: Send + 'static`

- **`Send`**：`T` 要能经 `join` **跨线程**传回；
- **`'static`**：`JoinHandle<T>` 可留存任意久，`T` 不能内含短期借用（**`T: 'static` 语义**见 [§3.4](#34-tstatic-到底是什么)）。

### 反例 + `move` 修复

> **为何必须手写 `move`？** 闭包默认 **最小化捕获**（能借就不拿所有权）→ 不加 `move` 往往是 **`&局部变量`**，过不了 **`F: 'static`**。详见 [§3.7](#37-spawn-两种传参借用-vs-move)。

```rust
let x = 10;
// 报错：F 不满足 'static——闭包默认借栈上 &x，不是 move 进 owned
let h = thread::spawn(|| dbg!(x));

// 修复：move 把 x 所有权移入闭包（owned 数据随闭包进子线程，不再依赖外层栈）
let h = thread::spawn(move || dbg!(x));
```

### 为什么 F 和 T 都要 `'static`？（双重 static）

| 对象 | 原因 |
|------|------|
| **`F`** | 闭包逻辑要在子线程里跑，线程寿命 **不受**创建者栈帧限制 |
| **`T`** | 返回值经 `JoinHandle` 保存，可能在 **join 之前**被带到函数外；不能带「比 handle 活得短」的引用 |

## 二、为什么必须 `Send`？

**`Send`**：类型可以**安全地把所有权从一个线程转移到另一个线程**。

`spawn` 时发生的事：

1. 在**当前线程**构造闭包 `F`；
2. `spawn` 把闭包 **move 到新 OS 线程**执行；
3. 子线程结束后，`join` 把 **`T` move 回**等待线程。

若闭包内含 **`Rc<T>`** 等非 `Send` 类型，跨线程 move 会破坏 `Rc` 不变量（非原子计数）→ **数据竞争 / 双重释放**。编译器用 **`F: Send`** 禁止。

**一句话**：**`Send` 保证闭包能安全「搬家」到新线程**。

## 三、为什么必须 `'static`？

**`'static` 不是一种「数据类型」**，而是**生命周期标注**：在此表示捕获的数据**不能依赖「可能比子线程先销毁」的栈帧**。

### 3.1 两套场景（纠正笼统说法）

| | **场景 A：进程退出** | **场景 B：函数提前返回** |
|--|---------------------|-------------------------|
| 典型情况 | `main` 走完、未 join 子线程 | `create_thread()` return，`main` 还在 `sleep` |
| 子线程命运 | 进程销毁 → **全体强杀** | 子线程**仍可能运行** |
| 局部栈 | 随进程一起没 | **创建函数栈已释放** |
| 与 `'static` | 不是设计 `'static` 的主因 | **这才是 `spawn` 要防的 UB** |

> **场景 A** 见 [1.1.1 §二](./1.1.1-spawn-join-basics.md)：`main` 结束则进程结束。  
> **场景 B** 才是日常踩坑：**函数作用域销毁 ≠ 子线程结束**。

### 3.1.5 易混点：编译期捕获 ≠ 运行时调度

很多人卡住在这句：**「编译期已经把该捕获的都捕获完了，怎么还会出现局部变量被 drop 了、子线程仍在访问？」**

—— 因为你在脑子里把 **两件独立的事** 合成了一件：

| 阶段 | 谁负责 | 在检查什么 |
|------|--------|------------|
| **① 编译期** | Rust 编译器 | 闭包**有没有合法拿到**变量（所有权转移 / 借用规则） |
| **② 运行时** | 操作系统调度 | 子线程**什么时候**真正执行闭包里的代码 |

**编译期只回答 ①**，不保证 ② 的顺序。`move` 闭包在编译期就把 `s` 的所有权**标记为移进闭包**；但子线程**何时**跑 `println!("{s}")` 完全由 OS 决定，编译器管不了。

**极端时间线（场景 B）**：

```rust
fn main() {
    let s = String::from("hello");
    thread::spawn(move || println!("{s}")); // ① 编译期：s 所有权已进闭包 ✅
    // ② 若此处 main 立刻 return（或外层函数 return）→ 栈帧销毁
    // ③ OS 可能还没调度子线程跑 println
}
```

若闭包捕获的是 **`&s`（借用）** 而不是 **`move` 后的 owned `s`**：

1. **编译期**：闭包类型里带着 **`&'a s`**，`'a` 只覆盖创建者栈帧 → 不满足 **`F: 'static`** → 直接**编译失败**（Rust 在此处就堵死了）。
2. **若没有 `'static` 约束（假想 C++）**：编译期「捕获完成」只表示闭包**记住了栈地址**；创建者函数 **return → 栈回收**；子线程**稍后才** dereference → **悬垂引用 UB**。问题不在「捕获没完成」，而在 **捕获的是借用的地址，而地址比子线程先失效**。

**`move` 自有类型（`String` 等）时**：编译期完成后，数据在**闭包内部**（通常随闭包一起被 move 进子线程），**不再依赖**外层栈帧上的那个槽位 —— 外层 `s` 绑定被移走，栈槽可以销毁，**子线程用的是闭包里的那份 owned 数据**，不会撞悬垂。

**`'static` 约束在堵什么**：不是重复检查「有没有 capture」，而是统一要求 —— 闭包类型里**不能含**「生命周期绑在可能比子线程更短的栈帧上」的引用；要么 **owned / `Arc` / 堆**，要么 **真·程序期数据**（`&'static str`、`static`），要么 **`thread::scope`** 把线程寿命绑回当前栈。

### 3.2 本质规则

**`F: 'static` 的本质**：`thread::spawn` 创建的子线程**生命周期不受创建它的函数 / 代码块限制**——创建者函数可以先 `return`、销毁栈上局部变量，子线程仍可能被 OS 调度。若闭包捕获 **`&'a T`** 且 `'a` 只覆盖该函数栈 → **悬垂引用**。

编译器**无法**在编译期模拟线程调度，故对 **`spawn` 统一要求 `'static`**。

### 3.3 合法方案

| 方案 | 做法 | 说明 |
|------|------|------|
| **1. move 所有权** | `spawn(move \|\| …)` | owned 数据随闭包进子线程，**不再借外层栈** |
| **2. 真·`'static` 数据** | `&'static str`、全局 `static` | 数据本身**整个程序**存活期内有效（见下 §「进程」指什么） |
| **3. 有界 scope** | **`thread::scope`** | 强制等子线程结束再销毁局部变量（[§七](#七绕过-staticthreadscope-为什么可以借局部变量)） |

#### 「进程」在 `'static` 里指什么？

**指整个程序从启动到退出的生命周期**（`main` 开始到进程结束），**不是**：

- ❌ 「闭包所在 OS 线程的内核线程对象」
- ❌ 「变量原来所在的那个栈帧 / 那个函数」

| 变量 | 生命周期绑在哪 | 能否直接给 `spawn` 闭包借？ |
|------|----------------|----------------------------|
| `static S: &str = "hello"` | **整个程序** | ✅ 借 `&'static str` 可以（数据在只读静态段） |
| `let s = String::from("hello")` | **当前栈帧**（如 `main` 块） | ❌ 不能借 `&s`；需 **`move`** 把 owned 移进闭包，或改 **`static` / `Arc`** |

口语里「生命周期和进程一样长」= **数据在程序运行期间不会被回收**；与变量**最初定义在哪个函数栈**无关 —— `main` 里的 `String` 仍只是栈上局部，**不等于**「进程级 `'static`」，所以 **`spawn` 仍要 `move` 或 `Arc`**。

**一句话**：**线程可能活过创建它的函数栈 → 强制 `'static`**；借局部栈 → **`thread::scope`**（[1.2.4 深拆](../1.2-scoped-threads/1.2.4-scoped-join-handle.md#static-vs-scope生命周期深拆)）。

> `T: 'static` 同理：`JoinHandle<T>` 可留存任意久，返回值不能内含比 handle 活得短的借用。  
> `move` 转移 **owned** 的 `String` 等即可，**不必**字面量写 `&'static`。

```rust
let num = 10;
// 报错：闭包借用 &num，生命周期只有 'a（当前栈），不是 'static
// let h = thread::spawn(|| println!("{num}"));

let h = thread::spawn(move || println!("{num}")); // num 所有权进闭包
```

### 3.4 `T: 'static` 到底是什么？

**规则**：类型 **`T`** 满足 **`T: 'static`**，当且仅当：

> **`T` 内部不存在任何非 `'static` 生命周期的引用（借用）。**

这与「是不是引用类型」无关——**任意类型**都可以（或不可以）满足 **`T: 'static`**：

| 概念 | 含义 |
|------|------|
| **`&'static T`** | **静态引用**：带生命周期标注的**引用类型**（如字面量 `"hello"` → `&'static str`） |
| **`T: 'static`** | **类型约束**：该类型里**所有**借用的生命周期都须 ≥ `'static`；**纯自有、无内部 `&` 的类型自动满足** |

**一句话**：

- 带 **`&`** 的是「引用自带生命周期」；
- **`T: 'static`** 是对**任意类型**的筛选——**有没有引用都可以套这条规则**。

| | **`&'a T` 里的 `'a`** | **`T: 'a` / `T: 'static`** |
|--|----------------------|---------------------------|
| 修饰对象 | **引用**能活多久 | **类型**里所有借用至少要活多久 |
| 特例 | `'a = 'static` → 静态引用 | **`T: 'static`** → 内部借用须活到程序结束；**无借用的自有类型也成立** |

### 3.5 三类典型：`String` / 借引用 / `Arc`

#### 情况 1：自有所有权、无内部引用 → 自动 `T: 'static`

`String`、`Vec<i32>`、`i32`、`Box<T>` 等**完全持有数据、内部不带 `&`** 的类型，均 **`T: 'static`**。

```rust
let s = String::from("abc");
thread::spawn(move || println!("{s}")); // move 后闭包只拥有 s，F: 'static ✅
```

#### 情况 2：捕获局部引用 `&'a T` → 违反 `F: 'static`

```rust
let s = String::from("abc");
let r = &s; // &'局部 String
thread::spawn(|| println!("{r}")); // 闭包内含非 'static 引用 → 编译失败 ❌
```

#### 情况 3：`Arc<T>` 共享所有权 → `Arc<T>: 'static`

`Arc` **不借**外层栈，只在堆上共享计数；`Arc<String>: 'static`：

```rust
use std::sync::Arc;
let s = Arc::new(String::from("abc"));
let s_clone = Arc::clone(&s);
thread::spawn(move || println!("{s_clone}")); // ✅
```

### 3.6 极易混淆：`&'static T` vs `T: 'static`

**完全不是一回事**：

```rust
// 1. &'static — 引用类型，指向只读静态段
let static_str: &'static str = "hello";

// 2. String: 'static — 类型约束：String 内部无短命引用；与「静态字符串」无关
let owned_str: String = String::from("hello");
```

| 写法 | 是什么 |
|------|--------|
| **`&'static str`** | **引用**，生命周期 = 整个程序 |
| **`String: 'static`** | **约束**：`String` 这种类型可以不含短命借用；普通 `String::from(...)` 变量仍满足 |

### 3.7 spawn 两种传参：借用 vs `move`

#### 闭包默认不会帮你 `move`（最小化捕获）

Rust 闭包捕获是 **「能借就不拿所有权」**，**不会**因为写了 `thread::spawn` 就自动 `move`：

| 闭包怎么用外部变量 | 编译器默认捕获什么 |
|--------------------|-------------------|
| 只读（如 `println!("{s}")`） | **`&T`** 引用 |
| 需要修改 | **`&mut T`** |
| 显式写 **`move`** | 把列出的变量 **所有权移进闭包** |

因此 **`spawn(|| …)` 不加 `move`** 时，闭包类型里往往是 **「局部变量的引用」** — 生命周期只跟创建者栈帧绑在一起，**不够 `'static`** → 编译器直接报错（如 `E0597` / `does not live long enough`）。

**心智模型（对照 spawn 的「必须 own」）**：

```text
thread::spawn 要的闭包  ≈  捕获物不能绑在「可能比子线程更短」的栈上
                              → 实质上要 own（或 Arc / &'static 等例外）

move 关键字              ≈  你手动下令：「把这些变量的所有权交给闭包」

没有 move                →  编译器只给 & / &mut  →  过不了 F: 'static
```

**一句话**：`spawn` 不会替你做 `move`；**`move` 是你满足 `'static` 的常用开关**，把栈上局部变成闭包内的 owned 数据。

#### 方式 A：传引用（捕获 `&T`）→ 违反 `F: 'static`

```rust
fn bad() {
    let data = vec![1, 2, 3];
    // 即使写 &data，默认捕获仍是「借 vec 的引用」→ 编译失败 ❌
    thread::spawn(|| dbg!(&data));
}
```

#### 方式 B：`move` 转移所有权 → 满足 `F: 'static`

```rust
fn good() {
    let data = vec![1, 2, 3];
    thread::spawn(move || dbg!(data)); // 闭包独占 Vec，内部无借用 → ✅
}
```

**`move` 的核心原理**：把**自有类型**的所有权移入闭包，闭包类型里**不再含**对创建者栈的引用 → **`F: 'static` 自然成立**，与是否写 `&'static` 无关。

---

## 四、常见错误：借用局部变量

```rust
use std::thread;

fn main() {
    let s = String::from("hello");

    // 错误：闭包捕获 &s，生命周期不是 'static
    let h = thread::spawn(|| {
        println!("{s}");
    });

    h.join().unwrap();
}
```

典型报错：`error[E0597]: 's' does not live long enough` — 闭包需要 **`'static`**，不能借有限生命周期的 `s`。

仓库对照：[move-closure-demo.rs](./code/1.1-threads-in-rust-move-closure-demo.rs)（正确写法）

---

## 五、正确写法：`move` 转移所有权

```rust
use std::thread;

fn main() {
    let s = String::from("hello");

    let h = thread::spawn(move || {
        println!("{s}"); // 闭包拥有 s
    });

    h.join().unwrap();
}
```

| 要点 | 说明 |
|------|------|
| **`move`** | 强制闭包**取得**捕获变量的所有权，不再借用 |
| **为何满足约束** | owned 的 `String` 随闭包进子线程，**不依赖**外层栈帧 |

---

## 六、多线程共享：只读 `Arc`

```rust
use std::sync::Arc;
use std::thread;

fn main() {
    let s = Arc::new(String::from("hello"));
    let mut handles = vec![];

    for _ in 0..2 {
        let s = Arc::clone(&s); // 只增引用计数
        handles.push(thread::spawn(move || {
            println!("{s}");
        }));
    }
    for h in handles {
        h.join().unwrap();
    }
}
```

- **`Arc<T>`**：`T: Send + Sync` 时 **`Arc` 可 Send**，多线程共享**只读**；
- 可变共享 → **`Arc<Mutex<T>>`**（[1.7](../1.7-mutex-rwlock/1.7-mutex-rwlock.md)）。

---

## 七、绕过 `'static`：`thread::scope` 为什么可以借局部变量？

**有界线程**：`thread::scope` 会**阻塞当前函数**，直到 scope 内**所有**子线程执行完毕，才继续往下走、才销毁被借用的局部变量 → 生命周期**闭环**，故闭包用 **`F: 'scope`** 而非 **`'static`**。

```rust
use std::thread;

fn safe_scope() {
    let s = String::from("局部变量"); // 存活于 safe_scope 栈帧
    thread::scope(|scope| {
        scope.spawn(|| println!("{s}")).join().unwrap(); // ✅ 借 s，无需 move
    });
    // scope 块结束：已自动 join 全部子线程，此时 s 才随函数栈销毁
}
```

| 机制 | `thread::spawn` | `thread::scope` |
|------|-----------------|-----------------|
| 创建者函数能否先 return | 能；子线程可继续跑 | scope **阻塞**到子线程全结束 |
| 局部变量销毁时机 | 函数 return 即 drop | **子线程结束后**才 drop 被借数据 |
| 闭包约束 | **`F: 'static`** | **`F: 'scope`** |
| HFT 倾向 | 线程池 / 长期任务（常 `move` / `Arc`） | 函数内有界并行，**少堆分配、生命周期可控** |

> **注意**：变量若定义在 **`scope` 闭包内部**，`scope.spawn` 仍常须 **`move`**（见 [1.2 demo](../1.2-scoped-threads/code/1.2-scoped-threads-demo.rs)）；上例是 **scope 外、函数内的局部变量**——这正是 scope 要解决的典型场景。

**最小可运行对比** → [1.1-threads-in-rust-static-vs-scope-demo.rs](./code/1.1-threads-in-rust-static-vs-scope-demo.rs)（含 `safe_scope` 风格 demo；注释含 `create_thread` 编译报错版）。

详见 [1.2 Scoped Threads](../1.2-scoped-threads/1.2-scoped-threads.md)、[1.2.4 `'static` vs `'scope` 深拆](../1.2-scoped-threads/1.2.4-scoped-join-handle.md#static-vs-scope生命周期深拆)。

## 八、HFT 语境与学习顺序

**结论**：**`spawn` + `'static` 是基础多线程模型，HFT 里会用到，但不是核心瓶颈**；HFT 更怕的是**锁、系统调用、分配、缓存失效**，而不是 `'static` 标注本身。

### HFT 传参与选型（贴合 `'static` 语义）

| 需求 | 推荐 | 原因 |
|------|------|------|
| 数据进子线程 | **`move` 自有类型** | 自动 **`F: 'static`**，无借用生命周期 |
| 多线程共享读 | **`Arc` + `move` clone** | **`Arc<T>: 'static`**，堆上共享 |
| 函数内借局部、要低延迟 | **`thread::scope`** | 允许借栈，**不必 `move` / 不必 `Arc` 堆分配** |

**一句话**：**`spawn` 优先 `move` 或 `Arc`**；不想受 `'static` 限制、追求可控生命周期 → **`scope`**。

### 建议学习顺序（面向系统 / HFT 方向）

1. **`std::sync`**（锁）、**`Arc`**、**`Mutex`**、**`Condvar`**、**`thread::scope`** → 共享状态、线程协作、安全访问  
2. **`tokio` / async**、**TCP/UDP** → 行情、订单、数据链路  
3. **过程宏** → 放最后；核心路径几乎不用，多用于序列化、配置、代码生成等辅助

**一句话**：**`F: 'static` 防的是「函数栈先销毁、子线程仍跑」**；HFT 函数内并行优先 **`thread::scope`**（生命周期可控、少 `Arc` 堆分配）。

## 九、场景对照表（错误 / 正确 / 原因）

| 场景 | ❌ 常见错误 | ✅ 正确写法 | 原因 |
|------|------------|------------|------|
| 局部 `String` 进子线程 | `spawn(\|\| println!("{s}"))` 无 move | `spawn(move \|\| …)` | 不能捕获 `&s`；须 **owned + Send** |
| 多个线程读同一份数据 | `Rc::clone` + `spawn` | **`Arc::clone` + `move`** | **`Rc` 非 Send** |
| 多个线程改同一份数据 | 多个 `move` 抢同一 `String` | **`Arc<Mutex<T>>`** | 要 **Sync + 互斥** |
| 短生命周期借用 | `spawn(\|\| use &local)` | **`thread::scope`** | `'static` 不允许借栈 |
| 全局配置 | — | **`static` / `LazyLock`** | 天然 `'static` |
| 子线程必在 fn 内结束 | 手动 join 列表 | **`thread::scope`** | 自动 join + 允许借用 |
| 返回值回主线程 | 返回 `&local` | 返回 **owned `T`** 且 **`T: Send`** | `join` 也要 **`T: Send`** |

---

## 十、背诵版

1. **`T: 'static`** = 类型内部**无短命引用**；**自有 `String`/`Vec`/`Arc` 等自动满足**，与 **`&'static str`**（静态**引用**）不是一回事。  
2. **`spawn` 防场景 B**：创建函数栈先销毁，子线程仍可能被调度（**不是** `main` 跑完子线程还在跑）。  
3. **`move` 自有类型** → 闭包无借用 → **`F: 'static` 天然成立**；共享 → **`Arc`**；借栈 → **`scope`**。  
4. HFT：关键路径优先 **scope / 无锁**；长期任务 **`move` / `Arc`**（[§八](#八hft-语境与学习顺序)）。
