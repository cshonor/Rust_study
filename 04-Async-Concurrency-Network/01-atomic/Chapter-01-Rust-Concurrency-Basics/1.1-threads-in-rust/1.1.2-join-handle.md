**一句话结论**：`std::thread::spawn(闭包)` → 返回 **`JoinHandle<T>`**，`T` = 子线程闭包的**返回值类型**。

```rust
use std::thread;

// 闭包无 return、只有语句 → T = () → JoinHandle<()>
let h: thread::JoinHandle<()> = thread::spawn(|| {
    println!("hello from thread");
});
let _ = h.join(); // Result<(), Box<dyn Any + Send + 'static>>
```

| 要点 | 说明 |
|------|------|
| **`T`** | 闭包 **`return` 的类型**；不写 return 时默认为 **`()`** |
| **所有权** | **`JoinHandle` 不可 `Clone`**；全进程唯一一份 handle，**只能被一个所有者** `move` 后 **`join(self)` 一次**（与 OS「只能 join 一次」一致） |

```rust
use std::thread;

fn main() {
    let h = thread::spawn(|| {
        println!("子线程正在运行");
        100 // 子线程返回值 → JoinHandle<i32>
    });

    let res = h.join().unwrap();
    println!("子线程返回值：{res}");
    println!("主线程收尾");
}
```

```text
子线程正在运行
子线程返回值：100
主线程收尾
```

## 一、JoinHandle 是什么

1. **持有对应子线程的 join 权**（句柄，非线程本体）；另有 **`.thread()`** 可查 `ThreadId`，**同步等待靠 `join()`**。
2. **`.join()`**：阻塞**当前**线程，等待子线程执行完毕，取回 **`Result<T, Box<dyn Any + Send + 'static>>`**：
   - 线程正常结束 → `Ok(T)`
   - 子线程 panic → `Err(Box<dyn Any + Send + 'static>)`（见 [§二 Err 三层拆解](#二err-类型三层拆解-boxdyn-any--send--static)）
3. **不调用 `join`、也不 drop 这个变量**：`main` 结束时进程退出，**子线程会被强制终止**（见 [§四](#四不用-join-会怎样)）。

> 补充：若 **`drop` 了 `JoinHandle` 且未 `join`**，子线程变为 **detached**，与 caller 脱钩；`main` 退出时同样可能来不及跑完。

## 二、`spawn` / `join` 类型拆解

```rust
pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,
```

| 泛型 | 含义 |
|------|------|
| `F` | 传入的线程函数 / 闭包 |
| `T` | 闭包 **return** 的类型 → `JoinHandle<T>`；无返回值时为 **`()`** |
| `Send + 'static` | **闭包捕获的变量必须能跨线程转移所有权**，不能借用栈上临时变量 |
| **`JoinHandle` 本身** | **无 `Clone`**；`join(self)` **消耗** handle，不能二次 join |

#### 有 / 无返回值：签名不变，`T` 自动推导

`spawn` 的 **`FnOnce() -> T + Send + 'static`** 写法**不管闭包有没有返回值都一样** — 你**不用**为「无返回值后台任务」写特殊 API；有没有 return 交给类型系统：

| 闭包写法 | 编译器推导的 `T` | `JoinHandle` | `join()` 成功时 |
|----------|------------------|--------------|-----------------|
| 只有语句、无 `return` | **`()`**（单元类型） | `JoinHandle<()>` | `Ok(())` |
| 末尾表达式 / `return x` | 表达式类型，如 `i32` | `JoinHandle<i32>` | `Ok(x)` |

```rust
use std::thread;

// 无返回值 → FnOnce() -> () + Send + 'static
let h1 = thread::spawn(|| {
    println!("后台打印");
}); // JoinHandle<()>

// 有返回值 → FnOnce() -> i32 + Send + 'static
let h2 = thread::spawn(|| 42); // JoinHandle<i32>

// 分离态 fire-and-forget：同样写法，drop(h1) 即可，不必改 spawn 语法
drop(h1);
```

**分离态 / 辅助任务**（日志、心跳、落库）多数是 **`T = ()`**：`join` 即便调用也只是 `Ok(())`，所以更常 **`drop(handle)` 放权** → [1.1.7 §分离态](./1.1.7-thread-lifecycle-four-states.md#分离态的心智模型放权)。**约束仍是 `Send + 'static`**（捕获物能否跨线程 move），与 `T` 是否为 `()` 无关 → [1.1.3](./1.1.3-send-static-spawn.md)。

#### `FnOnce() -> T` 的箭头：谁必须写、谁可省略

分场景 — **写闭包时**和**在 API 里写泛型约束时**不是同一回事：

| 场景 | 要不要写 `() -> T` | 原因 |
|------|-------------------|------|
| **调用 `spawn` 传闭包** | **不用**你在闭包上写 | 闭包体按平常写；编译器从 **`spawn` 的 where 约束 + 闭包体** 反推 `T` |
| **定义 `spawn` 这类 API 的 where** | **必须**写 `FnOnce() -> T` | 泛型边界要钉死：**无参调用** + **返回类型 `T`**，才能连到 `JoinHandle<T>` |
| **局部变量 + `impl Trait`** | 常可省略 `-> …` | 上下文足够时只写 **`impl FnOnce()`** 即可 |

```rust
use std::thread;

// ① 调用方：不写箭头，T 自动为 ()
thread::spawn(|| println!("hi"));

// ② 局部约束：省略返回值，编译器知 Output = ()
let f: impl FnOnce() = || println!("hi");
let _ = f();

// ③ 标准库 spawn 的定义侧（示意）— 必须写 FnOnce() -> T + Send + 'static
// pub fn spawn<F, T>(f: F) -> JoinHandle<T>
// where
//     F: FnOnce() -> T + Send + 'static,
//     T: Send + 'static,
```

**为何跨线程 API 要写全 `FnOnce() -> T + Send + 'static`？** 不是为难调用者，而是让编译器在**泛型边界**上严格校验：闭包**调用签名**、**返回值能否进 `JoinHandle<T>`**、**捕获物能否 `Send` 跨线程**、**生命周期是否 `'static`**。调用侧仍只写 `|| { … }` 即可。

语法拆解 → [1.1.5 §一–§三](./1.1.5-where-fnonce-syntax.md) · **trait / 参数 / 返回值三维** → [1.1.5 §地图](./1.1.5-where-fnonce-syntax.md#先建立地图三个独立维度) · 何时可省略 `-> T` → [1.1.5 §何时必须写箭头](./1.1.5-where-fnonce-syntax.md#何时必须写-fnonce---t何时可省略)。

完整签名：

```rust
pub fn join(self) -> Result<T, Box<dyn Any + Send + 'static>>
//                      ^Ok(返回值)  ^Err：子线程 panic 时的载荷
```

### Err 类型三层拆解：`Box<dyn Any + Send + 'static>`

子线程 **panic** 时，`join()` 拿不到 `T`，而是 **`Err(…)`**。

#### 设计动机：子线程 panic 载荷**完全不可控**

标准库在定义 `join` 时**无法预知**你在子线程里会抛什么 — 可能是 `&str`、`String`，也可能是自定义的 **`StrategyError`**（策略名、错误码、数据源标识）。若写成 `Result<T, String>` 或 `Result<T, StrategyError>`，就会**漏掉**其他 panic 形态，API 也无法扩展。

因此采用 **「先统一容器，再按需还原类型」** — 与量化里兼容多数据源错误的设计同构：

```text
子线程 panic 抛出 V（V 可能是 &str / String / StrategyError / …）
        ↓ Box 堆分配（V 大小编译期未知）
        ↓ dyn Any 类型擦除
join 侧固定拿到：Box<dyn Any + Send + 'static>
        ↓ downcast_ref::<T>()
按需还原：&str · String · StrategyError · …
```

| 层 | 写法 | 解决什么问题 |
|----|------|--------------|
| ① **`dyn Any`** | `std::any::Any` | **类型擦除** — 异构 panic 载荷装进**同一** trait 对象 |
| ② **`+ Send`** | trait 约束 | 载荷从**子线程**移回调 `join` 的线程 |
| ③ **`Box<…>`** | 堆指针 | `dyn Any` 是 **DST**，大小运行时才知道，**不能裸放栈上** |

#### 量化示例：`StrategyError` 与字符串错误同一套 `match`

```rust
use std::any::Any;
use std::panic;
use std::thread;

#[derive(Debug)]
struct StrategyError {
    strategy: String,
    code: u32,
    source: &'static str, // 如 "ctp" / "ib" / "sim"
}

fn handle_panic_payload(err: &Box<dyn Any + Send + 'static>) {
    if let Some(se) = err.downcast_ref::<StrategyError>() {
        eprintln!(
            "策略 {} 失败 [{}] code={}",
            se.strategy, se.source, se.code
        );
    } else if let Some(s) = err.downcast_ref::<&str>() {
        eprintln!("panic 消息: {s}");
    } else if let Some(msg) = err.downcast_ref::<String>() {
        eprintln!("panic 消息: {msg}");
    } else {
        eprintln!("未知 panic 类型: {:?}", err.type_id());
    }
}

fn main() {
    let h = thread::spawn(|| {
        panic::panic_any(StrategyError {
            strategy: "mean_revert_v2".into(),
            code: 1001,
            source: "ctp",
        });
    });

    if let Err(e) = h.join() {
        handle_panic_payload(&e); // 一条路径：先统一 Box，再 downcast 分支
    }
}
```

**要点**：

1. **不必**为 `&str` / `String` / `StrategyError` 写三套 `join` 签名 — `Err` 类型固定，分支在 **`downcast_ref`**。
2. 自定义 struct 须 **`Send + 'static`**（与 `panic_any` 要求一致），才能进 `Box<dyn Any + Send + 'static>`。
3. 生产更推荐 **`Result` 正常返回错误**；`panic_any` + downcast 适合**不可恢复**或遗留边界；日常 `panic!` 仍走 `&str`/`String` 分支即可。

#### `panic!` 与 `panic_any`：互补，不是替代

| | **`panic!` 宏** | **`std::panic::panic_any(value)`** |
|---|----------------|-------------------------------------|
| **载荷** | 字符串或可格式化成字符串的内容 | **任意** `Send + 'static` 类型 |
| **典型载荷** | `&'static str`、`String` | `StrategyError`、`i32`、自定义 struct … |
| **作用** | 快速抛可读消息 | **突破字符串限制**，把带上下文的自定义错误丢进 panic 流程 |
| **join 后还原** | `downcast_ref::<&str>` / `::<String>` | `downcast_ref::<StrategyError>` 等 |

```rust
use std::panic;

// panic! — 只能走字符串路线
panic!("CTP 连接超时");

// panic_any — 整颗 StrategyError 当载荷，不必 to_string
panic::panic_any(StrategyError {
    strategy: "arb_v3".into(),
    code: 2002,
    source: "ctp",
});
```

**记忆**：需要**策略名 / 错误码 / 数据源**等结构化上下文 → **`panic_any`**；只需一句人话 → **`panic!`** 足够。

#### 与 `anyhow`：把 join 的 `Err` 并进 `Result` + `?` 主流程

`join()` 的 **`Err(Box<dyn Any + Send + 'static>)`** 与生态 **`anyhow::Error`** 用的是**同一套类型擦除思路** — `anyhow::Error` 内部也借助 **`Box<dyn Any + Send + Sync + 'static>`** 等机制做 downcast / 上下文链（应用层错误汇总见 [Item 04 anyhow](../../../../01-ER/Chapter-01-Types/Item-04-idiomatic-error-types/README.md)）。

量化策略里若主流程已是 **`anyhow::Result`**（数据源错误、计算错误都用 **`?`**），子线程 panic 不必单独写一套分支 — **join 拿到 Box 后，转成 `anyhow::Error` 再 `?`**：

```rust
use anyhow::{Context, Result, anyhow};
use std::any::Any;
use std::panic;
use std::thread::{self, JoinHandle};

#[derive(Debug)]
struct StrategyError {
    strategy: String,
    code: u32,
    source: &'static str,
}

/// 把 join 的 panic 载荷统一成 anyhow，与策略里其它 ? 同型
fn join_or_anyhow<T>(h: JoinHandle<T>) -> Result<T> {
    h.join().map_err(|payload| panic_payload_to_anyhow(payload))
}

fn panic_payload_to_anyhow(payload: Box<dyn Any + Send + 'static>) -> anyhow::Error {
    if let Some(se) = payload.downcast_ref::<StrategyError>() {
        anyhow!(
            "策略线程 panic: {} [{}] code={}",
            se.strategy,
            se.source,
            se.code
        )
    } else if let Some(s) = payload.downcast_ref::<&str>() {
        anyhow!("策略线程 panic: {s}")
    } else if let Some(msg) = payload.downcast_ref::<String>() {
        anyhow!("策略线程 panic: {msg}")
    } else {
        anyhow!("策略线程 panic（未知载荷类型）")
    }
}

fn run_strategy() -> Result<()> {
    let bars = fetch_bars_from_ctp().context("拉 CTP 行情")?; // 普通 anyhow 错误

    let worker = thread::spawn(move || {
        if bars.is_empty() {
            panic::panic_any(StrategyError {
                strategy: "mean_revert_v2".into(),
                code: 1001,
                source: "ctp",
            });
        }
        // …
    });

    join_or_anyhow(worker).context("风控 worker 线程")?; // panic 也并进 ? 链
    Ok(())
}

fn fetch_bars_from_ctp() -> Result<Vec<u8>> {
    Ok(vec![])
}
```

**两条还原路径（可并存）**：

| 路径 | 何时用 |
|------|--------|
| **`downcast_ref::<StrategyError>()`** | 只要结构化字段，不必立刻进 `anyhow` |
| **包装成 `anyhow::Error` 再 `?`** | 与 **`fetch_*?` / 计算 `?`** 同一套错误流，日志 / context 一致 |

> **边界**：`panic` 表「不可恢复」语义不变 — `anyhow` 统一的是**主线程侧怎么处理 join 失败**，不是鼓励用 panic 代替 `Result`。Worker 里可恢复错误仍优先 **`mpsc` / `Result` 回传**；`panic_any` + anyhow 适合 worker 已 panic、主线程要把载荷**并回策略级 `?` 链**的场景。

```rust
use std::any::Any;
use std::thread;

let h = thread::spawn(|| {
    panic!("boom");
});

match h.join() {
    Ok(v) => println!("{v:?}"),
    Err(e) => {
        if let Some(s) = e.downcast_ref::<&str>() {
            eprintln!("子线程 panic: {s}"); // s 实际是 &&str，见 [§2.2](#22-downcast_refstr-类型与层级精读)
        }
    }
}
```

> 日常 `.join().unwrap()` = 子线程 panic 时**主线程也 panic**。要捕获 → **`match` + `downcast_ref`**（见 [§2.1](#21-panic-捕获与-downcast_ref)；`&&str` 等类型细节见 [§2.2](#22-downcast_refstr-类型与层级精读)）。

#### panic 与 `match` 的分工：抛 vs 接

先把**两个角色**拆开 — **`panic` 负责扔，`match` + `join` 负责接**；不要混成「match 是一种 panic」：

```text
子线程（抛出侧）                    主线程（捕获侧）
─────────────────                  ─────────────────
发现不可恢复错误                      handle.join()
  │                                    │
  ├─ panic! / panic_any ──► 线程终止     ├─ Ok(T)  → 正常返回值
  │   （主动抛出）                      │
  └─ 或运行时越界等 ──► 自动 panic       └─ Err(Box<dyn Any>)
                                           │
                                           match + downcast_ref
                                           （记录 / 备用策略 / 告警 …）
```

| 侧 | 手段 | 语义 |
|----|------|------|
| **抛出** | **`panic!`** / **`panic::panic_any(...)`** | **不可恢复** — 策略依赖的行情断了、状态无法继续，**直接终止该线程**并把载荷装箱 |
| **捕获** | **`join()` + `match` + `downcast_ref`** | **处理**已发生的 panic — 不管载荷来自**主动 `panic_any`** 还是**运行时自动 panic**（如数组越界），`join` 拿到的都是 **`Err(Box<dyn Any + Send + 'static>)`**，拆解方式相同 |

##### 主动 panic vs 运行时自动 panic：来源不同，**汇合点相同**

子线程里 panic 有**两条来源**，但 **`join()` 之后形态一致** — 都要靠你在主线程 **`match` / `if let` + `downcast_ref` 手动匹配**各类载荷：

| 来源 | 子线程里典型触发 | 常见载荷类型 | 谁「抛」的 |
|------|------------------|--------------|------------|
| **主动 panic** | `panic!`、`panic_any(StrategyError { … })`、业务判断「没法继续」 | `&str`、`String`、**自定义 struct** | **你写的代码** |
| **运行时自动 panic** | `.unwrap()` / `.expect()` 失败、数组越界、除零、`assert!` 失败 … | 多为 **`&'static str`**（如 `"index out of bounds"` — 编译期写死在程序里）或 **`String`** | **Rust 运行时 / 标准库** |

```text
子线程                              主线程（唯一处理入口）
────────                            ────────────────────
主动 panic_any(StrategyError)  ─┐
主动 panic!("逻辑异常")         ─┤
自动 [99] 越界                 ─┼──► join() ──► Err(Box<dyn Any>)
自动 opt.unwrap()              ─┘              │
                                               ├─ downcast StrategyError → 备用策略 / 告警
                                               ├─ downcast &str / String → 默认消息
                                               └─ 兜底 type_id()
```

**精准结论**：

1. **主动 / 自动只是子线程侧成因不同** — 不会走两套 join API。
2. **自动 panic 的载荷同样进 `Box<dyn Any>`** 传回主线程；不会因为是越界就「漏 join」。
3. **主线程必须自己写匹配链** — 既覆盖官方默认的 **`&str` / `String`**，也加 **`StrategyError`** 等自定义分支，才能做**全链路**处理（记错误码、切策略、告警 vs 仅打印越界信息）。

```rust
// 同一套 on_strategy_panic 处理两种来源 — 示意
let h1 = thread::spawn(|| {
    panic::panic_any(StrategyError { /* 主动 */ .. });
});
let h2 = thread::spawn(|| {
    let v = vec![1, 2, 3];
    let _ = v[99]; // 运行时自动 panic → join 仍是 Err(Box<dyn Any>)
});

for h in [h1, h2] {
    if let Err(payload) = h.join() {
        on_strategy_panic(&payload); // 同一匹配链
    }
}
```

> 口语里常说「用 panic 抛自定义错误」— 代码上自定义 struct 须走 **`panic_any(StrategyError { … })`**，**不能**写 `panic(StrategyError { … })`（`panic!` 宏只接受格式字符串路线）。

**量化完整链路**：定义带错误码、上下文的 **`StrategyError`** → 子线程里 **`panic_any`** → 主线程 **`match h.join()`** → **`downcast_ref::<StrategyError>`** 分支做针对性处理：

```rust
use std::any::Any;
use std::panic;
use std::thread;

#[derive(Debug)]
struct StrategyError {
    code: u32,
    strategy: String,
    detail: String, // 如 "行情 feed 断开"
}

fn on_strategy_panic(payload: &Box<dyn Any + Send + 'static>) {
    if let Some(se) = payload.downcast_ref::<StrategyError>() {
        log_error_code(se.code);
        switch_to_backup_strategy(&se.strategy);
        send_alert(&se.detail);
    } else if let Some(s) = payload.downcast_ref::<&str>() {
        eprintln!("panic 消息: {s}");
    } else if let Some(msg) = payload.downcast_ref::<String>() {
        eprintln!("panic 消息: {msg}");
    } else {
        eprintln!("未知 panic 载荷: {:?}", payload.type_id());
    }
}

fn log_error_code(_: u32) {}
fn switch_to_backup_strategy(_: &str) {}
fn send_alert(_: &str) {}

fn main() {
    let h = thread::spawn(|| {
        let feed_ok = false;
        if !feed_ok {
            // 行情断了 — 不可恢复，主动 panic_any
            panic::panic_any(StrategyError {
                code: 4001,
                strategy: "mean_revert_v2".into(),
                detail: "CTP 行情 feed 断开".into(),
            });
        }
    });

    match h.join() {
        Ok(_) => println!("worker 正常结束"),
        Err(payload) => on_strategy_panic(&payload),
    }
}
```

**收益**：**panic**（主动或自动）表达「这条线程已崩」；**match + downcast** 在主线程**统一收口** — 自定义 `StrategyError` 走业务分支，标准库 panic 走 `&str`/`String` 分支。可恢复错误（重试即可）仍用 **`Result` + channel**，不要滥用 panic → [§与 anyhow](#与-anyhow把-join-的-err-并进-result---主流程)。

### 2.1 panic 捕获与 `downcast_ref`

#### 完整可运行示例

```rust
use std::thread;

fn main() {
    let h = thread::spawn(|| {
        panic!("线程崩溃信息");
    });

    match h.join() {
        Ok(v) => println!("线程正常返回: {v:?}"),
        Err(e) => {
            if let Some(s) = e.downcast_ref::<&str>() {
                // s: &&str — downcast_ref::<&str> 返回 Option<&&str>
                eprintln!("子线程 panic(&str): {s}");
            } else if let Some(msg) = e.downcast_ref::<String>() {
                eprintln!("子线程 panic(String): {msg}");
            } else {
                eprintln!("子线程 panic 未知类型");
            }
        }
    }
}
```

仓库对应：[1.1-threads-in-rust-join-panic-demo.rs](./code/1.1-threads-in-rust-join-panic-demo.rs)

#### `join()` 返回值再强调

```rust
Result<T, Box<dyn Any + Send + 'static>>
```

| 分支 | 含义 |
|------|------|
| **`Ok(T)`** | 子线程**正常**结束，返回闭包返回值 |
| **`Err(Box<dyn Any>)`** | 子线程 **panic**，盒子里是 panic **抛出的值**（类型擦除） |

`downcast_ref` 的语法、`<T>` 类型参数、以及 **`s: &&str`** 为何能直接打印 → [§2.2](#22-downcast_refstr-类型与层级精读)。

#### 边界：非字符串 panic

```rust
thread::spawn(|| std::panic::panic_any(999i32)); // 载荷 i32（`panic!` 宏只接受格式字符串）
```

此时 `&str`、`String` 都匹配失败 → 走兜底分支；可用 **`err.type_id()`** 调试（见 demo）。

| 写法 | 常见载荷 | 说明 |
|------|----------|------|
| `panic!("错误")` | `&'static str` | 字面量编译期进 `.rodata`，**堆上无字符串分配** |
| 标准库越界 / `unwrap` 等 | **`&'static str`** | 如 `"index out of bounds: the len is …"` 中固定片段常为 static |
| `panic!("{}", 123)` | `String` | 格式化在**运行时堆分配** |
| `panic_any(StrategyError { … })` | 自定义 struct（须 **`Send + 'static`**）→ [§Err 设计动机](#设计动机子线程-panic-载荷完全不可控) |
| `panic_any(999)` | `i32` 等任意 `Send + 'static` 类型 |

#### 工具函数封装（统一打印）

```rust
use std::any::Any;
use std::thread::JoinHandle;

fn join_and_print<T>(handle: JoinHandle<T>) {
    match handle.join() {
        Ok(val) => println!("线程返回: {:?}", val),
        Err(panic_err) => print_panic_msg(&panic_err),
    }
}

fn print_panic_msg(err: &Box<dyn Any + Send>) {
    // 顺序有意为之：见 [§为何先 &str 再 String](#为何-downcast-顺序先-str-再-string)
    if let Some(s) = err.downcast_ref::<&str>() {
        // s: &&str — 含 &'static str（标准库原生 panic、panic!("字面量")）
        eprintln!("panic(&str): {s}");
    } else if let Some(s) = err.downcast_ref::<String>() {
        // 运行时格式化的动态字符串
        eprintln!("panic(String): {s}");
    } else {
        eprintln!("panic 未知类型，type_id: {:?}", err.type_id());
    }
}
```

#### 为何 downcast 顺序：先 `&str`，再 `String`？

`&str` 与 `String` 是**两种不同类型**，不会互相抢分支；顺序体现的是**优先覆盖哪类 panic**：

| 匹配顺序 | 覆盖场景 | 载荷从哪来 |
|----------|----------|------------|
| **① `downcast_ref::<&str>()`** | **标准库运行时 panic**（越界、`unwrap` 失败…）、**`panic!("静态字面量")`** | 多为 **`&'static str`** — 消息在编译期写进程序，**不为这条字面量单独堆分配 String** |
| **② `downcast_ref::<String>()`** | **`panic!("{}", var)`** 等需运行时拼接的消息 | **`String` 在堆上分配** |
| **③ 兜底** | `panic_any(StrategyError)`、`panic_any(999)` 等 | 自定义 struct / 其它 `Send + 'static` 类型 |

```rust
// 运行时自动 panic — 载荷通常是 &'static str（示意）
let v = vec![1, 2, 3];
let _ = v[99]; // → join 后常可 downcast_ref::<&str>() 读到 "index out of bounds" 类消息

// 主动格式化 panic — 载荷是 String
panic!("code={}", 4001); // → downcast_ref::<String>()
```

**`print_panic_msg` 先 `&str` 再 `String`**：先兜住**官方原生 / 静态字面量** panic（自动 + 简单 `panic!`），再处理**用户主动抛的动态字符串**。业务向的 **`on_strategy_panic`** 则通常在字符串分支**之前**加 **`StrategyError`** 分支 — 自定义 struct 与 `&str`/`String` 不冲突，全链路顺序常为：

```text
StrategyError（自定义）→ &str（static / 标准库）→ String（动态）→ type_id 兜底
```

#### 关键知识点

1. **`join()`** 阻塞当前线程，等待子线程结束；
2. **panic 不会跨线程自动传播**到主线程，须通过 **`join()` 的 `Err`** 捕获；
3. **分工**：**`panic!` / `panic_any` = 抛**（不可恢复）；**`match` + `downcast_ref` = 接** — **主动 panic 与运行时自动 panic 汇合为同一 `Err(Box<dyn Any>)`**，匹配链须同时覆盖 `&str`/`String` 与自定义 struct → [§主动 vs 自动 panic](#主动-panic-vs-运行时自动-panic来源不同汇合点相同)；
4. **`Box<dyn Any>`** = **统一容器**；`downcast_ref` 还原 `StrategyError` / 字符串等；
5. **`downcast` 顺序**：通用打印 **`&str` → `String` → 兜底**（先 static / 标准库，再动态字符串）；业务 handler 再加 **`StrategyError`** 等自定义分支 → [§为何先 &str 再 String](#为何-downcast-顺序先-str-再-string)。

#### 可选：`resume_unwind` 重新抛出

```rust
Err(e) => {
    std::panic::resume_unwind(e); // 主线程也跟着 panic
}
```

适合需要 **panic 连锁传递**、不必在主线程里消化错误的场景。

### 2.2 `downcast_ref::<&str>()`：turbofish 与类型层级精读

先把 **turbofish、类型、层级** 拆清楚，再回头看 §2.1 里的 `e.downcast_ref::<&str>()`。

#### （1）`Err(e)` 里 `e` 的真实类型

`h.join()` 失败时：

```rust
e: Box<dyn Any + Send + 'static>
```

`e` 是**装箱的 trait 对象**，里面装的是：**子线程 panic 时扔出来的那个值**（已被类型擦除，运行时靠 `downcast_ref` 恢复）。

| 子线程写法 | panic 抛出的原始值 `V` |
|------------|------------------------|
| `panic!("abc")` | `&'static str` |
| `panic!("num: {}", 123)` | `String` |

#### （2）`downcast_ref::<T>()`：turbofish 语法与 `<T>` 含义

`e.downcast_ref::<&str>()` 里的 **`::<&str>`** 就是 Rust 的 **turbofish** — 给**泛型方法**显式指定类型参数：

```rust
// 方法调用（日常写法）
err.downcast_ref::<&str>()

// 等价完整路径（方法名 + turbofish 都在）
std::any::Any::downcast_ref::<&str>(&err)
//              ^^^^^^^^^^^^  ^^^^^^^^
//              方法名         目标类型 T = &str
```

| 片段 | 是什么 | 作用 |
|------|--------|------|
| **`downcast_ref`** | **`Any` trait 的方法名** | 不是类型参数的一部分；turbofish **只填泛型 `T`** |
| **`::<&str>`** | **turbofish** | 告诉编译器：**要把盒内 `dyn Any` 当成 `&str` 来取引用** |
| **返回值** | **`Option<&T>`** | 运行时做**类型检查**：内部值真是 `T` → `Some(&T)`；否则 **`None`**（安全，不 panic） |

**语义一句话**：`dyn Any` 里的真实类型编译期不可见 → 你**显式声明**想要的 `T` → 标准库在**运行时**比对类型 id；匹配才借出 **`&T`**。

```rust
err.downcast_ref::<T>()
```

| 部分 | 含义 |
|------|------|
| **`<T>`** | **显式类型参数**（turbofish）：「我想把内部值当作 **`T`** 来取引用」 |
| **`downcast_ref`** | 返回 **`Option<&T>`** — 见上表 |

常见 turbofish 目标（与 [§为何先 &str 再 String](#为何-downcast-顺序先-str-再-string) 对应）：

```rust
payload.downcast_ref::<&str>()           // T = &str  → Option<&&str>
payload.downcast_ref::<String>()         // T = String → Option<&String>
payload.downcast_ref::<StrategyError>()  // T = 自定义 struct → Option<&StrategyError>
```

> **重点**：`downcast_ref` **本身就返回引用**（`&T`），不是把内部值「变成」引用；它是在已装箱的值上再借一层。

#### （3）拆解 `downcast_ref::<&str>`：目标类型 `T = &str`

以 `panic!("hello")` 为例，**一层层剥开**：

1. 子线程 `panic!("hello")` — 字面量 `"hello"` 类型：**`&'static str`**
2. 这个 **`&str`** 被装进 **`Box<dyn Any>`** → 就是 `join()` 拿到的 **`e`**
3. 调用 **`e.downcast_ref::<&str>()`**：
   - 检查：内部值是不是 **`&str`**？**是**
   - 返回：**`Some( &(内部的 &str) )`**

完整类型链：

```
e: Box<dyn Any>
  ↓ 内部真实值
V = &'static str
  ↓ downcast_ref::<&str>()
Option< &V > = Option< &&str >
```

因此：

```rust
if let Some(s) = e.downcast_ref::<&str>() {
    // s 的类型是 &&str —— 「字符串引用的引用」
}
```

#### （4）为什么 `&&str` 能直接打印？

Rust 有 **自动解引用（Deref coercion）**：`&&str` 在调用方法、打印、传参时会逐层解引用到 `&str` / `str`，所以 **`println!("{s}")` 完全正常**，不必写 `(*s)`。

极简对照：

```rust
fn main() {
    let s: &str = "hello";   // &str
    let ss: &&str = &s;      // &&str — 引用的引用

    println!("{}", ss);      // 自动解引用 → hello
}
```

对应 panic 捕获：

- `downcast_ref::<&str>()` → `Some(&&str)`
- 绑定到 `s` → **`s: &&str`**
- 打印 / 比较 / 传参 → 自动解引用，日常**感知不到**双层引用

#### （5）为什么不能写 `downcast_ref::<str>`？

```rust
// 错误写法
e.downcast_ref::<str>()
```

1. **`str` 是 DST（不定长类型）**，不能作为 `downcast` 的目标类型参数；
2. panic 抛出的本来就是 **`&str`**，不是裸 **`str`**。

**必须写 `<&str>`**。

#### （6）对比：`panic` 抛出 `String`

```rust
panic!("num: {}", 123); // 生成并抛出 String
```

- 内部值 **`V = String`**
- 应写：**`downcast_ref::<String>()`**
- 返回 **`Option<&String>`**（单层引用，因为 `T = String`，`&T = &String`）

两场景对照：

```rust
// 场景 1：panic!("字面量") → 内部是 &str
if let Some(s) = e.downcast_ref::<&str>() {
    // s: &&str
}

// 场景 2：panic!(格式化) → 内部是 String
else if let Some(msg) = e.downcast_ref::<String>() {
    // msg: &String
}
```

#### （7）一句话总结

1. `panic!("xxx")` 抛出的原始值是 **`&str`**；
2. **`downcast_ref::<&str>`** = 把内部值当作 **`&str`** 来取引用；
3. 绑定得到的 **`s` 类型是 `&&str`**（引用的引用）；
4. Rust 自动解引用，**正常打印 / 使用即可**，不必手动解一层。

### `JoinHandle<T>` 的 `T` vs `.join()` 的 `Result`：两个独立层次

很多人把句柄上的 **`T`** 和 **`.join()` 外层的 `Result`** 混成「业务里的 `Result<T, E>`」——其实是**两层**：

| 层 | 类型 | 含义 |
|----|------|------|
| **① 句柄泛型** | `JoinHandle<T>` | **`T` = 子线程闭包正常跑完时的返回值类型**（业务数据），与 `Result` 无关 |
| **② join 返回值** | `Result<T, Box<dyn Any + Send + 'static>>` | **外层 `Result`** 只表示「线程正常结束 vs panic」；`Ok` 里装的才是上面的 **`T`** |

#### 普通 `thread::spawn`

```rust
use std::thread;

// 闭包返回 i32 → JoinHandle<i32>
let h = thread::spawn(|| 1 + 2);
let res: Result<i32, _> = h.join();
println!("{}", res.unwrap()); // 3
```

闭包**无返回值**（只有语句）→ **`T = ()`** → `JoinHandle<()>`：

```rust
let h = thread::spawn(|| {
    println!("hello");
});
let _: Result<(), _> = h.join();
```

#### `ScopedJoinHandle<T>` 完全同理

```rust
thread::scope(|s| {
    let h = s.spawn(|| 99); // ScopedJoinHandle<'_, i32>
    let res: Result<i32, _> = h.join();
    assert_eq!(res.unwrap(), 99);
});
```

#### 直观对照：`Ok(T)` 里是业务返回值

```rust
let h = thread::spawn(|| -> String {
    "ok".to_string()
});

match h.join() {
    Ok(s) => println!("线程正常结束，返回：{s}"), // s: T = String
    Err(_panic_data) => println!("线程 panic"),
}
```

#### panic 时：`T` 不会出现

```rust
let h = thread::spawn(|| {
    panic!("出错了！");
});

let ret = h.join();
assert!(ret.is_err()); // Err(...)：拿不到 T
```

**背诵**：

1. **`JoinHandle<T>` / `ScopedJoinHandle<'_, T>` 的 `T`** = 闭包**正常**返回的类型；
2. **`.join()` 的 `Result`** = join **操作**的成功/失败（panic → `Err`），**不是**你业务逻辑里的 `Result<T, E>`；
3. 需要 **`T`** → 必须在 join 时 **`Ok` 分支 / `.unwrap()`** 取出来；panic 路径只有 `Err` 载荷。

## 五、两个易混点

1. **单词是 Join，不是 Enjoy**：口语 / 听写常把 **`Join`**（等待线程「汇合」）误听成 Enjoy。Rust 里对应 **`JoinHandle`**（线程句柄）与 **`.join()`**（阻塞等待子线程）；scope 内则是 **`ScopedJoinHandle`**，同样有 **`.join()`**，另见 [1.2.3 scope 自动 join](../1.2-scoped-threads/1.2.3-join-auto-join.md)。
2. **`JoinHandle` 不只有 `join()`**：还有 **`.thread()`**；**同步等待的核心 API 仍是 `join`**。
3. **句柄 ≠ 线程本体**：用来 **等待 + 取返回值 + 回收 OS 资源**。
4. **把 `JoinHandle<T>` 的 `T` 当成 `Result` 的业务泛型**：`T` 是闭包正常返回值；**`.join()` 外层 `Result`** 只区分「正常结束 / panic」（见 [§二 `T` vs `Result`](#joinhandlet-的-t-vs-join-的-result两个独立层次)）。

## 六、底层原理：OS → 标准库 → `join()`

#### （1）OS：`pthread_join`

| 概念 | 说明 |
|------|------|
| 未 join 的已结束线程（Unix） | 可能暂存为 **僵尸线程**，直到 `join` 回收 |
| **`pthread_join` / `WaitForSingleObject`** | 调用线程**内核阻塞**，直到目标线程结束 |

**OS 一句话**：`join` = 内核「等另一线程结束」原语；Rust 在其上封装 panic 传播与 `T` 回传。scope 自动 join 最终也走此路径（[1.2.3 §3](../1.2-scoped-threads/1.2.3-join-auto-join.md#3-分层拆解自动-join-做了什么是不是封装系统调用)）。

#### （2）教学心智模型 vs 标准库实装

`Arc` + `Condvar` 伪代码仅帮助理解阻塞/唤醒；**Unix 上 `join()` 通常直接 `pthread_join`**，阻塞在**内核**。读源码：`library/std/src/thread/`。

#### （3）`join(self)` 语义

1. 阻塞直到子线程结束；2. 取 `Result<T, _>`；3. **消耗 handle**，只能一次。

---

## 七、为什么对外几乎只暴露 `join`？

| 原因 | 说明 |
|------|------|
| **只能 join 一次** | `join(self)` 消耗 handle；重复 join 在 OS 上是 UB |
| **职责单一** | `spawn` 启动；`JoinHandle` 等待 + 回收 |
| **不提供 kill** | 强杀破坏内存安全 / 锁 |

---

## 八、drop 而不 `join`

> **Drop 机制与 detach**：[1.1.7 §主动放掉 handle 与 Drop](./1.1.7-thread-lifecycle-four-states.md#主动放掉-handle与-drop谁主动谁被动) · [15.3 Drop](../../../../00-Book/15-smart-pointers/15.3-使用Drop运行清理代码.md)

| 行为 | 后果 |
|------|------|
| **`drop(handle)` / handle 出作用域、且未 `join`** | 触发 **`impl Drop for JoinHandle`** → 线程 **detach**（分离态） |
| **`.join(self)`** | **消耗** handle，走 join 路径，**不**走 detach |
| **进程退出** | OS 强杀未完成线程 |

```rust
let h = thread::spawn(|| { /* … */ });
drop(h); // 等价于让 h 立刻出作用域 → Drop → detach

{
    let h = thread::spawn(|| { /* … */ });
} // 块结束同样 Drop(h)
```

**不能**写 `h.drop()`；分离逻辑在标准库 `JoinHandle` 的 **`Drop` 实现**里，你只控制 handle **何时**离开作用域。

## 九、极简总结

1. **`JoinHandle<T>` 的 `T`** = 子线程闭包正常返回的类型；**`.join()`** → `Result<T, panic载荷>`；panic 捕获见 [§2.1](#21-panic-捕获与-downcast_ref)，**`&&str` 层级**见 [§2.2](#22-downcast_refstr-类型与层级精读)。
2. 内核 `pthread_join`；句柄非线程本体；`join()` 阻塞 + 消耗 handle。
3. Condvar 模型仅助理解；scope 自动 join 同路径见 [1.2.3 §3](../1.2-scoped-threads/1.2.3-join-auto-join.md#3-分层拆解自动-join-做了什么是不是封装系统调用)。
