## 一句话

Rust **标准库没有**与操作系统内核线程状态**一一对应**的公开枚举；从 **`spawn`、`JoinHandle` 所有权、`join`、`main` 退出** 出发，可以把 `std::thread` 的生命周期归纳为 **四种核心状态**。

---

## 四种状态对照

| 状态 | 何时进入 | 含义 | 典型操作 |
|------|----------|------|----------|
| **① 可连接态（Joinable）** | `thread::spawn` 返回后，**仍持有** `JoinHandle` | 主线程（或任意 owner）仍拥有 **join 权** | `handle.join()` 阻塞等待 → `Ok(T)` 或 panic 的 `Err` |
| **② 分离态（Detached）** | **未** `join` 就 **`drop` 了 `JoinHandle`** | 子线程与 caller **脱钩**；**放权** — 不等待、不索取返回值，后台自管生命周期 | fire-and-forget 辅助任务；共享数据须 `Arc`/通道等同步 |
| **③ 终止态（Terminated）** | 子线程闭包**正常返回**或 **panic** 之后 | OS 线程已结束；若 handle 仍在 **①**，`join` 一次即回收 | `Ok(T)` / `Err(Box<dyn Any + …>)`；`join` **消耗** handle，不能二次 join |
| **④ 主线程退出强制终止** | **`main`（或进程入口）跑完**，仍有子线程在 **①** 且未完成 | 整个进程退出 → **未 join 的非 detached 线程被直接终止** | 与 C/Java 等「等子线程」模型不同；**必须** `join`、scope 或 detached 才能可控收尾 |

```text
spawn ──► ① Joinable（持有 JoinHandle）
              │
              ├─ join() ──► ③ Terminated（取 Result，handle 消耗）
              │
              └─ drop(handle) 未 join ──► ② Detached（无法再 join）

main 结束 + 仍有 ① 中未完成的线程 ──► ④ 强制终止（进程级）
```

## 分离态的心智模型：「放权」

**核心**：把线程及其 **`move` 进去的数据所有权、执行逻辑** 一并交出去 — 主线程 **既不 `join` 等待，也不通过 `JoinHandle` 索取返回值**，只要求它在后台把该做的事做完。

```text
主线程                          分离态子线程
  │                                  │
  ├─ spawn(move || { ... }) ────────►│ 拥有闭包捕获的数据
  ├─ drop(JoinHandle)  （放权）      │ 自管 loop / I/O / 收尾
  └─ 继续核心逻辑（零阻塞）           └─ 写完 / 发完 / 上报完 → 线程结束，OS 回收
```

与 **① 可连接** 对比：可连接 = 主线程仍握 **join 权**（可等、可取 `Result`）；分离 = **主动放弃** 这层控制权，代码路径更短、主逻辑更清爽。

### 「主动放掉 handle」与 `Drop`：谁主动、谁被动

说的是 **[`Drop` trait](../../../../00-Book/15-smart-pointers/15.3-使用Drop运行清理代码.md)**：`drop` 方法**不能**像普通函数那样手动调用（不能写 `handle.drop()`）；只能在变量**离开作用域**时，由所有权系统**自动**触发。

| 层次 | 谁在做 | 做什么 |
|------|--------|--------|
| **你（caller）** | **主动控制 `JoinHandle` 的生命周期** | 决定 handle **何时**离开作用域 |
| **编译器 / 运行时** | **被动调用** `JoinHandle` 的 `Drop` 实现 | 若尚未 `join`，在 drop 里把 OS 线程设为 **detach** |

常见写法都是在**绕开「不能主动调 Drop」**的前提下，**提前结束 handle 的生命周期**：

```rust
// 写法 A：std::mem::drop — 取得所有权，函数末尾自动 Drop（见 15.3 §四）
let handle = thread::spawn(|| { /* … */ });
drop(handle);

// 写法 B：单独块 — 块结束 handle 出作用域，同样触发 Drop
{
    let handle = thread::spawn(|| { /* … */ });
} // ← 此处 Drop(handle)，未 join 则 detach

// 写法 C：spawn 后不再绑定变量（少见，但等价于立刻 drop 临时值）
thread::spawn(|| { /* … */ }); // JoinHandle 临时值语句末尾即 Drop
```

```text
你：drop(handle) / 块结束     →  handle 离开作用域（主动）
                                      ↓
Rust：调用 impl Drop for JoinHandle  →  内部 detach 逻辑（被动）
                                      ↓
OS：子线程与 join 权脱钩            →  ② 分离态
```

**对照**：

- **`.join(self)`** — **消耗** handle，走 join 路径取 `Result`，**不会**在 Drop 里 detach（handle 已被 move 进 `join`）。
- **`drop` / 出作用域** — 未 join 就 Drop → **detach**；这正是「放权」在 API 层的落点。
- 与 [15.3 Drop](../../../../00-Book/15-smart-pointers/15.3-使用Drop运行清理代码.md) 一致：`std::mem::drop(x)` 不是「调用 x 的 drop 方法」，而是 **让 `x` 立刻出作用域**，从而合法触发 `Drop`。

精读 `JoinHandle` 与 OS detach：[1.1.2.5 §八](./1.1.2.5-join-os-drop-pitfalls.md#八drop-而不-join)。

### 典型辅助任务（都适合分离态）

| 场景 | 子线程自管什么 | 主线程为何不管 |
|------|----------------|----------------|
| **行情落时序库** | 拉完行情 → 写本地 TSDB → 连接/缓冲自行释放 | 无返回值；落库耗时不应阻塞下单 |
| **日志异步落盘** | 从 channel 取日志行 → 批量 `write` / `fsync` | 热路径只 `send`，不等磁盘 |
| **定时数据同步** | 按 cron 拉远端快照 → 写缓存 / DB | 同步失败重试在后台闭环 |
| **心跳 / 探活上报** | 固定间隔 POST 健康检查 | 上报与业务请求解耦 |
| **定时风控拉行情** | 见 [§量化分工](#场景量化交易里何时用分离态) | 主线程读快照即可，不 join |

**「写库自闭环」模式**（比共享内存快照更「放权」）：

```rust
let db_url = config.db_url.clone(); // 或 Arc<ConnectionPool>

let handle = thread::spawn(move || loop {
    let bars = fetch_market_bars();           // 子线程自己拉
    write_bars_to_tsdb(&db_url, &bars);       // 子线程自己写、自己处理错误/重试
    thread::sleep(Duration::from_secs(10));
});

drop(handle); // 分离：主线程不维护该线程生命周期，也不拿 JoinHandle 返回值
// 主线程只跑交易指令…
```

子线程 **`move` 走** 连接串 / 池 / 配置副本后，**落库全程在主线程视野外** — 既避免阻塞，也省掉「谁 join、谁收 `Result`」的样板代码。若要与主线程共享**只读**风控结论，再用 `Arc<Mutex<Snapshot>>`；若结果只进 DB、主线程从不读内存，则连共享快照都可省略。

### 何时仍用 join / scope（不放权）

| 需要 | 选 |
|------|-----|
| 子线程 **`Result` / 错误** 必须回到 caller | **① + `join()`** |
| 批任务结束才能继续（如 map-reduce 汇总） | **① + `join()`** 或 [1.2 scope](../1.2-scoped-threads/1.2-scoped-threads.md) |
| 进程退出前要 **确知** 后台写完 | 关停阶段 **`AtomicBool` 停 loop + 限时 `join`**（退出路径上短暂收回 join 权） |

## 场景：量化交易里何时用分离态

**分工**

| 线程 | 职责 | 状态选择 |
|------|------|----------|
| **主线程** | 接收用户指令、快速下单（延迟敏感） | 不能被后台 `join` 阻塞 |
| **后台线程** | 每 10s 拉全市场行情 → 风控校验（辅助、无返回值） | **分离态** 更合适 |

**为什么不用 join？** 风控是 **fire-and-forget**：你不关心它的 `()` 返回值，也不希望主线程每 10s 被卡一次；只要它在后台 **循环：拉行情 → 写共享快照 → sleep** 即可。

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

let risk = Arc::new(Mutex::new(RiskSnapshot::default()));

let risk_bg = Arc::clone(&risk);
let handle = thread::spawn(move || {
    loop {
        update_risk_from_market(&mut *risk_bg.lock().unwrap());
        thread::sleep(Duration::from_secs(10));
    }
});

// 进入分离态：主线程立刻继续处理下单
drop(handle);

// 主线程读 risk 快照决定是否放行订单…
```

**分离态的好处（本场景）**

1. **不阻塞** — 下单路径零等待 `join`。
2. **不要返回值** — 风控结果写入 `Arc<Mutex<RiskSnapshot>>`，不必通过 `JoinHandle` 传。
3. **生命周期跟进程** — 后台一直跑到 **进程结束** 或 **自己 panic 退出**；主线程专注核心交易逻辑。

**必须补上的工程纪律**

| 点 | 说明 |
|----|------|
| **共享状态** | `spawn` 要求 `Send + 'static` → 用 **`Arc<Mutex<_>>`** / 通道，不能借主线程栈上变量 |
| **后台 panic** | 分离后 **不会** 自动传到主线程；生产应打日志 / 告警，必要时用 **`AtomicBool` 熔断** |
| **优雅退出** | 真实系统会在关停时用 **`AtomicBool` + channel** 通知后台跳出 `loop`，再 `join` 或限时等待；教学 demo 简化为「跟进程一起走」 |
| **async 对照** | 高并发 I/O 场景常改 **`tokio::spawn` + 定时器**，语义类似「不阻塞主路径的辅助任务」→ [async §1.3](../../../02-async_tokio/ch01_async_intro/1.3-what-are-threads/1.3-what-are-threads.md) |

完整可运行 stub：[1.1-threads-in-rust-detached-risk-poll-demo.rs](./code/1.1-threads-in-rust-detached-risk-poll-demo.rs)（demo 里 sleep **2s** 便于观察，注释标明生产常用 **10s**）。

## 和其他语言的差异（④ 重点）

- **Rust**：`main` 返回 → **进程退出** → 仍在跑的 **joinable** 子线程**不会**被无限等待，而是随进程一起被干掉。
- **常见误区**：以为 `spawn` 后子线程一定会跑完；若既未 `join` 也未 detach，且 `main` 很快结束，长任务可能**一行都来不及打印**（见 demo [join-demo.rs](./code/1.1-threads-in-rust-join-demo.rs) 对照 [1.1.2.5 §四](./1.1.2.5-join-os-drop-pitfalls.md#四不用-join-会怎样)）。
- **可控收尾**：`join()` · [1.2 scope 自动 join](../1.2-scoped-threads/1.2.3-join-auto-join.md) · 或明确 detach 并接受无法取返回值。

---

## 速记

1. **有 handle = 可 join**；**drop handle = detach（放权）**。  
2. **线程结束 ≠ 已 join**；要 `Ok(T)` 必须在 **①** 时调用 `join`。  
3. **`main` 结束 = 进程结束**；别指望未 join 的子线程还能慢慢收尾。  
4. **辅助任务、无返回值、勿阻塞主路径**（落库、日志、同步、心跳、定时风控）→ 分离态；**要结果或批任务汇总** → join / scope。
