## 1.1.4 FnOnce / Fn / FnMut：定义规则 vs move 捕获

> 书 §1.1（续）· `spawn` 签名见 [1.1.3 Send/'static](./1.1.3-send-static-spawn.md) · [1.1.2 JoinHandle](./1.1.2-join-handle.md)

---

## 先纠正：两层逻辑必须分开

| 层 | 问的是什么 |
|----|------------|
| **① Trait 的定义规则** | `FnOnce` / `Fn` / `FnMut **本身**规定什么？ |
| **② 编译器对闭包的实现** | **哪种捕获方式**会让闭包**只**实现 `FnOnce`？ |

**`FnOnce` ≠ 一定 `move` 捕获**；**`move` 捕获闭包** → 通常**只**实现 `FnOnce`（场景因果，非 trait 字面定义）。

---

## 一、FnOnce 的核心语义（只看 trait）

**一句话**：调用会**消耗闭包自身**（`self`），故**最多调用一次**——与「有没有捕获外部变量」**无直接关系**。

简化定义：

```rust
trait FnOnce<Args> {
    type Output;
    fn call_once(self, args: Args) -> Self::Output; // 接收 self，不是 &self
}

trait FnMut<Args> {
    fn call_mut(&mut self, args: Args) -> Self::Output;
}

trait Fn<Args> {
    fn call(&self, args: Args) -> Self::Output;
}
```

| Trait | 调用方式 | 可调用次数（典型） |
|-------|----------|-------------------|
| **`FnOnce`** | `call_once(self)` — **拿走 F** | **一次**（F 被消耗） |
| **`FnMut`** | `call_mut(&mut self)` | 多次（可变借用自身） |
| **`Fn`** | `call(&self)` | 多次（共享借用自身） |

**层级**：实现了 **`Fn` → 自动有 `FnMut` + `FnOnce`**；实现了 **`FnMut` → 自动有 `FnOnce`**。  
约束 **`F: FnOnce()`** 最宽，**`F: Fn()`** 最严。

> `thread::spawn` 要求 **`FnOnce`**：子线程**拥有**闭包并执行**一次**（见 [1.1.5 §六](./1.1.5-where-fnonce-syntax.md#六trait-尖括号类型参数-vs-生命周期参数)：`FnOnce<()>` 的 `<()>` 是入参）。与 **`Send + 'static`** 是不同层（见 [1.1.3](./1.1.3-send-static-spawn.md)）。**`FnOnce() -> T` 写在 API 的 where 里**；调用 `spawn` 时闭包**不用**写箭头 → [1.1.5 §何时必须写箭头](./1.1.5-where-fnonce-syntax.md#何时必须写-fnonce---t何时可省略)。

---

## 二、为何常把 FnOnce 和 move 绑在一起？（场景因果）

1. **`move` 闭包拿走外部变量的所有权**（如 `move || s`，`s: String`）：
   - 第一次调用会把捕获的 `s` **随闭包一起 move 出**；
   - 编译器通常**只**为这类闭包实现 **`FnOnce`**，**不会**再实现 `FnMut` / `Fn`（否则第二次调用会重复 move，不合法）。

```rust
let s = String::from("hi");
let c = move || s; // 捕获 s 的所有权
// c 只实现 FnOnce；不能 c(); c();
```

2. **这是「捕获方式 → 实现哪些 trait」的推导**，不是 `FnOnce` trait 文档里写了「必须 move」。

---

## 三、反例：满足 FnOnce，但不涉及外部所有权

### 场景 1：无捕获闭包 — 同时实现 Fn / FnMut / FnOnce

```rust
let c = || 123;
c();
c();
c(); // 任意多次；仍满足 FnOnce 约束（spawn 可用）
```

无外部捕获 → 无「捕获变量被 move 消耗」问题 → 三个 trait 都能实现。

### 场景 2：手动实现 `FnOnce` 的类型（逻辑对，stable 不行）

**结论先讲透**：

| 说法 | 对错 |
|------|------|
| 手写 `impl FnOnce<()> for MyFunc` **语义** | ✅ 与 `\|\| 999` 一样：无参、返回 `i32`、**只能调用一次** |
| 在 **stable** Rust 里这么写 | ❌ **编译不过** |
| 原因 | `call_once` 在标准库中是 **`extern "rust-call"`**，直接 `impl Fn/FnMut/FnOnce` 须 **nightly** 特性 |

#### 标准库 `FnOnce`（简化）

```rust
pub trait FnOnce<Args> {
    type Output;
    fn call_once(self, args: Args) -> Self::Output;
}
```

| 部分 | 含义 |
|------|------|
| **`Args`** | 参数元组；无参 → **`()`** |
| **`call_once(self, …)`** | **拿走 `self` 所有权** → 最多调用一次 |
| **`Output`** | 返回类型 |

因此 **`FnOnce() -> i32`** = **`FnOnce<(), Output = i32>`**（语法糖见 [1.1.5](./1.1.5-where-fnonce-syntax.md)）。

#### 你想表达的代码（nightly 语义）

```rust
struct MyFunc;

impl FnOnce<()> for MyFunc {
    type Output = i32;
    fn call_once(self, _: ()) -> i32 {
        999
    }
}

fn run<F: FnOnce() -> i32>(f: F) {
    f(); // 等价 f.call_once(())
}

run(MyFunc); // 消耗 MyFunc 自身，返回 999
```

**语义完全正确**：`MyFunc` 是函数对象；`call_once(self)` 只够调用一次。

#### 为何 stable 编译不过？

标准库中 `call_once` 实为：

```rust
extern "rust-call" fn call_once(self, args: Args) -> Self::Output;
```

自定义类型直接 impl 这三个 trait 需要：

```rust
#![feature(fn_traits)]
#![feature(unboxed_closures)]
```

**stable 不允许**手写 `impl FnOnce` / `FnMut` / `Fn`——**只有闭包**（及少数内建形式如 **`fn()` 函数指针**）由编译器自动生成。

| 环境 | 手写 `impl FnOnce for MyFunc` |
|------|------------------------------|
| **stable** | ❌ |
| **nightly** + `fn_traits` | ✅ 可跑通 |

#### stable 等价写法

**写法 A：闭包（最常用）**

```rust
fn run<F: FnOnce() -> i32>(f: F) -> i32 {
    f()
}

run(|| 999); // 编译器自动 impl FnOnce() -> i32
```

**写法 B：`fn()` 函数指针（天生满足 `FnOnce`）**

```rust
fn returns_999() -> i32 {
    999
}
run(returns_999); // fn() -> i32 自动 impl Fn/FnMut/FnOnce
```

**写法 C：结构体 + 闭包包一层（模拟函数对象）**

```rust
struct MyFunc;
impl MyFunc {
    fn invoke(self) -> i32 {
        999
    }
}

run(move || MyFunc.invoke()); // stable：用闭包适配，不必手写 impl FnOnce
```

仓库 stable demo：[1.1-threads-in-rust-fnonce-demo.rs](./code/1.1-threads-in-rust-fnonce-demo.rs)

#### 与 `'static` 的关系（两个独立约束）

| 约束 | 问什么 |
|------|--------|
| **`FnOnce()`** | 能不能调用、**调用时是否消耗 `self`** |
| **`'static`** | 类型里有没有**非 `'static` 的引用**（见 [1.1.3 §3.4](./1.1.3-send-static-spawn.md#34-tstatic-到底是什么)） |

- **`FnOnce() -> i32` 本身不要求 `'static`**；
- **`thread::spawn`** 要 **`F: FnOnce() + Send + 'static`** — 两个条件**都要**；
- 手写的 **`MyFunc`**（无字段、无引用）→ **`MyFunc: 'static`**；nightly 下 `thread::spawn(MyFunc)` 在 **`Send`** 满足时合法。

**一句话**：**`FnOnce` 描述调用方式；`'static` 描述生命周期**——互不替代。

**只实现 `FnOnce`**（nightly 语境）是因为 **`call_once(self)` 消耗 `MyFunc` 实例**，不是因为动了什么外部 `String`。

## 四、逻辑链（背诵）

1. **`FnOnce` 本质**：`call_once(self)` → **F 自身**只够用一次。  
2. **`move` 捕获副作用**：拿走外部 **owned** 变量 → 编译器往往**只给 `FnOnce`**。  
3. **不能划等号**：  
   - 满足 `FnOnce` 的类型 ≠ 一定 `move` 了外部变量；  
   - `move` 闭包 → 几乎总是 `FnOnce`，且还要 **`Send + 'static`** 才能 `spawn`。

---

## 五、与 `thread::spawn` 的关系

```rust
pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,
```

| 约束 | 来源 |
|------|------|
| **`FnOnce()`** | 闭包交给子线程**执行一次**，入口消耗 **F 自身** |
| **`Send + 'static`** | 跨线程 + 不借短生命周期（见 [1.1.3](./1.1.3-send-static-spawn.md)） |

**常见组合**：`thread::spawn(move || { … })` — 同时满足 **FnOnce + Send + 'static**（三者回答不同问题）。`FnOnce() -> T` 语法见 [1.1.5](./1.1.5-where-fnonce-syntax.md)。

## 六、对照表

| 问题 | 答案 |
|------|------|
| FnOnce 定义里有没有写「必须 move 捕获」？ | **没有**；只写 `call_once(self)` |
| 能手写 `impl FnOnce for MyFunc` 吗？ | **stable ❌**（须 nightly `fn_traits`）；日常用 **闭包** 或 **`fn()` 指针** |
| move 闭包为何只能 FnOnce？ | 捕获的 owned 变量**随第一次调用被消耗** |
| `\|\| 123` 能 spawn 吗？ | 能（若满足 Send+'static）；spawn 仍只执行**一次** |
| spawn 为何用 FnOnce 不用 Fn？ | 线程入口**拿走**闭包所有权，只跑一遍 |
| `FnOnce` 和 `'static` 是一回事吗？ | **否**；前者调用语义，后者生命周期（[1.1.3 §3.4](./1.1.3-send-static-spawn.md#34-tstatic-到底是什么)） |
| 借局部变量能 spawn 吗？ | 不能（缺 'static）；用 [1.2 scope](../1.2-scoped-threads/1.2-scoped-threads.md) 的 `s.spawn` |

## 七、一句话区分

- **规则（FnOnce）**：只能调用一次，因为 **`call_once(self)` 消耗闭包自身**。  
- **场景（move 闭包）**：因**拿走外部变量所有权**，编译器**只**给 `FnOnce`。  
- **二者不等价**：实现 `FnOnce` 的类型，**不一定**涉及外部变量所有权。  
- **stable 实践**：用 **`|| …`** 或 **`fn()`**，不要手写 `impl FnOnce`（nightly 除外）。
