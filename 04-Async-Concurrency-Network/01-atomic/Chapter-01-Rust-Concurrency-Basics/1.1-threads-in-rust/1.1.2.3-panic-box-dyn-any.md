## 1.1.2.3 `Box<dyn Any>`：panic 载荷与 `panic_any` / `anyhow`

### Err 类型三层拆解：`Box<dyn Any + Send + 'static>`

子线程 **panic** 时，`join()` 拿不到 `T`，而是 **`Err(…)`**。

#### 设计动机：子线程 panic 载荷**完全不可控**

标准库在定义 `join` 时**无法预知**你在子线程里会抛什么 — 可能是 `&str`、`String`，也可能是自定义的 **`StrategyError`**（策略名、错误码、数据源标识）。若写成 `Result<T, String>` 或 `Result<T, StrategyError>`，就会**漏掉**其他 panic 形态，API 也无法扩展。

因此采用 **「先统一容器，再按需还原类型」** — 与量化里兼容多数据源错误的设计同构：

```text
子线程 panic 抛出 V（V 可能是 &str / String / StrategyError / …）
        ↓ Box 堆分配（V 大小编译期未知）
        ↓ dyn Any 类型擦除
join 侧固定拿到：Box<dyn Any + Send + 'static>
        ↓ downcast_ref::<T>()
按需还原：&str · String · StrategyError · …
```

| 层 | 写法 | 解决什么问题 |
|----|------|--------------|
| ① **`dyn Any`** | `std::any::Any` | **类型擦除** — 异构 panic 载荷装进**同一** trait 对象 |
| ② **`+ Send`** | trait 约束 | 载荷从**子线程**移回调 `join` 的线程 |
| ③ **`Box<…>`** | 堆指针 | `dyn Any` 是 **DST**，大小运行时才知道，**不能裸放栈上** |

#### 量化示例：`StrategyError` 与字符串错误同一套 `match`

```rust
use std::any::Any;
use std::panic;
use std::thread;

#[derive(Debug)]
struct StrategyError {
    strategy: String,
    code: u32,
    source: &'static str, // 如 "ctp" / "ib" / "sim"
}

fn handle_panic_payload(err: &Box<dyn Any + Send + 'static>) {
    if let Some(se) = err.downcast_ref::<StrategyError>() {
        eprintln!(
            "策略 {} 失败 [{}] code={}",
            se.strategy, se.source, se.code
        );
    } else if let Some(s) = err.downcast_ref::<&str>() {
        eprintln!("panic 消息: {s}");
    } else if let Some(msg) = err.downcast_ref::<String>() {
        eprintln!("panic 消息: {msg}");
    } else {
        eprintln!("未知 panic 类型: {:?}", err.type_id());
    }
}

fn main() {
    let h = thread::spawn(|| {
        panic::panic_any(StrategyError {
            strategy: "mean_revert_v2".into(),
            code: 1001,
            source: "ctp",
        });
    });

    if let Err(e) = h.join() {
        handle_panic_payload(&e); // 一条路径：先统一 Box，再 downcast 分支
    }
}
```

**要点**：

1. **不必**为 `&str` / `String` / `StrategyError` 写三套 `join` 签名 — `Err` 类型固定，分支在 **`downcast_ref`**。
2. 自定义 struct 须 **`Send + 'static`**（与 `panic_any` 要求一致），才能进 `Box<dyn Any + Send + 'static>`。
3. 生产更推荐 **`Result` 正常返回错误**；`panic_any` + downcast 适合**不可恢复**或遗留边界；日常 `panic!` 仍走 `&str`/`String` 分支即可。

#### `panic!` 与 `panic_any`：互补，不是替代

| | **`panic!` 宏** | **`std::panic::panic_any(value)`** |
|---|----------------|-------------------------------------|
| **载荷** | 字符串或可格式化成字符串的内容 | **任意** `Send + 'static` 类型 |
| **典型载荷** | `&'static str`、`String` | `StrategyError`、`i32`、自定义 struct … |
| **作用** | 快速抛可读消息 | **突破字符串限制**，把带上下文的自定义错误丢进 panic 流程 |
| **join 后还原** | `downcast_ref::<&str>` / `::<String>` | `downcast_ref::<StrategyError>` 等 |

```rust
use std::panic;

// panic! — 只能走字符串路线
panic!("CTP 连接超时");

// panic_any — 整颗 StrategyError 当载荷，不必 to_string
panic::panic_any(StrategyError {
    strategy: "arb_v3".into(),
    code: 2002,
    source: "ctp",
});
```

**记忆**：需要**策略名 / 错误码 / 数据源**等结构化上下文 → **`panic_any`**；只需一句人话 → **`panic!`** 足够。

#### 与 `anyhow`：把 join 的 `Err` 并进 `Result` + `?` 主流程

`join()` 的 **`Err(Box<dyn Any + Send + 'static>)`** 与生态 **`anyhow::Error`** 用的是**同一套类型擦除思路** — `anyhow::Error` 内部也借助 **`Box<dyn Any + Send + Sync + 'static>`** 等机制做 downcast / 上下文链（应用层错误汇总见 [Item 04 anyhow](../../../../01-ER/Chapter-01-Types/Item-04-idiomatic-error-types/README.md)）。

量化策略里若主流程已是 **`anyhow::Result`**（数据源错误、计算错误都用 **`?`**），子线程 panic 不必单独写一套分支 — **join 拿到 Box 后，转成 `anyhow::Error` 再 `?`**：

```rust
use anyhow::{Context, Result, anyhow};
use std::any::Any;
use std::panic;
use std::thread::{self, JoinHandle};

#[derive(Debug)]
struct StrategyError {
    strategy: String,
    code: u32,
    source: &'static str,
}

/// 把 join 的 panic 载荷统一成 anyhow，与策略里其它 ? 同型
fn join_or_anyhow<T>(h: JoinHandle<T>) -> Result<T> {
    h.join().map_err(|payload| panic_payload_to_anyhow(payload))
}

fn panic_payload_to_anyhow(payload: Box<dyn Any + Send + 'static>) -> anyhow::Error {
    if let Some(se) = payload.downcast_ref::<StrategyError>() {
        anyhow!(
            "策略线程 panic: {} [{}] code={}",
            se.strategy,
            se.source,
            se.code
        )
    } else if let Some(s) = payload.downcast_ref::<&str>() {
        anyhow!("策略线程 panic: {s}")
    } else if let Some(msg) = payload.downcast_ref::<String>() {
        anyhow!("策略线程 panic: {msg}")
    } else {
        anyhow!("策略线程 panic（未知载荷类型）")
    }
}

fn run_strategy() -> Result<()> {
    let bars = fetch_bars_from_ctp().context("拉 CTP 行情")?; // 普通 anyhow 错误

    let worker = thread::spawn(move || {
        if bars.is_empty() {
            panic::panic_any(StrategyError {
                strategy: "mean_revert_v2".into(),
                code: 1001,
                source: "ctp",
            });
        }
        // …
    });

    join_or_anyhow(worker).context("风控 worker 线程")?; // panic 也并进 ? 链
    Ok(())
}

fn fetch_bars_from_ctp() -> Result<Vec<u8>> {
    Ok(vec![])
}
```

**两条还原路径（可并存）**：

| 路径 | 何时用 |
|------|--------|
| **`downcast_ref::<StrategyError>()`** | 只要结构化字段，不必立刻进 `anyhow` |
| **包装成 `anyhow::Error` 再 `?`** | 与 **`fetch_*?` / 计算 `?`** 同一套错误流，日志 / context 一致 |

> **边界**：`panic` 表「不可恢复」语义不变 — `anyhow` 统一的是**主线程侧怎么处理 join 失败**，不是鼓励用 panic 代替 `Result`。Worker 里可恢复错误仍优先 **`mpsc` / `Result` 回传**；`panic_any` + anyhow 适合 worker 已 panic、主线程要把载荷**并回策略级 `?` 链**的场景。

```rust
use std::any::Any;
use std::thread;

let h = thread::spawn(|| {
    panic!("boom");
});

match h.join() {
    Ok(v) => println!("{v:?}"),
    Err(e) => {
        if let Some(s) = e.downcast_ref::<&str>() {
            eprintln!("子线程 panic: {s}"); // s 实际是 &&str，见 [1.1.2.4 §2.2](./1.1.2.4-panic-capture-downcast.md#22-downcast_refstrturbofish-与类型层级精读)
        }
    }
}
```

> 日常 `.join().unwrap()` = 子线程 panic 时**主线程也 panic**。要捕获 → **`match` + `downcast_ref`**（见 [1.1.2.4 §2.1](./1.1.2.4-panic-capture-downcast.md#21-panic-捕获与-downcast_ref)；`&&str` 等类型细节见 [1.1.2.4 §2.2](./1.1.2.4-panic-capture-downcast.md#22-downcast_refstrturbofish-与类型层级精读)）。

#### panic 与 `match` 的分工：抛 vs 接

先把**两个角色**拆开 — **`panic` 负责扔，`match` + `join` 负责接**；不要混成「match 是一种 panic」：

```text
子线程（抛出侧）                    主线程（捕获侧）
─────────────────                  ─────────────────
发现不可恢复错误                      handle.join()
  │                                    │
  ├─ panic! / panic_any ──► 线程终止     ├─ Ok(T)  → 正常返回值
  │   （主动抛出）                      │
  └─ 或运行时越界等 ──► 自动 panic       └─ Err(Box<dyn Any>)
                                           │
                                           match + downcast_ref
                                           （记录 / 备用策略 / 告警 …）
```

| 侧 | 手段 | 语义 |
|----|------|------|
| **抛出** | **`panic!`** / **`panic::panic_any(...)`** | **不可恢复** — 策略依赖的行情断了、状态无法继续，**直接终止该线程**并把载荷装箱 |
| **捕获** | **`join()` + `match` + `downcast_ref`** | **处理**已发生的 panic — 不管载荷来自**主动 `panic_any`** 还是**运行时自动 panic**（如数组越界），`join` 拿到的都是 **`Err(Box<dyn Any + Send + 'static>)`**，拆解方式相同 |

##### 主动 panic vs 运行时自动 panic：来源不同，**汇合点相同**

子线程里 panic 有**两条来源**，但 **`join()` 之后形态一致** — 都要靠你在主线程 **`match` / `if let` + `downcast_ref` 手动匹配**各类载荷：

| 来源 | 子线程里典型触发 | 常见载荷类型 | 谁「抛」的 |
|------|------------------|--------------|------------|
| **主动 panic** | `panic!`、`panic_any(StrategyError { … })`、业务判断「没法继续」 | `&str`、`String`、**自定义 struct** | **你写的代码** |
| **运行时自动 panic** | `.unwrap()` / `.expect()` 失败、数组越界、除零、`assert!` 失败 … | 多为 **`&'static str`**（如 `"index out of bounds"` — 编译期写死在程序里）或 **`String`** | **Rust 运行时 / 标准库** |

```text
子线程                              主线程（唯一处理入口）
────────                            ────────────────────
主动 panic_any(StrategyError)  ─┐
主动 panic!("逻辑异常")         ─┤
自动 [99] 越界                 ─┼──► join() ──► Err(Box<dyn Any>)
自动 opt.unwrap()              ─┘              │
                                               ├─ downcast StrategyError → 备用策略 / 告警
                                               ├─ downcast &str / String → 默认消息
                                               └─ 兜底 type_id()
```

**精准结论**：

1. **主动 / 自动只是子线程侧成因不同** — 不会走两套 join API。
2. **自动 panic 的载荷同样进 `Box<dyn Any>`** 传回主线程；不会因为是越界就「漏 join」。
3. **主线程必须自己写匹配链** — 既覆盖官方默认的 **`&str` / `String`**，也加 **`StrategyError`** 等自定义分支，才能做**全链路**处理（记错误码、切策略、告警 vs 仅打印越界信息）。

```rust
// 同一套 on_strategy_panic 处理两种来源 — 示意
let h1 = thread::spawn(|| {
    panic::panic_any(StrategyError { /* 主动 */ .. });
});
let h2 = thread::spawn(|| {
    let v = vec![1, 2, 3];
    let _ = v[99]; // 运行时自动 panic → join 仍是 Err(Box<dyn Any>)
});

for h in [h1, h2] {
    if let Err(payload) = h.join() {
        on_strategy_panic(&payload); // 同一匹配链
    }
}
```

> 口语里常说「用 panic 抛自定义错误」— 代码上自定义 struct 须走 **`panic_any(StrategyError { … })`**，**不能**写 `panic(StrategyError { … })`（`panic!` 宏只接受格式字符串路线）。

**量化完整链路**：定义带错误码、上下文的 **`StrategyError`** → 子线程里 **`panic_any`** → 主线程 **`match h.join()`** → **`downcast_ref::<StrategyError>`** 分支做针对性处理：

```rust
use std::any::Any;
use std::panic;
use std::thread;

#[derive(Debug)]
struct StrategyError {
    code: u32,
    strategy: String,
    detail: String, // 如 "行情 feed 断开"
}

fn on_strategy_panic(payload: &Box<dyn Any + Send + 'static>) {
    if let Some(se) = payload.downcast_ref::<StrategyError>() {
        log_error_code(se.code);
        switch_to_backup_strategy(&se.strategy);
        send_alert(&se.detail);
    } else if let Some(s) = payload.downcast_ref::<&str>() {
        eprintln!("panic 消息: {s}");
    } else if let Some(msg) = payload.downcast_ref::<String>() {
        eprintln!("panic 消息: {msg}");
    } else {
        eprintln!("未知 panic 载荷: {:?}", payload.type_id());
    }
}

fn log_error_code(_: u32) {}
fn switch_to_backup_strategy(_: &str) {}
fn send_alert(_: &str) {}

fn main() {
    let h = thread::spawn(|| {
        let feed_ok = false;
        if !feed_ok {
            // 行情断了 — 不可恢复，主动 panic_any
            panic::panic_any(StrategyError {
                code: 4001,
                strategy: "mean_revert_v2".into(),
                detail: "CTP 行情 feed 断开".into(),
            });
        }
    });

    match h.join() {
        Ok(_) => println!("worker 正常结束"),
        Err(payload) => on_strategy_panic(&payload),
    }
}
```

**收益**：**panic**（主动或自动）表达「这条线程已崩」；**match + downcast** 在主线程**统一收口** — 自定义 `StrategyError` 走业务分支，标准库 panic 走 `&str`/`String` 分支。可恢复错误（重试即可）仍用 **`Result` + channel**，不要滥用 panic → [§anyhow](#与-anyhow把-join-的-err-并进-result---主流程)。捕获细节 → [1.1.2.4](./1.1.2.4-panic-capture-downcast.md)。
