## 1.1.3.3 为什么必须 `'static`：场景、调度、独立线程

> 书 §1.1（3.3）· 索引：[1.1.3](./1.1.3-send-static-spawn.md) · 前：[1.1.3.2 Send](./1.1.3.2-send-why.md) · 后：[1.1.3.4 `T: 'static`](./1.1.3.4-t-static-types.md)

### 三、为什么必须 `'static`？

**`'static` 不是一种「数据类型」**，而是**生命周期标注**：在此表示捕获的数据**不能依赖「可能比子线程先销毁」的栈帧**。

#### 3.1 两套场景（纠正笼统说法）

| | **场景 A：进程退出** | **场景 B：函数提前返回** |
|--|---------------------|-------------------------|
| 典型情况 | `main` 走完、未 join 子线程 | `create_thread()` return，`main` 还在 `sleep` |
| 子线程命运 | 进程销毁 → **全体强杀** | 子线程**仍可能运行** |
| 局部栈 | 随进程一起没 | **创建函数栈已释放** |
| 与 `'static` | 不是设计 `'static` 的主因 | **这才是 `spawn` 要防的 UB** |

> **场景 A** 见 [1.1.1 §二](./1.1.1-spawn-join-basics.md)：`main` 结束则进程结束。  
> **场景 B** 才是日常踩坑：**函数作用域销毁 ≠ 子线程结束**。

#### 3.1.5 易混点：编译期捕获 ≠ 运行时调度

很多人卡住在这句：**「编译期已经把该捕获的都捕获完了，怎么还会出现局部变量被 drop 了、子线程仍在访问？」**

—— 根因**不是 CPU 指令乱序**，而是两件独立机制的矛盾：

| | **线程调度（OS）** | **栈帧回收（Rust 确定性）** |
|--|-------------------|---------------------------|
| 谁管 | 操作系统 | 作用域结束 / 函数 `return` |
| 特点 | **不可预测** — 子线程何时跑闭包，编译器管不了 | **确定** — 出了 `{}` 或函数 return，局部变量**立刻** drop |
| 冲突 | 可能「房间都拆了」才调度子线程去摸原来的栈地址 | 创建者栈已回收，借用的引用变悬垂 |

**房间比喻**（创建者函数 = 房间，子线程 = 门外的人，OS = 调度员）：

```text
你在房间里把东西「递」给闭包 / 子线程（capture）
递完你就关门、拆房间（函数 return → 栈帧回收）
门外的人什么时候伸手接、什么时候用 —— 调度员说了算

  调度员快  →  拆房间前就用完  →  借用碰巧安全（靠运气，语言不能赌）
  调度员慢  →  房间已拆才伸手  →  借用的 &局部  →  摸到空处（UB）
```

**`'static` + `move` 在干什么？**

**不是**去「纠正调度顺序」，而是**从根源禁止**「递出去（借引用）后自己先拆房间」：

| 做法 | 比喻 | 效果 |
|------|------|------|
| **`static` / `&'static`** | 东西放在整栋楼公区 | 不依赖某个房间；房间拆了公区还在 |
| **`move` 闭包** | 把东西直接塞给门外的人，让他自己带着 | 数据在闭包（子线程）里 **owned**；你拆房间不影响他手里那份 |
| **借 `&局部`（默认 capture）** | 只告诉门外的人「东西在 302 房间左上角」 | 房间一拆地址失效 — **`spawn` 编译拒绝** |

| 阶段 | 谁负责 | 在检查什么 |
|------|--------|------------|
| **① 编译期** | Rust 编译器 | 闭包**有没有合法拿到**变量（所有权转移 / 借用规则） |
| **② 运行时** | 操作系统调度 | 子线程**什么时候**真正执行闭包里的代码 |

**编译期只回答 ①**，不保证 ② 的顺序。`move` 闭包在编译期就把 `s` 的所有权**移进闭包**；但子线程**何时**跑 `println!("{s}")` 仍由 OS 决定 —— **没关系**，因为数据已不在外层栈上。

**极端时间线（场景 B · 借引用 · 假想无 `'static` 的语言）**：

```rust
fn main() {
    let s = String::from("hello");
    thread::spawn(|| println!("{s}")); // 闭包记住的是「&s 在 main 栈上」
    // main return → 栈帧回收（确定发生）
    // OS 稍后调度子线程 → 访问已回收内存（不确定何时，但一发生就 UB）
}
```

Rust 在编译期用 **`F: 'static`** 直接拒绝上例。合法写法：

```rust
thread::spawn(move || println!("{s}")); // s 跟闭包走；main 拆栈不影响闭包内的 owned s
// 或 static / Arc / thread::scope …
```

若闭包捕获的是 **`&s`（借用）** 而不是 **`move` 后的 owned `s`**：

1. **编译期**：闭包类型里带着 **`&'a s`**，`'a` 只覆盖创建者栈帧 → 不满足 **`F: 'static`** → 直接**编译失败**。
2. **若没有 `'static` 约束（假想 C++）**：问题不在「capture 没完成」，而在 **capture 的是房间地址，而房间比子线程先拆**。

**`'static` 约束在堵什么**：闭包类型里**不能含**绑在「可能比子线程更短」栈帧上的引用；要么 **move / `Arc` / 堆**，要么 **程序期数据**（`&'static str`、`static`），要么 **`thread::scope`** 把「拆房间」推迟到子线程全部结束。

#### 3.2 本质规则：为什么是「独立线程」就要 `'static`？

`thread::spawn` 创建的是 **与创建者线程没有生命周期绑定的独立 OS 线程** — 不是「父线程的子例程」：

| 特性 | 含义 |
|------|------|
| **无父子生命周期** | 创建它的函数可以 **立刻 return**、栈帧销毁；子线程 **仍可能被调度** |
| **`JoinHandle` 可任意流转** | 子线程返回的 handle 可以存进容器、传到 **完全不相关的另一个线程** 去 `join` |
| **编译器无法假设「谁先结束」** | 不能赌「子线程一定在创建者栈销毁前跑完」 |

因此 Rust **不敢**让闭包捕获任何 **依赖原栈帧** 的引用 — 统一要求 **`F: 'static`**：捕获物要么是 **程序期有效** 的数据，要么用 **`move` 把所有权彻底移进闭包**（数据跟着闭包 / 子线程走，不再绑创建者栈）。从 API 设计上 **堵死悬垂引用**，而不是运行时去「对齐调度顺序」。

```text
thread::spawn          →  独立线程，寿命与创建者栈脱钩  →  必须 'static / move
thread::scope          →  有界线程，scope 结束前必 join  →  可借 &'scope 局部（见 §七）
crossbeam::scope       →  同思路（生态 crate）；保证线程在作用域内结束  →   likewise 不需 'static
```

**对照 `scope`（标准库 [1.1.3.6 scope](./1.1.3.6-arc-scope-hft-cheatsheet.md#七绕过-staticthreadscope-为什么可以借局部变量) · [1.2](../1.2-scoped-threads/1.2-scoped-threads.md)）**：

| | **`thread::spawn`** | **`thread::scope` / `crossbeam::scope`** |
|--|---------------------|------------------------------------------|
| 线程与创建者栈 | **无绑定** — handle 可漂到别处 | **有绑定** — scope 块内 **阻塞到全部子线程结束** |
| 原栈帧何时销毁 | 创建者函数 **return 就 drop** | **子线程都结束后** 才继续、才 drop 被借局部 |
| 闭包能否借栈上局部 | ❌ 须 **`'static` / move** | ✅ 可 **`&'scope T`** — 编译器能证明 **栈帧在线程结束前不会被销毁** |
| 典型用途 | 线程池、后台 worker、fire-and-forget | 函数内有界并行、借局部数组做分片计算 |

编译器**无法**在 `spawn` 路径上模拟 OS 调度，故对 **`spawn` 一律最保守：`'static`**。

#### 3.3 合法方案

| 方案 | 做法 | 说明 |
|------|------|------|
| **1. move 所有权** | `spawn(move \|\| …)` | owned 数据随闭包进子线程，**不再借外层栈** |
| **2. 真·`'static` 数据** | `&'static str`、全局 `static` | 数据本身**整个程序**存活期内有效（见下 §「进程」指什么） |
| **3. 有界 scope** | **`thread::scope`** | 强制等子线程结束再销毁局部变量（[1.1.3.6 scope](./1.1.3.6-arc-scope-hft-cheatsheet.md#七绕过-staticthreadscope-为什么可以借局部变量)） |

#### 「进程」在 `'static` 里指什么？

**指整个程序从启动到退出的生命周期**（`main` 开始到进程结束），**不是**：

- ❌ 「闭包所在 OS 线程的内核线程对象」
- ❌ 「变量原来所在的那个栈帧 / 那个函数」

| 变量 | 生命周期绑在哪 | 能否直接给 `spawn` 闭包借？ |
|------|----------------|----------------------------|
| `static S: &str = "hello"` | **整个程序** | ✅ 借 `&'static str` 可以（数据在只读静态段） |
| `let s = String::from("hello")` | **当前栈帧**（如 `main` 块） | ❌ 不能借 `&s`；需 **`move`** 把 owned 移进闭包，或改 **`static` / `Arc`** |

口语里「生命周期和进程一样长」= **数据在程序运行期间不会被回收**；与变量**最初定义在哪个函数栈**无关 —— `main` 里的 `String` 仍只是栈上局部，**不等于**「进程级 `'static`」，所以 **`spawn` 仍要 `move` 或 `Arc`**。

**一句话**：**线程可能活过创建它的函数栈 → 强制 `'static`**；借局部栈 → **`thread::scope`**（[1.2.4 深拆](../1.2-scoped-threads/1.2.4-scoped-join-handle.md#static-vs-scope生命周期深拆)）。

> `T: 'static` 同理：`JoinHandle<T>` 可留存任意久，返回值不能内含比 handle 活得短的借用。  
> `move` 转移 **owned** 的 `String` 等即可，**不必**字面量写 `&'static`。

```rust
let num = 10;
// 报错：闭包借用 &num，生命周期只有 'a（当前栈），不是 'static
// let h = thread::spawn(|| println!("{num}"));

let h = thread::spawn(move || println!("{num}")); // num 所有权进闭包
```
