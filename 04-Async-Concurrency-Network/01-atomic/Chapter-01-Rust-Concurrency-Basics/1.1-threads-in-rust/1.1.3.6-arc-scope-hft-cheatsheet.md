### 六、多线程共享：只读 `Arc`

```rust
use std::sync::Arc;
use std::thread;

fn main() {
    let s = Arc::new(String::from("hello"));
    let mut handles = vec![];

    for _ in 0..2 {
        let s = Arc::clone(&s); // 只增引用计数
        handles.push(thread::spawn(move || {
            println!("{s}");
        }));
    }
    for h in handles {
        h.join().unwrap();
    }
}
```

- **`Arc<T>`**：`T: Send + Sync` 时 **`Arc` 可 Send**，多线程共享**只读**；
- 可变共享 → **`Arc<Mutex<T>>`**（[1.7](../1.7-mutex-rwlock/1.7-mutex-rwlock.md)）。

### 七、绕过 `'static`：`thread::scope` 为什么可以借局部变量？

**有界线程**：`thread::scope`（生态里 **`crossbeam::scope`** 同理）会**阻塞当前作用域**，直到 scope 内**所有**子线程执行完毕，才继续往下走、才销毁被借用的局部变量 → 生命周期**闭环**，故闭包用 **`F: 'scope`** 而非 **`'static`**。

与 **`spawn` 的独立线程**相对：scope 把「子线程何时结束」和「原栈帧何时 drop」**绑死在同一段作用域里**，编译器能证明 **借用的局部在子线程结束前一定有效** — 所以 **不必** 要求 `'static`，也 **不必** 为了过检而 `move`。

```rust
use std::thread;

fn safe_scope() {
    let s = String::from("局部变量"); // 存活于 safe_scope 栈帧
    thread::scope(|scope| {
        scope.spawn(|| println!("{s}")).join().unwrap(); // ✅ 借 s，无需 move
    });
    // scope 块结束：已自动 join 全部子线程，此时 s 才随函数栈销毁
}
```

| 机制 | `thread::spawn` | `thread::scope` / `crossbeam::scope` |
|------|-----------------|--------------------------------------|
| 与创建者栈的关系 | **无绑定** — 独立线程 | **有绑定** — scope 结束前必须 join 完 |
| 创建者函数能否先 return | 能；子线程可继续跑 | scope **阻塞**到子线程全结束 |
| `JoinHandle` 能否漂到别的线程 | ✅ 可以 | ❌ scope 内 handle，出 scope 前须 join |
| 局部变量销毁时机 | 函数 return 即 drop | **子线程结束后**才 drop 被借数据 |
| 闭包约束 | **`F: 'static`**（或 move owned） | **`F: 'scope`** — 可借栈上局部 |
| HFT 倾向 | 线程池 / 长期任务（常 `move` / `Arc`） | 函数内有界并行，**少堆分配、生命周期可控** |

> **注意**：变量若定义在 **`scope` 闭包内部**，`scope.spawn` 仍常须 **`move`**（见 [1.2 demo](../1.2-scoped-threads/code/1.2-scoped-threads-demo.rs)）；上例是 **scope 外、函数内的局部变量**——这正是 scope 要解决的典型场景。

### 八、HFT 语境与学习顺序

**结论**：**`spawn` + `'static` 是基础多线程模型，HFT 里会用到，但不是核心瓶颈**；HFT 更怕的是**锁、系统调用、分配、缓存失效**，而不是 `'static` 标注本身。

#### HFT 传参与选型（贴合 `'static` 语义）

| 需求 | 推荐 | 原因 |
|------|------|------|
| 数据进子线程 | **`move` 自有类型** | 自动 **`F: 'static`**，无借用生命周期 |
| 多线程共享读 | **`Arc` + `move` clone** | **`Arc<T>: 'static`**，堆上共享 |
| 函数内借局部、要低延迟 | **`thread::scope`** | 允许借栈，**不必 `move` / 不必 `Arc` 堆分配** |

**一句话**：**`spawn` 优先 `move` 或 `Arc`**；不想受 `'static` 限制、追求可控生命周期 → **`scope`**。

#### 建议学习顺序（面向系统 / HFT 方向）

1. **`std::sync`**（锁）、**`Arc`**、**`Mutex`**、**`Condvar`**、**`thread::scope`** → 共享状态、线程协作、安全访问  
2. **`tokio` / async**、**TCP/UDP** → 行情、订单、数据链路  
3. **过程宏** → 放最后；核心路径几乎不用，多用于序列化、配置、代码生成等辅助

**一句话**：**`F: 'static` 防的是「函数栈先销毁、子线程仍跑」**；HFT 函数内并行优先 **`thread::scope`**（生命周期可控、少 `Arc` 堆分配）。

### 九、场景对照表（错误 / 正确 / 原因）

| 场景 | ❌ 常见错误 | ✅ 正确写法 | 原因 |
|------|------------|------------|------|
| 局部 `String` 进子线程 | `spawn(\|\| println!("{s}"))` 无 move | `spawn(move \|\| …)` | 不能捕获 `&s`；须 **owned + Send** |
| 多个线程读同一份数据 | `Rc::clone` + `spawn` | **`Arc::clone` + `move`** | **`Rc` 非 Send** |
| 多个线程改同一份数据 | 多个 `move` 抢同一 `String` | **`Arc<Mutex<T>>`** | 要 **Sync + 互斥** |
| 短生命周期借用 | `spawn(\|\| use &local)` | **`thread::scope`** | `'static` 不允许借栈 |
| 全局配置 | — | **`static` / `LazyLock`** | 天然 `'static` |
| 子线程必在 fn 内结束 | 手动 join 列表 | **`thread::scope`** | 自动 join + 允许借用 |
| 返回值回主线程 | 返回 `&local` | 返回 **owned `T`** 且 **`T: Send`** | `join` 也要 **`T: Send`** |

---

### 十、背诵版

1. **`T: 'static`** = 类型内部**无短命引用**；**自有 `String`/`Vec`/`Arc` 等自动满足**，与 **`&'static str`**（静态**引用**）不是一回事。  
2. **`spawn` 防场景 B**：创建函数栈先销毁，子线程仍可能被调度（**不是** `main` 跑完子线程还在跑）。  
3. **`move` 自有类型** → 闭包无借用 → **`F: 'static` 天然成立**；共享 → **`Arc`**；借栈 → **`scope`**。  
4. HFT：关键路径优先 **scope / 无锁**；长期任务 **`move` / `Arc`**（[1.1.3.6 HFT](./1.1.3.6-arc-scope-hft-cheatsheet.md#八hft-语境与学习顺序)）。
