## 1.2.4 ScopedJoinHandle、`s.spawn` 约束与生命周期

> 书 §1.2 · [1.1.3 spawn 约束](../1.1-threads-in-rust/1.1.3-send-static-spawn.md) · 索引：[1.2](./1.2-scoped-threads.md) · 前：[1.2.3 join](./1.2.3-join-auto-join.md) · 后：[1.2.5 参数 s](./1.2.5-scope-param-s.md)

---

**一句话**：`s.spawn(闭包)` → 返回 **`ScopedJoinHandle<'scope, T>`**，语义类似 [1.1.2 的 `JoinHandle<T>`](../1.1-threads-in-rust/1.1.2-join-handle.md)，但生命周期绑在 **`'scope`** 上。

```rust
use std::thread;

fn main() {
    let result = thread::scope(|s| {
        let h = s.spawn(|| 10 + 20); // ScopedJoinHandle<'_, i32>
        h.join().unwrap()             // 也可不显式 join，scope 结束时会 join
    });
    println!("{result}");
}
```

### 1. 与 `JoinHandle` 的异同

| | **`thread::spawn`** | **`Scope::spawn`（scope 内）** |
|---|---------------------|--------------------------------|
| 句柄 | `JoinHandle<T>` | `ScopedJoinHandle<'scope, T>` |
| 闭包约束 | `FnOnce() -> T + Send + 'static` | `FnOnce() -> T + Send + 'scope` |
| 借用局部变量 | ❌ 不行（须 `'static`） | ✅ 可借 **`'scope` 内** 的数据 |
| 未 join 就离开 | drop → detached，可能泄漏线程 | **编译期禁止**：不能带着 handle 逃出 scope |
| 等待 | `.join()` 或 drop handle | scope 结束自动 join；也可手动 `.join()` |

### 2. 类型拆解（对比记忆）

```rust
// 普通 spawn
pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,

// scope 内 spawn（示意）
fn spawn<'scope, F, T>(&'scope self, f: F) -> ScopedJoinHandle<'scope, T>
where
    F: FnOnce() -> T + Send + 'scope,
    T: Send + 'scope,
```

关键差别在 **`'static` → `'scope`**：子线程只能活在这个 scope 里，因此可以借用「只要 scope 还在就不能被销毁」的栈上数据。

### 2.0 符号先拆清：`impl<'env> Scope<'env>`

教材或讲解里常写成 **`impl<'env> Scope<'env>`**。这里的 **`'env` 是生命周期参数**（env = environment，外部环境），**不是**变量名、**不是**类型泛型（如 `T`），专门标记「借用能活多久」。

#### 两种尖括号泛型（长得像，用途不同）

| 写法 | 种类 | 含义 | 例子 |
|------|------|------|------|
| **`<T, F>`** | **类型参数** | 任意数据类型 | `i32`、`String`、闭包类型 |
| **`<'env, 'a, 'b>`** | **生命周期参数** | 引用能活多久 | 只约束借用，不引入新类型 |

#### 为什么 `impl` 后面要写 `<'env>`？

结构体带生命周期参数时，**实现它的 `impl` 必须先声明同名生命周期**——语法硬性要求。

```rust
// 结构体：绑定生命周期 'a
struct Foo<'a> {
    data: &'a str,
}

// impl 实现 Foo，必须先写 <'a> 声明生命周期
impl<'a> Foo<'a> {
    fn get(&self) -> &'a str {
        self.data
    }
}
```

教材里的 **`Scope`** 同一规则（简化写法）：

```rust
struct Scope<'env> { /* … */ }

impl<'env> Scope<'env> {
    fn spawn<F, T>(...) -> ScopeJoinHandle<'env, T> { /* … */ }
}
```

**一句话规则**：结构体 `X<'a, …>` 含生命周期参数 → 写 `impl X<'a, …>` 时，尖括号里**先把生命周期声明出来**。

> 标准库实际是 **`Scope<'scope, 'env: 'scope>`**（两个生命周期），见 [§2.1 命名对照](#21-spawn-签名逐条拆解)。教材用单参数 **`Scope<'env>`** 是把「scope 外栈变量能活多久」单独拎出来讲，逻辑一致。

#### `'env` 精准含义：结构体参数，不是 trait 约束

**结论（缩小范围）**：`'env` **不是**「某个 trait 内部全部引用都不能超过 `'env`」——**`Scope` 是结构体，不是 trait**。  
`'env` 标记的是 **整个 scope 绑定的外部环境生命周期**：

> 在本 scope 中 **`s.spawn` 创建的所有子线程**，对外层变量的借用 **`&T` 的生命周期 ≤ `'env`**；  
> scope **阻塞到全部子线程结束** 后才离开 → `'env` 内的变量一定比这些线程活得久。

```rust
fn demo() {
    let msg = String::from("test"); // msg 的生命周期 ⊆ 'env
    thread::scope(|scope| {
        // scope: &Scope<'scope, 'env>（标准库；教材简写 &Scope<'env>）
        scope.spawn(|| {
            println!("{msg}"); // 借 &msg，生命周期 ≤ 'env ✅
        });
    });
    // 走到这里：所有子线程已 join，'env 结束，msg 此时才销毁
}
```

**底层逻辑**：

1. **`'env`** = scope **所在外层**（如 `demo` 函数）里，可被 scope 内线程借用的栈变量的存活区间；  
2. **`thread::scope` 强制规则**：所有 `spawn` 出的子线程 **join 完毕**，scope 块才结束；  
3. 因此：线程里对外层变量的引用，**上限就是 `'env`**，不会出现「引用先销毁、线程还在读」。

若没有 `'env` + scope 等待机制，`thread::spawn` 只能要求 **`'static`**；正是 **`'env` 标出外层环境存活区间**，配合阻塞 join，才允许 **直接借局部变量**（不必 `move` / `Arc`）。

| 符号 | 含义 |
|------|------|
| **`Scope<'scope, 'env>`** | **结构体**上的生命周期参数；`'env: 'scope` = 外部环境须活得比 scope 执行期长 |
| **`F: 'scope`**（标准库闭包约束） | 本 scope 内 spawn 的闭包及其捕获，寿命 ≤ **`'scope`**（可借 ≤ `'env` 的外层栈） |
| **安全逻辑** | scope join 全部子线程 → 被借变量 **≥** 所有线程 → 无悬垂引用 |

> 教材有时把闭包约束写成 **`F: 'env`**，是把「可借 scope **外**栈变量」口语化；标准库闭包 bound 是 **`F: 'scope`**，`'env` 出现在 **`Scope<'scope, 'env>`** 上。见 [§2.1 命名对照](#21-spawn-签名逐条拆解)。

#### 别混淆：`Trait<'a>` vs `Scope<'env>`

两种都带 **`'…` 生命周期参数**，约束对象**完全不同**：

| | **`trait BufRead<'buf>`**（trait 顶层生命周期） | **`struct Scope<'scope, 'env>`**（结构体参数） |
|--|--------------------------------------------------|------------------------------------------------|
| **是什么** | **Trait** 的泛型生命周期 | **`Scope` 结构体**的标记 |
| **约束谁** | 该 trait **所有方法**里出现的引用（如 `fn read(…) -> &'buf [u8]`） | 本 **scope 创建的全部子线程**对外层环境的 **借用** |
| **典型问题** | 「返回的切片不能超过 `'buf`」 | 「子线程借的 `&msg` 不能超过 `'env`」 |

```rust
// 情况 A：trait 自带 'buf —— 约束 trait 方法里的引用
trait BufRead<'buf> {
    fn read(&mut self) -> &'buf [u8];
}

// 情况 B：Scope 的 'env —— 约束 scope .spawn 线程对外层栈的借用
pub struct Scope<'scope, 'env: 'scope> { /* … */ }
```

**修正常见偏差**：

1. **`'env` 不属于 trait**，是 **`Scope` 结构体**的生命周期参数；  
2. 它约束的不是「某个 trait 内的引用」，而是 **「这个 scope .spawn 出来的线程」对外层环境的借用**。

标准库把「句柄 / 闭包不能逃出 scope」绑在更短的 **`'scope`** 上；**`'env: 'scope`** 保证外部环境活得足够长。见 [§4 `'scope` 从哪来？](#4-scope-从哪来threadscope-签名拆解)。

#### 为什么普通结构体很少见到这种写法？

大多数结构体**不存引用**，没有生命周期参数：

```rust
struct Person {
    name: String, // 所有权类型，无 &，不需要 'xxx
}

impl Person {
    fn new() -> Self { /* … */ }
}
```

只有结构体**需要借用外部数据**（存 `&T`、或像 `Scope` 一样允许子线程借栈）时，才会带上 **`'xxx`**。

#### 整条逻辑串起来

1. 用户在 scope **外**定义栈变量 → 有一段生命周期，教材命名为 **`'env`**；
2. **`Scope<'env>`** 记录这段生命周期；
3. 给 `Scope` 写方法时 → **`impl<'env> Scope<'env>`** 先声明生命周期；
4. **`spawn`** 返回 **`ScopeJoinHandle<'env, T>`**（教材）/ **`ScopedJoinHandle<'scope, T>`**（标准库），把同一套约束传给句柄；
5. **`F: 'env`**（教材）/ **`F: 'scope`**（标准库）：闭包借用不能超过外部变量存活时间。

**极简三句**：

1. **`impl<'env>`** 的 **`<'env>`** = 声明生命周期参数；**`Scope` 是结构体**，带 `'env` 就要在 impl 里先声明；  
2. **`'env`** = scope **外**栈环境存活区间；约束的是 **本 scope .spawn 的全部子线程**对外层的借用，**不是**某个 trait 内部的引用；  
3. 普通无引用的结构体不会出现这种写法；**`Scope`** 是「有界线程 + 允许借外层栈」的典型。

### 2.1 `s.spawn` 签名（逐条拆解）

> **命名对照（教材 vs 标准库）**：部分讲解把闭包约束写成 **`F: 'env`**、句柄写成 **`ScopeJoinHandle<'env, T>`**（把 `'env` 当「scope 外栈变量的生命周期」）。标准库实际写法是 **`F: 'scope`**、**`ScopedJoinHandle<'scope, T>`**；`'env` 出现在 **`Scope<'scope, 'env>`** 上，表示**外部环境须活得比 `'scope` 长**（`'env: 'scope`）。§2.0 用教材简化写法入门；下文按**标准库**名称讲解。

```rust
impl<'scope, 'env: 'scope> Scope<'scope, 'env> {
    pub fn spawn<F, T>(&self, f: F) -> ScopedJoinHandle<'scope, T>
    where
        F: FnOnce() -> T + Send + 'scope,
        T: Send + 'scope,
}
```

| # | 约束 | 与普通 `spawn` 的差别 |
|---|------|----------------------|
| **1** | `F: FnOnce() -> T` | 相同：一次性线程入口，产出 `T` |
| **2** | `F: Send` | 相同：闭包须 **move 到子线程** |
| **3** | `F: 'scope` | **代替 `F: 'static`**：闭包可借用 **scope 存活期内** 的外部栈数据 |
| **4** | `T: Send + 'scope` | **不要 `'static`**，但要 **`'scope`**（口语有时简写「仅 Send」；标准库仍要求 **`T: 'scope`**，返回值不能比 scope 活得更久还带借用） |

| 教材常见写法 | 标准库写法 | 含义 |
|--------------|------------|------|
| `F: 'env` | **`F: 'scope`** | 闭包可借 **scope 存活期内** 的数据，代替 **`F: 'static`** |
| `ScopeJoinHandle<'env, T>` | **`ScopedJoinHandle<'scope, T>`** | 句柄不能逃出 scope |
| `T: Send`（仅 Send） | **`T: Send + 'scope`** | 返回值可跨线程传回，且寿命 ≤ scope |

#### 为什么 `'scope` 能借局部变量？

标准库保证：**离开 scope 前 join 全部子线程** → 被借用的外部变量生命周期 **≥** 所有子线程运行期 → 不会出现悬垂引用。完整签名见 [1.2.5 参数 s](./1.2.5-scope-param-s.md)。

#### `Send` 两个 spawn 都要？

**`Send`** = 该类型的所有权可以**安全地从一线程转移到另一线程**。只要有「把闭包 move 给新线程」，就要求 **`F: Send`**，与有界 / 无界无关。常见 **非 Send**：`Rc<T>`、`*const T` / `*mut T`（默认）、`&mut T`（在多数跨线程场景）。

### 2.2 最简示例对比

**scope：允许借用（`F: 'scope`）**

```rust
let msg = "hello"; // 局部生命周期
thread::scope(|s| {
    s.spawn(|| println!("{msg}")); // 借 msg，不必 move
});
```

**普通 spawn：须 move（`F: 'static`）**

```rust
let msg = "hello".to_string();
let h = thread::spawn(move || println!("{msg}"));
h.join().unwrap();
```

**scope 里也可以 `move`（不冲突）**

```rust
let s = String::from("test");
thread::scope(|sc| {
    sc.spawn(move || println!("{s}")); // move 进闭包，仍满足 'scope
});
```

### 3. 数据竞争仍由编译器拦截

scope **不是**「随便共享可变数据」：

- 多个子线程**只读**借用同一 `&T` → 通常可以；
- 任一子线程需要 **`&mut T`** → 编译器保证同一时刻只有一个活跃的可变借用 → **编译期防数据竞争**（详细规则见 [1.4.4 借用规则](../1.4-borrowing-data-races/1.4.4-borrowing-rules.md)）。

### 4. `JoinHandle` vs `ScopedJoinHandle`：四条核心差别

1. **生命周期绑定**
   - **`JoinHandle<T>`**：无 `'scope` 参数；句柄可存进 `Vec`、从函数**返回**、活得比创建它的栈帧更久；
   - **`ScopedJoinHandle<'scope, T>`**：绑定 scope 生命周期 **`'scope`**，**只能在 scope 闭包内**使用，编译器禁止带出闭包。

2. **安全规则（最大区别）**

   普通 `thread::spawn` **强制 move / `'static`**，不允许借栈：

   ```rust
   let x = 10;
   thread::spawn(|| println!("{x}")); // 编译错误：不能借用 x
   ```

   scope 的 `s.spawn` **允许借用** scope 外变量：

   ```rust
   let x = 10;
   thread::scope(|s| {
       s.spawn(|| println!("{x}")); // OK：'scope 保证 x 不会先于子线程销毁
   });
   ```

   原理：类型系统 + `'scope` 证明「所有子线程在 scope 结束前结束 → 被借用的 `x` 不会提前 drop」。

3. **未 join 就「丢」句柄时**
   - **`JoinHandle`**：drop 且未 join → 线程 **detach**；`main` 直接结束 → 子线程可能被 OS 强杀（[1.1.2 §四](../1.1-threads-in-rust/1.1.2-join-handle.md#四不用-join-会怎样)）；
   - **`ScopedJoinHandle`**：不能逃出 scope；scope 收尾时 **Scope 批量 join**，**阻塞当前线程**直到子线程全部结束（[1.2.3 §3](./1.2.3-join-auto-join.md#3-分层拆解自动-join-做了什么是不是封装系统调用)）。

4. **`.join()` 底层**
   两种句柄都有 **`.join()`**；手动 / 自动 join 最终走**同一套** OS 等待 API（`pthread_join` / `WaitForSingleObject`），差别只在 std **包装层**与**生命周期**。

## 两段 spawn 泛型约束对照

> 普通 `thread::spawn` 逐条见 [1.1.3.1 签名](../1.1-threads-in-rust/1.1.3.1-spawn-signature-constraints.md)。

| 约束点 | `thread::spawn` | `s.spawn`（scope 内） |
|--------|-----------------|------------------------|
| 闭包特征 | `FnOnce() -> T` | 同左 |
| 跨线程 move | **`F: Send`** | **`F: Send`** |
| 闭包生命周期 | **`F: 'static`** | **`F: 'scope`**（教材或写 **`F: 'env`**，见 [1.2.4 §2.1](./1.2.4-scoped-join-handle.md#21-spawn-签名逐条拆解)） |
| 能否借栈引用 | ❌ 禁止，须 **move** | ✅ 允许，通常不必 move |
| 返回值 `T` | **`Send + 'static`** | **`Send + 'scope`**（不要 `'static`） |
| 返回句柄 | `JoinHandle<T>` | `ScopedJoinHandle<'scope, T>` |
| 自动 join | ❌ | ✅ scope 收尾批量 join |

**为什么 scope 不用 `'static`？**  
Scope **强制等待**：所有子线程在 scope 结束前 join → 外部栈变量一定活得比子线程久 → 编译器用 **`'scope`** 绑定即可，不必抬到全程序 `'static`。

## `'static` vs `'scope`：生命周期深拆

> 普通 `spawn` 的 `'static` 详见 [1.1.3.3](../1.1-threads-in-rust/1.1.3.3-static-why-design.md)。

### 1. 普通 `thread::spawn` 为什么要求 `'static`？

```rust
pub fn spawn<F, T>(f: F) -> JoinHandle<T>
where
    F: FnOnce() -> T + Send + 'static,
    T: Send + 'static,
```

**`F: 'static` 本质**：子线程**生命周期不受创建它的函数 / 代码块限制**——创建者函数可先 `return`、销毁栈，子线程仍可能被调度（**不是**「`main` 跑完子线程还在跑」；进程退出会强杀所有线程）。若闭包捕获 **`&'a T`**（`'a` 只覆盖该函数栈）→ 悬垂引用。精读：[1.1.3 §三](../1.1-threads-in-rust/1.1.3-send-static-spawn.md#三为什么必须-static)。

合法出路：

| 方案 | 做法 |
|------|------|
| **move 所有权** | `spawn(move \|\| …)` — owned 数据随闭包进子线程，**不再借栈** |
| **真·`'static` 数据** | `&'static str`、全局 `static` |

```rust
let num = 10;
// thread::spawn(|| println!("{num}")); // 报错：F 不满足 'static

let h = thread::spawn(move || println!("{num}")); // num 所有权进闭包
h.join().unwrap();
```

> **易错**：`move` 后是 **owned 数据进闭包**，不是「一定挪到堆上」；`i32` 拷贝、`String` 转移所有权即可满足 **`'static`**。

### 2. `s.spawn` 为什么用 `'scope` 而不是 `'static`？

```rust
pub fn scope<'env, F, T>(f: F) -> T
where
    F: for<'scope> FnOnce(&'scope Scope<'scope, 'env>) -> T,
```

**`ScopedJoinHandle<'scope, T>`**（标准库名；口语 **`ScopeJoinHandle`**）绑定 **`'scope`**。标准库保证：**所有 `s.spawn` 线程在 scope 结束前 join** → 外部栈变量寿命 **≥ `'scope`** → 可安全借用。

```rust
let num = 10;
thread::scope(|s| {
    s.spawn(|| println!("{num}")); // 借 &num，不必 move
});
// 全部子线程已 join，num 才销毁
```

**一句话**：`thread::spawn` 线程可能活过当前栈 → **`'static`** + 通常 **`move`**；`s.spawn` 线程必死在 scope 内 → **`'scope`** + 可借栈。

### 3. 一句话对照表

| | `thread::spawn` | `s.spawn` |
|---|-----------------|-----------|
| 线程寿命 | 可能**活过**创建者栈帧 | **一定死在** scope 结束前 |
| 生命周期约束 | **`F: 'static`**，**`T: Send + 'static`** | **`F: 'scope`**，**`T: Send + 'scope`** |
| 捕获局部变量 | ❌ 禁止借用；须 **move** 或 `'static` 数据 | ✅ 可**直接借用**外部栈变量 |

### 4. `'scope` 从哪来？`thread::scope` 签名拆解

```rust
pub fn scope<'env, F, T>(f: F) -> T
where
    F: for<'scope> FnOnce(&'scope Scope<'scope, 'env>) -> T,
```

- **`'env`**：scope **外面**那层环境（例如 `fn main` 里的局部变量）；
- **`'scope`**：scope **闭包执行期间**——所有 `s.spawn` 子线程**必须在此结束前 join**；
- **`ScopedJoinHandle<'scope, T>`** 把句柄和返回值绑在 **`'scope`** 上，**不能逃出**闭包。

因此编译器能证明：被借用的栈变量（如外面的 `num`）生命周期 **≥ `'scope` ≥ 所有子线程** → 不会悬垂。

```rust
let num = 10; // 栈变量，生命周期覆盖整个 scope 调用
thread::scope(|s| {
    s.spawn(|| println!("{num}")); // 借 &num，'scope 由 Scope 约束
});
// 到这里：全部子线程已 join，num 才安全销毁
```

**一句话**：**`s.spawn` — 线程一定死在 scope 结束前 → 绑定局部 `'scope`，允许借栈引用。**

### 5. 生命周期标注 ≠ 传入的一定是引用

生命周期泛型（`'static` / `'scope`）是**安全检查规则**：约束「**若存在借用，被借的数据须活够久**」——**不**表示闭包里只能传引用。

#### （1）传引用（无 `move`，scope 典型）

```rust
let s = String::from("hello");
thread::scope(|sc| {
    sc.spawn(|| println!("{s}")); // 闭包捕获 &String，生命周期 'scope
});
```

只借、不转移所有权；安全靠 scope **自动 join** + `'scope` 证明。

#### （2）传所有权（`move`，两种 spawn 都常见）

```rust
let s = String::from("hello");
thread::scope(|sc| {
    sc.spawn(move || println!("{s}")); // 无 &，String 所有权进子线程
});
```

- **`move`** 后闭包内**没有**对外部变量的引用，仍满足 **`'scope`**（或普通 spawn 的 **`'static`**）；
- **`thread::spawn` 绝大多数场景**都是这种「所有权转移、无引用」——因为**不允许借栈**。

#### （3）区分记忆

| 概念 | 说明 |
|------|------|
| **`JoinHandle<T>` / `ScopedJoinHandle<'scope, T>` 的 `T`** | 可以是 **`String`、`i32`**（owned），也可以是 **`&'scope i32`**（借用） |
| **何时产生引用？** | **不写 `move`、直接捕获外部变量** → 闭包往往捕获 **`&T`** |
| **何时无引用？** | **`move`** → 转移所有权；普通 spawn 靠此满足 **`'static`** |

### 6. 补充细节（背诵）

1. **`'static`**：最长生命周期标注，**不是类型**；spawn 侧表示「不能依赖短栈借用」。  
2. **`'scope`**：**局部临时**生命周期，只覆盖 scope 闭包执行阶段。  
3. **生命周期 = 规则**，你可以选 **借用（无 move）** 或 **转移所有权（move）**；scope 两种都合法，普通 spawn 实际上只能后者（或 `'static` 数据）。  
4. 普通 spawn **没有** scope 的强制 join 保护 → **彻底禁用**借局部栈，只能 **move** 或 `'static` 数据。
