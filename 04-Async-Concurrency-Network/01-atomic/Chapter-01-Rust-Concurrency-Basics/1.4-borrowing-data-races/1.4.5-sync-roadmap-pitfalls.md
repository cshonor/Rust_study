## 二、三类检查机制对比

| 工具 | 检查时机 | 冲突表现 | 跨线程 | 核心用途 |
|------|----------|----------|--------|----------|
| **`&T` / `&mut T`** | 编译期 | 编译失败 | 配合 `Sync` / scope | 只读共享或独占写，**零运行时开销** |
| **`Cell` / `RefCell`** | 运行时 | 双重可变借 → **panic** | ❌ 非 `Sync` | 单线程内部可变 |
| **`Mutex` / `RwLock`** | 运行时 | 冲突 → **阻塞等待** | ✅ 为多线程设计 | 多线程共享可变 |

**记忆**：编译期 = 最安全、限制最多、无锁开销；运行时 = 更灵活、有开销、靠逻辑/原语兜底。

### 为什么 `RefCell` 不能跨线程，`Mutex` 可以？

- **`RefCell`**：只在**单线程**内做运行时借用检查，**没有**跨线程同步 → 多线程共享必然数据竞争；
- **`Mutex`**：内部用 OS 同步原语，**加锁 / 解锁**保证互斥 → 专为多线程设计（见 [1.7](../1.7-mutex-rwlock/1.7-mutex-rwlock.md)）。

## 联动问答：scope 与 spawn 和本节的关系

### Q1：`scope` 和借用规则什么关系？

两层安全，**各司其职**：

1. **scope（[1.2](../1.2-scoped-threads/1.2-scoped-threads.md)）**：生命周期 — 可借栈变量，子线程必先 join，无悬垂引用；
2. **借用规则（本节）**：并发读写 — 不管是否 scope，都不允许「同时写 + 读/写」。

### Q2：`thread::spawn` + `move` 和数据竞争有关吗？

- 每个线程 **move 走一份所有权** → 不共享可变内存 → **天然无**数据竞争；
- 要**共享**同一份数据 → `Arc` + 锁 / 原子，回到本节与 [1.7](../1.7-mutex-rwlock/1.7-mutex-rwlock.md)。

## 三、新手易错点

1. **以为 scope 允许并发修改** — scope 只解决生命周期，并发 `&mut` 仍编译报错（[1.4.4](./1.4.4-borrowing-rules.md)）。
2. **把数据竞争当普通算错** — 是 **UB**，优化器下行为不可控。
3. **以为 `unsafe` 可以乱写** — 须手动保证规则，否则 UB / Unsound（[1.4.3](./1.4.3-ub-unsound.md)）。
4. **想在多线程间共享 `&mut T`** — 安全 Rust 做不到；共享可变用 **Mutex / 原子**。
5. **「多线程读同一 `&mut`」** — 不可能通过编译；要么共享读（`&T`），要么加锁。
