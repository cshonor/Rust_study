## 3.3.4 `Ordering::SeqCst`（顺序一致）

> 书 §3.3 · 前：[3.3.3 AcqRel](./3.3.3-acqrel.md) · 索引：[3.3-memory-order-options.md](./3.3-memory-order-options.md) · 后：[3.3.5 对照与选型](./3.3.5-ordering-compare-select.md) · Demo：[3.3-memory-order-options-demo.rs](./code/3.3-memory-order-options-demo.rs)

---

| 项 | 内容 |
|----|------|
| **保证** | 含 **Acquire/Release** 约束 + 所有 **SeqCst** 操作共有一条 **全局严格总顺序** |
| **效果** | 所有线程对 SeqCst 操作看到**同一条时间线** |
| **场景** | **极少** — 如书中「双旗帜」：需根据**两个不同原子**写入的**全局先后**做决策 |
| **审查** | 代码里大量 SeqCst 常是**警告信号**（未分析清需求或偷懒） |

---

## SeqCst 比 Release/Acquire 多什么？

**Release/Acquire** 只在你关心的 **「Release store → Acquire load 读到」** 这一对之间建立 HB。

**SeqCst** 额外要求：所有标记为 SeqCst 的原子操作，在所有线程眼中按**同一条全局顺序**发生 —— 即使它们在不同变量上、没有直接的 RA 配对。

---

## 何时才需要？

| 情况 | 建议 |
|------|------|
| 单标志 + 普通数据同步 | **Release/Acquire** 足够 |
| 自旋锁 | **Acquire** lock / **Release** unlock |
| 需比较 **原子 A** 与 **原子 B** 在全局谁先写 | 才考虑 **SeqCst** |
| 不确定 | 先写 RA，证明不够再加 SeqCst |

## 性能提示

- **x86** 上 SeqCst 与 RA 的机器码差距有时不大；
- **ARM** 等弱内存模型上 SeqCst 开销更明显；
- Rust 语义按**抽象模型**，不能因「我的 CPU 好像不乱序」就全用 Relaxed / 省略 Ordering。
