## 一、为什么需要内存顺序？

现代 **CPU** 与 **编译器** 为性能会做 **指令重排（Instruction Reordering）**。若对共享数据并发访问却不加约束，各线程看到的**有效顺序**可能与源码不一致。

| 场景 | 重排 |
|------|------|
| **单线程** | 通常**无害** — 只要保证程序**可观察行为**与源码语义一致 |
| **多线程** | **有害** — 「先写数据、再写标志位」可能被乱序，他线程看到标志已 set、数据仍未就绪 |

---

## 二、典型反例（伪代码）

```rust
// 线程 1：发布数据
data = 42;
flag.store(true, Ordering::Relaxed); // 编译器/CPU 可能把 store 提前

// 线程 2：订阅
if flag.load(Ordering::Relaxed) {
    print(data); // 可能读到未初始化的值！
}
```

- **`Relaxed`** 只保证对 **`flag` 这一原子变量** 的读写不可分割；
- **不**保证 `data = 42` 在 `flag.store` **之前**对其他线程可见。

→ 标志位当「数据已就绪」信使时，必须用 **`Release`（写标志）+ `Acquire`（读标志）** — 见 [3.3](./3.3-memory-order-options.md)。

---

## 三、谁在重排？

| 层级 | 行为 |
|------|------|
| **编译器** | 优化、删「无用」读写、调换无关语句顺序 |
| **CPU** | 乱序执行、Store Buffer、缓存 —— 不同架构（x86 / ARM）默认强弱不同 |

Rust 用 **内存模型 + `Ordering`** 在抽象层统一行为，让你**不必**为每种 CPU 写不同同步逻辑（详见 [3.2](./3.2-the-memory-model.md)）。

---

## 四、与第 2 章的衔接

| 概念 | 第 2 章 | 第 3 章（本节起） |
|------|---------|-------------------|
| 原子变量上的 DR | 原子操作消除 | 仍须选对 `Ordering` |
| 默认教学用 `Relaxed` | 只保单变量总修改顺序 | 多变量 / 普通内存同步不够 |
| 每次原子 API | 必须传 `Ordering` | 本节解释**为何必传** |

**一句话**：重排是性能手段；多线程下要用 **`Ordering` 限制重排**，保证可见性与顺序。
