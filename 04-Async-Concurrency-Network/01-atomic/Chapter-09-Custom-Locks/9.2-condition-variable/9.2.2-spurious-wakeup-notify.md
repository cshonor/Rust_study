## 虚假唤醒（Spurious Wakeup）

**`wait` 可能无明确 notify 也返回** —— 与 [1.8.4](../Chapter-01-Rust-Concurrency-Basics/1.8-parking-condvar/1.8.4-spurious-notify.md) 相同。

**必须**在 **`while` / `loop`** 里检查条件：

```rust
while queue.is_empty() {
    guard = not_empty.wait(guard).unwrap();
}
```

手写 Condvar **不能**省略循环 —— 否则优化 syscall 时易引入逻辑 bug。

---

## notify 最佳实践

| 规则 | 原因 |
|------|------|
| **先改共享状态，再 notify** | 唤醒者看到真条件 |
| **push 后 `drop(guard)` 再 notify**（可选但常见） | 减少持锁时间；消费者 re-lock 后再处理 |
| **单产出 → `notify_one`** | 避免惊群 |
| **广播式状态 → `notify_all` + while 重检** | 多等待者都需重查 |

→ [Condvar 贯通 §3–5](../Condvar与条件变量-贯通笔记.md)

---

## 惊群（Thundering Herd）

**`notify_all`** → 所有等待者 runnable → **同时抢 Mutex** → 仅一个成功，其余再睡。

优化线：

- 应用层：**`notify_one`** 当每轮只产生一份 work；
- OS 层：**futex REQUEUE**（第 8 章）；
- 第 9 章手写：精确 **wake 个数** + 等待计数。
