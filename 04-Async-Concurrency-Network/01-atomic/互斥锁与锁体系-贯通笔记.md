## 1. Mutex 底层原理与 Rust 的特殊设计

### 1.1 数据与锁绑定（相对 C/C++）

| C/C++ 常见形态 | Rust `Mutex<T>` |
|----------------|-----------------|
| 锁与数据分离，易忘记加锁 | **锁包裹数据**；必须先 `lock()` 得 `MutexGuard` 才能访问 |
| 裸指针绕过 | 类型系统强制「有锁才有 `&mut T`」 |

### 1.2 状态与阻塞

- 概念上：**未锁定 / 已锁定**。  
- 争用失败 → 线程**阻塞**，内核调度走 CPU（非忙等）。

### 1.3 底层 OS 支持（第 8 章展开）

| 平台 | 典型原语 |
|------|----------|
| Linux | **futex** |
| macOS | `os_unfair_lock` 等 |
| Windows | SRWLock、`WaitOnAddress` 等 |

高效阻塞依赖这些**用户态快速路径 + 内核等待**的组合。

## 2. 锁守卫（Guard）与作用域

### 2.1 RAII：无 `.unlock()`

- `lock()` → **`MutexGuard`**（实现 **`Deref` / `DerefMut`**，用起来像 `&mut T`）。  
- **`Drop` 时自动解锁** — 避免忘记释放。

### 2.2 经典陷阱：`if let` 延长锁（第 1 章必考）

**错误**（Guard 活到整个 `if let` 结束）：

```rust
if let Some(item) = list.lock().unwrap().pop() {
    process(item); // 仍持锁！
}
```

**正确**：

```rust
let item = list.lock().unwrap().pop();
if let Some(item) = item {
    process(item); // 锁已释放
}
```

→ 代码：`use_mutex.rs`、`use_mutex_guard_lifetime.rs`

## 3. 三种锁的选型

| 类型 | 行为 | 适用 | 注意 |
|------|------|------|------|
| **`Mutex<T>`** | 独占；阻塞休眠 | **绝大多数**共享可变状态 | 尽快 `drop(guard)` |
| **`RwLock<T>`** | 多读 **或** 一写 | **读多写少** | 详见 **[RwLock与读写锁体系-贯通笔记.md](./RwLock与读写锁体系-贯通笔记.md)**（写者饥饿、第 9 章写优先） |
| **自旋锁**（第 4 章） | 忙等 + `spin_loop` | 临界区**极短**、多核 | 持锁久 → 烧 CPU；见 `spin_lock.rs` |

**对比**：自旋 **无 syscall**；`Mutex` **有**休眠/唤醒成本。第 5 章 Channel 用 **park** 等消息，不用自旋等锁。

## 4. 异常、死锁与处理

### 4.1 锁中毒（Lock Poisoning）

- 持锁线程 **panic** → `Mutex` 标记 **poisoned**。  
- 后续 `lock()` 得 **`Err`** — 防止读到半写入的不一致状态。  
- 处理：`lock().unwrap_or_else(|e| e.into_inner())` 等（按业务决定是否恢复）。

### 4.2 死锁

**借用检查器不能防止死锁。**

| 类型 | 说明 |
|------|------|
| **递归加锁** | 同线程再次 `lock` 同一 `Mutex`；POSIX 可能 UB/永久阻塞；Windows CRITICAL_SECTION 可递归但在 Rust 会产生多个 `&mut T` → **UB**；安全实现**禁止**递归 |
| **规避** | 锁粒度最小；多把锁时**全局固定加锁顺序** |

## 5. 性能损耗与优化（第 9 章思路）

### 5.1 损耗来源

1. **缓存行争用**（多核抢同一 cache line）  
2. **内核 syscall**（休眠/唤醒昂贵）

### 5.2 优化 1：三态，减少无谓 syscall

| 状态 | 含义 |
|------|------|
| 0 | 未锁定 |
| 1 | 已锁定，**无等待者** |
| 2 | 已锁定，**有等待者** |

- **解锁**：仅当状态为 **2** 时才 `wake_one`。  
- **加锁失败**：仅当把 1→2 成功后才 `wait`。  
- **无竞争**路径：**零 syscall**。

### 5.3 优化 2：混合锁（Hybrid）

- 真正 `wait` 前先 **自旋** 若干次（如 100× `spin_loop`）。  
- 锁在窗口内释放 → 直接抢到，省调度。

### 5.4 优化 3：假共享（False Sharing）

- CPU 按 **缓存行（常 64B）** 缓存。  
- 多个无关锁挤在同一行 → 一核改锁使它核行失效。  
- **高频多锁**：`#[repr(align(64))]` 分散到不同行。  
- **锁与保护数据**：紧凑相邻有时更利于一次拉入缓存（视访问模式而定）。

## 7. 面试一句话

**Rust 的 `Mutex<T>` 用类型系统把数据和锁绑在一起，`MutexGuard` RAII 自动解锁；争用走内核阻塞；读多写少用 `RwLock`；极短临界区用自旋；panic 中毒、禁止递归死锁；高性能锁靠三态减 syscall、混合自旋、缓存行对齐防假共享。**
