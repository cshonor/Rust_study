# 1.1 What Is Async?（什么是异步？）

> Demo：[1.1-what-is-async/1.1-what-is-async-join-demo.rs](./1.1-what-is-async/1.1-what-is-async-join-demo.rs)

---

## 一、核心梳理（详细展开）

### 1.1.1 高视角：并发的本质

现代操作系统通过**在少量 CPU 核心上快速切换执行流**（线程/任务），让你感觉「成千上万个程序在同时跑」。

但 CPU 同一时刻**只能真正执行一个逻辑流**（单核视角）。所谓「并发」，本质是：

- **切得快** → 看起来并行
- **等待多** → 切换收益大（I/O 密集最合适）

### 1.1.2 同步阻塞（Synchronous Blocking）

**定义**：发起 I/O 后，线程原地阻塞，什么都不干，直到 I/O 完成。

- **比喻**：烧水时，你**站在水壶旁边死等**，水不开不走，这段时间不能做别的事。
- **代码特征**：

```rust
// 同步读文件（std::fs）
let content = std::fs::read_to_string("data.txt").unwrap();
// 这一行没读完，下一行不会执行
println!("done");
```

- **问题**：I/O 等待时**线程被浪费**；要高并发就得开很多线程，内存与上下文切换成本都很高。

### 1.1.3 异步非阻塞（Asynchronous Non-blocking）

**定义**：发起 I/O 后**先让出执行权**，去跑其它任务；I/O 就绪后再从断点继续。

- **比喻**：烧水时**点火后去洗菜**；水开（事件就绪）再回来关火。
- **核心**：利用 **Dead Time（等待时间）** 做有用的事。
- **Rust / Tokio 特征**：

```rust
async fn read_file() -> String {
    tokio::fs::read_to_string("data.txt").await.unwrap()
}
```

- **`.await`**：在 I/O 未就绪时**协作式让出**，运行时去 poll 其它任务。
- **就绪后**：任务被 **Waker 唤醒**，从 `.await` 之后继续执行。

### 1.1.4 `tokio::join!`：并发等待多个异步操作

**作用**：在**不额外为每个任务开 OS 线程**的前提下，**同时推进**多个异步任务，并等待**全部完成**。

| 行为 | 说明 |
|------|------|
| 启动 | 括号内各 Future **同时开始** poll |
| 结束 | **全部** `Ready` 后 `join!` 才返回 |
| 墙钟时间 | ≈ **最慢** 那个任务的耗时，而非相加 |

**示例**：`task_a` sleep 1s、`task_b` sleep 2s：

- 串行 `await`：约 **3s**
- `join!(task_a(), task_b())`：约 **2s**

## 二、摩尔定律与 I/O 现实（为什么异步越来越重要）

### 2.1 摩尔定律放缓

- 2010 年后，**单核频率**提升明显放缓（常见 ~3GHz 量级）。
- 算力增长更多靠**多核**，但核心数仍有限。
- **结论**：不能指望「CPU 越来越快」单独解决服务端扩展问题。

### 2.2 服务端的真实负载：I/O 密集

现代后端**大量时间花在等 I/O**：

- HTTP / RPC
- 数据库
- 磁盘与缓存（Redis 等）

这些操作**CPU 占用低**，但**延迟高**（毫秒～秒级）。

### 2.3 异步的目标

**不是**让 CPU 算得更快，而是：

- 在 I/O 密集场景下，用**少量线程**（常接近 CPU 核数）承接**海量并发连接/请求**；
- 提高**吞吐**（单位时间完成请求数）；
- 改善**响应**（减少排队与空等）。

一句话：

> 线程很贵，I/O 等待很闲 → **异步把闲时间变成有效工作量**。

## 三、本仓库示例说明

本章用 **`tokio::join!` + `tokio::time::sleep`** 模拟并发等待；真实 HTTP 并发见 **§1.6**（`1.6-improving-http-request-performance/`）。

### 文件：`1.1-what-is-async-join-demo.rs`

程序分两段：

1. **串行**：`task_a().await` 再 `task_b().await` → 约 3s。
2. **`join!`**：两任务并发 → 约 2s。

每段打印 `Instant::elapsed()`，便于对照。

### 运行方式

在已配置 Tokio 的 crate 中执行该文件，例如将 demo 拷为 `examples/ch01_1_1_join.rs` 后：

```bash
cargo run --example ch01_1_1_join
```

或在含 `tokio = { version = "1", features = ["rt-multi-thread", "macros", "time"] }` 的临时工程中 `cargo run`。

### 预期输出（量级）

```
=== 同步执行（总耗时 ~3s）===
Task A 开始
Task A 完成
Task B 开始
Task B 完成
同步总耗时：3002ms

=== 异步 join!（总耗时 ~2s）===
Task A 开始
Task B 开始
Task A 完成
Task B 完成
异步 join! 总耗时：2001ms
```

### 关键结论

| 方式 | 时间关系 | 线程 |
|------|----------|------|
| 串行 `await` | 相加 | 仍由运行时调度，但任务**未重叠** |
| `join!` | 取 max | 默认 `#[tokio::main]` 为多线程运行时，但**不要求**每个任务占一个 OS 线程 |
