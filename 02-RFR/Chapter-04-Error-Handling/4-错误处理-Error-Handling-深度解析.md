# 第 4 章：错误处理 (Error Handling) —— 深度解析

《Rust for Rustaceans》第 4 章从 **API 契约** 角度讨论失败：不仅要用 `Result` 与 `?`，更要思考**调用者将如何与你返回的错误交互**——是区分变体分别处理，还是接受「失败即可」的擦除类型。

---

## 1. 错误的表示方式 (Representing Errors)

常见两条路线：**枚举式（具体、可分支）** 与 **不透明 / 擦除（统一、易组合）**。

### 1.1 枚举式错误 (Enumeration)

当调用者**需要按错误种类采取不同行动**时，用 `enum` 表达失败原因。例如复制管道：输入端断开与输出端断开，后续策略可能完全不同。

让自定义错误在生态里「好用」，通常建议做到：

1. **`std::error::Error`**：提供 `source()` 等，便于链接因果链（与 backtrace/日志生态协作）。
2. **`Display` + `Debug`**：`Display` 宜**单行、小写、句末无多余标点**，便于嵌入更大日志；`Debug` 可承载更多诊断信息（路径、端口等）。
3. **尽量 `Send + Sync`**：否则在异步/多线程中传递错误会非常别扭。
4. **尽量 `'static`**：减少向上传播时与借用生命周期的纠缠；也便于与常见不透明错误类型组合。

### 1.2 不透明错误 (Opaque / Type erasure)

当调用者**通常不需要区分**上百种内部失败细节（例如「解码失败」作为整体对外），枚举会拖垮 API 表面积。此时可用**单一不透明类型**，常见形态如：

`Box<dyn std::error::Error + Send + Sync + 'static>`

- **组合性**：函数内对不同来源的 `Result` 使用 `?`，可通过 `From` 等统一到同一对外错误类型。
- **为何常带 `'static`**：擦除后具体类型不可见；加上 `'static` 等约束后，调用方仍可用 **`Error::downcast_ref`** 等尝试**向下转型**到确知的具体错误类型（需要时再取用）。

## 2. 特殊错误情形 (Special Error Cases)

### 2.1 `Result<T, ()>` 与 `Option<T>`

二者不等价：

- **`Err(())`**：强调「操作失败」，常隐含「需要处理或重试」等语义。
- **`None`**：表示「此处无值」，**未必**是异常或失败。

是否把 `Result<T, ()>` 简化为 `Option<T>`，应优先服从**语义**而非字数。

### 2.2 Never 类型 `!`

- 表示**不可能产生的值**；用于「分支上永不发生」的类型推断（如 `match` 中 `panic!` / `continue` 与正常分支汇合）。
- 若 API 为 `Result<T, !>`，编译器可知 **`Err` 不可达**（与 `Option`/`Result` 的收窄优化相关语境）。

### 2.3 `std::thread::Result` 与 `Box<dyn Any + Send + 'static>`

线程 join 的错误类型不是 `dyn Error`，而是 **`Any`**：因为 `panic!` 可携带**任意类型**的 payload，未必实现 `Error`。

## 3. 传播错误 (Propagating Errors)

### 3.1 `?` 与 `From`

`?` 在「提前返回」之外，还会按需要触发 **`From`** 做错误类型转换。因此可为库级枚举实现 `From<io::Error>` 等，让内部 `io::Result` 经 `?` 自然抬升到统一错误类型。

**`From` 与 `Into`（书中常见提示）**：

- 实现侧：优先实现 **`From<T>`**，标准库会提供 **`Into<U>`** 的 blanket 实现。
- 约束侧：泛型边界有时写 **`Into<Foo>`** 更宽（对调用方更友好）。

### 3.2 `Try` trait 与 `try { ... }` 块（语言演进）

- **`Try`**：`?` 可视为围绕 `Try` 概念的语法糖；除 `Result` 外，也与 `Option`、以及部分异步/`Poll` 语境下的「可提前失败」类型相关（具体以当前 Rust 版本与稳定性为准）。
- **`try` 块**：将 `?` 的**提前返回范围**限制在块内，避免「函数中间 `?` 直接退出导致末尾清理逻辑被跳过」；属于仍在推进的特性，使用前应查阅当前版本的 **stability / edition** 说明。

---

## 小结

错误是 **API 的一部分**：在「可分支的详尽枚举」与「可组合的不透明错误」之间按调用者需求取舍；用 **`Error` / `Display` / `Send` / `Sync` / `'static`** 提升可组合性；注意 **`Option` vs `Result`** 语义、`!` 与线程 **`Any`** 等特殊形态；传播层善用 **`From` + `?`**，并了解 **`Try` / `try` 块**对控制流与清理代码的影响。
