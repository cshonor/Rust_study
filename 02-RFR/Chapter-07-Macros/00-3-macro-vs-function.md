## 一、核心对比表（背诵重点）

| 对比维度 | **宏 Macro** | **普通函数 Function** |
|----------|--------------|-------------------------|
| **执行时机** | **编译期展开**，生成代码 | **运行期调用**，执行逻辑 |
| **参数规则** | 任意 **Token 流**，无固定类型/个数 | **固定签名**，类型与数量严格约束 |
| **代码能力** | 可生成 `struct` / `impl` / 语句块 | 仅执行业务逻辑，**不能生成新项** |
| **运行开销** | 无宏调用开销（展开后嵌入） | 有调用开销（可内联优化） |
| **调试体验** | 报错常指向展开后代码；需 `cargo expand` | 栈追踪清晰，定位直观 |
| **作用域可见** | 须**提前定义**或 `use` 导入 | 同模块可后定义（部分前向引用） |
| **类型检查时机** | **先展开**，再类型检查 | 定义处即类型校验 |
| **适用场景** | 样板、可变语法、derive、DSL | 逻辑复用、强类型、日常业务 |

---

## 二、取舍原则（核心口诀）

> **能用泛型 + trait + 函数实现的，绝不使用宏。**

宏增加编译复杂度与调试成本；仅当函数/泛型**无法满足**代码生成需求时才选用。

```text
1. 泛型 + trait + 函数  ── 能表达？→ 用
2. macro_rules!          ── 只是 token 形状/样板？→ 用
3. 过程宏                ── 须解析/变换 AST？→ 用，并接受编译成本
```

→ 详 [06 你真的需要宏吗](./06-so-you-think-you-want-a-macro.md)

---

## 三、宏的劣势

| 问题 | 说明 |
|------|------|
| **类型检查滞后** | 错误要等到展开后才暴露，信息常晦涩 |
| **编译变慢** | 每处宏调用执行生成逻辑；过程宏 + 大 derive 尤甚 → [05 代价](./05-cost-of-procedural-macros.md) |
| **IDE 支持弱** | 补全、跳转不如普通函数 |
| **隐藏逻辑** | 展开结果不可见时难维护 |

---

## 四、什么时候必须用宏？

| 场景 | 原因 |
|------|------|
| 批量生成重复代码 | ORM 实体、`derive` 序列化 — 手写 impl 不现实 |
| 自定义语法 DSL | SQL、路由、模板 — 函数签名固定不了 |
| 操作类型 / trait 实现 | 泛型无法覆盖的元编程（按字段名生成） |

**仍优先**：能拆成「宏只生成样板 + 函数承载逻辑」就不要把业务写进宏里。

## 五、调试工具

```bash
cargo install cargo-expand   # 首次
cargo expand                 # 查看宏展开后的 Rust 代码
cargo expand --lib           # 指定 crate 库根
```

展开结果是**等价源码视图**，帮助理解宏输出 — 不是编译器内部 Token 步骤。  
→ [第 13 章 · cargo-expand](../Chapter-13-Rust-Ecosystem/01-tools.md)

## 六、核心速记

1. 宏 = **编译期**生成代码；函数 = **运行期**执行。  
2. 宏能生成 **项（item）**；函数不能。  
3. 类型检查在宏**展开之后**。  
4. **函数优先**；宏是最后手段。  
5. 排错靠 **`cargo expand`**。
