# 第 6 章：测试 (Testing) —— 深度解析

《Rust for Rustaceans》第 6 章从「会写 `#[test]`」推进到**可维护、可证明、可度量**的测试体系：理解 Cargo/`rustc` 的测试脚手架、善用 `#[cfg(test)]` 与文档测试、用 Clippy/模糊测试/属性测试补洞，并用 **Miri**、**Loom** 与严谨的 **Benchmark** 对付并发与优化带来的「幽灵问题」。

---

## 1. Rust 内置测试机制

### 1.1 测试脚手架 (Test harness)

`cargo test` 会驱动 `rustc` 以 **`--test`** 等方式生成**测试二进制**：内置一个 `main`，负责发现并运行所有 `#[test]` 函数。

- **`harness = false`**（在 `[[test]]` 等目标上）：关闭默认脚手架，**自行实现 `main`**，完全控制执行顺序与集成/模糊测试等流程。

### 1.2 `#[cfg(test)]` 的高阶用法

不仅为了「不把测试代码编进 release」，更是为了**更好的隔离与可观测性**：

- **Mock / 替身**：在 `#[cfg(test)]` 下替换网络、数据库等依赖（可配合 `mockall` 等）。
- **测试专用 API**：需要断言内部不变式又不想污染公共 API 时，可在 `#[cfg(test)]` 下提供 `pub` 辅助方法供同 crate 测试使用。
- **测试记账**：如统计某路径调用次数，仅在测试配置下为结构体增加计数字段。

### 1.3 文档测试 (Doctests)

`///` 中的代码块会作为测试编译运行。

- **隐藏行**：行首 `# ` 的代码仍会参与编译/测试，但**不出现在**渲染后的文档中，适合隐藏样板。
- **`compile_fail`**：断言某段代码**必须不能通过编译**（例如验证某类型在特定边界下不满足 `Send` 等），是 Rust 文档测试的特色能力之一。

---

## 2. 额外工具：Lint 与测试生成

### 2.1 Lint（Clippy）

将 **Clippy** 纳入 CI：除风格外，大量 **correctness** 类 lint 能抓住「能编译但明显错误」的模式。

### 2.2 测试生成

- **模糊测试 (Fuzzing)**：对解析器等喂随机/变异输入直至 panic；生态常用 **`cargo-fuzz`**（libFuzzer 等后端）。
- **基于属性的测试 (PBT)**：用 **`proptest`** 等生成大量输入，对照**朴素但可信的参考实现**或不变量（Invariant），发现边界 bug。

---

## 3. 测试增强：Miri 与 Loom

### 3.1 Miri

**MIR 解释器**：`cargo miri test` 较慢，但能检测 **未初始化读、悬垂解引用、违反 stacked borrows / 别名规则** 等 UB，把 `unsafe` 问题从「生产偶现」拉到「测试可复现」。

### 3.2 Loom

并发 bug 依赖**线程交错与内存可见性**。**Loom** 接管 `Mutex`、原子等原语的语义模型，**系统化探索**多种调度顺序，暴露仅在极端交错下触发的竞争与死锁。

---

## 4. 性能测试 (Benchmarks) 常见陷阱

1. **方差 (Variance)**：单次计时易受降频、后台任务干扰；应用 **`criterion`** 等做多次采样与统计区间。
2. **编译器优化「测没了」**：未使用的结果可能被 DCE；对被测结果使用 **`std::hint::black_box`**，抑制对「测量目标」的过度优化。
3. **测到 I/O 或无关开销**：热循环里 `println!`、RNG 等会主导计时，掩盖真实算法成本。

---

## 小结

「能跑」不等于「正确」或「安全」。本章主线：**脚手架与 `cfg(test)` 组织测试** → **Clippy / Fuzz / PBT 扩覆盖** → **Miri / Loom 攻并发与 unsafe** → **`black_box` + 统计学基准** 让性能数字可信。这是从写几个单元测试走向**工程级 Rust** 的常走路径。
