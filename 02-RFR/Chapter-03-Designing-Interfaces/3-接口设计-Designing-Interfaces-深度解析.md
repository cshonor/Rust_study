# 第 3 章：接口设计 (Designing Interfaces) —— 深度解析

《Rust for Rustaceans》第 3 章讨论如何设计**既地道（idiomatic）又难误用**的专业级 API。核心可归纳为四条准则：**不意外 (Unsurprising)**、**灵活 (Flexible)**、**显而易见 (Obvious)**、**受限 (Constrained)**。类型系统不仅是内存安全工具，也是**表达契约、引导正确使用**的手段。

---

## 1. 不意外 (Unsurprising)

优秀接口应符合社区直觉：用户「盲猜」用法时也应大概率猜对。

### 1.1 命名与惯例

- 复用约定俗成的名字与语义，例如：`iter` 应对应**借用迭代器**；`into_inner` 应**消耗包装类型**并取出内部值。若 `iter` 却消耗所有权，会破坏预期、增加挫败感。

### 1.2 实现通用 Trait

用户期望类型在常见场景下「开箱即用」：

- **`Debug`**：几乎所有公开类型都应实现（哪怕实现较简），便于日志与调试。
- **`Send` / `Sync`**：异步与多线程的默认假设；若不能实现，应在文档中**明确说明原因与替代用法**。
- **比较与哈希**：`PartialEq`、`Eq`、`Hash` 等，对作为 `HashMap` 键、测试断言等场景很重要。
- **`Copy`**：**谨慎**。一旦 `Copy`，未来很难加入非 `Copy` 成员（如 `String`）；也可能带来「隐式复制」上的意外。仅在类型稳定且语义上确为「平凡小值」时优先考虑。

### 1.3 人体工程学：空白实现 (Blanket impls)

为自定义 Trait 考虑对 **`&T`、`&mut T`、`Box<T>`** 等的 blanket 实现，使用户在**仅持有引用**时也能调用 Trait 方法，减少 API 分叉。

---

## 2. 灵活 (Flexible)

在不过度牺牲表达力的前提下，**少强加不必要约束**，把选择权交给调用方。

### 2.1 泛型与接受「多种借用形态」

- 优先 **`impl AsRef<str>`、`impl AsRef<Path>`** 等，而非钉死 `&str`，便于传入 `String`、`PathBuf` 等。
- **借用 vs 拥有**：若函数**确实需要所有权**，应明确要求调用方传入拥有值，而不是收 `&T` 再在内部 `clone()`「偷偷」分配。若两种路径都可能，可考虑 **`Cow<'a, T>`** 等显式模型。

### 2.2 静态 vs 动态

- 在非热点路径上，**`dyn Trait`** 可减小二进制体积与编译压力；热点路径再权衡单态化与 vtable 成本。
- 设计 Trait 时尽量考虑**对象安全**，便于用户把行为放进 `Vec<Box<dyn Trait>>` 等异构集合。

### 2.3 资源与 I/O：`Drop` 的局限

- `Drop` **不能返回 `Result`**，也不适合做异步收尾。
- 对涉及 I/O 的类型，通常应提供显式 **`close(self) -> Result<_, _>`**（或 `shutdown` 等），在文档中说明契约；`Drop` 中可做「尽力而为」的兜底清理，但不应作为唯一错误通道。

---

## 3. 显而易见 (Obvious)

用**类型系统 + 文档**降低认知负担，让「正确用法」成为阻力最小的路径。

### 3.1 文档

除功能说明外，应明确：

- **Panic 条件**（何时 `unwrap` 会炸、哪些不变式被破坏会 panic）。
- **错误语义**（`Result` 各变体含义、可重试性）。
- **`unsafe` 的安全不变式 (safety invariants)**：调用方必须保证什么。

### 3.2 类型引导

- **语义化类型**：避免多个相邻 `bool` 参数；用 **`enum`** 或新类型表达状态与选项。
- **状态机 / 类型状态模式**：用 **ZST（零大小类型）** 作阶段标记，使非法状态**无法通过类型构造**（例如只有 `Rocket<Launched>` 才有 `accelerate`）。
- **`#[must_use]`**：对不应被忽略的返回值（含 `Result`、配置构建器等）提示调用方处理。

---

## 4. 受限 (Constrained)

在可演进前提下**收紧可见承诺**，避免用户依赖不稳定细节。

### 4.1 可见性

- 尽量少 `pub`；多用 **`pub(crate)`** 等缩小边界，保留未来重构空间。

### 4.2 `#[non_exhaustive]`

- 对面向用户的 `struct` / `enum` 使用 `#[non_exhaustive]`，避免外部**穷尽匹配**或依赖**全字段构造**，从而在 semver 兼容范围内增字段/增变体。

### 4.3 密封 Trait (Sealed trait)

- 通过「私有模块中的父 Trait」等惯用法，使 Trait **对外可用、对外不可 impl**，便于控制实现集合、做优化或保持不变式。

### 4.4 隐藏契约

- **重导出 (re-export)** 第三方类型、以及 **`Send`/`Sync` 等自动实现** 会构成隐式 API 承诺；可用**针对性测试**（如 `assert_send_sync` 辅助）锁定「公开类型始终满足某组约束」的意图。

---

## 小结

四条准则相互牵制：**不意外**降低学习成本，**灵活**减少过度约束，**显而易见**把正确用法写进类型与文档，**受限**保护长期演进。组合使用 Trait、泛型、可见性与模式（状态机、密封、`non_exhaustive`），可以做出「想写错都难」的工业级接口。
