# 1.2 Variables in Depth（变量深入）

> 所属：**Talking About Memory** · [← 章索引](./README.md)

同一套「变量」概念，在不同问题下用**两个互补模型**思考 — Rust 给开发者的「双重视角」。

## 新手从哪一层入门？

**从高层（数据流）开始。** Book、所有权、`&` / `&mut`、生命周期、函数传参、迭代器 — 全是这一层：

- 你写 `let r = &x`，**不必**先想「r 指向栈上哪条地址」
- 只需理解：**借来了读/写权限**；编译器检查生命周期、有没有和 `&mut` 冲突

高层模型**故意屏蔽**地址、槽大小、堆栈细节，让你先建立「值怎么流动、谁能在何时访问」的安全思维。

等你写 `unsafe`、看 LLVM IR、做布局/低延迟优化时，再切**底层（内存槽）**：变量 = 固定地址的一块存储，改值常是改槽内比特、地址不变。

| | 高层 · 数据流 | 底层 · 内存槽 |
|---|----------------|----------------|
| **视角** | 逻辑 / 编译器安全 | 硬件 / 物理存储 |
| **变量是什么** | 数据流里的**节点** | 栈或静态区里**固定大小的块** |
| **引用是什么** | 带规则的**读/写权限** | 往往涉及**地址**（机器上像指针） |
| **典型场景** | 日常业务、学语法、HFT 业务逻辑 | `unsafe`、裸指针、FFI、`04_Compilers-and-LLVM-Learning/04_Learn-LLVM-17`、极致优化 |
| **你要关心的** | 谁拥有、谁借用、能借多久 | 槽在哪、多大、地址变不变、比特怎么变 |

两层**不矛盾**：同一行 `let mut x = 5; x = 10;` 在高层是「绑定可变、重新赋值」；在底层是「同一槽、改里面存的数」。HFT 常见路径：**高层保安全 → 热点路径再切底层抠性能**。

## 高层模型：数据流 (Data Flow)

**何时用**：生命周期、借用、跨函数传引用。

- 变量是数据流的**节点**；借用检查器追踪值如何经引用在节点间流动。
- **冲突规则**：同一数据上若同时存在违反协议的**独占流**（`&mut`）与**共享流**（`&`），编译拒绝。

## 底层模型：内存槽 (Slots)

**何时用**：`unsafe`、原始指针、布局、`04_Compilers-and-LLVM-Learning/04_Learn-LLVM-17` 读 IR。

### 内存槽到底是什么？（纯 Rust，一句话）

**内存槽** = 编译器为某个 `let` 绑定划出的 **固定地址 + 固定大小** 的一块存储；从声明到离开作用域，**槽的位置和大小不变**，变的只是槽里存的比特。

```rust
let mut x: u32 = 5;
// 栈上开 4 字节槽，假设地址 0x1000
println!("{:p}", &x); // 0x1000
x = 10;               // 槽还在 0x1000，只是把里面的 5 改成 10
println!("{:p}", &x); // 还是 0x1000
```

- **槽** = 那块 4 字节的物理存储（place / location）
- **变量名 `x`** = 编译器符号表里「名字 → 这个槽」的标签（不是运行时指针）
- **`mut`** = 允许**改写这个槽里的内容**（不是移动槽、不是改槽的大小）

`String` 同理：`let mut s = String::from("hi")` 在栈上占一个**固定大小**的槽（64 位下通常 24 字节：`ptr + len + cap`）。`s.push('!')` 时 **s 这个槽的地址不变**，变的是槽里的 `len`（以及可能触发的堆扩容）。

- 变量对应栈（或静态区等）上的**槽位**，大小由类型在编译期确定。  
  例：`let x: usize` → 栈上固定大小槽位。
- **重新赋值**通常不改变「`&x` 指向该槽位」的地址语义（槽位仍在），变化的是槽内比特。
- 与「引用指向的**对象整体被移动/替换**」要在具体语境下区分——见 [04 所有权](./04-ownership.md)。

## 两模型如何配合

| 问题 | 优先模型 |
|------|----------|
| 「为什么这里不能同时有 `&` 和 `&mut`？」 | 数据流 |
| 「这个 `*const T` 指向的地址还合法吗？」 | 槽位 + 有效性 |
| 「优化器能否合并两次 `*r` 读取？」 | 数据流上的别名协议 → [05 共享引用](./05-shared-references.md) |

## `mut` 与 `&mut`：都是在「开修改槽位的权限」

**先回答最常见的问题：改槽里的内容，要不要 `mut`？**

| 改法 | 写法 | 需要什么 |
|------|------|----------|
| **直接改变量** | `let mut x = 5;` 然后 `x = 10;` | 声明时 **`let mut x`** |
| **通过借用改** | `let mut x = 5;` 然后 `*(&mut x) = 10;` | 声明时 **`let mut x`** + 借用时 **`&mut`** |
| **遮蔽（换一个新槽）** | `let x = 5;` 然后 `let x = 10;` | **不要** `mut`（这不是改旧槽，是开新槽换绑名字） |

所以：**只要你是「改同一个槽里的内容」，无论直接写变量名还是走 `&mut`，原变量都必须先 `let mut` 解锁。**  
`&mut` 不是 `mut` 的替代品，而是「在仍需要 `let mut` 的前提下，多一条临时修改路径」。

| | 作用 |
|---|------|
| **`let mut x`** | 允许**直接改这个槽**（`x = 10`），也允许借出 **`&mut x`** |
| **`let x`（不可变）** | 槽内比特**焊死**：不能 `x = 10`，也不能 `&mut x` |
| **`&mut T`（可变借用）** | 允许**通过这条借用路径**改**它指向的那个槽**；前提是原变量已是 `mut` |

```rust
let mut s = String::from("a");
s.push('b'); // 需要 s 是 mut：push 要改栈上 {ptr,len,cap}，可能还要改堆
```

**常见误区**：「不可变变量只能借 `&mut` 来改」——**不对**。`&mut` 的前提是 **`let mut` 已解锁**；若只有 `let x`，连 `&mut x` 都拿不到。

## 想「变」：三种不同机制

| 机制 | 写法 | 改的是啥 | 要 `mut` 吗 |
|------|------|----------|-------------|
| **变量遮蔽 (shadowing)** | `let x = 5;` 再 `let x = 10;` | **新开一个槽**，符号表把名字 `x` **改绑**到新槽 | **不要** |
| **重新赋值** | `let mut x = 5;` 再 `x = 10;` | **同一个槽**里的比特 | **要** |
| **可变借用改槽** | `let mut y = 5;` 再 `*(&mut y) = 10` | **同一个槽**里的比特 | **要**（原变量） |

```rust
// 遮蔽：新槽位，地址会变
let x = 5;
let p1 = &x as *const i32;
let x = 10;              // 新 let → 新槽；旧槽还在栈帧里，只是名字 x 不再指向它
let p2 = &x as *const i32;
// p1 != p2（通常）

// 可变引用：同一槽位，地址不变
let mut y = 5;
let p3 = &y as *const i32;
let r = &mut y;
*r = 10;
let p4 = &y as *const i32;
// p3 == p4
```

遮蔽后若还持有**指向旧槽**的引用，借用检查器会拒绝（生命周期/遮蔽规则）：

```rust
let x = 5;
let r = &x;
let x = 10;   // 遮蔽
// println!("{r}"); // 编译错误：r 仍指向已被遮蔽的旧绑定
```

→ 可变借用细节见 [06 可变引用](./06-mutable-references.md)；内部可变性（无 `mut` 也能改）见 [07](./07-interior-mutability.md)。

## 变量名是什么？像指针吗？

**不像运行时指针。** 变量名是编译器**符号表**里的条目：**名字 → 当前作用域内生效的那个槽位**。

- `let x = 5`：在栈上开槽 1，符号表 `x → 槽 1`
- `let x = 10`（遮蔽）：在栈上开槽 2，符号表 **`x → 槽 2`**（不是改槽 1 里的数）

可以想成字典：`key` = 变量名，`value` = 当前绑定的槽；遮蔽是**换绑 value**，不是改旧槽里的比特。

---

## 被遮蔽的旧槽何时释放？

**不会马上释放。** 栈按**栈帧 / 作用域**整块回收：遮蔽只让名字不再指向旧槽，旧槽仍占着栈空间，直到**整个 `{ ... }` 或函数结束**才和帧内其它槽一起弹栈。

这对性能是利好：遮蔽几乎零分配成本；也配合借用检查，避免「名字已换绑、旧引用仍指向旧槽」的悬垂访问（能编译过的代码里这类情况会被拦住）。
