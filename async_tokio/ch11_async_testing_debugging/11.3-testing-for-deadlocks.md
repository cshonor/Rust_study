# 11.3 Testing For Deadlocks（测试死锁）

> Demo：[11.3-testing-for-deadlocks/11.3-testing-for-deadlocks-timeout-demo.rs](./11.3-testing-for-deadlocks/11.3-testing-for-deadlocks-timeout-demo.rs)

## 死锁、活锁与「假超时」

| 现象 | 特征 | 测试上如何发现 |
|------|------|----------------|
| **Deadlock** | 任务互相等锁，无进展 | `timeout` 内未完成 → 失败 |
| **Livelock** | CPU 在跑但业务无进展 | 需断言业务指标 + timeout |
| **慢路径** | 最终会完成 | 调大 timeout 或优化 |

书中强调：测试用 **`tokio::time::timeout`** 不是根因分析工具，而是**判定「挂起过久」**的可靠信号。

## 典型写法

```rust
use tokio::time::{timeout, Duration};

#[tokio::test]
async fn work_finishes_in_time() {
    let result = timeout(Duration::from_millis(500), async {
        do_something().await
    })
    .await;
    assert!(result.is_ok(), "应在 500ms 内完成，否则疑似死锁或过慢");
}
```

## 与 `#[tokio::test]` 超时

`tokio::test` 可配 `#[tokio::test(flavor = "current_thread", start_paused = true)]` 配合 `advance`，避免真实等待（见 Tokio 测试文档）。

## 定位建议

1. 超时后打印 `tracing` span / 任务 dump（若启用）；
2. 缩小用例：两任务 + 两把锁的最小复现；
3. 检查 `Mutex` 锁顺序、跨 `await` 持锁（**禁止**）。

## Demo

本目录 demo 用 **10s sleep + 250ms timeout** 演示「超时即失败」；**不是**真死锁复现。真死锁用例应在测试里构造 `tokio::sync::Mutex` 交叉 `lock().await`。
