# Item 1 · 01 基础类型严谨性

← [Item 1 目录](../README.md)

> **总原则（背一句）**：强静态类型 · 编译期检查 · **无隐式数值转换** · **无 null**（用 `Option`）。

## 定义

Rust 是**强静态类型**语言，所有基础类型有**明确位数与符号**，类型安全由编译器强制保证。

## 整数类型

| 分类 | 类型 |
|------|------|
| 固定大小 | `i8` / `i16` / `i32` / `i64` / `i128`，`u8` / `u16` / `u32` / `u64` / `u128` |
| 平台相关 | `isize` / `usize`（等于指针宽度，32 或 64 位） |

命名口诀：`i` = 有符号（signed），`u` = 无符号（unsigned），后缀数字 = **位数**。例如 `i32` = 32 位有符号整数，`u64` = 64 位无符号整数。

### 固定大小 vs Python 式 `int`（HFT 视角）

| | Rust 标量 | Python `int` 等动态类型 |
|--|-----------|---------------------------|
| 大小 | **编译期定死**（如 `i32` 恒 4 字节） | 可**自动扩容**，对象头 + 堆分配 |
| 布局 | 栈上或结构体里**紧凑排列**，大小可预测 | 运行时决定，难做精细内存规划 |
| 性能 | 无装箱、无动态类型分派 | 便利，但热路径有额外开销 |

对 HFT 的意义：

1. **精准控制内存布局**——订单簿档位、行情字段用 `u32` / `u64` / `f64` 等你选定的宽度，结构体大小在编译期可知，利于缓存行对齐、减少 **cache miss**。
2. **延迟更可控**——热路径没有「整数变大再分配」的惊喜；算子和内存访问成本稳定。
3. **按需选型，不浪费**——价格用 `f64` 够用就不必上更大类型；成交量用 `u32` 够就不必默认 `u64`。

```rust
// 示意：紧凑行情档位（布局由你声明的类型决定）
struct Level {
    price: f64,   // 8 字节，IEEE 754
    qty: u32,     // 4 字节，够用的就不要 u64
}
// 编译器知道 Level 的确切大小，可内联、可向量化、可放进数组连续存储
```

不是「越小越好」，而是**业务范围内选最小够用的固定类型**，并在类型层面写清楚语义（也与 Item 1「用类型表达设计」一致）。

浮点固定宽度见 [02-scalar-types.md](./02-scalar-types.md)（`f32` / `f64` 同为定长，无 Python 式任意精度）。

## 关键规则

- ❌ **无隐式类型转换**（`u8 → u32` 也必须 `as` 或 `From`/`Into`）
- ❌ 不同类型**不能直接**比较、赋值（见下节）
- ✅ 必须**显式**处理截断、溢出（见下节）

```rust
let a: u8 = 10;
let b: u32 = a as u32; // 必须显式
// let c: u32 = a;     // ❌ mismatched types

let x: i32 = 42;
let y: i16 = x; // ❌ mismatched types
```

---

## 「不能直接比较」是什么意思？

**直接比较**：编译器**不会**先把较小类型隐式提升再比，不同类型不能写 `a == b`。

```rust
let a: u8 = 10;
let b: u32 = 10;
// a == b;  // ❌ mismatched types
```

**间接比较**：先**手动**转成同一类型，再比较——这是允许的，也是推荐做法。

```rust
let a: u8 = 10;
let b: u32 = 10;

// 方式 1：as 提升到较宽类型
assert_eq!(a as u32, b);

// 方式 2：From / Into（语义更清晰时优先）
assert_eq!(u32::from(a), b);
```

| | 直接 | 间接 |
|--|------|------|
| 含义 | 隐式统一类型后比较 | 显式转换后再比较 |
| 编译器 | ❌ 拒绝 | ✅ 允许 |
| 风险 | — | `as` 窄化可能截断，要心里有数 |

### HFT 场景

订单簿里成交量可能是 `u32`，累计成交量用 `u64`——对比、累加前**先统一类型**，避免混比混算：

```rust
let lot_vol: u32 = 100;
let cum_vol: u64 = 1_000_000;

// 先提升再比 / 再算
if cum_vol == lot_vol as u64 {
    // ...
}
```

## 溢出：debug / release 与显式处理

### 编译 profile 叫什么？

| 日常说法 | Cargo 命令 | 说明 |
|----------|------------|------|
| **开发 / debug** | `cargo run`、`cargo build`（默认） | 常叫 develop，和 **debug profile** 是一回事 |
| **发布 / release** | `cargo build --release`、`cargo run --release` | 生产优化，默认**不**为整数溢出插入 panic |

### 默认行为（未用 `wrapping_*` / `checked_*` 等时）

| Profile | `255u8 + 1` 这类运算 |
|---------|----------------------|
| **Debug** | 溢出 → **panic**（帮你抓 bug） |
| **Release** | 溢出 → **wrapping（环绕）**：`u8` 的 `255 + 1` → `0`，只保留低位 |

Release 下环绕 ≈ 你说的「补低位 / 回绕」，**不会**报错，数值会悄悄变错——实盘里价格、成交量算错就是真损失。

Rust 因此要求：**别靠 profile 碰运气**，用显式 API 固定语义，debug / release 行为一致。

### 三种显式写法（背一个选法）

| 方法族 | 溢出时 | 适用 |
|--------|--------|------|
| `wrapping_add` 等 | 环绕，得到「错但确定」的结果 | 你**故意**要模运算 / 环形计数 |
| `checked_add` 等 | 返回 `None` | 金融、HFT：**绝不能静默溢出** |
| `overflowing_add` 等 | `(结果, 是否溢出)` | 既要结果又要标志 |

---

## 例子 1：`wrapping_add`（确定要环绕）

成交量用 `u16` 做环形计数器，**业务上**就是 65535 之后回到 0：

```rust
let vol: u16 = 65535;
let new_vol = vol.wrapping_add(1);
assert_eq!(new_vol, 0); // debug / release 都是 0，行为一致
```

含义：向编译器声明——「我就要环绕逻辑」，不靠 release 的默认环绕碰运气。

---

## 例子 2：`checked_add`（HFT 容错，推荐）

价差、名义金额等**不能**静默溢出，用 `Option` 走错误分支：

```rust
let bid: u64 = 100;
let spread: u64 = u64::MAX;

match bid.checked_add(spread) {
    Some(ask) => println!("卖价是 {}", ask),
    None => {
        // 实盘：告警、拒单、熔断，别用诡异数值下单
        panic!("价差计算溢出，需告警");
    }
}
```

不管 debug 还是 release，`None` 时都会进你写的分支，不会出现「debug 正常、release 变负数/回绕」的双面行为。

### 和 `as` 截断的区别

- **`as` 变窄**：值太大时**截断位数**（如 `u32` → `u8`）。
- **算术溢出**：同一类型加减乘超出表示范围；处理用 `wrapping_*` / `checked_*`，不要和 `as` 混为一谈。

## 面试补一句

「无隐式转换」指数值/基础类型层面；另有 Deref 强制转换、trait 对象等，别和整数混谈。
