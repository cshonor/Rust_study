# 9.4 系统调用（SYSCALL）的封装

> 章索引：[第 9 章](./README.md) · 前：[9.3](./9.3-std-memory-mgmt.md) · 后：[9.5 · FD](./9.5-fd-and-handles.md)

## 一句话

用户态使用 **硬件/内核资源** 的唯一合法路径是 **SYSCALL** — **`std::sys`** 对 Linux **`syscall!`** / **`libc`**、Windows **Win32 API** 等做 **平台抽象**，上层 **`fs` / `net` / `process`** 均建在此 **安全封装** 之上。

---

## 原书要点

### 为何需要封装

| 裸 syscall | std 封装 |
|------------|----------|
| 寄存器/参数 **架构相关** | 统一 **`io::Result`** |
| **unsafe** | 边界集中 · API safe |
| errno / GetLastError | **`io::Error`** |

```text
std::fs::File::open
  └── sys::unix::fs::open  (path → OsStr → C bytes)
        └── libc::open  或  syscall(SYS_openat, …)
```

### 与 FFI 关系

**SYSCALL 层往往经 C 运行时（libc）** — 与 [9.1 · FFI](./9.1-rust-c-interop.md) 衔接。
