# 1.4 回顾

> 章索引：[第 1 章](./README.md) · 前：[1.3 `std` 库](./1.3-std-crate.md) · 后：[第 2 章 Rust 特征小议](../chapter02_rust_features_summary/README.md)

---

## 一句话

**`core` → `alloc` → `std`** 分层解耦：**底层可在无 OS 环境运行（内核态）**，**`std` 为日常用户态应用提供 SYSCALL 之上的丰富系统能力** — 使 Rust 适配各 CPU 架构与 OS 平台。

---

## 原书总结（三层对照）

| 库 | 定位 | 依赖 | 典型场景 |
|----|------|------|----------|
| **`core`** | 与 CPU 架构无关的可移植 **语言核心** | 无 | 内核 · 嵌入式 · 一切程序的基础类型 |
| **`alloc`** | **堆上** 智能指针与集合 | **仅 core** | 内核+分配器 · 无完整 OS 的动态内存 |
| **`std`** | **用户态** 标准库 · **SYSCALL** | core + alloc | 线程 · 文件 · 网络 · 普通应用 |

```text
         内核态 / 用户态
core  ────────────────────  可移植 · 无堆 · 无 OS
alloc ────────────────────  +堆 · 仍无 OS
std   ─── 仅用户态 ───────  +OS 资源 · 重导出整合
```

---

## 设计哲学（原书）

| 要点 | 含义 |
|------|------|
| **解耦** | 写 OS 不必拖入 `std`；写应用不必重复实现 `Option` |
| **可移植** | `core`/`alloc` 跨架构；`std` 按平台封装 SYSCALL |
| **unsafe 集中** | 约 **90% unsafe** 与 **`core::mem` / `core::ptr` / `core::alloc::Layout`**（及 `alloc` 分配实现）相关 — 第 3 章深入 |
| **读源码路径** | 应用 API → **`std`** → **`alloc`** → **`core`** |

---

## 依赖关系（读源码时）

1. **算法下沉** — `Vec` 遍历、比较在 **`core::slice`**；容量在 **`alloc::raw_vec`**。
2. **I/O 上浮** — `TcpStream::read` 在 **`std::sys`** 调平台 API。
3. **Trait 统一** — `Send`/`Sync`/`Drop` 在 **`core`**；`Mutex` 在 **`std`** 包 OS 锁。

---

## 选型速查（HFT / 系统编程）

| 需求 | 选层 |
|------|------|
| 固件、内核、零堆热路径 | **`core`**（± 自备 **`alloc`**） |
| 内存池、`Vec`、无文件网络 | **`no_std` + `alloc`** |
| 行情接入、多线程、mpsc、落盘 | **`std`** |
| Wasm 模块（07 专题） | 多为 **`wasm32` + 精简 std/alloc** — 对照 [07-Wasm](../../07-Programming-WebAssembly-with-Rust/chapter03_rust_wasm/3.2-installing-rust.md) |

## 读完后应能回答

3. 原书说 **90% unsafe** 落在哪三个模块方向？
