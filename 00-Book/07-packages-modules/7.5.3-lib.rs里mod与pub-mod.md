# 7.5.3 `lib.rs` 里 `mod` vs `pub mod`：私有登记 vs 对外开放

← [7.5 分文件](./7.5-将模块分割进不同文件.md) · [7.1.5 `lib.rs` 是 Crate 出口](./7.1.5-lib.rs是Crate出口.md) · [7.5.2 登记 vs 收拢](./7.5.2-pub-mod登记与pub-use收拢.md)

对应 compile-fail demo：`7.1-packages-crates-demo/bin_plus_lib/compile_fail/private_mod/`

---

## 结论（先给你）

> **完全没错。** 术语用 **crate**、**mod**；在 `lib.rs` 写 `mod 名字;` 还是 `pub mod 名字;`，就是在区分「只给本 crate 内部用」和「允许 crate 外访问」。

---

## 一、核心规则

在 **`lib.rs`（lib crate 的 root）** 里：

| 写法 | 编译器 | 本 crate 内其他 mod | crate 外（bin / 别的 Package / `tests`） |
|------|--------|---------------------|-------------------------------------------|
| **`mod 文件夹名;`** | ✅ 识别、正常编译 | ✅ 可访问（路径如 `crate::helper::…`） | ❌ **看不见** |
| **`pub mod 文件夹名;`** | ✅ 识别、正常编译 | ✅ 可访问 | ✅ 可 `use 包名::文件夹名::…` |

### 1. 只写 `mod 名字;`（不带 `pub`）

- **作用**：告诉编译器**把该文件夹/文件挂进模块树**，代码正常编译；同 crate 里的 `scanner`、`parser` 等可以调用。
- **限制**：**外部（其他 crate、依赖你的 bin）访问不到** — 相当于「crate 内私有模块」。

### 2. 写 `pub mod 名字;`（带 `pub`）

- 编译器同样识别、正常编译。
- 允许 **crate 外** 通过 `use 包名::模块名::…` 访问该子模块。

> 这和 [7.1.5](./7.1.5-lib.rs是Crate出口.md) 一致：`lib.rs` 是 **library crate** 的大门；不加 `pub` 的 `mod`，外面永远进不来。开门之后还要在子模块里 **`pub` 成员** — 见 [7.5.6](./7.5.6-pub-mod开门与pub成员拿物.md)。

---

## 二、clox 风格举例

目录（单 Package、一个 lib crate）：

```
src/
├── scanner/mod.rs
├── helper/mod.rs    # 工具辅助，只想内部用，不对外暴露
└── lib.rs
```

### 示例 1：私有模块（只内部用，对外隐藏）

`lib.rs`：

```rust
mod helper;        // 不带 pub — crate 内可见，crate 外不可见

pub mod scanner;   // 对外公开 scanner 子模块
```

| | |
|---|---|
| ✅ | 编译器识别 `helper`；同 crate 的 `scanner`、`parser` 等可 `use crate::helper::util_func` |
| ❌ | 外部代码、别的 crate **无法** `use my_clox::helper::…` |

### 示例 2：公开模块（允许外部使用）

```rust
pub mod helper;
pub mod scanner;
```

| | |
|---|---|
| ✅ | 编译器识别 + **crate 外** 可 `use my_clox::helper::…` |

## 三、模块**内部**的可见性（避坑）

父模块在 `lib.rs` 里写成 **`mod helper;`（私有）** 时，`helper/mod.rs` 里还要再分两层：

| 成员写法 | 谁能用 |
|----------|--------|
| `fn inner_func() {}` | 仅 **helper 内部** |
| `pub fn util_func() {}` | **同 crate 任意 mod**（须写路径 / `use`） |
| 子模块 `mod inner;` | 仅 helper 内；要跨 mod 用 → 子模块里再 `pub` |

`helper/mod.rs` 示例：

```rust
// 同 crate 里其他模块可以调用（scanner 等）
pub fn util_func() {}

// 仅 helper 模块内部能用
fn inner_func() {}
```

**要点**：`lib.rs` 的 `mod` / `pub mod` 管的是 **子模块整棵子树是否对 crate 外开放**；子树里每个函数、结构体、字段仍各自遵守默认私有 + `pub` 规则 → [7.2](./7.2-定义模块来控制作用域与私有性.md)、[7.2.2](./7.2.2-结构体与字段私有性.md)。

## 四、和 `pub use` 联动

| 情况 | 结果 |
|------|------|
| `lib.rs` 里 **`mod helper;`**（私有） | 即使 `helper/mod.rs` 里写 `pub use xxx::Y`，**crate 外依旧看不到** — `pub use` 只能把名字抬到 **当前模块出口**；父模块若私有，出口在 crate 边界外不可达 |
| 想对外只暴露一个函数、不暴露整个 `helper` 模块 | `lib.rs`：`mod helper;` + `pub use helper::util_func;` → [7.1.5 示例](./7.1.5-lib.rs是Crate出口.md) |

`pub use` 详解 → [7.4.2 `use` vs `pub use`](./7.4.2-use与pub-use.md) · 与 `pub mod` 分工 → [7.5.2](./7.5.2-pub-mod登记与pub-use收拢.md)

## 五、compile-fail 验证

`compile_fail/private_mod/`：

`lib.rs`：

```rust
mod math;   // 私有 — 不 pub
```

`math/mod.rs` 里有 `pub fn add(…)`，但 **模块 `math` 本身未从 lib 出口放出**。

`main.rs`（另一个 bin crate，依赖本 lib）：

```rust
use private_mod_fail::math::add;  // ❌ 编译错误：math 是私有模块
```

→ 函数 `add` 在 `math` 里虽是 `pub`，**子模块未 `pub mod` 登记到 crate 根**，crate 外仍不可见。

## 一句话总结

| 目标 | `lib.rs` 写法 |
|------|----------------|
| 只让编译器识别、**仅本 crate 内部用、不让外部访问** | `mod 名字;` |
| 需要对外开放，允许别处 `use 包名::名字::…` | `pub mod 名字;` |
