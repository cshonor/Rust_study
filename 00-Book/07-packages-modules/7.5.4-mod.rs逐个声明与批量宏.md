# 7.5.4 `mod.rs` 逐个声明 vs 宏批量：没有标准魔法

← [7.5 分文件](./7.5-将模块分割进不同文件.md) · [7.5.1 mod 与 use](./7.5.1-mod.rs里的mod与use.md) · [7.4.2 `use` vs `pub use`](./7.4.2-use与pub-use.md)

---

## 结论（先给你）

> **Rust 标准语法没有「一行批量导入所有 `.rs`」的魔法，只能一个个写 `mod 文件名;`。** 想少写重复代码，要靠第三方宏（`automod`、`mods`）—— 可选，生产代码仍以手写 `mod` 为主。

---

## 一、标准写法：`mod.rs` 里逐个声明（99% 项目）

目录（clox `scanner` 风格）：

```
src/scanner/
├── mod.rs
├── token.rs
├── lex.rs
└── error.rs
```

### `scanner/mod.rs`

```rust
// 1. 逐个声明同级 .rs（必须写，否则编译器不纳入模块树）
mod token;   // → token.rs
mod lex;     // → lex.rs
mod error;   // → error.rs

// 2. 内部简写（可选，仅本文件）
use self::token::{Token, TokenType};
use self::lex::scan;

// 3. 对外重导出（可选，推荐）
pub use token::{Token, TokenType};
pub use lex::scan;
```

| 要点 | 说明 |
|------|------|
| **每个 `.rs` 一行 `mod 名字;`** | 没有例外；编译器**不会**自动扫描目录 |
| **顺序** | 无所谓 |
| **`mod` vs `pub mod`** | 不加 `pub` = 子模块仅 crate 内经路径访问；`pub mod` = 外部可 `scanner::token` → [7.5.3](./7.5.3-lib.rs里mod与pub-mod.md) |
| **最佳实践** | **优先 `pub use` 暴露 API**，少 `pub mod` 内部文件（封装更好）→ [7.4.2](./7.4.2-use与pub-use.md) |

## 二、可复制 `mod.rs` 模板（照着填）

把 `scanner` / `token` / `lex` / `error` 换成你的模块名即可：

```rust
//! scanner 模块入口 — 登记子文件 · 内部 use · 对外 pub use

// ── 1. 登记：每个同级 .rs 一行 mod（标准写法，必会）──
mod token;
mod lex;
mod error;

// ── 2. 内部：简化本 mod.rs 里的路径（可选）──
use self::token::{Token, TokenType};
use self::lex::scan;

// ── 3. 对外：重导出必要 API（推荐，不暴露文件结构）──
pub use token::{Token, TokenType};
pub use lex::scan;

// ── 4. 模块级入口（可选）──
pub fn scan_source(source: &str) -> Vec<Token> {
    scan(source)
}
```

**子文件 `token.rs` 等** 里要用兄弟模块的类型，仍须**各自** `use crate::scanner::lex::…` 或全路径 — `mod.rs` 的 `use` **不继承** → [7.4.1](./7.4.1-use作用域仅限当前文件.md)

## 三、想「批量」？第三方宏（可选）

### 1. `mods` — 批量写 `mod`，仍要列名字

少写重复的 `mod` 关键字；**文件名还是要你列出来**，不能自动发现新文件。

`Cargo.toml`：

```toml
[dependencies]
mods = "1.0"
```

`scanner/mod.rs`：

```rust
mods::mods! {
    // 私有（默认）— 展开为 mod token; mod lex; mod error;
    token, lex, error;
    // 若要对 crate 外公开子模块：
    // pub token, lex;
}
```

等价于：

```rust
mod token;
mod lex;
mod error;
```

还支持分组可见性：

```rust
mods::mods! {
    pub a, b;
    pub(crate) c, d;
    token, lex, error;
}
```

| 优点 | 缺点 |
|------|------|
| 语法更短；可 `pub` / `pub(crate)` 分组 | 仍要维护模块名列表；多一个依赖 |

→ [crates.io/mods](https://crates.io/crates/mods)

### 2. `automod` — 按目录自动发现 `.rs`（常见于测试）

**不用逐个列名字**；宏扫描目录下所有 `.rs`（排除 `mod.rs` / `lib.rs` / `main.rs`），展开成一堆 `mod xxx;`。

`Cargo.toml`（测试目录常用 **dev-dependency**）：

```toml
[dev-dependencies]
automod = "1.0"
```

典型用法（路径相对 **含 `Cargo.toml` 的目录**）：

```rust
// tests/regression.rs
mod regression {
    automod::dir!("tests/regression");
}
// 展开 → mod issue1; mod issue2; …
```

要生成 **`pub mod`**：

```rust
automod::dir!(pub "tests/regression");
```

| 优点 | 缺点 |
|------|------|
| 丢新 `.rs` 进目录即参与编译，适合回归测试集 | 需依赖；**精细控制每个模块可见性较难**；路径相对 Cargo 根，和 `mod.rs` 同目录场景要设计好包一层 |

> **注意**：`automod` 的 API 是 `automod::dir!("相对 Cargo.toml 的路径")`，**不是**无参的 `automod::dir!()`。业务 `src/scanner/mod.rs` 生产代码里，业界仍以手写 `mod` 为主；`automod` 多见于 `tests/` 批量用例。

→ [crates.io/automod](https://crates.io/crates/automod)（dtolnay）

## 四、标准「干净 mod.rs」全貌

```rust
// 1. 声明所有同级文件（固定写法）
mod token;
mod lex;
mod error;

// 2. 内部导入（简化路径，仅 mod.rs 本文件）
use token::Token;
use lex::scan;

// 3. 对外重导出（暴露必要内容，隐藏 token.rs / lex.rs 路径）
pub use token::{Token, TokenType};
pub use lex::scan;

// 4. 对外入口
pub fn start() {
    let _ = scan("");
}
```

## 五、对照表

| 方式 | 要不要列文件名 | 标准库 | 适用 |
|------|----------------|--------|------|
| **`mod foo;` 逐个写** | ✅ 每个文件一行 | ✅ | **日常生产代码（必会）** |
| **`mods!`** | ✅ 列表里列名，语法更短 | ❌ 第三方 | 子模块多、想分组 `pub(crate)` |
| **`automod::dir!(…)`** | ❌ 扫目录自动 | ❌ 第三方 | 测试目录、回归用例批量挂载 |

---

## 一句话总结

| 记这个 | |
|--------|---|
| **标准（必会）** | `mod.rs` 里**每个 `.rs` 一行 `mod 名字;`**，无批量魔法 |
| **内部简写** | `use 名字::xxx;` |
| **对外暴露** | `pub use 名字::xxx;`（推荐，不暴露文件结构）→ [7.5.5 两位置何时 pub mod](./7.5.5-顶层lib与子mod里pub-mod何时写.md) |
| **懒人批量** | `mods`（仍列名）或 `automod`（扫目录，多用于测试） |
