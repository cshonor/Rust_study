## 核心结论（先背）

1. **默认 Drop 只管内存**；**手写 `impl Drop` 会替换默认逻辑**，自动调用时**先跑你的 `drop`**，再**自动回收各字段内存** —— 不是「叠加」，也不会丢内存释放。
2. **`drop` 只能被动调用**：变量**离开作用域**时编译器自动插入；**禁止**写 `value.drop()`。
3. **作用域 · 所有权 · 提前释放 · 容器/panic** 是核心考点；`Drop` + RAII = Rust 内存安全与并发安全的基础。

## 一、为什么每个类型都有 Drop，还要单独讲？

### 1. 两种实现形式

```rust
pub trait Drop {
    fn drop(&mut self);
}
```

| 形式 | 谁实现 | 做什么 |
|------|--------|--------|
| **隐式默认** | 编译器自动生成 | 递归释放字段、归还栈/堆内存（`i32`/`String`/`Vec`/普通 struct…） |
| **手写自定义** | 开发者 `impl Drop` | 关文件、释放 socket、解锁、FFI `free`、mmap 解除映射… |

👉 **默认只管内存；手写管一切外部资源。** 它是 RAII 在 Rust 的载体，和内存安全、并发、系统编程强绑定，不是 C++ 析构函数的简单翻版。

### 2. 硬性规则：`Copy` 与自定义 `Drop` 互斥

**一个类型不能同时 `Copy` + 手写 `Drop`。**

- `Copy` 语义 = 按位复制、旧值仍有效
- `Drop` 语义 = 销毁时释放资源、旧值不可再用

二者矛盾 → 设计类型时要先想清楚「能不能 Copy」。→ [15.3.1 §三](./15.3.1-Drop顺序与进阶场景.md#三copy-与自定义-drop-互斥)

## 二、默认 Drop vs 手写 Drop：替换，不是叠加

**一句话：没错，默认 Drop 只管内存；一旦手写 `impl Drop`，编译器用你的逻辑替换默认版本，自动调用时优先执行你的代码。**

### 1. 不手写 `impl Drop`（编译器默认）

编译器自动生成析构逻辑：

- 逐个**递归销毁** struct / enum / tuple 的每个字段
- 只做**内存回收**：栈出栈、堆释放
- **不会**关文件、解锁、断 socket 等额外操作

适合：`i32`、`Vec`、`String`、纯数据业务 struct。

### 2. 手写 `impl Drop`（完全替换默认路径）

| 阶段 | 发生什么 |
|------|----------|
| ① 变量出作用域 | 调用**你写的** `fn drop(&mut self)`（关 fd、解锁、FFI `free`…） |
| ② 你的 `drop` 返回后 | 编译器**仍自动 drop 各字段**（`Vec`/`String` 等内存照常释放） |

```text
变量出作用域 → 你的 drop（外部资源）→ 编译器 drop 各字段（内存）
```

**没有「默认 + 自定义一起跑」的叠加模式** —— 只有「全默认」或「先你的、再字段」。

### 3. 代码验证：`FileHandle`

```rust
struct FileHandle {
    fd: i32,           // 模拟 fd（外部资源）
    buf: Vec<u8>,      // 纯内存字段
}

impl Drop for FileHandle {
    fn drop(&mut self) {
        println!("手动执行：关闭 fd = {}", self.fd);
        // 只处理外部资源；不要手动 drop self.buf
    }
}

fn main() {
    let _fh = FileHandle { fd: 10, buf: vec![1, 2, 3] };
    println!("变量创建完成");
} // → 打印关闭 fd → 编译器自动释放 buf
```

→ demo：`cargo run -- custom`

### 4. 三个必记规则

1. **不要在 `drop` 里手动 drop 字段** —— 编译器会后补；手动调用 → **double free**
2. **替换，非叠加** —— 不手写 = 全走默认；手写 = 先你的逻辑，再字段内存
3. **`MutexGuard` 同理** —— 手写 `Drop` 解锁 → 返回后回收守卫自身内存 → [16.3](../16-fearless-concurrency/16.3-共享状态并发.md)

## 三、`drop`：调用时机与调用方式

### 核心：**永远被动调用，禁止主动调 `drop()`**

#### 1. 正常触发（最主流）

变量**离开当前作用域**时，编译器自动调用 `drop(&mut self)`。

作用域边界：`{}` 块、函数结束、`if`/`while`/`for` 块结束。

```rust
struct Foo;

impl Drop for Foo {
    fn drop(&mut self) {
        println!("Foo 被销毁");
    }
}

fn main() {
    {
        let f = Foo;
    } // 离开作用域 → 自动 drop
    println!("作用域结束");
}
// 输出：Foo 被销毁 → 作用域结束
```

#### 2. 其他自动触发

| 场景 | 行为 |
|------|------|
| **所有权 move 走** | 原变量位失效 → drop 原值（若未 Copy） |
| **容器销毁** | `Vec`/`Box` 先 drop 元素，再释放容器自身 |
| **`panic!` + unwind** | 栈展开时仍逐层 `drop`（除非 `abort`）→ [9.1 panic](../09-error-handling/9.1-panic-与不可恢复的错误.md) |

#### 3. 为什么不能写 `f.drop()`？

```rust
let mut f = Foo;
f.drop(); // ❌ explicit call to destructor `drop` is disallowed
```

1. **防止双重释放** —— 手动调一次，出作用域再调一次 → double free
2. **防止僵尸对象** —— `drop` 是 `&mut self`，对象还在作用域里却已清理，后续使用 = 未定义行为

→ demo：`cargo test explicit_drop_disallowed`（编译失败用例，见 lib 注释）

## 四、提前释放：合法替代方案

既然不能 `self.drop()`，Rust 提供：

### 方案 1：`std::mem::drop`（标准用法）

**唯一合法**的「提前触发销毁」：拿走所有权 → 值立刻失效 → 运行 `Drop`。

```rust
use std::mem;

let f = Foo;
println!("开始");
mem::drop(f); // 提前销毁
println!("结束");
// 再用 f → 编译错误
```

`mem::drop` 是**函数**（prelude 里），不是 trait 方法；通过 **move** 结束生命周期。

### 方案 2：缩小 `{}` 作用域

```rust
{
    let guard = lock();
    // 仅此处持锁
} // guard drop → 解锁
// 耗时逻辑放外面
```

→ ER Item 11 锁模式 · demo：`cargo run -- early` · `cargo run -- guard`

## 五、关键细节（底层/系统路线）

### 1. 同一作用域：后进先出（LIFO）

后创建的变量**先**被 drop（栈式）。

→ demo：`cargo run` §顺序 · [15.3.1 §一](./15.3.1-Drop顺序与进阶场景.md)

### 2. 嵌套类型与容器

- `struct Outer(Inner)`：**先** `Outer::drop`，**再** `Inner::drop`
- `Vec<T>`：元素按**声明顺序**（正序）drop，再释放堆缓冲区

→ [15.3.1 §二](./15.3.1-Drop顺序与进阶场景.md)

### 3. 与 Deref / 锁联动

`MutexGuard` / `RwLockGuard` **手写 `Drop`**：守卫出作用域 → 自动解锁。

```rust
{
    let guard = m.lock().unwrap();
    *guard += 1;
} // Drop → unlock
```

→ [15.2 §四 MutexGuard](./15.2-通过Deref将智能指针当作引用.md#四标准库典型实现天天碰) · [16.3 共享状态并发](../16-fearless-concurrency/16.3-共享状态并发.md) · **Atomics and Locks** 前置

### 4. `ManuallyDrop`（进阶）

暂时**禁用**自动 `Drop`（FFI、内存池、无锁结构）→ `std::mem::ManuallyDrop<T>`。

→ [15.3.1 §四](./15.3.1-Drop顺序与进阶场景.md#四manuallydrop底层--高性能) · demo：`cargo run -- manual`

### 5. `Drop` 里不能 `Result` / 不能 `.await`

- 释放失败 → 提供 `release() -> Result` 让用户主动调；`Drop` 只做**防泄漏兜底**
- 异步关闭 → 专门模式（Item 11 §5）

---

## 六、一句话总结

1. **默认 Drop = 仅内存**；**手写 = 先你的外部资源清理，再自动 drop 字段**。
2. **`drop` 被动调用**；禁止 `x.drop()`；提前释放用 `mem::drop(x)` 或 `{}`。
3. **替换非叠加**；`drop` 里**不要**手动 drop 字段。
4. **Drop + RAII** = 锁自动解、文件自动关 —— 并发与系统编程的安全基石。
