## 核心结论（先纠正误区）

**普通引用 `&T` 并不靠 `Deref` trait 做点号自动解引用** — 那是**语言原生**剥 `&` 层。

**智能指针**（`Box` / `Rc` / …）是普通结构体，**必须 `impl Deref`** 才参与点调用 / 传参时的自动解包。

日常标准库智能指针都已实现 `Deref`，所以 `&s` 和 `Box::new(s)` 点调用手感一样 — **底层机制不同**。

---

## 一、`.` 点号自动解引用（Deref 强制转换）

调用 `xxx.method()` 时，编译器**循环**解包，直到找到匹配方法（**不限层数**）。每一步只有两条路：

| 当前包装 | 解包方式 |
|----------|----------|
| **`&T` / `&mut T` 原生引用** | **语言内置**剥一层 `&` — **不调用 `Deref`** |
| **实现了 `Deref` 的类型**（Box / Rc / 自定义 MyBox…） | 调用 **`Deref::deref`** → `&内部类型` |

> 官方术语：**Deref coercion** — 既包含 trait 解包，也包含方法查找时对引用的原生剥离（二者在方法解析里交替进行）。

### 1. 原生引用：没有 `Deref`，语法自带剥 `&`

```rust
let s = String::from("hi");
let r1 = &s;    // &String
let r2 = &&s;   // &&String
let r3 = &&&s;  // &&&String

r1.len();  // ✅
r2.len();  // ✅
r3.len();  // ✅
```

```text
&&&String  → 剥 &  →  &&String  → 剥 &  →  &String  → 剥 &  →  String  →  len()
```

👉 **`&String` 未实现 `Deref`** — 全是语言内置逻辑。

### 2. 智能指针：必须 `impl Deref` 才参与点解包

```rust
let b = Box::new(String::from("hi"));
b.len();
```

```text
Box<String>  →  无 len  →  Deref::deref  →  &String
              →  剥 &（原生）  →  String  →  len()
```

**没实现 `Deref` 的自定义包装 — 点号不会自动拆：**

```rust
struct RawWrap<T>(T);
let w = RawWrap(String::from("abc"));
// w.len();  // ❌ 编译报错 — RawWrap 无 Deref
```

补上 `impl Deref for RawWrap<T> { type Target = T; … }` 后 `w.len()` 才通。

→ demo：`MyBox` 有 Deref vs 无 Deref 对比见 [15.2 §二](./15.2-通过Deref将智能指针当作引用.md#二最简示例mybox)

### 3. 多层混搭：`&&Box<Rc<String>>.len()` 完整链

```rust
use std::rc::Rc;

let nested = &&Box::new(Rc::new(String::from("deep")));
nested.len();
```

逐步拆解（方法查找循环）：

```text
&&Box<Rc<String>>     无 len
  → 剥 &              &Box<Rc<String>>
  → Deref (Box)       &Rc<String>
  → 剥 &              Rc<String>   （或保持 &Rc，视自动引用步骤）
  → Deref (Rc)        &String
  → 剥 &              String
  → len()             ✅
```

→ demo：`cargo run -- dot-paths`

### 4. 手动 `*` 同样分两条路

| 表达式 | 机制 |
|--------|------|
| `*(&s)` / `*rx` | **原生**剥引用 |
| `*box_val` | 语法糖 ≈ `*(box_val.deref())` — 走 **`Deref`** |

### 5. 何时发生 / 何时不发生

| ✅ 会自动解包 | ❌ 不会随便解包 |
|--------------|----------------|
| **`.` 方法调用** | 赋值 `let x = smart_ptr` |
| **函数传参**需要 `&T` | 多数模式匹配 |
| 字段访问（配合 deref） | 非引用上下文期望精确类型 |

可变场景配套 **`DerefMut`** — 规则与 `Deref` 对称 → [15.2.1 §二](./15.2.1-Deref嵌套可变与编译坑.md#二derefmut-完整规则)

### 6. 日常常识

1. 标准库 `Box` / `Rc` / `Arc` / `Ref` / `RefMut` / 锁守卫 **都已实现 `Deref`** — 无感使用。  
2. **自己造的封装**才要考虑要不要 `impl Deref`。  
3. 两种路线混用时，观感都是「点一下就行」，记住**底层不一样**。

## 二、`&mut T` 修改值：何时要手动 `*`？

### ✅ 不用手动 `*`（点语法走 §一 两条路线）

```rust
let mut p = Box::new(Point { x: 0, y: 0 });
p.x = 10;           // 字段：deref_mut + 原生
let mut v = Box::new(vec![1, 2]);
v.push(3);          // 方法：DerefMut 链
```

### ⚠️ 必须手动 `*`（整体赋值）

```rust
let mut b = Box::new(10);
*b = 20;            // 必须 * — 改堆上 i32，不是改 Box 变量

let mut r = &mut 42;
*r = 100;           // 原生 &mut 同理
```

| 写法 | 含义 |
|------|------|
| `ref.field = v` / `ref.method()` | §一 自动解包 |
| `*ref = new_value` | **整体替换** — 必须 `*` |

## 三、`unsafe` 与解引用的关系

| 操作 | `unsafe`？ |
|------|-----------|
| `&T` / `&mut T` 手动 `*` | ❌ |
| 智能指针 `*` / `.` / `Deref` coercion | ❌ |
| **裸指针** `*const T` / `*mut T` 手动 `*` | ✅ 必须 |

| 误区 | 正解 |
|------|------|
| 「解引用都要 unsafe」 | 只有**裸指针**手动 `*` |
| 「只有裸指针才 unsafe」 | 还有 `unsafe fn`、trait、`static mut`、`union` 等 |

→ [RFR Ch09 §02 裸指针](../../02-RFR/Chapter-09-Unsafe-Code/02-raw-pointers.md)

---

## 四、一句话速记

**点号自动拆包装：原生 `&` 靠语言自带，智能指针靠 `Deref`；标准库已写好 Deref，手感与普通引用一致。**

1. 整体赋值 → `*ref =` 必写。  
2. 安全引用 + 智能指针 → 与 `unsafe` 无关。  
3. 裸指针 `*` → 必须 `unsafe`。
