# 4.3.2 切片与借用规则（`&` / `&mut`）

← 主笔记：[4.3 切片](./4.3-切片slice.md) · 通用规则：[4.2 引用与借用](./4.2-引用与借用.md) · demo：[4.3-slices-demo](./4.3-slices-demo/)

本章只讲：**切片场景下，不可变借用和可变借用怎么互斥**（`first_word` / `clear` 那条线）。

---

## 两条铁则（复习）

| 借用 | 数量 | 能做什么 |
|------|------|----------|
| **`&T` 不可变** | 可多个 | 只读 |
| **`&mut T` 可变** | 同时最多一个 | 独占写 |

> **同一时刻**：不能既有存活的 **`&`**，又申请 **`&mut`** 指向同一块数据。

完整条文、NLL、`&mut` 互斥小例 → [4.2](./4.2-引用与借用.md)。

---

## 切片活着，能不能改？

**结论**：**切片存活期间，不能对「它借的那块数据」做需要 `&mut` 的修改**；等借用结束（出作用域或 NLL 判定最后使用之后）才能改。

> 不是「因为有切片类型就不能改」，而是 **切片 = 不可变借用 `&`**；有存活的 `&`，就不能再对同一块数据申请 `&mut`。

### 哪些操作被挡住

```rust
let mut s = String::from("hello");
let slice: &str = &s[..];   // 不可变借用 s 堆上的字节

s.clear();    // ❌ 要 &mut self
s.push('!');  // ❌ 要 &mut self
// slice 还活着 → s 的堆内容被「只读锁」
```

凡是要 **`&mut self`** / **`&mut T`** 的写操作（`clear`、`push`、`truncate`、`arr[i] = ...` 等），在切片借用未结束前都会被编译器拒绝。

### 「取消掉」= 借用结束（不是手动 API）

Rust 没有「手动取消切片」的接口。借用会在下面情况**自动结束**：

| 方式 | 例子 |
|------|------|
| **出作用域** | `{ let slice = &s[..]; }` 花括号结束 → 可以 `s.clear()` |
| **NLL（最后使用之后）** | `println!("{slice}"); s.clear();` — 若 `slice` 之后再没用到，编译器可能认为借用已结束 |

```rust
let mut s = String::from("hello world");
{
    let slice = &s[..];
    println!("{slice}");
} // slice 出作用域 → 借用结束
s.clear(); // ✓
```

### 能改 vs 不能改

| 情况 | 能不能改 |
|------|----------|
| 没有存活 `&str` / `&[T]` 借这块数据 | ✅ 可以 `&mut` 修改 |
| 有存活 `&str = &s[..]` | ❌ 不能 `clear` / `push` 等要 `&mut` 的操作 |
| 多个 `&str` 同时借同一块 | ✅ 可以（多个只读借用允许） |
| `let idx: usize = 5`（只是数字） | ⚠️ 编译能过，但 `clear` 后按下标切可能 panic → [4.3.1](./4.3.1-usize下标与悬空索引.md) |
| `let p: &str = "abc"`（静态字面量） | ✅ **不借 `s`**，不影响 `s` 的修改 |

### 数组切片同理

```rust
let mut arr = [1, 2, 3];
let sl: &[i32] = &arr[..];

arr[0] = 99; // ❌ 要 &mut arr，和 sl 冲突
// sl 出作用域或最后使用之后：
arr[0] = 99; // ✓
```

数据源在栈还是堆不影响规则——**看谁在被借、谁在被改**。

### 为什么要这样设计

若允许 `clear` 而 `slice` 仍指着堆里的旧字节 → `slice` 悬空。编译器在修改发生**之前**就拦住，比 C 里「改完才发现指针失效」安全。

## 案例：`word: &str` 挡住 `s.clear()`

```rust
fn first_word(s: &String) -> &str { /* &s[0..i] */ }

let mut s = String::from("hello world");
let word = first_word(&s); // 不可变借用 s 的堆字节
s.clear();                 // ❌ E0502
println!("{word}");
```

### `clear()` 在要什么

`fn clear(&mut self)` → `s.clear()` = 向 `s` 申请**整段堆数据的独占可变借用**。

### 报错链（一步步）

1. `word` 活着 → 占用 `s` 堆内存的**不可变借用**；
2. `clear()` 要 **`&mut s`**；
3. 不可变借用未结束 → 编译器拒绝。

### 何时能 `clear`

```rust
let mut s = String::from("hello world");
{
    let word = first_word(&s);
} // word 出作用域，不可变借用结束
s.clear(); // ✓
```

验证：`rustc --edition 2021 compile_fail/slice_blocks_clear.rs`。

---

## 关键：`&str` 借的是堆字节，不是「不算借用」

```rust
let mut s = String::from("hello world");
let word: &str = &s[0..5];
```

- `s`：栈上 ptr/len/cap，**字节在堆**；
- `word`：**胖指针指向 `s` 在堆里的那一段**。

借用检查看**谁在用这块堆内存**，不看表面类型：

| 写法 | 阻碍 `clear`？ |
|------|----------------|
| `let r: &String = &s` | ✅ |
| `let word: &str = &s[..]` | ✅ **一样** |
| `let idx: usize = 5` | ❌ 不拦（见 [4.3.1](./4.3.1-usize下标与悬空索引.md)） |

> **`&str` 切片 ≈ 给 `s` 挂上不可变借用**；`word` 活着就拿不到 `&mut` 去 `clear`。

若 `word` 仍指着堆里的旧内容就清空 → `word` 会悬空 → 编译器提前禁止。

---

## `&String` vs `&str`（效果）

```rust
let r: &String = &s;      // 借整个串对象
let word: &str = &s[..];   // 借堆里一段字节
```

对 **`clear`** 而言：**二者都会锁 `s` 的堆数据**，只要还有一个不可变借用存活，`clear` 就失败。

---

## 与 `usize` 的对比（一句话）

`idx: usize` **不产生借用** → `clear` 能通过 → 后面 `&s[0..idx]` 可能运行时越界。  
`word: &str` **产生借用** → `clear` 编译失败 → 从根上避免悬空切片。

---

## 小结

| 要点 | 一句 |
|------|------|
| 切片活着 | 不能 `&mut` 改它借的那块数据，等借用结束才能改 |
| 铁则 | 有存活 `&` 就不能再 `&mut` 同数据 |
| `clear` | 要 `&mut self`，与 `&str` 切片冲突 |
| `&str` | 借堆字节，和 `&String` 一样挡 `clear` |
| `usize` | 不借，规则管不到 |
