# 第 8 章：异步编程 (Asynchronous Programming) —— 深度解析

《Rust for Rustaceans》第 8 章从**编译器与运行时模型**解释 `async/await`：不只讲「怎么用某个 executor」，而是说明写下 `async` 时，背后如何变成**状态机**、为何需要 **`Pin`**、**`Poll` / `Waker`** 如何协作，以及 **阻塞与 `spawn`** 的系统性后果。若已读过《Async Rust》之类材料，本章可作为**内功**对齐术语与心智模型。

---

## 1. 为什么要异步？(What’s the Deal with Asynchrony?)

CPU 很快，程序大量时间花在 **等待**（I/O、网络、磁盘、人机事件）。

| 模型 | 直觉 | 代价 / 权衡 |
|------|------|----------------|
| **同步** | 调用线程阻塞直到 I/O 完成 | 简单；每连接一线程会放大阻塞成本。 |
| **多线程** | 阻塞时换线程跑别的 | 线程有成本；并行带来 **锁、原子、数据竞争** 等心智负担。 |
| **异步** | 等待时把线程让给其他任务 | 用 **`Poll::Pending` / `Ready`** 与调度协作，单线程或少量线程也能高并发；代价是 **状态机、`Pin`、取消与背压** 等复杂度。 |

Rust 异步的核心接口仍是 **`Future::poll`**：未完成则返回 **`Pending`** 并登记「稍后谁该叫醒我」。

---

## 2. 人体工程学：`async` 与状态机

手写 `Future` 往往是手写 **`enum` 状态机** + 手工保存跨 `.await` 的局部——易错难读。

- **`async fn` / `.await`**：语法糖，编译期把函数体**降阶**为实现了 `Future` 的匿名类型（可理解为**协程/生成器式**的让出点：在 `.await` 处切分状态）。
- **让出 (yield)**：当子 `Future` 未就绪时，当前 `Future` 返回 `Pending`；**局部变量**被打包进该状态结构体的字段，供下次 `poll` 恢复。
- **性能注意**：状态结构体可能**很大**（嵌套 `await` 会把子 `Future` 内嵌进父状态），带来栈压力或 memcpy；实践中常用 **`Box::pin`** 等把大 `Future` 挪到堆上并固定地址（与 `Pin` 配合）。

---

## 3. `Pin` 与 `Unpin`

**自引用**：`async` 体内可能出现「某字段引用另一字段」的布局；`.await` 后这种布局必须跨 `poll` 调用保持有效。

**移动的灾难**：若 `Future` 在内存中整体 **move**，被内嵌指针仍指向**旧地址**，则 UB。

**`Pin<P<Pointer<T>>>` 的契约**：对 **`!Unpin`** 的 `T`，`Pin` 与指针类型共同约束「不在 `poll` 期间非法移动 `T`」这类语义（精确规则以官方文档为准：`Unpin` 时可放松）。

- **`Unpin`**：类型**不依赖**「当前地址不变」这一不变式（如许多纯拥有型值），则自动/可标记为 `Unpin`；此时 `Pin` 的额外约束大多不构成负担。

---

## 4. 沉睡与唤醒：执行器与 `Waker`

若 `Pending` 后**忙轮询**，CPU 会烧干。

- **`Context` 与 `Waker`**：`poll(cx, …)` 中的 `Waker` 代表「当事件就绪时请通知执行器」。I/O 层（reactor：`epoll` / `kqueue` / IOCP 等）在就绪时调用 **`wake()`**。
- **「唤醒」≠ 立即运行**：`wake` 通常把任务标为 **runnable** 并推进执行器队列；何时再次 `poll` 由调度决定。
- **Poll 契约**：返回 **`Pending` 时必须** 确保将来某处会 **`wake`**（除非任务被永久取消且协议允许）；否则任务饿死。

---

## 5. 不要在异步运行时里阻塞

在 executor 工作线程上调用 **`thread::sleep`**、长时间 CPU 死算、同步阻塞 I/O 等，会**占用 worker**，使同池其它任务饥饿；线程池小时可造成**全局停顿**。

书中常见表述是**经验法则**：尽量保证 `poll` 路径上**长时间**不让线程独占（具体阈值依运行时与 SLA 而定，不必机械理解成「1ms 硬指标」）。

---

## 6. `spawn` 与顶层任务

若在单循环里顺序 `.await` 每个连接处理逻辑，则**并发度为 1**。

**`spawn`**：把处理每个连接的 `Future` 变成**独立顶层任务**，由运行时线程池调度，才能实现 accept 循环与业务处理的重叠。

---

## 小结

本章价值在于把 **`async` → 状态机 → 可能自引用 → `Pin`** 与 **`Pending` → `Waker` → 队列化 `poll`** 串成一条因果链。理解后，面对复杂的 `Future` 签名、生命周期与 `Send` 约束时，更容易判断是「状态太大」「错误阻塞」还是「唤醒缺失」类问题。
