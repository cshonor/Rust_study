# 第 7 章：宏 (Macros) —— 深度解析

《Rust for Rustaceans》第 7 章把宏定位为**让编译器替你生成代码**的元编程工具，并系统区分 **`macro_rules!` 声明宏** 与 **过程宏 (proc-macros)**：前者像受约束的「模式匹配 + 转录」，后者像在编译期运行的「小型代码生成器」。

---

## 1. 声明式宏 (Declarative Macros)

使用 **`macro_rules!`** 定义，调用带 **`!`**（如 `vec!`、`println!`）。直觉上接近**结构化**的查找/替换，但发生在 **TokenTree** 层面，而非 C 预处理器的纯文本拼接。

### 1.1 何时使用？

- **削样板**：多份结构相同、仅细节不同的代码（多组测试骨架、多类型相似 `impl` 等）。

**泛型 vs 宏（书中黄金法则）**：

- 若差异主要是**类型参数**能表达的，**优先泛型**（更符合惯例、错误信息更好）。
- 若差异在于**语法形状 / 重复片段的结构本身**无法被泛型优雅覆盖，再考虑宏。

### 1.2 如何工作？

- 宏在解析管线中展开；输入被切成 **TokenTree**，**不必**已是完整合法 Rust 程序，但须是合法 token 流。
- 展开结果必须是**合法 Rust 片段**（表达式/项/类型等），因此比「任意文本拼接」更难写出半截语法。

### 1.3 匹配器与转录器 (Matchers & Transcribers)

- **匹配器**：用片段分类符（如 `:expr`、`:ident`、`:ty`）与重复运算符（`*` / `+` / `?`）描述输入形状。
- **转录器**：匹配成功后，把捕获的**元变量**填入模板生成代码。

### 1.4 卫生 (Hygiene) 与 `$crate`

- **局部绑定**：声明宏对 **`let` 绑定**等通常是**卫生**的，宏内名字不会意外捕获调用点同名变量。
- **项级生成**：生成的 `fn` / `type` / `mod` 等会进入正常模块系统，调用方可见——否则很多「生成 API」的宏无法工作。
- **`$crate`**：在宏展开中引用当前定义宏的 crate，避免跨 crate 展开时路径漂移；也可配合 `::core::...` 等显式根路径。

---

## 2. 过程宏 (Procedural Macros)

过程宏是 **Rust 函数**：入参 **`TokenStream`**，出参 **`TokenStream`**，中间可用任意逻辑（常配合 **`syn`** / **`quote`**）解析与再生成代码。

### 2.1 三类过程宏

| 种类 | 形态 | 典型用途 |
|------|------|----------|
| **派生 (derive)** | `#[derive(Foo)]` | 为类型**追加** `impl` 等（不改变用户手写项的主体文本；可与辅助属性配合）。 |
| **类函数 (function-like)** | `foo!(...)` | 编译期复杂变换、DSL（如字面量编译期求值）。 |
| **属性 (attribute)** | `#[bar(...)]` 修饰项 | 框架路由、测试封装、字段级变换（如 `pin_project` 类生成）。 |

**注意**：过程宏的「卫生」**不自动**等同声明宏对局部名的处理；常需通过 **`Span`** 等 API 精细控制标识符解析位置（如 `Span::call_site()` / `mixed_site()` / `def_site()` 等语义随版本演进，以官方文档为准）。

### 2.2 代价 (Cost)

- **依赖重量**：`syn` + `quote` 等会显著增加 **proc-macro crate 自身**的编译负担。
- **生成代码体积**：不慎生成大量单态/冗长实现会拖慢下游 crate 的编译与优化。

### 2.3 Span 与错误体验

- 每个 token 带 **`Span`**（源位置关联）。
- 用 **`compile_error!`**、`proc_macro::Diagnostic`（若可用）等把错误**钉在用户写的位置**，而不是宏内部生成的匿名代码处，可大幅提升宏 API 的可用性。

---

## 小结

- **日常重复**：优先 **`macro_rules!`**，遵守「能泛型则泛型」。
- **框架级/强 DSL**：再上 **过程宏**，并主动管理 **编译时间** 与 **Span/错误信息**。
