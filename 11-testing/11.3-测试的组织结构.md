# 11.3 测试的组织结构

对应 demo 项目：`11-testing/11.3-test-organization-demo/`

Rust 社区通常将测试分为两类：
- **单元测试（unit tests）**：更小、更聚焦；通常和被测代码放在同一个 `src/*.rs` 文件中；可以测试私有实现细节。
- **集成测试（integration tests）**：站在“库的使用者”角度；位于项目根目录 `tests/`；只能测试公有 API；更强调多个模块组合后的行为。

---

## 单元测试（放在 `src/`，常用 `#[cfg(test)] mod tests`）

### `#[cfg(test)]` 的作用

`#[cfg(test)]` 告诉 Rust：**只有在执行 `cargo test` 时才编译/运行这段测试代码**，正常 `cargo build` 不会把它编译进去。

典型结构：

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        assert_eq!(2 + 2, 4);
    }
}
```

### 可以测试私有函数

Rust 的模块可见性规则允许在同文件的测试模块里通过 `use super::*;` 调用私有函数，因此你可以选择测试实现细节（也可以不测，Rust 不强迫）。

---

## 集成测试（放在项目根目录 `tests/`）

### `tests/` 目录的规则

- 在 crate 根目录创建 `tests/`（与 `src/` 同级）。
- `tests/` 下**每个 `*.rs` 文件都会被 Cargo 当作一个独立 crate 来编译**。
- 因为它们是“外部 crate”，所以需要显式 `use 你的库crate名;`，并且只能调用 `pub` 的 API。

运行某个集成测试文件（只跑该文件里的测试）：

```bash
cargo test --test integration_test
```

---

## 集成测试共享代码：`tests/common/mod.rs`

如果你把共享的帮助函数写成 `tests/common.rs`，Cargo 会把它当作一个集成测试 crate，导致测试输出里出现 `running 0 tests` 的那一段。

推荐做法是放到子目录模块里：
- `tests/common/mod.rs`

这样它不会作为独立集成测试 crate 出现在输出中，但仍可在其它集成测试文件里 `mod common;` 后使用。

---

## 二进制 crate 的集成测试

如果项目只有 `src/main.rs`（纯二进制 crate）而没有 `src/lib.rs`，那么 `tests/` 下的集成测试无法 `use` 你在 `main.rs` 里定义的函数。

常见工程实践：
- 把主要逻辑放到 `src/lib.rs`
- `src/main.rs` 负责解析参数/调用库逻辑

这样既便于复用，也便于写集成测试。
