## 一、本节在学什么

并非所有协议都适合「整包 `serde_json::from_slice`」：

- **HTTP 请求行**、SMTP 命令、自定义文本命令 → 用 **切分 + 模式匹配** 更自然
- **PEG（Parsing Expression Grammar）** 无歧义、可组合；Rust 里 **`nom`** 是常用的**解析器组合子**库

本节目标：能读懂并写出**最小**文本解析器（书中：**HTTP 请求行**），并知道如何调试失败输入。

---

## 二、PEG 与 `nom` 模型

**一句话**：把大语法拆成小解析器函数，用组合子**顺序拼接**或**分支选择**。

```text
输入 &str / &[u8]
  → 子解析器 A：消费前缀，返回 (剩余, 值)
  → 子解析器 B：在剩余上继续
  → …
  → Ok((剩余, 组装好的结果)) 或 Err(失败位置)
```

本仓库使用 **nom 7**（函数式组合子为主；书上部分示例为 macro 风格，概念相通）。

`Cargo.toml`：`nom = "7"`。

---

## 三、常用组合子

| 宏 / 函数 | 作用 | 示例场景 |
|-----------|------|----------|
| **`tag`** | 匹配固定字面量 | `tag("GET ")`、`tag(" HTTP/1.1")` |
| **`take_until1`** | 读到分隔符前（至少 1 字节） | 方法名读到空格、路径读到空格 |
| **`tuple`** | 顺序执行多个解析器 | 方法 + 空格 + 路径 + … |
| **`alt`** | 多分支选一 | `GET` \| `POST` |
| **`many0` / `many1`** | 重复 0 次 / 至少 1 次 | 重复 header 行 |

返回类型 **`IResult<I, O>`**：

- `Ok((remaining_input, output))` — 成功
- `Err(...)` — 失败，含**失败偏移**与剩余输入

---

## 四、HTTP 请求行（书中示例）

解析目标：

```http
GET /api/users HTTP/1.1
```

得到：**方法**、**URL 路径**、**HTTP 版本**；行尾常为 `\r\n`。

### 仓库实现

```rust
use nom::{
    bytes::complete::{tag, take_until1},
    sequence::tuple,
    IResult,
};

fn parse_request_line(input: &str) -> IResult<&str, (&str, &str, &str)> {
    let (input, (method, _, path, _, version)) = tuple((
        take_until1(" "),
        tag(" "),
        take_until1(" "),
        tag(" "),
        take_until1("\r\n"),
    ))(input)?;
    Ok((input, (method, path, version)))
}
```

```rust
// 用法
let line = "GET /api/users HTTP/1.1\r\n";
let (rest, (method, path, version)) = parse_request_line(line).unwrap();
// method=GET  path=/api/users  version=HTTP/1.1  rest=""
```

仓库对应：[4.2-parsing-textual-http-line-demo.rs](./code/4.2-parsing-textual-http-line-demo.rs)

## 五、调试 `nom`

解析失败时，看**剩余输入**比猜规则快。

| 手段 | 作用 |
|------|------|
| 打印 `Err(e)` | nom 7 的 `VerboseError` 可定位片段 |
| **`dbg!` / 单测** | 对每个子解析器喂最小用例 |
| **`cargo expand`** | 展开宏，看清生成的解析链（macro 风格时） |

策略：**先写最小子解析器单测**（只测 `tag("GET ")`），再 `tuple` 组合。

## 六、对照：`nom` vs `pom`

| | **nom** | **pom** |
|---|---------|---------|
| 风格 | 宏 + 组合子 / 函数式 API | **Trait + 运算符**（`\|` 或，`*` 顺序） |
| 生态 | HTTP、二进制协议示例多 | 更接近「用 Rust 写解析表达式」 |
| 学习曲线 | 宏调试较难 | 运算符较直观 |

两者都是 PEG；**选一种深入即可**。文本行解析 `nom` 资料更多。

---

## 七、对照：文本解析 vs Serde

| | **本节（nom 文本）** | **[4.1 Serde JSON](../4.1-serde-serialization/4.1-serde-serialization.md)** |
|---|----------------------|------------------------------------------------------|
| 输入 | 空格分隔行、命令动词 | 完整 JSON 文档 |
| 典型协议 | HTTP 行、SMTP、自定义 CLI | REST body、配置同步 |
| 扩展 | 加 header 块、多行 `many0` | 加字段、改 `rename` |

实际服务常混合：请求行用 `nom`，body 用 Serde（或 `hyper` 代劳）。

---

## 八、新手易错

1. **`complete` vs `streaming`**：本 demo 用 `bytes::complete::*`，假设输入**已全部在内存**；TCP 流式应用选 `streaming` 并处理 `Incomplete`。
2. **分隔符写错**：HTTP/1.1 行尾是 `\r\n` 不是单独 `\n`；`take_until1(" ")` 要求至少一个非空格字符。
3. **空格与 `tag(" ")`**：方法后通常一个空格；多个空格需 `space1` 或 `many1(char(' '))`。
4. **成功后的 `rest`**：若 `rest` 非空，说明只消费了一行，后面还有 header/body 要接着解析。
5. **用 Serde 硬解析请求行**：可行但别扭；行级文本协议用 PEG 更清晰。
