## 1. 协程与生成器（Coroutines and Generators）

### 动机

**Future + 组合子** 易出现：

- 控制流难跟、类型嵌套深  
- 调试栈不直观  

**协程 / 生成器**：在函数体内 **`yield`** 挂起，恢复时从断点继续，写法更接近同步代码。

### 无栈 vs 有栈

| 类型 | 状态保存 | 特点 |
|------|----------|------|
| **无栈（Stackless）** | 编译为状态机（如 `async fn`） | 轻量；Rust **`async/await` 即此类** |
| **有栈（Stackful）** | 独立协程栈 | 可在任意深度 `yield`；资源占用更高，调用方更「像线程」 |

```text
async/await（无栈）  ←  本书 Ch7 主线、async_tokio/
may / 用户态线程（有栈）  ←  本节、Go 风格
```

### Nightly：`Generator` + `yield`

标准库实验特性（需 **nightly** + `#![feature(generators)]` 等）：

- **`Generator`** trait  
- **`yield`** 产出下一项  

书中用生成器实现 **Collatz 序列** — 与 `Iterator` 对比学习「谁驱动谁」。

**仓库对照**：[`async_tokio/ch05_coroutines_generators/`](../../../02-async_tokio/ch05_coroutines_generators/)（另一本书的协程章，概念互通）。

## 2. 使用 May 处理协程

**`may`**：Rust 生态中较成熟的 **有栈协程** 库。

| 能力 | 说明 |
|------|------|
| **`go!` 宏** | 类似 Go 的 `go`，启动协程，写法简单 |
| **`may_minihttp`** | 基于 may 的迷你 HTTP 栈 |

与 **Hyper + Future** 对比（书中观点）：

| | Hyper / Tokio | may_minihttp |
|---|---------------|--------------|
| 代码结构 | 显式 Future、组合子 | 更接近同步写法 |
| 性能 | 成熟、广泛使用 | 书中基准表现优异（视场景而定） |
| 生态 | 主流 | 小众，选型需评估维护度 |

**学习价值**：理解「有栈协程」如何隐藏异步；**生产 Web** 仍以 **Tokio + hyper/axum** 为主流。

## 3. Awaiting the future（`async/await` 抽象）

### 问题：`futures` 手写时代

多层 **`and_then`** → **回调地狱** + 复杂关联类型。

### 解法：`futures-await`（书中，历史）

| 宏 | 作用 |
|----|------|
| **`#[async]`** | 函数体变为异步，返回 `impl Future` |
| **`await!`** | 等待 Future 完成 |

允许在异步函数里使用 **`?`** 传播错误，**线性**控制流。

### 现代：语言内置 `async/await`（Rust 1.39+）

```rust
async fn fetch() -> Result<String, Error> {
    let body = client.get(url).await?.text().await?;
    Ok(body)
}
```

**`futures-await` 已被标准语法取代** — 读附录时把其视为 **async/await 的预研版** 即可。

**仓库对照**：

- [Ch07](../../stage07_tokio_async_net/)  
- [`async_tokio/ch02`](../../../02-async_tokio/)  
- [`ch10_noop_waker_block_on`](../../../02-async_tokio/ch10_dependency_free_async_server/) — 理解谁 `poll` 你的 Future

## 4. 数据并行（Data parallelism）

### 任务并行 vs 数据并行

| | 任务并行 | 数据并行 |
|---|----------|----------|
| 划分方式 | 不同**任务** | 同一数据集**分块**独立处理 |
| 典型 | 线程、`async` 多连接 | 数组/map/filter 并行 |
| 本书工具 | Tokio、线程 | **`rayon`** |

### `rayon`

| API | 用途 |
|-----|------|
| **`rayon::scope`** | 分叉子任务，作用域结束自动 join |
| **`par_iter()`** | 并行迭代器，替代 `iter()` |
| **`par_sort`** 等 | 并行排序、划分 |

```rust
// 概念
use rayon::prelude::*;
let sum: i32 = huge_vec.par_iter().map(|x| expensive(*x)).sum();
```

书中示例：

- 并行 **二分查找**（`scope` 分治）  
- **`par_iter`** 过滤集合  
- **`rustc-test` / criterion** 基准：海量计算时并行明显快于顺序  

**与网络的关系**：解析大文件、批量校验、压缩、加密等 **CPU 密集** 步骤可用 rayon；**I/O 等待** 仍用 async，二者常组合（`spawn_blocking` + rayon）。

**仓库对照**：[`atomic/rayon/`](../../../01-atomic/rayon/) · 根 README 中 `study_rayon`。

## 5. 使用 Pest 进行解析

[Ch04](../../stage04_serde_parse/) 介绍了 **`nom`**（组合子 + 宏）；附录给出 **PEG 的另一条路：`pest`**。

| | **nom** | **pest** |
|---|---------|----------|
| 规则写法 | Rust 宏 / 组合子函数 | 独立 **`.pest` 文法文件** |
| 生成 | 编译期展开宏 | **`pest_derive`** 派生解析器 |
| 错误信息 | 依赖调试技巧 | 书中称 **失败提示更友好** |
| 维护 | 复杂协议需仔细拆子解析器 | 文法集中、易读 |

### 工作流

1. 编写 `grammar.pest`（PEG 规则）  
2. `#[derive(Parser)]` + `#[grammar = "..."]`  
3. 对输入 `pair` 树做语义动作（构建 AST）

书中用 Pest **重写 HTTP 请求行解析** — 与 Ch04 `nom` 示例对照，体会风格差异。

**选型**：团队熟 PEG 文法 → pest；纯 Rust 组合、嵌入式 → nom；二进制 → 仍常 nom `bits!` 或手写。

## 6. 其他实用工具（Miscellaneous）

### `bitflags`

网络协议常出现 **C 风格位标志**（2 的幂）：

```rust
bitflags::bitflags! {
    struct Flags: u32 {
        const FIN = 0b001;
        const SYN = 0b010;
        const ACK = 0b100;
    }
}
```

- 类型安全、支持 `|` `&` `contains()`  
- 用于 TCP 标志位、socket 选项、自定义协议头

### `url`

解析 URL 字符串为结构化字段：

| 字段 | 示例 |
|------|------|
| scheme | `https` |
| host | `example.com` |
| port | 可选 |
| path / query / fragment | `/a?b=1#frag` |
| username / password | 可选（不推荐明文存密码） |

客户端构造请求、爬虫、重定向处理、与 **reqwest** 联用时先 `Url::parse` 再改 query。
