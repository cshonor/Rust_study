## 一、本节在学什么

在 [Ch7](../stage07_tokio_async_net/7.2-heading-to-tokio/7.2-heading-to-tokio.md) 的 `TcpListener::accept().await` 之后，把 **`TcpStream` 升级为 TLS 流**，再跑 HTTP（Hyper/axum）或自定义协议。

书中 **`tokio-tls`** + **`TlsAcceptor`** → 现代 **`tokio-rustls`** / **`tokio-native-tls`**。

---

## 二、异步 TLS 升级路径

```text
TcpListener::bind(...).await
    ↓
accept().await  →  TcpStream
    ↓
tls_acceptor.accept(stream).await  →  TlsStream
    ↓
hyper / axum / 自定义 AsyncReadExt
```

| 现代组合 | 说明 |
|----------|------|
| **`tokio-rustls`** | Tokio + **rustls**（推荐，纯 Rust） |
| **`tokio-native-tls`** | Tokio + 系统 OpenSSL/Schannel |

```rust
// 概念（伪代码）
let tls_stream = tls_acceptor.accept(tcp_stream).await?;
// 在 tls_stream 上 serve HTTP 或读写字节
```

---

## 三、书中要点：PKCS#12 与 native-tls

**`tokio-tls` → `native-tls`** 加载服务端身份时，常要求 **PKCS#12** 打包（证书 + 私钥 + 链），不能随意只丢两个 PEM。

| 栈 | 常见证书格式 |
|----|--------------|
| native-tls 服务端 | `.pfx` / `.p12` |
| rustls 服务端 | PEM cert + PEM key（或 DER） |

```bash
openssl pkcs12 -export -in cert.pem -inkey key.pem -out server.pfx
```

选型不同，部署步骤不同 — 读文档时确认 **你用的 TLS crate**。

---

## 四、典型生产组合

```text
公网 HTTPS API：
  axum + hyper + tokio-rustls
  证书：ACME fullchain.pem + privkey.pem

内网 mTLS 实验：
  rustls 双向证书 + tokio-rustls

遗留 Windows/IIS 互操作：
  tokio-native-tls + PKCS#12
```

与 Ch6 Rocket `tls` feature：配置证书路径即可；底层可能是 native-tls 或 rustls，视 feature 而定。

仓库 demo 打印栈要点：[8.1.3-securing-tokio-overview-demo.rs](./code/8.1.3-securing-tokio-overview-demo.rs)

## 五、新手易错

1. **先掌握 Ch7 再加 TLS**：同时踩 async 与证书两个问题难排查。
2. **HTTP 与 HTTPS 混用端口**：TLS acceptor 只应接 443（或配置的 TLS 端口）。
3. **证书链不完整**：移动端/旧客户端 handshake 失败。
4. **在 TLS 流上再包一层错误协议**：升级后读写的是**密文通道**，勿对明文 socket 写 HTTP。
5. **热重载证书**：续期后需 reload acceptor 或进程重启（视框架支持）。
