## 1. 远程过程调用（RPC）与 gRPC

### RPC 简介

**RPC（Remote Procedure Call）**：像调用本地函数一样，通过网络调用远端机器上的过程/方法。

| 概念 | 说明 |
|------|------|
| 调用方 | 客户端 stub：序列化参数 → 发请求 → 收响应 → 反序列化 |
| 服务方 | 服务端 skeleton：收请求 → 执行业务 → 返回结果 |
| 价值 | 微服务拆分、语言无关接口（配合 IDL） |

你不再手写「第 3 字节是命令号」，而是**按接口定义**生成客户端/服务端代码。

### gRPC 与 Protocol Buffers

**gRPC** 是 Google 推出的 RPC 框架，本书重点：

| 层次 | 技术 |
|------|------|
| 传输 | 默认 **HTTP/2**（跑在 TCP 上） |
| 能力 | 一元调用、**服务端流/客户端流/双向流** |
| IDL | **Protocol Buffers（`.proto`）** |
| 载荷 | protobuf **二进制**编码（比 JSON 更紧凑、有 schema） |

```protobuf
// 概念示例：打车应用
syntax = "proto3";

message CabLocationRequest {
  double latitude = 1;
  double longitude = 2;
  string cab_id = 3;
}

service FooBarService {
  rpc RecordLocation(CabLocationRequest) returns (Empty);
  rpc GetNearbyCabs(NearbyRequest) returns (NearbyResponse);
}
```

### Rust 实现流程（书中：`protoc-rust-grpc`）

1. **编写 `.proto`**：消息类型 + `service` 与 `rpc` 方法。  
2. **构建时生成代码**：`build.rs` 调用 **`protoc-rust-grpc`**（或同类工具）+ `protoc`，在 **`cargo build` 时**生成 Rust 模块。  
3. **实现服务端**：为生成的 trait 编写具体逻辑（记录位置、查询附近车辆等）。  
4. **实现客户端**：调用生成的 stub 发起 RPC。

```text
.proto  →  build.rs + protoc  →  生成的 Rust 模块
                                    ↓
                          server impl / client stub
                                    ↓
                          HTTP/2 上 gRPC 帧（TCP）
```

**与现代生态对照**（书外补充，便于查资料）：

| 书中 / 较老栈 | 当前常见栈 |
|---------------|------------|
| `protoc-rust-grpc` | **`tonic`** + **`prost`**（gRPC on Rust 主流） |
| 手写 HTTP/2 | 仍由库处理；你主要写 `.proto` 与业务 impl |

学习本章重在 **IDL → 代码生成 → 客户端/服务端分工**；具体 crate 名可随项目选型替换。

**延伸**：[stage06 HTTP](../../stage06_http_programming/README.md) · [stage07 Tokio](../../stage07_tokio_async_net/README.md)（异步 gRPC 常基于 Tokio）。

## 2. SMTP（简单邮件传输协议）

### 协议要点

| 项目 | 说明 |
|------|------|
| 用途 | 互联网**发信**的标准文本协议 |
| 传输 | 通常 **TCP 25**（提交邮件）；用户收信常用 POP3/IMAP，本章侧重 **发** |
| 形态 | 文本命令/响应（`HELO`、`MAIL FROM`、`RCPT TO`、`DATA` 等） |

适合：**告警邮件**、报表、注册验证（生产常配合 SPF/DKIM、限流与反垃圾）。

### 使用 `lettre` 发送邮件

不必手写 SMTP 状态机，用 **`lettre`**：

1. 定义业务结构（如 **`CrashReport`**）。  
2. 为其实现 **`SendableEmail`**（或按版本用 builder / `Message` API 组装 MIME）。  
3. **`SmtpTransport`** / **`SmtpTransportBuilder`**：配置主机、端口、**TLS**、用户名密码。  
4. **`send`** 发出。

```rust
// 概念流程（API 随 lettre 版本略有差异）
// SmtpTransport::relay("smtp.example.com")?
//     .credentials(Credentials::new("user", "pass"))
//     .build()
//     .send(&email)?;
```

**注意**：云厂商常要求 **587 + STARTTLS** 或 **465 SMTPS**；凭据勿写进仓库，用环境变量。

**与 Ch4**：邮件正文可以是纯文本/HTML；复杂附件涉及 MIME 编码，仍属「应用层格式」而非 Serde JSON。

## 3. FTP 与 TFTP 协议

### FTP（文件传输协议）

| 项目 | 说明 |
|------|------|
| 模型 | **基于文本**的命令/响应；**控制连接**（常 TCP **21**）+ **数据连接**（传文件，主动/被动模式） |
| 能力 | 登录、列目录、上传/下载、改名等 |

书中用 **`rust-ftp`** 写客户端示例：

```text
连接 ftp.example.com:21
→ 登录（用户名/密码）
→ 打印当前目录（PWD/LIST）
→ put：用 std::io::Cursor 作内存缓冲区上传文件
```

**特点**：协议年代久，防火墙/NAT 下**被动模式（PASV）** 常踩坑；新项目更常用 **SFTP**、**S3**、**HTTPS** 传文件，但读 FTP 有助于理解「控制通道 vs 数据通道」。

### TFTP（简单文件传输协议）

| 对比 FTP | TFTP |
|----------|------|
| 传输 | **UDP**（非 TCP） |
| 认证 | **无** |
| 复杂度 | 极简（读/写块、ACK） |
| 场景 | **嵌入式**、**PXE/BOOTP** 网络启动、设备固件 |

书中用 **`tftp_server`** 快速在本机绑定端口，提供文件给 TFTP 客户端下载。

```text
TFTP 客户端  --UDP-->  tftp_server（Rust）
              读请求 / 数据块 / ACK
```

**与 Ch3**：直接对应 `UdpSocket` 上的**无连接、有报文边界**模型；比 FTP 更接近「最小文件传输」。

## 协议形态对照（本章总表）

| 协议 | 传输 | 编码 | Rust 生态（书中） | 典型场景 |
|------|------|------|-------------------|----------|
| **gRPC** | TCP + HTTP/2 | protobuf | `protoc-rust-grpc` / 现代 `tonic` | 微服务 API、流式数据 |
| **SMTP** | TCP | 文本 + MIME | `lettre` | 告警、通知邮件 |
| **FTP** | TCP（双连接） | 文本命令 | `rust-ftp` | 传统文件交换 |
| **TFTP** | **UDP** | 二进制块 | `tftp_server` | 引导、嵌入式 |
