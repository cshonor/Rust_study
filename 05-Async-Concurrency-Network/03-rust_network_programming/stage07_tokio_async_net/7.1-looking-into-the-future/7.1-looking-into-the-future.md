## 7.1 Looking into the Future（探讨 Future / 期约机制）

## 一、本节在学什么

把 [Ch3](../stage03_std_tcp_udp/3.1-simple-tcp-server-client/3.1-simple-tcp-server-client.md) 的**阻塞 I/O**（线程在 `read`/`accept` 上卡住）换成 **Future + 事件循环**：

- **Future**：尚未完成的计算；**惰性**，不会自己跑
- **Executor**：在 reactor 里反复 `poll`，直到 `Poll::Ready`
- **async/await**：编译器把 `async fn` 展开为状态机，现代写法替代手写组合子

目标：理解「为什么 `read().await` 不阻塞线程池里其它任务」。

---

## 二、Future 与 `Poll`

**一句话**：Future 是**占位符**；`poll` 返回 **`Ready(v)`** 或 **`Pending`**（稍后再试）。

| 书中（较旧 `futures`） | 现代 `std::task::Poll` | 含义 |
|------------------------|------------------------|------|
| `Async::Ready(v)` | `Poll::Ready(v)` | 完成，得到 `v` |
| `Async::NotReady` | `Poll::Pending` | I/O 未就绪，让出线程 |
| 错误 | `Output = Result<…>` 或 `Ready(Err)` | 失败路径 |

```rust
// async fn 编译后就是实现 Future 的状态机
async fn fetch() -> String {
    // ...
    todo!()
}
```

### 三种执行方式（书中）

| 方式 | 行为 | 适用 |
|------|------|------|
| 当前线程阻塞执行 | 卡住线程直到完成 | 测试、`block_on` 桥接 |
| 线程池（CPU pool） | offload 计算 | CPU 密集，非 I/O 主路径 |
| **Reactor / 事件循环** | Pending 时调度其它 task | **网络服务主流** |

**Tokio** = Reactor（`mio`）+ task 调度 + 可选 `spawn_blocking`。

底层栈：

```text
tokio（运行时 + tokio::net）
  → mio（Poll / epoll，Ch3 已介绍）
  → 非阻塞 fd + 事件循环
```

## 三、组合子与 `async/await`

多层 **`and_then`** 易成回调地狱；现代以 **线性 `async/await`** 为主，宏仍用于并发：

| 工具 | 作用 |
|------|------|
| `.await` 顺序 | 上一步完成再执行下一步 |
| `map` / `then` | 变换结果（组合子或 `async` 块内处理） |
| **`join!`** | 多个 future **全部**完成（并发等待） |
| **`select!`** | 多路竞争，**谁先完成用谁** |
| `timeout` | 超时取消或分支 |

```rust
use tokio::time::{sleep, Duration};

async fn step1() -> u32 {
    sleep(Duration::from_millis(50)).await;
    1
}

async fn step2(v: u32) -> u32 {
    sleep(Duration::from_millis(50)).await;
    v + 2
}

// 现代等价于 and_then 链
async fn work() -> u32 {
    let a = step1().await;
    step2(a).await
}
```

`join!` 让两个 sleep **并行**计时（总耗时 ≈ 最慢那个，而非相加）：

```rust
tokio::join!(task("A", 1), task("B", 2));
// A 1s + B 2s 并行 → 约 2s，不是 3s
```

## 四、对照：阻塞 vs 异步

| | **Ch3 `std::net` + `thread::spawn`** | **本章 Future + Tokio** |
|---|--------------------------------------|---------------------------|
| 等待 I/O | 线程卡住 | task Pending，线程干别的 |
| 并发模型 | 每连接一线程 | 单/少线程管大量连接 |
| API | `read` / `accept` 阻塞 | `read().await` / `accept().await` |
| 心智负担 | 低 | 需理解 `async`、Send、生命周期 |

---

## 五、新手易错

1. **Future 惰性**：创建了不 `.await` / 不 `spawn`，里面代码可能**根本不跑**。
2. **`.await` 会 yield**：持 `std::sync::Mutex` 锁跨越 `.await` 可能死锁；见 [7.1.1](../7.1.1-streams-and-sinks/7.1.1-streams-and-sinks.md)。
3. **`join!` vs 顺序 `.await`**：前者并发，后者串行；别混用概念。
4. **阻塞 syscall 在 async 里**：会卡住整个 worker 线程；用 `tokio::task::spawn_blocking` 或专用 crate。
