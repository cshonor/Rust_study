## 6.1 Introducing Hyper（Hyper 框架简介）

## 一、本节在学什么

把 [Ch3](../stage03_std_tcp_udp/) 的 **TCP 字节流** 之上，加上 **HTTP 语义**（方法、头、状态码、Body），并认识 **Hyper** —— Rust 生态里偏底层的 HTTP 实现，常与 **Tokio** 组合写高并发服务器。

| 层次 | 本章关注点 |
|------|------------|
| 应用 | HTTP/1.1（及 H2 概念） |
| 传输 | TCP（常见）或 TLS（HTTPS，见 [stage08](../stage08_network_security/)） |
| Socket | Ch3 `TcpListener` / `TcpStream` |

---

## 二、HTTP 在栈中的位置（复习）

```text
应用：HTTP 语义（方法、头、状态码、Body）
  ↓
传输：TCP（常见）或 TLS（HTTPS）
  ↓
… Ch1 分层、Ch3 Socket …
```

| 版本 | 特点 |
|------|------|
| **HTTP/1.1** | 文本头 + `Content-Length` 或 chunked 体；一问一答为主，可 keep-alive |
| **HTTP/2** | 多路复用、头部压缩；**gRPC 默认跑在 H2 上**（见 [Ch05](../stage05_app_protocol/)） |

本章 API 以 **HTTP/1.1 风格** 为主；现代 `hyper` 已支持 H1/H2，概念不变。

**一句话**：HTTP 是「在 TCP 上约定好的请求—响应文本/二进制格式」；Hyper 负责解析与生成这些报文，并驱动连接上的异步 I/O。

## 三、最小 HTTP 响应：理解 Hyper 之下是什么

在引入 Hyper 之前，先用 **Ch3 的 `std::net`** 手写一条 HTTP/1.0 响应，看清报文形态：

```text
HTTP/1.0 200 OK\r\n
Content-Type: text/plain\r\n
Content-Length: 42\r\n
\r\n
Hello from raw TCP HTTP demo\n
```

| 部分 | 含义 |
|------|------|
| 状态行 | `HTTP/1.0 200 OK` |
| 头字段 | `Content-Type`、`Content-Length` 等，以 `\r\n` 结尾 |
| 空行 | `\r\n` 分隔头与体 |
| Body | 实际载荷字节 |

仓库 demo 在 `127.0.0.1:18080` 监听，读入请求（demo 不解析）、写回固定响应：

```rust
// 概念（见 6.1-introducing-hyper-raw-tcp-demo.rs）
let resp = format!(
    "HTTP/1.0 200 OK\r\nContent-Type: text/plain\r\nContent-Length: {}\r\n\r\n{}",
    body.len(),
    body
);
stream.write_all(resp.as_bytes())?;
```

**要点**：生产环境不要手写拼报文 —— 交给 **Hyper**（或 Rocket/reqwest 等更高层库）处理分帧、keep-alive、H2 等细节。

## 四、Hyper 简介

**Hyper** 是 Rust 生态里成熟、底层的 **HTTP 实现/库**（偏框架基石，而非「自带路由的全家桶」）。

| 特点 | 说明 |
|------|------|
| 角色 | 解析/生成 HTTP、驱动连接；常与 **Tower** 中间件组合 |
| 并发模型 | 服务端基于 **Tokio + async/await** |
| 适用 | 高并发、需精细控制连接与协议的场景（网关、代理、自建服务） |

**现代对照**：书内可能仍出现 `tokio-core`、`net2`、`futures` 0.1 等旧名；新项目用 **`tokio` 1.x + `hyper` 1.x`**，心智模型不变（事件循环、非阻塞 I/O、`Service`/`Handler`）。

```text
客户端请求 → Hyper 解析 HTTP → 你的 service 返回 Response
```

## 五、单线程 Hyper 服务器

**一句话**：一个 Tokio 运行时线程处理 **accept + 读写的 poll**。

| | 说明 |
|---|------|
| **优点** | 模型简单，便于理解 `Service` 回调链与异步 I/O |
| **缺点** | 单核 CPU 打满后吞吐受限，无法吃满多核 |

这与 Ch3「**每连接一线程 + 阻塞 read**」不同：Hyper 侧是 **单线程多连接 + 非阻塞 I/O**（详见 [stage07](../stage07_tokio_async_net/)）。

---

## 六、多核扩展：`SO_REUSEPORT` 与多监听

书中扩展路径（概念仍适用）：

1. **多个 Tokio 运行时 / 多线程 runtime**：在多个线程上跑异步任务。  
2. **`SO_REUSEPORT`**（书中用 `net2` 等设置）：多个线程/进程在**同一端口** `bind`，内核把新连接**分摊**到不同监听套接字 → 连接级负载均衡。  
3. 结果：书中示例吞吐可**近翻倍**（视硬件与负载而定）。

```text
         ┌─ thread 1 ─ Hyper + Tokio
:8080 ───┼─ thread 2 ─ Hyper + Tokio   （REUSEPORT）
         └─ thread N ─ …
```

| 模型 | Ch3 阻塞多线程 | Hyper + Tokio（单 reactor） | Hyper + REUSEPORT |
|------|----------------|------------------------------|-------------------|
| I/O | 阻塞 | 非阻塞 poll | 非阻塞 + 多监听 |
| 吃多核 | 连接数 ≈ 线程数 | 单核为主 | 多 reactor 分摊连接 |
| 复杂度 | 低 | 中 | 较高 |

**延伸**：仓库 `02-async_tokio/`、[stage07](../stage07_tokio_async_net/) 继续深化 Tokio 与并发 HTTP。

## 七、Hyper vs 其它「写服务」选型

| Crate | 层级 | 典型场景 |
|-------|------|----------|
| **Hyper** | 底层 HTTP + 异步 I/O | 网关、精细协议控制 |
| **Rocket** | 全栈 Web（[§6.2](../6.2-introducing-rocket/6.2-introducing-rocket.md)） | 路由宏、REST、模板 |
| **axum / actix-web** | 生产常用框架 | 与 Hyper/Tokio 生态衔接 |

不做网关时：**先掌握 [§6.3 reqwest](../6.3-introducing-reqwest/6.3-introducing-reqwest.md) 调 API**，Hyper 可与 stage07 一起后移。

## 八、新手易错

1. **HTTP 仍是字节流**：在 TCP 上可能一次 `read` 只收到半条头；必须按 `\r\n\r\n` 与 `Content-Length`/chunked 组帧（Hyper 替你做了）。
2. **混淆 HTTP 版本与 TLS**：HTTPS = HTTP over TLS，不是另一个应用协议名；TLS 见 stage08。
3. **单 reactor 打满 CPU ≠ 程序「慢」**：可能是单核瓶颈，需多线程 runtime 或 REUSEPORT。
4. **书内旧 crate 名**：对照官方 `hyper` + `tokio` 1.x 示例阅读，勿照抄 `tokio-core` API。
