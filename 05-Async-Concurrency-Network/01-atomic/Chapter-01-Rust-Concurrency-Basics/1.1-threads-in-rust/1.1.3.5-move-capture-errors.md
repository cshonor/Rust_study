### 3.7 spawn 两种传参：借用 vs `move`

#### 闭包默认不会帮你 `move`（最小化捕获）

Rust 闭包捕获是 **「能借就不拿所有权」**，**不会**因为写了 `thread::spawn` 就自动 `move`：

| 闭包怎么用外部变量 | 编译器默认捕获什么 |
|--------------------|-------------------|
| 只读（如 `println!("{s}")`） | **`&T`** 引用 |
| 需要修改 | **`&mut T`** |
| 显式写 **`move`** | 把列出的变量 **所有权移进闭包** |

因此 **`spawn(|| …)` 不加 `move`** 时，闭包类型里往往是 **「局部变量的引用」** — 生命周期只跟创建者栈帧绑在一起，**不够 `'static`** → 编译器直接报错（如 `E0597` / `does not live long enough`）。

**心智模型（对照 spawn 的「必须 own」）**：

```text
thread::spawn 要的闭包  ≈  捕获物不能绑在「可能比子线程更短」的栈上
                              → 实质上要 own（或 Arc / &'static 等例外）

move 关键字              ≈  你手动下令：「把这些变量的所有权交给闭包」

没有 move                →  编译器只给 & / &mut  →  过不了 F: 'static
```

**一句话**：`spawn` 不会替你做 `move`；**`move` 是你满足 `'static` 的常用开关**，把栈上局部变成闭包内的 owned 数据。

#### 方式 A：传引用（捕获 `&T`）→ 违反 `F: 'static`

```rust
fn bad() {
    let data = vec![1, 2, 3];
    // 即使写 &data，默认捕获仍是「借 vec 的引用」→ 编译失败 ❌
    thread::spawn(|| dbg!(&data));
}
```

#### 方式 B：`move` 转移所有权 → 满足 `F: 'static`

```rust
fn good() {
    let data = vec![1, 2, 3];
    thread::spawn(move || dbg!(data)); // 闭包独占 Vec，内部无借用 → ✅
}
```

**`move` 的核心原理**：把**自有类型**的所有权移入闭包，闭包类型里**不再含**对创建者栈的引用 → **`F: 'static` 自然成立**，与是否写 `&'static` 无关。

### 四、常见错误：借用局部变量（编译器如何拦住）

**故意写错** — 不加 `move`，让闭包默认借 `main` 里的局部变量（仓库可对照编译：[spawn-no-move-compile-fail.rs](./code/1.1-threads-in-rust-spawn-no-move-compile-fail.rs)）：

```rust
use std::thread;

fn main() {
    let local_var = String::from("我是 main 里的局部变量");

    // 不加 move → 闭包捕获 &local_var → F: 'static 不满足
    let handle = thread::spawn(|| {
        println!("子线程打印: {local_var}");
    });

    handle.join().unwrap();
}
```

**编译器报错（实测 `E0373`，措辞因版本略有差异）**：

```text
error[E0373]: closure may outlive the current function, but it borrows `local_var`,
              which is owned by the current function
  ...
note: function requires argument type to outlive `'static`
help: to force the closure to take ownership of `local_var`, use the `move` keyword
      thread::spawn(move || {
                      ++++
```

| 报错在说什么 | 含义 |
|--------------|------|
| **`borrows local_var`** | 闭包**默认最小化捕获** — 这里是 **`&local_var`**，不是 owned |
| **`may outlive the current function`** | 子线程闭包**可能比** `main` 这段栈**活得更久**（OS 调度不可预测） |
| **`requires … outlive 'static`** | `thread::spawn` 签名要求 **`F: 'static`** — 不能带绑在短命栈上的借用 |
| **`help: use move`** | 编译器直接提示修复：把所有权移进闭包 |

口语版（与上表同义）：**「子线程闭包要求捕获物得是 `'static`，但你这是 main 的局部变量引用。」**

**若没有这层检查（例如强行 `unsafe` 绕开）**会怎样？`main` 结束时 `local_var` 被 **确定** drop；OS **不确定**何时调度子线程去读它 → 悬垂引用 → 垃圾值 / 段错误。Rust **在编译期拒绝**这种写法；合法出路只有：

| 修复 | 做法 |
|------|------|
| **`move`** | 所有权进闭包 / 子线程，外层栈回收不影响（[§五](#五正确写法move-转移所有权)） |
| **`static`** | 数据放程序期存储，不随 `main` 栈帧销毁 |
| **`thread::scope`** | 强制 join 完再销毁局部变量（[1.1.3.6 scope](./1.1.3.6-arc-scope-hft-cheatsheet.md#七绕过-staticthreadscope-为什么可以借局部变量)） |

正确写法对照：[move-closure-demo.rs](./code/1.1-threads-in-rust-move-closure-demo.rs)

### 五、正确写法：`move` 转移所有权

```rust
use std::thread;

fn main() {
    let s = String::from("hello");

    let h = thread::spawn(move || {
        println!("{s}"); // 闭包拥有 s
    });

    h.join().unwrap();
}
```

| 要点 | 说明 |
|------|------|
| **`move`** | 强制闭包**取得**捕获变量的所有权，不再借用 |
| **为何满足约束** | owned 的 `String` 随闭包进子线程，**不依赖**外层栈帧 |
