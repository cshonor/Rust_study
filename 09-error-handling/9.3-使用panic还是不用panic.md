# 9.3 使用 panic! 还是不用 panic!

决定用 `panic!` 还是返回 `Result`，本质是在决定：**这个失败是“调用者可预期并可能恢复”的，还是“代表 bug / 有害状态，继续运行更危险”的**。

对应 demo 项目：`09-error-handling/9.3-panic-or-result-demo/`

---

## 默认选择：返回 `Result`

当错误是“预期可能发生”的（例如用户输入格式错误、文件不存在、网络限流等），更合适的做法是返回 `Result`，把处理方式交给调用者。

---

## 什么时候 `panic!` 更合适

- **示例 / 原型 / 测试**：用 `unwrap` / `expect` 作为占位符，或让测试直接失败
- **你比编译器知道更多**：逻辑上保证必然是 `Ok`，但编译器无法推断（例如硬编码的合法 IP）
- **有害状态（harmful state）**：假设/契约/不变量被破坏，继续运行可能产生更严重问题  
  - 典型例子：越界索引导致标准库 panic（避免潜在安全漏洞）

---

## 用类型系统把“有效性”编码进去：`Guess`

与其在每个地方重复检查 “1..=100”，不如创建一个新类型：

- 只能通过 `Guess::new(value)` 构造
- 构造时统一验证范围
- 之后函数签名接收 `Guess`，就能相信值一定合法

如果越界代表调用者的 bug，可以选择在 `new` 里 `panic!`，并在 API 文档里说明契约。

---

## demo 运行

```bash
cd 09-error-handling/9.3-panic-or-result-demo

# unwrap 适用：硬编码合法 IP
cargo run -- ip

# Result 适用：解析用户输入（错误可预期）
cargo run -- parse 42
cargo run -- parse not_a_number

# Guess：把校验前置到构造（越界会 panic）
cargo run -- guess 50
cargo run -- guess 120
```
